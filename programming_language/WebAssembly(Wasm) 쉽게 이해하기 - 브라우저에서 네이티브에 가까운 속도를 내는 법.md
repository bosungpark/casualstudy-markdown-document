## AI 요약

### 1. WebAssembly란?

**WebAssembly(줄여서 Wasm)**는 C, C++, Rust 같은 언어를 컴파일해서 만드는 **저수준 바이너리 포맷**이다.
브라우저 안에서 **네이티브에 가까운 속도(near-native)**로 실행되며, JavaScript와 나란히 동작한다.

핵심은 한 문장으로 요약된다:
> **Wasm은 JavaScript를 "대체"하는 것이 아니라 "보완"한다.**

| 구분 | JavaScript | WebAssembly |
|------|-----------|-------------|
| 수준 | 고수준, 동적 타입 | 저수준, 어셈블리에 가까움 |
| 형태 | 텍스트(소스) | 컴팩트한 바이너리 |
| 작성 방식 | 사람이 직접 작성 | 다른 언어를 컴파일한 결과물 |
| 강점 | 유연함, DOM/Web API 제어 | 연산 성능 |
| 역할 | 앱 로직, 브라우저 API 다루기 | 성능이 중요한 무거운 계산 |

Wasm은 W3C 표준이며, 다음 4가지 목표를 가진다.

1. **빠르고 효율적이며 이식 가능(portable)** — 플랫폼에 상관없이 네이티브에 가까운 속도
2. **읽고 디버깅 가능** — 사람이 읽을 수 있는 텍스트 포맷(`.wat`)을 함께 제공
3. **안전(secure)** — 샌드박스 환경에서 실행되며 브라우저의 동일 출처 정책을 따름
4. **웹과 호환** — 기존 웹 기술과 매끄럽게 통합

### 2. 왜 빠른가? — JS와의 차이

JavaScript는 실행 시점에 한 줄씩 해석(interpret)하거나 JIT 컴파일하는 과정을 거친다.
반면 Wasm은 이미 **기계어에 가깝게 컴파일된 바이너리**이기 때문에, 브라우저가 파싱·컴파일하는 비용이 훨씬 적다.

```
[기존 웹]
소스(JS) ──파싱──> 해석/JIT ──> 실행
        (느림, 텍스트 파싱 비용 큼)

[WebAssembly]
소스(C/Rust) ──사전 컴파일──> .wasm(바이너리) ──> 거의 바로 기계어로 실행
                                          (빠름)
```

### 3. 컴파일 툴체인 — 무엇으로 .wasm을 만드나

Wasm은 직접 손으로 쓰는 게 아니라 **다른 언어를 컴파일한 결과물**이다.

| 툴체인 | 입력 언어 | 특징 |
|--------|----------|------|
| **Emscripten** | C / C++ | 가장 성숙한 도구. Clang+LLVM으로 컴파일하고 JS "글루 코드"까지 생성 |
| **Rust → Wasm** | Rust | Rust 공식 WG가 지원, 메모리 안전성 + 성능 |
| **AssemblyScript** | TypeScript 유사 문법 | 웹 개발자에게 친숙, 기존 TS 툴링(ESLint 등) 사용 가능 |
| **직접 작성(.wat)** | 텍스트 포맷 | 어셈블리를 직접 쓰거나 생성 후 바이너리로 변환 |

Emscripten의 흐름:

```
C/C++ 소스 ──> Clang + LLVM ──> .wasm 바이너리
                               + JS 글루 코드(Web API 연결용)
                               + HTML 문서
```

> Wasm은 DOM에 직접 접근할 수 없기 때문에, Emscripten은 SDL·OpenGL·POSIX 같은
> 익숙한 라이브러리를 Web API로 감싸주는 글루 코드를 자동 생성한다.

### 4. 텍스트 포맷(.wat) — 사람이 읽을 수 있는 Wasm

바이너리(.wasm)와 1:1로 대응되는 사람이 읽을 수 있는 표현이 `.wat`이다.
두 정수를 더하는 예시:

```wat
(module
  (func $add (param $a i32) (param $b i32) (result i32)
    local.get $a
    local.get $b
    i32.add
  )
  (export "add" (func $add))
)
```

스택 기반 가상 머신이라, "값을 스택에 올리고(local.get) → 더하고(i32.add) → 결과를 돌려준다"는 흐름이다.

### 5. 4가지 핵심 구성 요소

Wasm의 개념은 JavaScript API에 그대로 1:1로 반영된다.

| 개념 | 설명 |
|------|------|
| **Module(모듈)** | 컴파일된 Wasm 바이너리. 상태가 없고(stateless) `postMessage()`로 공유 가능. ES 모듈처럼 import/export를 선언 |
| **Memory(메모리)** | 크기 조절 가능한 `ArrayBuffer`. 저수준 접근이 가능한 선형(linear) 바이트 배열 |
| **Table(테이블)** | 함수 포인터처럼 raw 바이트로 저장할 수 없는 참조들을 담는 크기 조절 가능한 배열 |
| **Instance(인스턴스)** | Module + 런타임 상태(Memory, Table, import 값). 실제로 로드되어 실행 가능한 형태 |

### 6. 웹 플랫폼 속에서의 위치

기존 웹 플랫폼은 (1) 코드를 실행하는 가상 머신(VM) + (2) Web API(DOM, WebGL, Web Audio 등)로 구성된다.
지금까지 VM은 JavaScript 전용이었는데, **Wasm이 두 번째 VM 런타임으로 추가**되었다.

```
        ┌─────────────────────────────────────┐
        │            Web APIs                  │
        │  (DOM, WebGL, IndexedDB, Web Audio)  │
        └───────────▲──────────────▲───────────┘
                    │              │
          ┌─────────┴───┐   ┌──────┴─────────┐
          │ JS 런타임   │   │ Wasm 런타임     │
          │ (VM)        │   │ (VM)           │
          └─────────────┘   └────────────────┘
              둘 다 같은 Web API에 접근
```

### 7. 브라우저에서 로드하고 실행하기

```javascript
// 스트리밍 방식 (큰 모듈에 권장)
const { instance } = await WebAssembly.instantiateStreaming(
  fetch('module.wasm'),
  {
    imports: {
      jsFunction: (x) => x * 2   // Wasm이 호출할 JS 함수 주입
    }
  }
);

// export된 Wasm 함수를 JS 함수처럼 호출
const result = instance.exports.add(10, 20); // 30
```

JS와 Wasm은 **양방향 호출**이 가능하다.
- JS → Wasm: `instance.exports.someFunc()`
- Wasm → JS: 인스턴스화할 때 넘긴 `imports`를 통해 호출

### 8. 주로 어디에 쓰나

JavaScript만으로는 느린, 연산이 무거운 작업에 적합하다.

- 3D 게임 / 그래픽
- VR / AR
- 컴퓨터 비전, 이미지·영상 편집
- 과학 계산, 시뮬레이션
- JS 파싱·컴파일 부담이 큰 대형 애플리케이션
- 모바일 등 리소스가 제한된 환경

### 9. 한계와 미래

- **현재 한계**: Wasm은 DOM이나 Web API에 직접 접근할 수 없어 JS를 중간 다리로 거쳐야 한다.
- **메모리 모델**: 명시적 수동 관리(C/C++/Rust 방식). GC 언어는 아직 1차 타깃이 아님.
- **미래**: Web API 직접 접근, 가비지 컬렉션 언어 지원 등의 제안이 진행 중.

## 내가 얻은 인사이트

### 아키텍처 관점

1. **"대체"가 아니라 "역할 분담"이라는 설계 철학**
   - Wasm을 처음 접하면 "JS를 밀어내는 기술"로 오해하기 쉽다. 하지만 실제 설계는 *VM을 하나 더 추가*하는 방향이다.
   - JS는 유연함과 DOM 제어를, Wasm은 무거운 연산을 맡는 분업 구조. 새로운 런타임을 도입할 때 기존 생태계를 깨지 않고 옆에 끼워넣는 좋은 사례다.

2. **DOM 접근 불가가 만든 "글루 코드"라는 현실**
   - Wasm이 Web API에 직접 닿지 못한다는 제약 때문에 Emscripten이 JS 글루 코드를 자동 생성한다.
   - 즉, 순수 Wasm 성능 이득과 JS↔Wasm 경계를 넘나드는 호출 비용(call overhead) 사이에 트레이드오프가 존재한다. 경계를 자주 넘으면 오히려 느려질 수 있어, "한 번 넘어가서 큰 덩어리 연산을 하고 돌아온다"는 패턴이 유리하다.

### 보안 관점

3. **샌드박스 + 선형 메모리 모델**
   - Wasm은 격리된 선형 메모리(ArrayBuffer) 안에서만 동작하고 동일 출처 정책을 따른다. 네이티브 수준의 성능을 주면서도, 임의 메모리 접근으로 호스트를 망가뜨리지 못하게 막은 설계다.
   - 이 특성 덕분에 최근에는 브라우저를 넘어 서버사이드(WASI) 플러그인 샌드박스로도 확장되고 있다. "신뢰할 수 없는 코드를 안전하게 실행"이라는 문제에 대한 범용 해법으로 보인다.

### 실무 적용 관점

4. **언어 선택의 자유 — 진입점이 여러 개**
   - C/C++(Emscripten), Rust, TypeScript 유사 문법(AssemblyScript)까지 진입점이 다양하다.
   - 웹 개발자라면 AssemblyScript로 부담 없이 시작하고, 성능·메모리 안전성이 중요하면 Rust를 택하는 식으로 점진적 도입이 가능하다.

5. **"전부 Wasm으로" 보다 "병목만 Wasm으로"**
   - 모든 코드를 Wasm으로 옮기는 건 글루 코드·경계 비용 때문에 비효율적이다.
   - 프로파일링으로 진짜 병목(이미지 처리, 암호 연산 등)만 골라 Wasm으로 대체하는 게 현실적인 전략이다.
