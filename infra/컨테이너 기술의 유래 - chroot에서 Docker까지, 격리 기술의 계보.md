## AI 요약

### 1. 컨테이너 기술이란?

컨테이너는 **어느 날 갑자기 발명된 것이 아니라**, 40년 넘게 누적된 "프로세스 격리(isolation)" 기술의 결과물이다. 핵심은 하나의 OS 커널을 공유하면서도, 각 프로세스 그룹에게 *"나 혼자 시스템을 쓰고 있다"* 는 착각을 주는 것이다.

| 가상화 방식 | 격리 단위 | 커널 | 오버헤드 | 대표 기술 |
|------------|----------|------|---------|----------|
| 하드웨어 가상화(VM) | 가상 머신(게스트 OS) | 게스트마다 별도 | 무거움 | VMware, KVM |
| OS 수준 가상화(컨테이너) | 프로세스 그룹 | 호스트와 공유 | 가벼움 | Docker, LXC |

컨테이너가 가벼운 이유는 게스트 OS를 통째로 띄우지 않고, **호스트 커널의 격리 기능**(namespaces + cgroups)을 그대로 빌려 쓰기 때문이다.

### 2. 계보(타임라인) — 격리는 이렇게 진화했다

```
1979 chroot ─────────► 파일시스템 루트만 격리 (가장 원시적인 "감옥")
        │
2000 FreeBSD Jails ──► 파일시스템 + 네트워크(별도 IP) + 사용자 격리
        │
2001 Linux VServer ──► 커널 패치로 자원 분할 (OS-level virtualization)
        │
2004 Solaris Zones ──► 자원 제어 + zone 경계 + ZFS 결합
        │
2005 OpenVZ ─────────► 패치된 리눅스 커널 기반 격리/자원관리
        │
2006 cgroups ────────► [Google] 자원 "제한·계측·격리"의 표준 등장 ★
        │
2008 LXC ────────────► cgroups + namespaces 결합한 첫 완성형 컨테이너 매니저 ★
        │
2011 Warden ─────────► [CloudFoundry] 클라이언트-서버 모델, 라이프사이클 API
        │
2013 LMCTFY ─────────► [Google] "컨테이너를 인식하는" 앱 스택 (→ 후에 libcontainer로 기여)
        │
2013 Docker ─────────► 이미지 + 레지스트리 + CLI 생태계로 대중화 ★★★
        │
2014~ Kubernetes ────► 컨테이너 오케스트레이션의 사실상 표준으로
```
★ 표시가 오늘날 컨테이너의 **3대 기술적 뿌리**다.

### 3. 핵심 격리 메커니즘 3가지

컨테이너는 마법이 아니라, 리눅스 커널 기능 3개의 조합이다.

| 메커니즘 | 등장 | 격리하는 것 | 한 줄 비유 |
|---------|------|------------|-----------|
| **namespaces** | 점진적 | "보이는 것" (PID, 네트워크, 마운트, 호스트명, 사용자) | 각자 다른 방에 두고 옆방을 못 보게 함 |
| **cgroups** | 2006 (Google) | "쓸 수 있는 자원" (CPU, 메모리, I/O) | 각 방의 전기·수도 사용량에 상한선을 둠 |
| **union filesystem** | Docker가 대중화 | "디스크 이미지" (레이어 적층) | 투명 필름을 겹쳐 하나의 이미지처럼 보이게 함 |

```
       컨테이너 = namespaces + cgroups + union filesystem
       ───────────────────────────────────────────────
        "격리"        "본다"     "쓴다"      "담는다"
```

### 4. 왜 Docker가 게임을 바꿨나?

LXC(2008)가 이미 "컨테이너 매니저"를 만들었는데, 왜 컨테이너 시대는 2013년 Docker부터 시작됐다고 할까?

- **기술이 아니라 경험을 발명했다.** Docker 이전의 컨테이너는 *커널 전문가의 도구*였다. Docker는 `Dockerfile`, 이미지, 레지스트리(Docker Hub), `docker run` 한 줄이라는 **개발자 워크플로우**를 만들었다.
- **이식성(portability) 표준화.** "내 노트북에선 됐는데"를 이미지 단위로 박제했다.
- Docker는 처음엔 LXC를 백엔드로 썼지만, 이후 **libcontainer**(자체 구현)로 교체하며 LXC 의존을 끊었다.

> 저자의 결론: 컨테이너는 이제 Gartner의 "생산성 안정기(Plateau of Productivity)"에 진입했다. 엔터프라이즈의 90%가 사용하는 **기반 인프라**가 되었고, 혁신의 초점은 컨테이너 자체의 동작 원리에서 → 보안·상태 관리·AI 워크로드 같은 "day 2" 문제로 옮겨갔다.

## 내가 얻은 인사이트

### 역사·설계 관점

1. **격리는 "전부 아니면 전무"가 아니라 스펙트럼이다.**
   - chroot(파일시스템만) → Jails(네트워크 추가) → cgroups(자원 추가)로, 격리할 수 있는 "차원"이 하나씩 늘어난 역사다. 컨테이너 보안 사고의 상당수는 *어떤 차원이 격리되지 않았는가*에서 나온다는 점을 거꾸로 시사한다.

2. **핵심 기술은 거의 Google에서 나왔다.**
   - cgroups(2006), LMCTFY(2013), 그리고 그 경험이 녹아든 Kubernetes까지. Google의 내부 인프라(Borg) 운영 경험이 오픈소스 생태계의 뼈대가 됐다. "내부에서 풀던 진짜 문제"가 표준이 된 전형적 사례다.

### 실무 관점

3. **"컨테이너 == Docker"라는 등식을 의심하게 된다.**
   - Docker는 *대중화*의 주역이지 *발명가*가 아니다. 이 구분을 알면, 이후 Kubernetes가 dockershim을 제거하고 containerd/CRI-O로 갈아탄 흐름(이 레포의 [dockershim 디프리케이션 노트](Don't%20Panic%20-%20Kubernetes%20and%20Docker%20(dockershim%20디프리케이션).md))이 "배신"이 아니라 *원래 분리 가능했던 계층의 정상화*로 읽힌다.

4. **새 기술을 평가할 때 "계보"를 먼저 그린다.**
   - 컨테이너처럼 보이는 신기술(예: Wasm, microVM)을 만나면, "이건 namespaces/cgroups/union-fs 중 무엇을 대체/보완하는가?"를 물으면 위치를 빠르게 잡을 수 있다. 격리의 3축이 좋은 평가 좌표계가 된다.

### 트레이드오프 관점

5. **가벼움의 대가는 "커널 공유"다.**
   - 컨테이너가 VM보다 가벼운 이유(호스트 커널 공유)는 곧 보안 경계가 VM보다 얇다는 뜻이기도 하다. 그래서 2018년 이후 Kata Containers·gVisor 같은 *"VM의 격리 + 컨테이너의 가벼움"* 절충안이 등장했다. 모든 격리 기술은 결국 **격리 강도 ↔ 오버헤드**의 트레이드오프 위 어딘가에 놓인다.
