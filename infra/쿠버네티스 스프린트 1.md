# 쿠버네티스 스프린트

> Kubernetes의 핵심 오브젝트를 ① 그림으로 이해하고, ② 실제 앱 동작(Probe)으로 이해하는 첫 스프린트 정리.
> 강의 흐름(일프로 "쿠버네티스 어나더 클래스")을 이해하기 쉽게 재구성했다.

---

## 들어가며 — 오늘 배우는 게 왜 중요한가

### 파레토 법칙 (80/20)

> **상위 20%의 원인이 전체 결과의 80%를 만든다.**

오늘 배우는 오브젝트는 전체 Kubernetes 오브젝트의 **약 20%** 지만, **앞으로 실무에서 쓰게 될 오브젝트의 80%** 를 차지한다. → "언제 다 배우나" 부담 가질 필요 없이, 이 핵심부터 잡으면 실무의 대부분이 커버된다. **첫 단추를 잘 끼우는 단계.**

### 오브젝트를 이해하는 3단계 학습법

```
1단계  오브젝트를 "그려보며" 이해하기   ← 이 노트 1부
2단계  실제 "앱을 동작"시키며 이해하기   ← 이 노트 2부 (Probe)
3단계  쿠버네티스 "내부 동작" 이해하기   ← 다음 스프린트
```

기능마다 잘 이해되는 방법이 다르기 때문에 단계를 나눴다.
- 어떤 기능은 → **그림**을 그려야 이해된다.
- 어떤 기능은 → **실제 앱 동작**을 봐야 정확히 이해된다.
- 어떤 기능은 → **내부 동작**을 알아야 제대로 이해된다.

### 강사의 이해도 성장기 (이 순서가 효과적인 근거)

| 어떻게 공부했나 | 이해도 |
|----------------|--------|
| 책으로 개념 학습 | **30%** |
| 실무 프로젝트에 적용 (앱 동작과 함께 봐야 이해되는 영역 존재) | **60%** |
| 내부 동작까지 파고듦 | **90%** (남은 10%는 겸손함으로) |

# 1부. 오브젝트 그려보며 이해하기

> 이 단계의 목표: 각 오브젝트가 **어떤 역할을 하는지 대략적인 느낌**만 잡는 것. 처음엔 혼란스러운 게 정상이다.

## 강의 환경 구성

1. 마스터 노드에 접속해 디렉토리 생성 → 나중에 파드가 이 경로에 파일을 저장할 용도(PV).
2. 대시보드에서 야믈(YAML) 파일들을 하나씩 붙여넣어 오브젝트 생성.
3. 네임스페이스를 `kubernetes-anotherclass-123` 으로 변경.
4. 파드가 모두 정상 기동되면 성공.

## 핵심 오브젝트 한눈에 보기

### Namespace — 오브젝트를 그루핑

- 여러 오브젝트를 **묶어주는** 역할. metadata에 `name`, `label`.
- **삭제하면 그 안의 모든 오브젝트가 함께 삭제된다.** → 실습 정리 시 네임스페이스 하나만 지우면 끝. (단, PV는 네임스페이스에 속하지 않아 별도 삭제 필요.)

### Deployment — 파드를 만들고 업그레이드

- 소속 `namespace` 를 지정하면 그 네임스페이스에 속한다.
- 한 네임스페이스 안에서 **같은 종류 오브젝트끼리는 이름이 중복되면 안 된다.** (Service와 Deployment는 종류가 달라 이름이 같아도 됨.)
- `replicas: 2` → 파드 2개. `strategy: RollingUpdate` 는 Deployment의 핵심(업데이트 방식, 2부 실습에서).
- `template` 부터가 파드 생성 명세:
  - `nodeSelector` — 파드를 띄울 노드 선택
  - `containers.image` — Docker Hub의 이미지를 받아서 사용
  - `env` + `configMap` — 앱 환경변수 (값은 ConfigMap이 제공)
  - `resources` — CPU/메모리 할당량(request)과 상한(limit). **설정 안 하면 파드가 노드 자원을 다 써버린다.**
  - `volumeMounts` — 파드 내부 디렉토리(mountPath)를 만들고, 이름으로 PVC와 연결
  - `secret` 데이터 소스 — Secret 오브젝트와 연결

### Probe (3종) — 앱 상태 체크 *(자세한 건 2부)*

- **Startup Probe** — 앱이 기동됐는지 체크. 기동 실패하면 재기동. 성공하면 나머지 둘을 시작.
- **Readiness Probe** — 앱에 **트래픽을 연결할지** 결정.
- **Liveness Probe** — 앱이 정상이 아니면 **재시작**할지 판단.

### Service — 파드에 트래픽 연결

- metadata 3종 세트(`namespace`/`name`/`label`) + `selector`/`port`/`type`.
- 셀렉터로 파드를 찾아 트래픽을 연결한다.

### ConfigMap / Secret — 파드에 값 제공

- **ConfigMap** — 환경변수 등 외부에서 앱에 전달하는 데이터. `data` 에 값.
- **Secret** — ConfigMap과 비슷하나 **더 중요한 값**(민감 정보)에 사용. `stringData` 를 쓰면 그 내용으로 파드 안에 **파일**이 만들어진다.

### PVC / PV — 저장 공간

- **PVC (PersistentVolumeClaim)** — 파드가 PV를 지정할 때 사용. 저장 용량, accessMode(읽기/쓰기) 등. *이번 실습에선 `selector` 만 실제로 쓰임.*
- **PV (PersistentVolume)** — 실제 볼륨 지정. `local` 패스를 볼륨으로 사용, `nodeAffinity` 로 대상 노드(마스터 노드) 지정. → 그래서 환경 구성 때 디렉토리를 미리 만들었던 것.

### HPA (HorizontalPodAutoscaler) — 부하 따라 스케일링

- 대상으로 **Deployment** 지정.
- 최소 2개 ~ 최대 4개 파드 유지.
- `metrics`: 파드 CPU 평균 **40%** 초과 시 스케일 아웃.
- `behavior`: 한번 늘어난 뒤 **600초(10분)** 동안 추가 증가 억제 (2→3 직후 바로 4로 튀는 것 방지).

## 오브젝트 레벨: 클러스터 vs 네임스페이스

오브젝트의 가장 큰 범위는 **클러스터**다. 크게 두 분류:

| 레벨 | 오브젝트 예시 |
|------|--------------|
| **클러스터 레벨** | Namespace, **PersistentVolume(PV)**, Node |
| **네임스페이스 레벨** | Deployment, Service, ConfigMap, Secret, PVC, HPA … |

- 각 오브젝트는 자기 레벨이 정해져 있다. → Deployment를 PV처럼 클러스터 레벨에 만들 수 없다.
- PV가 네임스페이스에 안 속하는 이유 → 네임스페이스 삭제 시 PV는 **따로 한 번 더 삭제**해야 하는 이유.

## 레이블 · 셀렉터 · 네이밍 (가장 신경 써야 할 부분)

> 강사가 "시간이 지나서야 챙기기 시작했다, 운영에 올라간 뒤엔 수정이 매우 부담스럽다"며 **운영 전에 충분히 고민하라**고 강조한 영역.

### 레이블(Label) — 오브젝트에 붙이는 정보 + 연결 수단

옷의 라벨처럼, 오브젝트가 많아져도 레이블만 보면 **어떤 앱의 무슨 구성인지** 바로 파악할 수 있다. Kubernetes가 **권고하는 표준 레이블**이 있고, Prometheus가 이를 잘 따른다.

| 레이블 | 의미 | 예시 |
|--------|------|------|
| `app.kubernetes.io/part-of` | 애플리케이션 **전체 이름**(서비스 대표명) | prometheus |
| `app.kubernetes.io/component` | 서비스를 구성하는 분리된 **기능** | metrics, exporter |
| `app.kubernetes.io/name` | 애플리케이션 **실제 이름** | prometheus, kube-state-metrics, node-exporter |
| `app.kubernetes.io/instance` | **설치 단위 식별자** (같은 앱 여러 개 설치 시 구분) | prometheus-k8s |
| `app.kubernetes.io/version` | 앱 **버전** (버전 바뀌면 같이 수정 필요) | 1.0.0 |
| `app.kubernetes.io/managed-by` | 배포·관리한 **도구** | helm |

- `component` vs `name` 헷갈림 주의: 한 component(예: exporter = "메트릭 제공") 밑에 여러 name(kube-state-metrics, node-exporter)이 올 수 있다.
- 레이블 앞에 `app.kubernetes.io/` 같은 **prefix(도메인)** 를 붙이기도 함(선택). K8s 문서 예제에서 자주 보임.
- 레이블에는 **정보+기능** 역할을 하는 것과 **정보만** 주는 것이 있다.

### 네이밍 — K8s 권고는 없음, 강사 권고 예시

K8s는 오브젝트 네이밍 규칙(이름 중복 불가, 특수문자는 `-`/`.` 만 등)은 있지만 **"어떻게 지어라"는 권고는 없다.** 강사 권고:

- **Namespace 이름** = 그 안에 들어갈 앱들의 **범위**를 나타내는 이름 (예: `monitoring`). 강의에선 `kubernetes-anotherclass-123` (123은 수업 넘버링 — 이전 수업 오브젝트와 충돌 방지).
- **오브젝트 이름** = `앱 이름 + 인스턴스 식별자` 조합 → 보통 **레이블의 instance 값** 이 곧 오브젝트 이름이 됨 (유니크해짐). 예: Deployment `api-tester-123`.
- **사용 목적**을 이름 뒤에 덧붙이면 좋다:
  - ConfigMap이 앱 초기화용 데이터면 → 뒤에 `-rule` (Prometheus는 성능 계산 공식을 담아 이렇게 함)
  - 클러스터 내부용 Service → `-internal`, 외부 접속용 → `-external`
  - 특정 임계치용 HPA → 목적에 맞는 이름

### 셀렉터(Selector) & 레이블 매칭 — 오브젝트 연결

레이블은 정보 파악뿐 아니라 **셀렉터와 매칭되어 두 오브젝트를 연결**한다.

> **매칭 규칙: 셀렉터의 내용이 모두 레이블에 포함되어야 한다.**
> - 레이블에 추가 항목(예: version)이 더 있는 건 OK.
> - 반대로 셀렉터에만 version이 더 있으면 → 연결 안 됨.
> - instance가 유니크하므로 셀렉터에 instance만 넣어도 원하는 파드를 정확히 매칭 가능 (나머지는 선택).

연결되는 관계 예시: Deployment↔ReplicaSet, ReplicaSet/Service ↔ Pod, PVC ↔ PV.

### 오브젝트를 연결하는 2가지 방식

1. **Selector ↔ Label** 매칭 — ReplicaSet→Pod, Service→Pod, PVC→PV, Pod의 `nodeSelector`→Node 레이블, PV의 `nodeAffinity`→Node 레이블.
2. **오브젝트 내부 속성으로 대상 직접 지정** — HPA가 Deployment 이름 지정, Pod가 ConfigMap/Secret/PVC를 직접 지정.

> 셀렉터 표현 방식은 오브젝트마다 조금씩 다르다(Service, PVC, nodeAffinity 등). **외울 필요 없이** 잘 된 템플릿 하나를 두고 복사·수정해서 쓰면 된다.

### Deployment를 만들면 일어나는 일 (생성 흐름)

```
Deployment (api-tester-123)
    │  쿠버네티스가 자동 생성
    ▼
ReplicaSet (api-tester-123-<랜덤>)   ← replicas 값 복사, 복제본 관리
    │  replicas 수만큼 생성 + 항상 그 수 유지(파드 강제 삭제해도 다시 생성)
    ▼
Pod (api-tester-123-<랜덤>-<랜덤>)    ← 실제 앱 실행
```

> 역할 분담: **Deployment = 파드 업데이트 관리**, **ReplicaSet = 복제본 수 관리.**

# 2부. 앱 동작으로 이해하기 — Probe

## 들어가기 전: 기능 적용은 신중하게

> 좋은 관리 기능이 많다고 다 적용하면 **주객전도**. 주인공은 애플리케이션이고, 인프라는 앱을 잘 관리하기 위해 존재한다.

- 예: **NetworkPolicy**(파드 간 방화벽)는 이미 별도 보안팀/방화벽이 있다면 "굳이 또?"가 될 수 있어 **안 써도 되는** 애매한 기능.
- 반면 **Probe**는 **반드시 써야 하는** 기능인데, 앱에 대한 이해 없이 적용하면 **의도치 않은 파드 장애**를 만난다. → "새 툴 하나 더 배울 시간에 Probe를 깊게 파라."

## Probe 기본 동작

설정: 3종 Probe가 각각 URL(예: `/ready`)을 8080 포트로 **일정 간격**으로 호출, 성공/실패 임계치를 가진다. (앱에는 해당 헬스체크 URL이 미리 구현돼 있어야 함.)

```
파드 생성
   │
   ▼
[기동 중]  Startup Probe 동작 (10초 간격 호출)
   │   - 기동 중엔 응답 못 받아 계속 실패
   │   - 10번 실패 전에 1번이라도 성공하면 → 성공 간주
   ▼
Startup 성공 → Startup 중지, Readiness/Liveness 시작
   │
   ├─ Readiness Probe ─ 성공 시 외부 트래픽 연결(서비스 활성화)
   │                    실패 시 연결 해제
   └─ Liveness Probe ── 앱 생존 체크. 실패(예: 2회) 시 파드 재기동
```

### 흔한 의문 (그리고 답)

- "어차피 같은 URL 쓸 건데 왜 Probe마다 API를 따로 적지?"
- "장애 나면 Liveness·Readiness 같이 실패할 텐데 Readiness가 따로 필요한가?"

→ **목적이 다르기 때문.** (아래 "왜 생겼나"에서 해소)

## 로그로 보는 Probe 동작 (실습)

**준비:** HPA의 `minReplicas` 를 1로 변경. (HPA가 Deployment에 영향을 줘서 replicas를 1로 바꿔도 다시 2로 되돌리기 때문 → 로그/API 관찰 혼란 방지.)

Grafana 실시간 로그로 파드 재기동 과정을 관찰하면:

1. 앱 초기화 시작 → Spring/서블릿 초기화 → DB 연결(로그용 시뮬레이션).
2. 기동 중 **Startup Probe는 계속 실패** (5초 간격). 기동 완료되면 **성공**.
   - ※ 실제론 기동 전엔 앱이 API를 못 받으므로 Startup 로그가 안 찍히는 게 정상. 강의에선 학습용으로 일부러 로그를 찍게 구성(WAS/Tomcat이면 access log에서 확인 가능).
3. 기동 완료 후 사용자 초기화(`ConfigMap data loading` 등 외부 데이터로 추가 초기화).
4. 이 구간에 **Liveness는 성공, Readiness는 (의도적으로) 실패**.
5. 추가 데이터 작업 완료 → **둘 다 성공**, 이후 10초 간격으로 헬스체크 반복.

## 왜 Probe가 생겼나 — 앱 동작 관점

Kubernetes는 앱을 편하게 관리하려고 만들어졌다. 앱의 생애에는 자동화하고 싶은 요구가 있다:

| 앱 단계 | 상태 | 필요한 자동화 |
|---------|------|--------------|
| **앱 초기화** (Spring/DB 연결 등) | API 못 받음 | 초기화 끝났는지 체크 + 외부 API 차단 |
| **유저 초기화** (초기 데이터 로딩/검증 등) | 기동은 됨, API 받을 수 있음 | 앱 살아있는지 체크 + 외부 API 여전히 차단 |
| **완전 기동** | 정상 | 상태 체크 계속 + **외부 API 자동 허용** |
| **장애 발생** | 비정상 | 자동 **재기동** |

K8s 이전엔 이걸 운영자가 **수동**으로 했다 (API 찔러보고 → LB 스위치 수정해 트래픽 유입 → 모니터링 보다 장애 알람 받으면 접속해 재기동). Probe가 이 수동 작업들을 자동화한 것:

- **Startup Probe** → 앱 초기화 완료 판단. (성공하면 중지하고 나머지 둘 활성화)
- **Liveness Probe** → 생존 체크, 실패 시 **파드 재기동**.
- **Readiness Probe** → 성공해야 Service↔Pod 연결(트래픽 유입). 실패하면 연결 해제, 다시 성공하면 재연결.
  - ※ Service와 Pod는 Selector/Label로 매핑돼 있어도 Kubernetes가 **미리 연결해두진 않는다.** Readiness 성공 시점에 비로소 연결.

### 핵심 인사이트

- **Readiness ≠ Liveness인 이유:** 앱은 기동됐지만(=Liveness 성공) **유저 초기화 동안 외부 트래픽은 받기 싫을 때** Readiness를 일부러 실패시킨다. 이 의도가 없다면 둘이 같은 API를 써도 무방하지만, **의도가 있으면 API를 구분**해야 한다.
- Readiness의 API 결과는 "내가 의도한 초기화가 다 끝났다"를 알려줘야 한다.
- Startup/Liveness/Readiness 모두 단순 200 OK가 아니라 **각자 목적에 맞는 점검 로직**을 넣으면 좋다. 단, **Readiness/Liveness는 앱이 죽을 때까지 계속 호출되므로 로직을 가볍게** 만들 것.
- 앱 개발자와 K8s 관리자가 다르면 이 지점을 **충분히 협의**해야 한다.

## API로 직접 확인하기 (시나리오)

1. **앱 초기화 중** → 외부 API 호출 **실패**.
2. **유저 초기화 중** → 외부 API는 여전히 **실패**, 그러나 파드 내부에서 보낸 API는 **성공** (앱은 기동된 상태).
3. **완전 기동 후** → 외부 API **성공** (Readiness 성공 → 서비스 활성화).
4. **트래픽 중단 API 전송** → Readiness 3회 실패 → **Service-Pod 연결 해제** → 외부 API 실패.
5. **트래픽 재개** → 다시 연결 → 외부 API 성공.
6. **장애 발생 API 전송** → Liveness 3회 실패 → **파드 재기동** (기동 로그 다시 출력).

## 일시적 장애 상황에서 Probe 활용 (중요)

서비스 중 **일시적 장애**(순간 부하 급증, 렉, 스레드풀 포화 등)가 생기면:

- 외부 API가 계속 들어와 "불난 곳에 기름 붓기" → 시스템이 멈출 수도, **시간이 지나면 스스로 회복**할 수도 있다.
- 그런데 이 와중에 Liveness/Readiness가 실패하면 **Kubernetes가 파드를 재기동** → 회복할 수 있었던 앱이 강제로 죽고 **처리 중이던 작업이 모두 실패**한다. (Probe가 오히려 독이 되는 상황)

**완화 방법:**

| Probe | 설정 방향 |
|-------|-----------|
| **Readiness** | 실패 시 외부 트래픽을 차단해 부담을 덜어주므로 **그대로 둬도 OK** |
| **Liveness** | **체크 주기를 Readiness와 다르게**, 실패까지 걸리는 시간을 **더 길게** → 재기동을 지연시켜 스스로 회복할 시간을 준다 |

- 얼마나 지연시킬지는 앱마다 **시스템 관리 정책**으로 정해야 한다.
- 중요한 시스템은 보통 **파드를 이중화**하므로, 일시 장애 파드를 빨리 재기동하는 것만이 답은 아니다. **회복할 시간을 더 주는 것**도 고려할 것.

> 핵심: 내 앱을 잘 관찰하고, Probe를 어떻게 적용할지 깊이 고민하라.

# 3부. ConfigMap과 Secret

> "가볍게 이해하고 넘어가면 나중에 목소리 커질 일이 생기는" 오브젝트. 개념은 쉬워 보이지만 실무에선 **영역 파괴의 주범(ConfigMap)** 이자 **이름값 못 하는 기대(Secret)** 다.

## 기본 동작 복습

ConfigMap과 Secret 모두 파드에 바로 연결되어 **데이터를 주입**한다. 주입 방식은 파드의 연결 속성으로 결정:

| 연결 속성 | 결과 | 확인 방법 |
|-----------|------|-----------|
| `envFrom` | 데이터가 **환경변수**로 주입 | 파드 안에서 `env` 명령 |
| `volume` + `volumeMounts` | 데이터가 **파일**로 마운트 | 마운트 패스 파일 조회 |

## ConfigMap 데이터 — 환경변수의 다양한 쓰임

`data` 의 모든 key-value가 컨테이너 **환경변수로 주입**된다. 이미지에 넣어둔 java 실행 명령이 컨테이너 생성 후 자동 실행되며 이 환경변수를 읽는다. (값이 없으면 `null` 로 주입.)

환경변수는 단순히 "개발/검증/운영 구분"만이 아니라 쓰임이 다양하다:

| 분류 | 예시 | 설명 |
|------|------|------|
| **인프라 환경 값** | `SPRING_PROFILES_ACTIVE=dev` | 앱이 어느 환경에서 도는지 기동 시점에 알려줌. Spring은 이 값으로 환경별 `application-{profile}.yaml` 선택 (K8s 이전부터 쓰던 방식) |
| **앱 기능 제어 값** | `APPLICATION_ROLE` | 앱의 역할 지정. 한 앱의 여러 기능을 통째로 띄울 수도, **스케일링 모드**로 기능별로 따로 띄울 수도 (MSA 반영 오픈소스 多) |
| **외부 환경 주입 값** | `POSTGRESQL_FILEPATH` | Secret 데이터로 연결할 파일 경로. 경로 바꿀 때 **앱 재빌드 없이 ConfigMap만 수정** |

> 고민거리: 그럼 `application.yaml` 파일을 다 없애고 ConfigMap으로만 관리해야 하나? 둘 다 쓴다면 기준은? → (아래 "영역 파괴" 참고)

## ConfigMap — "영역 파괴의 주범"

**K8s 이전 vs K8s 환경의 배포 차이:**

```
[K8s 환경]
환경마다 파드 생성 → 모두 같은 이미지(Docker Hub) 다운
              → 환경마다 다른 값은 ConfigMap으로 주입

[K8s 이전]  여러 담당자가 각자 영역에서 환경변수를 관리
- 인프라 담당자: VM 환경변수 / OpenJDK 설치
- DevOps 담당자: Jenkins 환경변수로 실행 스크립트 구성
- 개발자: 환경별 properties 파일
```

K8s 이전엔 인프라/DevOps/개발자가 **각자의 영역**에서 환경변수를 관리했다. ConfigMap이 나오면서 이 **모든 역할을 하나로 통합**할 수 있게 됐다.

- ✅ 관리 포인트가 하나로 통일 → 슈퍼 개발자라면 반가운 일.
- ⚠️ 하지만 큰 프로젝트엔 영역별 담당자가 있고, 모두가 K8s를 다루는 것도, 모든 인프라가 K8s 위에 있는 것도 아니다.
- → ConfigMap에 값을 넣으려면 각 영역 담당자에게 "왜 여기 넣어야 하는지, 기존 구성이 어떻게 바뀌는지"를 설명·조율해야 한다. 사람에 따라 **자기 영역을 침범당한다**고 느낄 수 있고, 조율 결과에 따라 들어갈 값이 달라진다.

> 그래서 ConfigMap은 목적은 간단해 보여도 **영역을 넘나들고 정답이 없다**. 값 넣기가 어려운 건 자연스러운 일이며, 생각이 깊을수록 ConfigMap을 오래 들여다보게 된다.

**properties를 다 ConfigMap으로 빼야 하나?** → 기능적으론 가능하지만, **관리하는 사람의 입장**에서 전체적으로 어떻게 관리하는 게 유리한지가 핵심. (신규 프로젝트는 다 몰아넣어 볼 수도, 기존 전환은 영향도가 크니 **자주 바뀔 값부터 하나씩** 빼면 됨.)

## Secret — "이름 때문에 기대가 컸던" 오브젝트

### 기본: base64 인코딩 ≠ 암호화

- `stringData` 는 **쓰기 전용** 속성. 저장되면 ConfigMap과 같은 `data` 속성에 **value가 base64 인코딩**되어 들어간다.
- 파드에 파일로 마운트되면 다시 **디코딩**되어 원래 값이 보인다.
- ⚠️ base64는 **암호화가 아니다.** 누구나 쉽게 디코딩 가능. → 보안 기능을 기대하면 실망한다. "보안 작업을 줄여주는" 느낌이 아니다.

> ConfigMap에 비밀번호를 넣어도 되나? → **된다.** 암호화 방법은 다양해서 ConfigMap에 중요 데이터를 넣어도 안전하게 쓸 수 있다. 데이터 암호화는 이 오브젝트들과 **별개**로 생각할 것.

### Secret의 진짜 쓸모: `type`

Secret은 ConfigMap과 달리 **`type` 속성**이 있다. 타입에 따라 기능이 완전히 달라진다:

| type | 용도 | 파드 연결 속성 |
|------|------|---------------|
| `Opaque` (기본) | 투명. **ConfigMap과 동일**, 조금 중요한 데이터면 대체용으로 | env / volume |
| `kubernetes.io/dockerconfigjson` | **Private Docker Registry** 인증 정보. K8s가 이 정보로 이미지 다운로드 | `imagePullSecrets` |
| `kubernetes.io/tls` | 파드마다 다른 **TLS 인증서** 심을 때 (Sprint 3 Ingress에서 사용) | volume |

→ 즉, Secret은 각 타입마다 **정해진 key**를 채우는 **템플릿 역할**을 한다. (이 외에도 몇 가지 타입 더 있음. 단, **데이터 암호화를 제공하는 타입은 없다.**)

### 그럼 중요 데이터 암호화는 어떻게? (3가지 방법)

1. **접근 제어** — Secret을 파이프라인에 태우지 말고 **클러스터 내에서 직접 생성·관리**. K8s 관리 권한을 철저히 적용하면 아무나 Secret을 조회하거나 파드에 못 들어간다.
   - (단, 일반 오브젝트는 YAML 배포 위해 형상관리/배포 서버를 거쳐야 하므로 클러스터 내에서만 관리 불가. **비밀번호 든 Secret에 한해** 이 방식 사용.)
2. **문자 자체 암호화** — 특정 키로 문자를 암호화해 그 암호문을 Secret(또는 ConfigMap)으로 관리. 파이프라인 중간에 누가 봐도 실제 값은 모름. **앱 안에 복호화 로직**이 있어야 함(직접 코드 또는 WAS 제공).
3. **서드파티 도구** — HashiCorp **Vault** 등. 관리자가 Vault에 비밀번호 입력 → 앱 기동 시 Vault에 요청 → **지정된 파드에만** 값 제공.

> 2·3번의 장점: 누가 클러스터에 접근해도 앱의 중요 데이터 노출을 막을 수 있다. 암호화는 Secret 하나에 기대지 말고 **상황에 맞게 골라 쓸 것.**

### 주의: Secret을 환경변수(envFrom)로 넣지 말 것

기능적으론 Secret을 `envFrom` 으로 환경변수로 넣을 수 있지만, 파드에서 `env` 만 치면 누구나 DB 정보를 보게 되고 **앱 기동 로그에 환경변수가 통째로 찍히는 경우**도 많다. → 중요 데이터가 의도치 않게 노출되니 **지양**하고 **volume(파일) 방식**을 쓸 것.

## 한 장 요약

- 🎯 핵심 오브젝트 20%가 실무의 80%. 첫 단추를 잘 끼우자.
- 🧩 **오브젝트**: Namespace(그루핑/삭제 시 일괄 삭제) · Deployment→ReplicaSet→Pod(업데이트/복제본 관리) · Service(트래픽 연결) · ConfigMap/Secret(값 제공) · PVC/PV(저장) · HPA(스케일링).
- 🪜 **레벨**: 클러스터 레벨(Namespace, PV, Node) vs 네임스페이스 레벨(나머지).
- 🏷 **레이블·셀렉터·네이밍**: 표준 레이블(part-of/component/name/instance/version/managed-by), 셀렉터⊆레이블 매칭, 운영 전에 충분히 고민. 연결은 ①Selector-Label ②내부 속성 지정 2가지.
- ❤️‍🩹 **Probe**: Startup(기동 체크)→Readiness(트래픽 연결)/Liveness(생존→재기동). 목적이 다르므로 의도가 있으면 API를 구분하고, 헬스체크 로직은 가볍게. **일시 장애** 시 Liveness 주기를 길게 잡아 성급한 재기동을 막을 것.
- ⚙️ **ConfigMap**: 환경변수/파일로 값 주입(인프라 환경·기능 제어·외부 주입 값). 영역을 넘나드는 만큼 **담당자와 조율**이 핵심 — "영역 파괴의 주범".
- 🔐 **Secret**: base64는 암호화가 아님. 진짜 가치는 **type**(dockerconfigjson/tls 등 템플릿 역할). 중요 데이터 암호화는 ①접근제어 ②문자 암호화 ③Vault로 별도 해결. 환경변수 주입은 지양.
