# 쿠버네티스 스프린트

> Kubernetes의 핵심 오브젝트를 ① 그림으로 이해하고, ② 실제 앱 동작(Probe)으로 이해하는 첫 스프린트 정리.
> 강의 흐름(일프로 "쿠버네티스 어나더 클래스")을 이해하기 쉽게 재구성했다.

---

## 들어가며 — 오늘 배우는 게 왜 중요한가

### 파레토 법칙 (80/20)

> **상위 20%의 원인이 전체 결과의 80%를 만든다.**

오늘 배우는 오브젝트는 전체 Kubernetes 오브젝트의 **약 20%** 지만, **앞으로 실무에서 쓰게 될 오브젝트의 80%** 를 차지한다. → "언제 다 배우나" 부담 가질 필요 없이, 이 핵심부터 잡으면 실무의 대부분이 커버된다. **첫 단추를 잘 끼우는 단계.**

### 오브젝트를 이해하는 3단계 학습법

```
1단계  오브젝트를 "그려보며" 이해하기   ← 이 노트 1부
2단계  실제 "앱을 동작"시키며 이해하기   ← 이 노트 2부 (Probe)
3단계  쿠버네티스 "내부 동작" 이해하기   ← 다음 스프린트
```

기능마다 잘 이해되는 방법이 다르기 때문에 단계를 나눴다.
- 어떤 기능은 → **그림**을 그려야 이해된다.
- 어떤 기능은 → **실제 앱 동작**을 봐야 정확히 이해된다.
- 어떤 기능은 → **내부 동작**을 알아야 제대로 이해된다.

### 강사의 이해도 성장기 (이 순서가 효과적인 근거)

| 어떻게 공부했나 | 이해도 |
|----------------|--------|
| 책으로 개념 학습 | **30%** |
| 실무 프로젝트에 적용 (앱 동작과 함께 봐야 이해되는 영역 존재) | **60%** |
| 내부 동작까지 파고듦 | **90%** (남은 10%는 겸손함으로) |

# 1부. 오브젝트 그려보며 이해하기

> 이 단계의 목표: 각 오브젝트가 **어떤 역할을 하는지 대략적인 느낌**만 잡는 것. 처음엔 혼란스러운 게 정상이다.

## 강의 환경 구성

1. 마스터 노드에 접속해 디렉토리 생성 → 나중에 파드가 이 경로에 파일을 저장할 용도(PV).
2. 대시보드에서 야믈(YAML) 파일들을 하나씩 붙여넣어 오브젝트 생성.
3. 네임스페이스를 `kubernetes-anotherclass-123` 으로 변경.
4. 파드가 모두 정상 기동되면 성공.

## 핵심 오브젝트 한눈에 보기

### Namespace — 오브젝트를 그루핑

- 여러 오브젝트를 **묶어주는** 역할. metadata에 `name`, `label`.
- **삭제하면 그 안의 모든 오브젝트가 함께 삭제된다.** → 실습 정리 시 네임스페이스 하나만 지우면 끝. (단, PV는 네임스페이스에 속하지 않아 별도 삭제 필요.)

### Deployment — 파드를 만들고 업그레이드

- 소속 `namespace` 를 지정하면 그 네임스페이스에 속한다.
- 한 네임스페이스 안에서 **같은 종류 오브젝트끼리는 이름이 중복되면 안 된다.** (Service와 Deployment는 종류가 달라 이름이 같아도 됨.)
- `replicas: 2` → 파드 2개. `strategy: RollingUpdate` 는 Deployment의 핵심(업데이트 방식, 2부 실습에서).
- `template` 부터가 파드 생성 명세:
  - `nodeSelector` — 파드를 띄울 노드 선택
  - `containers.image` — Docker Hub의 이미지를 받아서 사용
  - `env` + `configMap` — 앱 환경변수 (값은 ConfigMap이 제공)
  - `resources` — CPU/메모리 할당량(request)과 상한(limit). **설정 안 하면 파드가 노드 자원을 다 써버린다.**
  - `volumeMounts` — 파드 내부 디렉토리(mountPath)를 만들고, 이름으로 PVC와 연결
  - `secret` 데이터 소스 — Secret 오브젝트와 연결

### Probe (3종) — 앱 상태 체크 *(자세한 건 2부)*

- **Startup Probe** — 앱이 기동됐는지 체크. 기동 실패하면 재기동. 성공하면 나머지 둘을 시작.
- **Readiness Probe** — 앱에 **트래픽을 연결할지** 결정.
- **Liveness Probe** — 앱이 정상이 아니면 **재시작**할지 판단.

### Service — 파드에 트래픽 연결

- metadata 3종 세트(`namespace`/`name`/`label`) + `selector`/`port`/`type`.
- 셀렉터로 파드를 찾아 트래픽을 연결한다.

### ConfigMap / Secret — 파드에 값 제공

- **ConfigMap** — 환경변수 등 외부에서 앱에 전달하는 데이터. `data` 에 값.
- **Secret** — ConfigMap과 비슷하나 **더 중요한 값**(민감 정보)에 사용. `stringData` 를 쓰면 그 내용으로 파드 안에 **파일**이 만들어진다.

### PVC / PV — 저장 공간

- **PVC (PersistentVolumeClaim)** — 파드가 PV를 지정할 때 사용. 저장 용량, accessMode(읽기/쓰기) 등. *이번 실습에선 `selector` 만 실제로 쓰임.*
- **PV (PersistentVolume)** — 실제 볼륨 지정. `local` 패스를 볼륨으로 사용, `nodeAffinity` 로 대상 노드(마스터 노드) 지정. → 그래서 환경 구성 때 디렉토리를 미리 만들었던 것.

### HPA (HorizontalPodAutoscaler) — 부하 따라 스케일링

- 대상으로 **Deployment** 지정.
- 최소 2개 ~ 최대 4개 파드 유지.
- `metrics`: 파드 CPU 평균 **40%** 초과 시 스케일 아웃.
- `behavior`: 한번 늘어난 뒤 **600초(10분)** 동안 추가 증가 억제 (2→3 직후 바로 4로 튀는 것 방지).

## 오브젝트 레벨: 클러스터 vs 네임스페이스

오브젝트의 가장 큰 범위는 **클러스터**다. 크게 두 분류:

| 레벨 | 오브젝트 예시 |
|------|--------------|
| **클러스터 레벨** | Namespace, **PersistentVolume(PV)**, Node |
| **네임스페이스 레벨** | Deployment, Service, ConfigMap, Secret, PVC, HPA … |

- 각 오브젝트는 자기 레벨이 정해져 있다. → Deployment를 PV처럼 클러스터 레벨에 만들 수 없다.
- PV가 네임스페이스에 안 속하는 이유 → 네임스페이스 삭제 시 PV는 **따로 한 번 더 삭제**해야 하는 이유.

## 레이블 · 셀렉터 · 네이밍 (가장 신경 써야 할 부분)

> 강사가 "시간이 지나서야 챙기기 시작했다, 운영에 올라간 뒤엔 수정이 매우 부담스럽다"며 **운영 전에 충분히 고민하라**고 강조한 영역.

### 레이블(Label) — 오브젝트에 붙이는 정보 + 연결 수단

옷의 라벨처럼, 오브젝트가 많아져도 레이블만 보면 **어떤 앱의 무슨 구성인지** 바로 파악할 수 있다. Kubernetes가 **권고하는 표준 레이블**이 있고, Prometheus가 이를 잘 따른다.

| 레이블 | 의미 | 예시 |
|--------|------|------|
| `app.kubernetes.io/part-of` | 애플리케이션 **전체 이름**(서비스 대표명) | prometheus |
| `app.kubernetes.io/component` | 서비스를 구성하는 분리된 **기능** | metrics, exporter |
| `app.kubernetes.io/name` | 애플리케이션 **실제 이름** | prometheus, kube-state-metrics, node-exporter |
| `app.kubernetes.io/instance` | **설치 단위 식별자** (같은 앱 여러 개 설치 시 구분) | prometheus-k8s |
| `app.kubernetes.io/version` | 앱 **버전** (버전 바뀌면 같이 수정 필요) | 1.0.0 |
| `app.kubernetes.io/managed-by` | 배포·관리한 **도구** | helm |

- `component` vs `name` 헷갈림 주의: 한 component(예: exporter = "메트릭 제공") 밑에 여러 name(kube-state-metrics, node-exporter)이 올 수 있다.
- 레이블 앞에 `app.kubernetes.io/` 같은 **prefix(도메인)** 를 붙이기도 함(선택). K8s 문서 예제에서 자주 보임.
- 레이블에는 **정보+기능** 역할을 하는 것과 **정보만** 주는 것이 있다.

### 네이밍 — K8s 권고는 없음, 강사 권고 예시

K8s는 오브젝트 네이밍 규칙(이름 중복 불가, 특수문자는 `-`/`.` 만 등)은 있지만 **"어떻게 지어라"는 권고는 없다.** 강사 권고:

- **Namespace 이름** = 그 안에 들어갈 앱들의 **범위**를 나타내는 이름 (예: `monitoring`). 강의에선 `kubernetes-anotherclass-123` (123은 수업 넘버링 — 이전 수업 오브젝트와 충돌 방지).
- **오브젝트 이름** = `앱 이름 + 인스턴스 식별자` 조합 → 보통 **레이블의 instance 값** 이 곧 오브젝트 이름이 됨 (유니크해짐). 예: Deployment `api-tester-123`.
- **사용 목적**을 이름 뒤에 덧붙이면 좋다:
  - ConfigMap이 앱 초기화용 데이터면 → 뒤에 `-rule` (Prometheus는 성능 계산 공식을 담아 이렇게 함)
  - 클러스터 내부용 Service → `-internal`, 외부 접속용 → `-external`
  - 특정 임계치용 HPA → 목적에 맞는 이름

### 셀렉터(Selector) & 레이블 매칭 — 오브젝트 연결

레이블은 정보 파악뿐 아니라 **셀렉터와 매칭되어 두 오브젝트를 연결**한다.

> **매칭 규칙: 셀렉터의 내용이 모두 레이블에 포함되어야 한다.**
> - 레이블에 추가 항목(예: version)이 더 있는 건 OK.
> - 반대로 셀렉터에만 version이 더 있으면 → 연결 안 됨.
> - instance가 유니크하므로 셀렉터에 instance만 넣어도 원하는 파드를 정확히 매칭 가능 (나머지는 선택).

연결되는 관계 예시: Deployment↔ReplicaSet, ReplicaSet/Service ↔ Pod, PVC ↔ PV.

### 오브젝트를 연결하는 2가지 방식

1. **Selector ↔ Label** 매칭 — ReplicaSet→Pod, Service→Pod, PVC→PV, Pod의 `nodeSelector`→Node 레이블, PV의 `nodeAffinity`→Node 레이블.
2. **오브젝트 내부 속성으로 대상 직접 지정** — HPA가 Deployment 이름 지정, Pod가 ConfigMap/Secret/PVC를 직접 지정.

> 셀렉터 표현 방식은 오브젝트마다 조금씩 다르다(Service, PVC, nodeAffinity 등). **외울 필요 없이** 잘 된 템플릿 하나를 두고 복사·수정해서 쓰면 된다.

### Deployment를 만들면 일어나는 일 (생성 흐름)

```
Deployment (api-tester-123)
    │  쿠버네티스가 자동 생성
    ▼
ReplicaSet (api-tester-123-<랜덤>)   ← replicas 값 복사, 복제본 관리
    │  replicas 수만큼 생성 + 항상 그 수 유지(파드 강제 삭제해도 다시 생성)
    ▼
Pod (api-tester-123-<랜덤>-<랜덤>)    ← 실제 앱 실행
```

> 역할 분담: **Deployment = 파드 업데이트 관리**, **ReplicaSet = 복제본 수 관리.**

# 2부. 앱 동작으로 이해하기 — Probe

## 들어가기 전: 기능 적용은 신중하게

> 좋은 관리 기능이 많다고 다 적용하면 **주객전도**. 주인공은 애플리케이션이고, 인프라는 앱을 잘 관리하기 위해 존재한다.

- 예: **NetworkPolicy**(파드 간 방화벽)는 이미 별도 보안팀/방화벽이 있다면 "굳이 또?"가 될 수 있어 **안 써도 되는** 애매한 기능.
- 반면 **Probe**는 **반드시 써야 하는** 기능인데, 앱에 대한 이해 없이 적용하면 **의도치 않은 파드 장애**를 만난다. → "새 툴 하나 더 배울 시간에 Probe를 깊게 파라."

## Probe 기본 동작

설정: 3종 Probe가 각각 URL(예: `/ready`)을 8080 포트로 **일정 간격**으로 호출, 성공/실패 임계치를 가진다. (앱에는 해당 헬스체크 URL이 미리 구현돼 있어야 함.)

```
파드 생성
   │
   ▼
[기동 중]  Startup Probe 동작 (10초 간격 호출)
   │   - 기동 중엔 응답 못 받아 계속 실패
   │   - 10번 실패 전에 1번이라도 성공하면 → 성공 간주
   ▼
Startup 성공 → Startup 중지, Readiness/Liveness 시작
   │
   ├─ Readiness Probe ─ 성공 시 외부 트래픽 연결(서비스 활성화)
   │                    실패 시 연결 해제
   └─ Liveness Probe ── 앱 생존 체크. 실패(예: 2회) 시 파드 재기동
```

### 흔한 의문 (그리고 답)

- "어차피 같은 URL 쓸 건데 왜 Probe마다 API를 따로 적지?"
- "장애 나면 Liveness·Readiness 같이 실패할 텐데 Readiness가 따로 필요한가?"

→ **목적이 다르기 때문.** (아래 "왜 생겼나"에서 해소)

## 로그로 보는 Probe 동작 (실습)

**준비:** HPA의 `minReplicas` 를 1로 변경. (HPA가 Deployment에 영향을 줘서 replicas를 1로 바꿔도 다시 2로 되돌리기 때문 → 로그/API 관찰 혼란 방지.)

Grafana 실시간 로그로 파드 재기동 과정을 관찰하면:

1. 앱 초기화 시작 → Spring/서블릿 초기화 → DB 연결(로그용 시뮬레이션).
2. 기동 중 **Startup Probe는 계속 실패** (5초 간격). 기동 완료되면 **성공**.
   - ※ 실제론 기동 전엔 앱이 API를 못 받으므로 Startup 로그가 안 찍히는 게 정상. 강의에선 학습용으로 일부러 로그를 찍게 구성(WAS/Tomcat이면 access log에서 확인 가능).
3. 기동 완료 후 사용자 초기화(`ConfigMap data loading` 등 외부 데이터로 추가 초기화).
4. 이 구간에 **Liveness는 성공, Readiness는 (의도적으로) 실패**.
5. 추가 데이터 작업 완료 → **둘 다 성공**, 이후 10초 간격으로 헬스체크 반복.

## 왜 Probe가 생겼나 — 앱 동작 관점

Kubernetes는 앱을 편하게 관리하려고 만들어졌다. 앱의 생애에는 자동화하고 싶은 요구가 있다:

| 앱 단계 | 상태 | 필요한 자동화 |
|---------|------|--------------|
| **앱 초기화** (Spring/DB 연결 등) | API 못 받음 | 초기화 끝났는지 체크 + 외부 API 차단 |
| **유저 초기화** (초기 데이터 로딩/검증 등) | 기동은 됨, API 받을 수 있음 | 앱 살아있는지 체크 + 외부 API 여전히 차단 |
| **완전 기동** | 정상 | 상태 체크 계속 + **외부 API 자동 허용** |
| **장애 발생** | 비정상 | 자동 **재기동** |

K8s 이전엔 이걸 운영자가 **수동**으로 했다 (API 찔러보고 → LB 스위치 수정해 트래픽 유입 → 모니터링 보다 장애 알람 받으면 접속해 재기동). Probe가 이 수동 작업들을 자동화한 것:

- **Startup Probe** → 앱 초기화 완료 판단. (성공하면 중지하고 나머지 둘 활성화)
- **Liveness Probe** → 생존 체크, 실패 시 **파드 재기동**.
- **Readiness Probe** → 성공해야 Service↔Pod 연결(트래픽 유입). 실패하면 연결 해제, 다시 성공하면 재연결.
  - ※ Service와 Pod는 Selector/Label로 매핑돼 있어도 Kubernetes가 **미리 연결해두진 않는다.** Readiness 성공 시점에 비로소 연결.

### 핵심 인사이트

- **Readiness ≠ Liveness인 이유:** 앱은 기동됐지만(=Liveness 성공) **유저 초기화 동안 외부 트래픽은 받기 싫을 때** Readiness를 일부러 실패시킨다. 이 의도가 없다면 둘이 같은 API를 써도 무방하지만, **의도가 있으면 API를 구분**해야 한다.
- Readiness의 API 결과는 "내가 의도한 초기화가 다 끝났다"를 알려줘야 한다.
- Startup/Liveness/Readiness 모두 단순 200 OK가 아니라 **각자 목적에 맞는 점검 로직**을 넣으면 좋다. 단, **Readiness/Liveness는 앱이 죽을 때까지 계속 호출되므로 로직을 가볍게** 만들 것.
- 앱 개발자와 K8s 관리자가 다르면 이 지점을 **충분히 협의**해야 한다.

## API로 직접 확인하기 (시나리오)

1. **앱 초기화 중** → 외부 API 호출 **실패**.
2. **유저 초기화 중** → 외부 API는 여전히 **실패**, 그러나 파드 내부에서 보낸 API는 **성공** (앱은 기동된 상태).
3. **완전 기동 후** → 외부 API **성공** (Readiness 성공 → 서비스 활성화).
4. **트래픽 중단 API 전송** → Readiness 3회 실패 → **Service-Pod 연결 해제** → 외부 API 실패.
5. **트래픽 재개** → 다시 연결 → 외부 API 성공.
6. **장애 발생 API 전송** → Liveness 3회 실패 → **파드 재기동** (기동 로그 다시 출력).

## 일시적 장애 상황에서 Probe 활용 (중요)

서비스 중 **일시적 장애**(순간 부하 급증, 렉, 스레드풀 포화 등)가 생기면:

- 외부 API가 계속 들어와 "불난 곳에 기름 붓기" → 시스템이 멈출 수도, **시간이 지나면 스스로 회복**할 수도 있다.
- 그런데 이 와중에 Liveness/Readiness가 실패하면 **Kubernetes가 파드를 재기동** → 회복할 수 있었던 앱이 강제로 죽고 **처리 중이던 작업이 모두 실패**한다. (Probe가 오히려 독이 되는 상황)

**완화 방법:**

| Probe | 설정 방향 |
|-------|-----------|
| **Readiness** | 실패 시 외부 트래픽을 차단해 부담을 덜어주므로 **그대로 둬도 OK** |
| **Liveness** | **체크 주기를 Readiness와 다르게**, 실패까지 걸리는 시간을 **더 길게** → 재기동을 지연시켜 스스로 회복할 시간을 준다 |

- 얼마나 지연시킬지는 앱마다 **시스템 관리 정책**으로 정해야 한다.
- 중요한 시스템은 보통 **파드를 이중화**하므로, 일시 장애 파드를 빨리 재기동하는 것만이 답은 아니다. **회복할 시간을 더 주는 것**도 고려할 것.

> 핵심: 내 앱을 잘 관찰하고, Probe를 어떻게 적용할지 깊이 고민하라.

# 3부. ConfigMap과 Secret

> "가볍게 이해하고 넘어가면 나중에 목소리 커질 일이 생기는" 오브젝트. 개념은 쉬워 보이지만 실무에선 **영역 파괴의 주범(ConfigMap)** 이자 **이름값 못 하는 기대(Secret)** 다.

## 기본 동작 복습

ConfigMap과 Secret 모두 파드에 바로 연결되어 **데이터를 주입**한다. 주입 방식은 파드의 연결 속성으로 결정:

| 연결 속성 | 결과 | 확인 방법 |
|-----------|------|-----------|
| `envFrom` | 데이터가 **환경변수**로 주입 | 파드 안에서 `env` 명령 |
| `volume` + `volumeMounts` | 데이터가 **파일**로 마운트 | 마운트 패스 파일 조회 |

## ConfigMap 데이터 — 환경변수의 다양한 쓰임

`data` 의 모든 key-value가 컨테이너 **환경변수로 주입**된다. 이미지에 넣어둔 java 실행 명령이 컨테이너 생성 후 자동 실행되며 이 환경변수를 읽는다. (값이 없으면 `null` 로 주입.)

환경변수는 단순히 "개발/검증/운영 구분"만이 아니라 쓰임이 다양하다:

| 분류 | 예시 | 설명 |
|------|------|------|
| **인프라 환경 값** | `SPRING_PROFILES_ACTIVE=dev` | 앱이 어느 환경에서 도는지 기동 시점에 알려줌. Spring은 이 값으로 환경별 `application-{profile}.yaml` 선택 (K8s 이전부터 쓰던 방식) |
| **앱 기능 제어 값** | `APPLICATION_ROLE` | 앱의 역할 지정. 한 앱의 여러 기능을 통째로 띄울 수도, **스케일링 모드**로 기능별로 따로 띄울 수도 (MSA 반영 오픈소스 多) |
| **외부 환경 주입 값** | `POSTGRESQL_FILEPATH` | Secret 데이터로 연결할 파일 경로. 경로 바꿀 때 **앱 재빌드 없이 ConfigMap만 수정** |

> 고민거리: 그럼 `application.yaml` 파일을 다 없애고 ConfigMap으로만 관리해야 하나? 둘 다 쓴다면 기준은? → (아래 "영역 파괴" 참고)

## ConfigMap — "영역 파괴의 주범"

**K8s 이전 vs K8s 환경의 배포 차이:**

```
[K8s 환경]
환경마다 파드 생성 → 모두 같은 이미지(Docker Hub) 다운
              → 환경마다 다른 값은 ConfigMap으로 주입

[K8s 이전]  여러 담당자가 각자 영역에서 환경변수를 관리
- 인프라 담당자: VM 환경변수 / OpenJDK 설치
- DevOps 담당자: Jenkins 환경변수로 실행 스크립트 구성
- 개발자: 환경별 properties 파일
```

K8s 이전엔 인프라/DevOps/개발자가 **각자의 영역**에서 환경변수를 관리했다. ConfigMap이 나오면서 이 **모든 역할을 하나로 통합**할 수 있게 됐다.

- ✅ 관리 포인트가 하나로 통일 → 슈퍼 개발자라면 반가운 일.
- ⚠️ 하지만 큰 프로젝트엔 영역별 담당자가 있고, 모두가 K8s를 다루는 것도, 모든 인프라가 K8s 위에 있는 것도 아니다.
- → ConfigMap에 값을 넣으려면 각 영역 담당자에게 "왜 여기 넣어야 하는지, 기존 구성이 어떻게 바뀌는지"를 설명·조율해야 한다. 사람에 따라 **자기 영역을 침범당한다**고 느낄 수 있고, 조율 결과에 따라 들어갈 값이 달라진다.

> 그래서 ConfigMap은 목적은 간단해 보여도 **영역을 넘나들고 정답이 없다**. 값 넣기가 어려운 건 자연스러운 일이며, 생각이 깊을수록 ConfigMap을 오래 들여다보게 된다.

**properties를 다 ConfigMap으로 빼야 하나?** → 기능적으론 가능하지만, **관리하는 사람의 입장**에서 전체적으로 어떻게 관리하는 게 유리한지가 핵심. (신규 프로젝트는 다 몰아넣어 볼 수도, 기존 전환은 영향도가 크니 **자주 바뀔 값부터 하나씩** 빼면 됨.)

## Secret — "이름 때문에 기대가 컸던" 오브젝트

### 기본: base64 인코딩 ≠ 암호화

- `stringData` 는 **쓰기 전용** 속성. 저장되면 ConfigMap과 같은 `data` 속성에 **value가 base64 인코딩**되어 들어간다.
- 파드에 파일로 마운트되면 다시 **디코딩**되어 원래 값이 보인다.
- ⚠️ base64는 **암호화가 아니다.** 누구나 쉽게 디코딩 가능. → 보안 기능을 기대하면 실망한다. "보안 작업을 줄여주는" 느낌이 아니다.

> ConfigMap에 비밀번호를 넣어도 되나? → **된다.** 암호화 방법은 다양해서 ConfigMap에 중요 데이터를 넣어도 안전하게 쓸 수 있다. 데이터 암호화는 이 오브젝트들과 **별개**로 생각할 것.

### Secret의 진짜 쓸모: `type`

Secret은 ConfigMap과 달리 **`type` 속성**이 있다. 타입에 따라 기능이 완전히 달라진다:

| type | 용도 | 파드 연결 속성 |
|------|------|---------------|
| `Opaque` (기본) | 투명. **ConfigMap과 동일**, 조금 중요한 데이터면 대체용으로 | env / volume |
| `kubernetes.io/dockerconfigjson` | **Private Docker Registry** 인증 정보. K8s가 이 정보로 이미지 다운로드 | `imagePullSecrets` |
| `kubernetes.io/tls` | 파드마다 다른 **TLS 인증서** 심을 때 (Sprint 3 Ingress에서 사용) | volume |

→ 즉, Secret은 각 타입마다 **정해진 key**를 채우는 **템플릿 역할**을 한다. (이 외에도 몇 가지 타입 더 있음. 단, **데이터 암호화를 제공하는 타입은 없다.**)

### 그럼 중요 데이터 암호화는 어떻게? (3가지 방법)

1. **접근 제어** — Secret을 파이프라인에 태우지 말고 **클러스터 내에서 직접 생성·관리**. K8s 관리 권한을 철저히 적용하면 아무나 Secret을 조회하거나 파드에 못 들어간다.
   - (단, 일반 오브젝트는 YAML 배포 위해 형상관리/배포 서버를 거쳐야 하므로 클러스터 내에서만 관리 불가. **비밀번호 든 Secret에 한해** 이 방식 사용.)
2. **문자 자체 암호화** — 특정 키로 문자를 암호화해 그 암호문을 Secret(또는 ConfigMap)으로 관리. 파이프라인 중간에 누가 봐도 실제 값은 모름. **앱 안에 복호화 로직**이 있어야 함(직접 코드 또는 WAS 제공).
3. **서드파티 도구** — HashiCorp **Vault** 등. 관리자가 Vault에 비밀번호 입력 → 앱 기동 시 Vault에 요청 → **지정된 파드에만** 값 제공.

> 2·3번의 장점: 누가 클러스터에 접근해도 앱의 중요 데이터 노출을 막을 수 있다. 암호화는 Secret 하나에 기대지 말고 **상황에 맞게 골라 쓸 것.**

### 주의: Secret을 환경변수(envFrom)로 넣지 말 것

기능적으론 Secret을 `envFrom` 으로 환경변수로 넣을 수 있지만, 파드에서 `env` 만 치면 누구나 DB 정보를 보게 되고 **앱 기동 로그에 환경변수가 통째로 찍히는 경우**도 많다. → 중요 데이터가 의도치 않게 노출되니 **지양**하고 **volume(파일) 방식**을 쓸 것.

# 4부. PVC/PV · Service/Deployment · HPA

### 배울수록 커지는 공부 범위 — PVC/PV로 보는 예시

> 처음엔 "Kubernetes만 따라가면 될 줄 알았는데", 공부하다 보니 추가로 배워야 할 게 계속 늘어난다.

강사의 경험담(개발자 출신):

1. 처음엔 Kubernetes가 제공하는 오브젝트들의 **사용법**을 익혔다. → 만들 줄은 알겠는데, **"내가 맞게 쓰고 있는 건가?"** 라는 의문이 생기기 시작.
2. 개발자 시절엔 볼륨은 인프라 담당자가 알아서 해줬고 **요청만 하면 됐다.** 할당받은 스토리지가 어떻게 생겼는지 알 필요가 없었다.
3. 하지만 PVC/PV를 **제대로** 쓰려면 **스토리지 개념**을 알아야 한다. — Probe 수업에서 앱 기동 동작을 알아야 Probe 속성을 잘 쓸 수 있었던 것과 같은 이치.
4. 실제 저장소를 제공하는 **솔루션**들이 있고, **PV는 이 솔루션과 연동하는 플러그인 역할**을 한다. → 솔루션 사용법도 어느 정도 알아야 한다.
5. 요즘은 **Kubernetes에 최적화된 네이티브 스토리지**도 많고, 이 솔루션들은 대체로 **쿠버네티스 위에 올라간다.** 운영 환경을 구축하다 보면 자의든 타의든 이런 스토리지를 써야 하는 상황이 생긴다.
6. Kubernetes를 모르면 인프라 담당자라도 이 솔루션을 설치·관리하기(또는 시키기) 힘들다. → 결국 **쿠버네티스를 하는 사람이 어떤 스토리지를 쓸지 정하고 직접 관리하는 추세.**

> 그래서 쿠버네티스를 하다 보면 전체 그림을 다 알아야 하고, **알면 알수록 배울 게 많다**고 말하는 것.

### 이번 강의의 범위와 이후 로드맵

| 어디서 | 무엇을 |
|--------|--------|
| **이번 강의** | 오브젝트들의 **기본 내용**. PV는 `local` 특성으로 **마스터 노드를 스토리지 볼륨으로 사용** — 많이 쓰는 케이스면서, 스토리지 개념·별도 솔루션 설치 없이 다룰 수 있어 한 강의로 소화 가능 |
| **해설편** | 몇몇 스토리지 플러그인 |
| **Sprint 3** | 스토리지 개념 + 볼륨 오브젝트의 상세 내용 |

> ⚠️ 실무에서 자주 쓰는 내용만 담았지만 **절대 가볍지 않다.** "이 오브젝트 좀 쓸 줄 안다"고 넘기지 말 것.

## PVC / PV — `local` 속성 사용법

이전 시간과 같은 그림에서, 이번엔 파드의 볼륨에 Secret이 아닌 **PVC를 연결**한다.

### 쓰이진 않지만 같아야 하는 속성

- PVC의 속성(용량·accessModes 등)과 PV의 비슷한 속성은 **이번 강의에선 실제 볼륨 속성으로 적용되지 않는다.** 필수 값이라 넣은 것이고 Sprint 3에서 자세히 다룬다.
- 단, **적용되지 않더라도 두 오브젝트의 이 속성 내용은 같아야 한다.** 둘은 Selector-Label로 연결돼 있지만, 이 속성 내용이 다르면 **연결 자체가 안 된다.**

### `local` + `nodeAffinity` — 노드를 스토리지로 쓰는 법

- PV에 `local` 속성을 쓰면 **path**를 지정하고, 이때 **`nodeAffinity` 속성이 필수**가 된다 (어느 노드에 파드를 생성할지 결정).
- nodeAffinity 내용이 복잡해 보이지만 **"PV에 local을 쓸 때의 루틴"** 이라 생각하고 통으로 쓰면 된다. 자주 변경할 일 없는 속성은 한 줄 한 줄 외워봤자 금방 까먹는다 — 필요할 때 자세히 보면 됨.
- 노드를 만들면 기본적으로 Node 오브젝트에 레이블이 달려 있고, nodeAffinity가 있는 PV에 연결된 파드는 **그 노드 위에 만들어진다.**
- 테스트 환경에서 별도 스토리지 구축이 번거로우니 **노드 중 한 곳(마스터 노드)을 임시 스토리지로 지정**하는 것. 마스터 노드에 path를 만들어 놓았기 때문에 **파드가 다른 노드에 생성되면 안 되고**, 그래서 nodeAffinity가 꼭 필요하다.

### 근본 질문: 왜 별도의 저장 공간이 필요한가

- 앱이 **컨테이너 안에만** 파일을 저장하면 **파드가 죽을 때 파일도 같이 삭제**된다.
- 임시 파일이면 상관없지만, 영구 보관 파일이라면 파드가 죽었다 다시 만들어져도 기존 파일이 보여야 한다.
- 구성해 둔 노드의 path에 파일을 저장하면 파드가 죽어도 데이터가 유지되고, 새 파드가 생기면 Kubernetes가 **기존과 똑같이 path를 연결**해 준다.

### hostPath — 더 쉽지만 조심해야 하는 방법

사실 local 속성은 PVC/PV를 설명하려고 쓴 것이고, 더 쉬운 방법이 있다: 파드 볼륨에 **`hostPath`** 로 path를 직접 지정하고, 파드에 `nodeSelector` 를 주면 노드도 지정된다. 더 간결하고 실제로 더 많이 쓰지만 **주의사항**이 있다.

- 공식 문서 경고: **"가능하면 hostPath를 사용하지 말 것."**
- 테스트용으로 편하다 보니 저장 용도로 오해하기 쉽지만, 원래 용도는 **노드에 있는 정보를 앱이 조회**하는 것.
  - 예: 지금까지 Grafana에서 모든 앱 로그를 볼 수 있었던 이유 — 파드들의 로그가 노드의 특정 path에 저장되고, **Loki의 Promtail 파드가 hostPath로 그 경로를 조회**하고 있었기 때문.
  - 그래서 문서에도 "필요한 파일/디렉토리 범위만 지정하고 **read-only로 마운트**하라"고 쓰여 있다.
- 테스트 환경에선 임시 저장용으로 많이 쓰지만, **운영 환경에서 저장 용도로는 절대 금지.** 보안 문제를 떠나 **노드의 저장 공간은 한정**돼 있다 — 한 파드가 디스크를 다 써서 노드가 죽으면 **그 노드 위의 다른 파드까지 전부 죽는다.**

**"모든 노드에 같은 경로로 NAS를 마운트하면 되지 않나?"** (강사가 예전에 실제로 했던 생각)

- 노드도 파드처럼 **언제든 죽을 수 있는 대상**이다. Kubernetes는 노드가 죽으면 파드를 다른 노드로 옮겨주고, 노드 수를 늘리고 줄이는 게 빈번해도 되도록 설계돼 있다.
- NFS 연결이 명령어 하나로 간단하더라도, **수동 연결이 하나 생기는 순간 이 자동화 설계가 무너지기 시작**한다. 예외 사항이 쌓이면 운영자 부담이 되고, 연결을 빠트린 노드에 파드가 올라가면 장애다.

### 왜 파드에 볼륨을 직접 붙이지 않고 PVC/PV로 나눴나

- 쿠버네티스 개념상 **파드를 만드는 주체는 개발자**다. 볼륨 솔루션은 종류가 많고 기능마다 써야 하는 속성이 달라지는데, 개발자가 파드를 만들며 거기까지 신경 쓰는 건 아니라고 본 것.
- 그래서 **인프라 담당자가 PV를 관리**하는 컨셉이고, 개발자는 **PVC로 필요한 자원을 요청**만 한다. 이 **인터페이스** 덕분에 PV의 솔루션이 변경돼도 파드까지 손댈 일이 없다.
- (다만 현실은 개발자/인프라 구분 없이 "쿠버네티스 아는 사람"이 다 하고 있어서 이 컨셉이 잘 와닿지 않을 수 있다.)

### 동작 확인 (순서만 — 상세는 자료실)

1. 파일 생성 API 2개 호출 → 각각의 폴더에 파일 생성 확인.
2. **세 군데 위치**(컨테이너 내부 경로들 + 노드의 path)에서 파일 확인.
3. 파드 삭제 후 같은 위치에서 파일이 유지되는지 다시 확인.
4. hostPath로도 똑같이 반복.

## Deployment — 업데이트 전략

### 언제 업데이트가 일어나나

- Deployment의 `strategy` 속성이 업데이트 방식을 지원하며 두 가지가 있다.
- **`template` 밑의 내용이 하나라도 변경되면 업데이트가 진행**된다. template 내용은 모두 파드로 들어가는 내용이므로, 변경이 있으면 그 내용으로 새 파드를 만드는 것.
  - 레이블만 바뀌어도 업데이트되고, 이미지를 V2→V1로 **다운그레이드해도** 업데이트 동작은 진행된다.

### 업데이트 시 오브젝트 변화

```
Deployment
   │ 업데이트 발생
   ▼
새 ReplicaSet 생성 (이름 뒤 난수만 다름) → replicas 수만큼 새 파드 생성
기존 ReplicaSet은 replicas가 0으로 변경 → 이전 파드 삭제
                └─ ReplicaSet 자체는 남는다 → 새 버전에 문제가 생겼을 때 롤백에 사용
```

### Recreate vs RollingUpdate

| | Recreate | RollingUpdate |
|---|---|---|
| 동작 | 기존 파드 **모두 삭제**와 동시에 새 파드 생성 | 새 파드 1개 생성 → **기동 완료되자마자** 기존 파드 1개 삭제, 반복 |
| 서비스 중단 | **있음** (첫 파드 기동 시간만큼) | **없음** |
| 자원 | 추가 없음 | 업데이트 중 **150%** 로 증가 |
| 특이사항 | 파드별 기동 시간 차이 발생 가능 | 업데이트 중 **두 버전이 동시에 호출**됨 |

- 두 버전 동시 호출이 문제라면 **Blue/Green 배포**: 두 버전이 동시에 호출되지 않지만 자원 **200%** + **별도 배포 솔루션** 필요.
- "그럼 블루그린이 제일 좋네?" → 운영에선 한 앱의 파드가 20개일 수도, 여러 앱을 동시에 업데이트할 수도 있어 **자원 200%는 쉽게 볼 문제가 아니다.** 상황에 맞는 배포 선택이 중요 (Sprint 2는 배포만 다룬다).

### maxUnavailable / maxSurge (RollingUpdate 세부 옵션, 디폴트 각 25%)

- **maxUnavailable** — 업데이트 동안 **서비스 상태로 유지해야 할 파드**에서 몇 개까지 빠져도 되는지.
- **maxSurge** — 새 파드를 **최대 몇 개까지 동시에** 만들지.

예제로 이해하기:

| 설정 | 동작 |
|------|------|
| 둘 다 **100%** | 파드 모두 삭제(서비스 불가 100% = 중단 발생) + replicas의 100%만큼 동시 생성 → **RollingUpdate로 Recreate와 같은 효과** (속성 설명용 예시일 뿐, 실제론 Recreate를 쓰면 됨) |
| **maxUnavailable 0 / maxSurge 100%** (강사가 종종 쓰는 설정) | 새 파드 2개 동시 생성 + 업데이트 중 **서비스 가능한 파드 수를 계속 2개로 유지** |

- 헷갈리는 포인트: maxUnavailable 0은 "기존 파드를 안 죽인다"가 아니라 **"서비스 되는 파드 수를 유지한다"** 는 뜻. 앱마다 기동 시간이 달라서 새 파드 하나가 기동 완료되면 기존 파드 하나가 죽는 식으로, 끝날 때까지 서비스 파드 2개가 유지된다.
- 파드 수가 많을 때 기본값(25%)을 그대로 쓰면: 배포 시작하자마자 새 파드 2개 생성 + 기존 1개 삭제 → 5개로 서비스하던 게 순간 4개가 된다 (20개였다면 차이가 더 큼). 그래서 이 설정을 쓴다.
- 대가로 자원 200%를 쓰지만 **업데이트 시간이 단축**되고, 파드 간 기동 시간이 비슷하면 **블루그린에 가까운 효과**도 난다.

### 동작 확인

1. RollingUpdate(기본 설정) → 2. maxUnavailable/maxSurge 수치를 바꿔가며 테스트 → 3. Recreate → 4. Rollback 확인.

## Service — 역할이 많아서 개념 잡기가 가장 어렵다

Kubernetes에서 서비스 개념이 어려운 이유 = **하는 역할이 많아서.** 하나씩 보자.

### ① 서비스 퍼블리싱 — NodePort로 외부 트래픽 연결

- 서비스는 셀렉터-레이블로 파드와 연결되고 `type` 이 있다. 지금까지 쓴 타입이 **NodePort**.
- 노드에 포트가 만들어지면서 **외부에서 보낸 트래픽을 클러스터 내부의 파드로 전달**한다.
- `ports` 속성: **targetPort** = 컨테이너(앱)의 포트, **nodePort** = 실제로 열리는 외부 포트 번호.

### ② ClusterIP와 Service Discovery — 내부 호출

- 타입을 넣지 않으면 디폴트는 **ClusterIP**: **오직 클러스터 내부의 파드에서만** 접근하는 용도.
- 서비스의 `port` (필수 속성, 예: 80)를 지정하면, **Service Discovery** 기능으로 Kubernetes **내부 DNS**에 서비스 이름이 자동 등록된다. → `http://서비스이름:80/...` 으로 호출하면 내부적으로 targetPort로 포워딩되어 파드에 전달.
- 파드/서비스 모두 생성 시 IP가 부여되지만, **파드는 삭제 시 IP가 바뀌므로 항상 서비스를 통해 호출**해야 한다. 서비스도 IP를 기억하기 힘드니 DNS(이름)를 쓰는 것.
- **다른 네임스페이스**의 파드가 호출할 땐 `서비스이름.네임스페이스이름` 까지 붙여야 한다.

### 자주 쓰는 패턴 — 포트에 이름 붙이기 (의외로 모르는 사람이 많음)

파드에도 `ports` 속성을 줄 수 있다: `name`(포트 성격) + `containerPort`(앱이 실제 노출하는 포트).

- 흔한 오해: "파드에 containerPort를 줘야 앱이 트래픽을 받는다" → **아니다.** containerPort는 기본적으로 **정보성 속성**이다.
- 그런데 서비스의 targetPort에 `8080` 대신 **`http` 같은 포트 이름**을 넣을 수 있다. 그러면 서비스 호출 시 파드에서 **이름이 매핑된 포트를 찾아** 그 포트로 트래픽을 보낸다.
- 효과: **컨테이너 포트가 변경돼도 파드만 수정하면 되고 서비스는 건드릴 필요가 없다.** 이때 containerPort는 정보성이 아니라 중요한 역할을 하게 된다.

### ③ 서비스 레지스트리 & ④ 로드밸런싱

- Deployment 업데이트 과정에서 파드가 죽고 새로 생겨도, **Kubernetes가 서비스에 연결되는 파드 IP를 즉시 제거/등록**해 준다. → 서비스만 연결해 놓으면 파드 생사에 신경 쓸 게 없다 = **서비스 레지스트리** 기능.
- 서비스에 트래픽을 날리면 **연결된 여러 파드로 알아서 분배**해 주는 **로드밸런싱** 기능도 있다.

### 동작 확인

1. **Service Discovery**: 별도 파드 없이, 기존 파드 내부에서 `서비스이름:80` 으로 API 호출.
2. **containerPort의 정보성 확인**: 파드의 ports(name) 설정을 삭제하고 서비스 targetPort를 `http` → `8080` 으로 바꾼 뒤 호출해 보기.

## HPA — 스케일링의 이상과 현실

### 기본 구조와 계산 공식

- `scaleTargetRef` 로 **Deployment를 지정**, 부하 시 min/maxReplicas 범위 내에서 replicas를 변경.
- `metrics` 에 CPU **평균 60%** 기준 스케일링 설정.
- **파드 resources의 `requests` 가 100% 계산의 기준값**이다. 예: requests CPU `100m` 일 때 앱이 70m 사용 중 → 그 파드는 70%.
  - 파드 2개 중 하나만 70%면 **평균 35%** → 스케일링 없음. 한 파드가 150% 정도 올라가야 평균 60%를 넘어 스케일 아웃 시작.
- 계산 공식: **변경될 파드 수 = ceil( 현재 파드 수 × 평균 CPU / 목표 CPU )**. 예: 계산 결과 2.51 → **올림 처리**되어 파드 3개.

### limits가 HPA 임계치에 주는 영향

- `limits` CPU 200m → 한 파드가 **최대 200%** 까지 올라갈 수 있다.
- 만약 limits = requests(100m)라면 한 파드는 100%를 넘을 수 없으므로, HPA 임계치를 낮추지 않는 한 **파드 하나의 부하만으로는 스케일링이 일어나지 않는다.**
- 포인트: **limits가 HPA 평균 임계치 도달 가능성에 큰 영향을 준다.**

### 메모리 메트릭은 왜 거의 안 쓰나

- 메모리는 CPU처럼 "쓰면 오르고 안 쓰면 내려가는" 자원이 아니다. 앱 기동 시 내부적으로 최대 사용량을 정해 두고, 사용량이 차오르면 **GC가 정리해서 수치를 낮추는** 식으로 움직인다.
- 즉 **사용 상황에 따라 변하는 값이지 부하에 따른 증감 대상이 아니라서**, HPA 옵션으로 제공되지만 거의 쓸 일이 없다.

### CPU만으로 충분한가

- 결국 기본 기능으론 CPU뿐인데, **CPU만으로 모든 앱의 부하 상태를 판단하긴 어렵다.** 앱마다 "부하 중"의 기준이 다르다 — 내부 큐 적체, 리퀘스트량, DB 커넥션 수, 스레드 수 등.
- 통상 이런 것들이 쌓일 때 CPU가 오르긴 하지만, **CPU가 오르기 전에 DB 커넥션이 부족해서** 파드를 늘려야 할 수도 있다.
- 제대로 쓰려면 다양한 메트릭을 얻는 **별도 솔루션**이 필요하지만, 내 앱을 CPU 기준으로 테스트해봤을 때 괜찮으면 HPA만 써도 된다.

### 이상 vs 현실

- **이상적인 그래프**: 부하가 꾸준히 증가 → 평균 60% 초과 → 스케일 아웃 → 새 파드 기동 완료로 평균 하락 → 트래픽 감소 → 스케일 인. ("Kubernetes 쓰면 자동 스케일링 됩니다" 홍보용 그래프)
- **현실**: 부하가 **급격히** 몰려오면 CPU가 치솟고 **파드가 모두 죽는다.** CPU 100% 초과로 스케일 아웃은 되지만, 죽은 기존 파드는 셀프 힐링으로 재시작 중 + 새 파드는 기동 중 → **그 시간만큼 서비스 중단.**
- 결론: **미리 예상하지 못한 트래픽에는 장사가 없다.** 자동 스케일링은 **보조 수단**으로 생각하고,
  - 서비스별 **피크 시간대를 분석해 미리 자원을 늘려놓거나**,
  - **대기열(큐) 아키텍처**를 구성하는 게 가장 확실하다.
- 그래도 파드 수가 많을수록 평균이 고르게 퍼져 이상적인 스케일링이 실제로 나오기도 하므로, 없는 것보단 낫다. (성능 테스트와 스케일링은 해설편에서 자세히.)

### behavior — 잦은 스케일링 방지 (유용!)

- Java 앱은 복잡한 로직의 API 하나로도 CPU가 **순간적으로 크게 튀기** 쉽다. 그때마다 스케일 아웃되면 새 파드가 만들어지고 바로 죽는 과정에서 CPU만 비효율적으로 쓴다.
- **scaleUp 안정화 윈도우(예: 120초)**: 2분 동안 CPU 60% 이상이 **유지**돼야 스케일 아웃.
- **scaleDown 안정화 윈도우(예: 600초)**: 부하가 줄어도 언제 또 올지 모르니 **10분 기다렸다가** 파드 수를 줄임.
- 더 세밀한 정책도 가능: 삭제될 파드가 5개라도 **1분에 하나씩** 삭제.
- (여담: 용어상 scale up/down은 한 서버의 자원 증감, scale in/out이 서버 개수 증감인데 속성 이름이 왜 ScaleUp/Down인지는 강사도 의문.)

### 동작 확인

1. behavior가 적용된 상태 → **2분 이상** 부하를 줘야 스케일 아웃되는 것 확인.
2. behavior 설정 삭제 → 부하 발생 시 **바로** 스케일 아웃되는 것 확인.

# 5부. 스프린트 1 마지막 강의 — 컴포넌트 동작으로 이해하기

> ☕ **(쉬어가기) 강사의 정규직 → 프리랜서 전향기.** SI 중소기업에서 개발+운영+제안서까지 여러 역할을 하다가, 표준(oneM2M) 문서를 스스로 분석하며 **주도적으로 일하는 프리랜서**를 만난 게 인식이 바뀐 계기. 과장 진급 후 관리자가 되는 대신 "개발을 오래 하는 돌파구"로 프리랜서를 택했다. 교훈: **프리랜서는 환상으로 시작하면 안 되고, "나는 정말 주체적으로 일하는 사람인가"를 스스로에게 물어볼 것.** 그래야 나이가 들어도 누가 일을 주지 않아도 스스로 일을 만들 수 있다.

## 클러스터 구성 요소 복습

- VM에 마스터 노드를 만들고 kubeadm으로 클러스터를 만들면, 특정 경로의 YAML 파일들로 **컨트롤 플레인 컴포넌트** 파드들이 생성된다 (Kubernetes가 돌아가는 데 필요한 역할들).
- 다른 VM으로 **워커 노드**를 만들어 조인하면(마스터와 동일한 것들 설치) 워커 노드 컴포넌트 영역이 생기고, Kubernetes는 **kube-proxy만 추가로** 만들어 둔다. 이 영역이 **사용자 앱을 올리는 공간**.
- ⚠️ 강의에선 마스터 노드에 앱을 올렸지만, 앱의 CPU가 치솟아 마스터 노드가 죽으면 **클러스터 전체가 안 돌아간다.** 운영에선 **앱은 워커 노드에** 올리고, 자원이 더 필요하면 워커 노드를 추가한다.
- 대시보드, 메트릭 서버처럼 K8s 기본 기능을 확장하는 앱들 = **애드온(Add-on) 파드**. 앞으로도 계속 설치하게 되고 그만큼 어려워지지만 다 필요한 것들이다.

## 오브젝트 vs 컨트롤러 vs 리소스

지금까지 전부 "오브젝트"라고 불렀지만 정확히는 나뉜다:

| 용어 | 의미 |
|------|------|
| **오브젝트** | 하나하나가 **인프라 개념으로서 단독 기능** (네트워크·볼륨·환경변수 같은 인프라 요소를 정의한 것) |
| **컨트롤러** | **타 컨트롤러나 오브젝트를 제어**(자동화) 하는 기능 (예: Deployment, ReplicaSet, HPA) |
| **리소스** | 이 전체를 통칭. 다시 **네임스페이스 레벨 / 클러스터 레벨**로 나뉨 |

- 전체 리소스는 **컨트롤 플레인 컴포넌트**들에 의해 동작한다. 단, **실제 컨테이너는 containerd가 만들고** Kubernetes는 생성 **요청**만 한다 — K8s가 직접 뭘 하는 게 아니라 노드에 설치된 기술들을 사용하는 것.

## 가성비 공부법 (강사 조언)

- 이런 기술 스택이 있어서, 어디까지 파느냐에 따라 학습 난이도가 달라진다: ① 리소스 개념/파드 만드는 법 → ② K8s가 파드를 만들기 위해 어떤 동작을 하는지 → ③ 리눅스의 어떤 기술로 컨테이너를 실체화하는지.
- 내부 깊은 영역은 통상 **Kubernetes가 알아서 잘 동작**한다. 깊이 알면 장애 때 빨리 해결하는 데 좋지만, "깊이 공부했고 + 안 까먹었고 + 마침 K8s 업무를 맡았고 + 우연히 장애가 났을 때" 얘기라 **써먹을 확률 대비 가성비가 낮다.**
- 강사 경험담: 처음에 오버레이 네트워크를 열심히 공부했지만, 지금 쿠버네티스는 그때보다 잘하는데 오버레이 네트워크를 더 잘 알지는 않고, 정리해 둔 걸 들여다볼 일도 없다. → **정말 궁금증이 몰려올 때만** 깊이 공부하고, 평소엔 이 강의 수준의 **전체 구성과 큰 역할**만 파악하면 충분히 오래 써먹는다.
- 한 오브젝트의 **제대로 된 개념과 활용도를 넓혀가는 것**이 K8s를 더 잘, 더 빠르게 하는 길이지, 처음부터 무거운 마음으로 깊이 파는 건 지양.

## 컴포넌트 동작으로 이해하기

> 몰라도 진도에는 문제 없는 내용이니 편하게 볼 것.

### 파드가 생성되는 과정 + Probe

```
kubectl(또는 대시보드) 로 Deployment 생성
   ▼
kube-apiserver 가 API 수신 → etcd 에 데이터 저장  (이제 조회/대시보드에 나타남)
   ▼
kube-controller-manager  ← DB를 계속 모니터링
   ├─ Deployment 발견 → "ReplicaSet 생성" API 전송
   └─ ReplicaSet 발견 → "Pod 생성" API 전송      ※ 아직 '데이터'만 있고 실제 컨테이너는 없음
   ▼
kube-scheduler  ← 노드들의 자원을 모니터링
   └─ DB에 노드 미지정 파드 발견 → 파드를 띄울 노드 결정(스케줄링, 사용자의 nodeSelector 등도 참고)
   ▼
kubelet  ← 자기 노드 정보가 있는 파드를 모니터링
   └─ 발견 → 컨테이너 런타임(containerd)에 컨테이너 생성 요청 → containerd가 컨테이너 생성
   ▼
Probe 설정이 있으면 → kubelet 이 설정대로 컨테이너에 헬스체크 API를 주기적으로 호출
```

### Service (NodePort) 동작

- 서비스를 파드에 연결하면 kubelet이 **kube-proxy**에 네트워크 생성을 요청 → kube-proxy가 **iptables**에 매핑 룰 추가 ("31231 포트로 들어오면 이 서비스로 가라").
- iptables는 리눅스로 들어오는 **모든 패킷을 관리**한다. 사용자가 API를 날리면 이 룰을 타고 컨테이너로 전달되는데, 이 실제 트래픽 전달 구간은 **CNI(칼리코)** 가 담당.

### Secret 동작 — 메모리에 마운트

- 컨테이너 내부에 마운트된 Secret 파일들은 **노드의 메모리 영역**에 마운트된다.
- 메모리는 전원이 꺼지면 지워지므로: 디스크 교체 시 실수로 데이터가 남거나, 디스크를 탈취당해도 일반 데이터처럼 복구할 수 없다 → 보안상 안전. (발생 가능하지만 딱히 와닿는 상황은 아님.)
- 오히려 실사용 관점: Secret을 많이 만들면 **노드 메모리가 부족**해진다 (그럴 정도로 만들 일은 잘 없지만 불안 요소).
- Secret 내용을 수정해도 컨테이너 안 파일이 바로 바뀌지 않는다 — **kubelet이 주기적으로 체크**하다가 변경사항을 업데이트하므로 **약간의 딜레이**가 있다.

### HPA 동작 — 최악의 경우 85초

```
containerd  ← 컨테이너의 실제 자원 사용량을 알고 있음
   ▲ kubelet 이 CPU/메모리를 10초마다 조회
   ▲ metrics-server (별도 설치 필수!) 가 60초마다 수집·저장
   ▲ kube-controller-manager 가 HPA 임계값과 메트릭을 15초마다 비교 → 스케일링 발생
```

- metrics-server가 없으면 HPA를 연결해도 **아무 일도 일어나지 않는다.**
- 수집 구간 타이밍이 잘 맞으면 즉시 스케일링, 최악의 경우 부하 발생 → 스케일링까지 **85초**(10+60+15).

## kubectl로 직접 확인 + 트러블슈팅

### 리소스/로그/파일 위치

- `kubectl api-resources` — 전체 리소스 목록: 이름/약어/버전/NAMESPACED(false면 클러스터 레벨).
- 주요 컴포넌트 파드는 **kube-system 네임스페이스**에 있고, `kubectl logs` 로 조회 가능 (설치가 잘 됐다면 볼 일은 적음).
- 마스터 노드의 주요 파일 위치:
  - K8s 설정 파일 위치에 **kube-apiserver 접속 인증서**가 있고, 설치 때 이걸 복사해 둔 것을 **kubectl이 참조**해 API를 날린다.
  - **컨트롤 플레인 컴포넌트를 띄우는 YAML** 파일들이 있는 경로(manifests)가 있다.
  - **모든 파드의 로그**가 저장되는 경로가 있으며(네이밍 룰 + 컨테이너별 파일), 파드 폴더의 데이터에 **링크로 연결**된 구조(중복 저장 아님). 보통은 `kubectl logs` 를 쓰지만 특정 시간대·문자열 검색이 필요할 때 직접 들어가 보기도 한다. (우리는 Loki의 Promtail이 이 파일들을 읽으므로 들어갈 일이 별로 없음.)

### 트러블슈팅 루틴 (이것만 알면 90% 커버)

1. **kubelet 상태 확인** — running이 아니면 restart/start. 안 올라가면 **상세 로그**를 보고 원인 구글링(10분).
2. 안 나오면 **VM 재부팅**. "한두 번 이상한 건 고쳐 쓰는 것보다 껐다 켜는 게 낫다."
3. 자주 재발하면 **VM 삭제 후 재설치.** "처음엔 잘 되다가 안 되는 거면 분명히 뭔가 건드린 것. **시스템은 거짓말을 하지 않는다.**" 스크립트 덕에 재설치 부담이 적으니, 본인이 추가한 작업은 **스크립트에 넣거나 한곳에 정리**해 둘 것 (정리 안 한 설정은 어차피 까먹고 못 써먹는다).
4. 그래도 같은 문제면 **해결될 때까지 에러 로그 구글링.** "구글엔 모든 답이 있다. 단지 내가 찾는 걸 포기했을 뿐." (실무 트러블슈팅도 결국 구글링. 똑똑한 친구보다 포기하지 않는 친구가 믿음직스럽다.)
5. **containerd·노드 상태** 확인.
6. **파드 상태** 확인 — Running이 아닌 게 있으면 **이벤트 로그** 확인 (전체는 너무 많으니 네임스페이스/타입 지정). 예: ConfigMap을 지우고 파드를 죽이면 "configmap not found" 이벤트가 찍힌다.
7. 이벤트가 없다면 리소스 문제가 아니라 **앱 기동 문제**일 수 있다 → **앱 로그** 확인. 자주 쓰는 옵션: 마지막 N줄(`--tail`), 실시간(`-f`), 최근 N분(`--since`).

### iptables의 NodePort 매핑 확인

- iptables에서 NodePort 매핑 내용을 조회하면 선언한 노드포트와 목적지가 보이고, **서비스 이름이 주석으로** 친절하게 달려 있다. (K8s가 알아서 잘 해주는 부분이라 찾아볼 일은 별로 없음.)

## 스프린트 1 마무리

> "지금까지 실습한 환경을 다시 만들면 얼마나 걸릴까?" → 아마 **1시간 이내.** 실무에서 K8s를 처음 맡은 사람이 이 정도 테스트 환경(앱 개발 포함)을 만들려면 **한 달 이상** 걸린다.
> 공부한 자료를 잘 정리해 두면 시간이 지나도 금방 적응하고 시작점이 앞선다 — **표준(K8s)이 있는 DevOps/인프라 분야의 장점**이며, 어느 프로젝트를 가도 비슷한 일을 할 확률이 높아 실력과 월급을 점진적으로 높일 수 있는 분야.

## 한 장 요약

- 🎯 핵심 오브젝트 20%가 실무의 80%. 첫 단추를 잘 끼우자.
- 🧩 **오브젝트**: Namespace(그루핑/삭제 시 일괄 삭제) · Deployment→ReplicaSet→Pod(업데이트/복제본 관리) · Service(트래픽 연결) · ConfigMap/Secret(값 제공) · PVC/PV(저장) · HPA(스케일링).
- 🪜 **레벨**: 클러스터 레벨(Namespace, PV, Node) vs 네임스페이스 레벨(나머지).
- 🏷 **레이블·셀렉터·네이밍**: 표준 레이블(part-of/component/name/instance/version/managed-by), 셀렉터⊆레이블 매칭, 운영 전에 충분히 고민. 연결은 ①Selector-Label ②내부 속성 지정 2가지.
- ❤️‍🩹 **Probe**: Startup(기동 체크)→Readiness(트래픽 연결)/Liveness(생존→재기동). 목적이 다르므로 의도가 있으면 API를 구분하고, 헬스체크 로직은 가볍게. **일시 장애** 시 Liveness 주기를 길게 잡아 성급한 재기동을 막을 것.
- ⚙️ **ConfigMap**: 환경변수/파일로 값 주입(인프라 환경·기능 제어·외부 주입 값). 영역을 넘나드는 만큼 **담당자와 조율**이 핵심 — "영역 파괴의 주범".
- 🔐 **Secret**: base64는 암호화가 아님. 진짜 가치는 **type**(dockerconfigjson/tls 등 템플릿 역할). 중요 데이터 암호화는 ①접근제어 ②문자 암호화 ③Vault로 별도 해결. 환경변수 주입은 지양.
- 📦 **PV local/hostPath**: local+nodeAffinity(필수)로 노드를 임시 스토리지로. 컨테이너에만 저장하면 파드 죽을 때 파일도 죽는다. hostPath는 원래 **노드 정보 조회용**(예: Promtail) — 운영에서 저장 용도 금지(노드 디스크 한정). PVC(개발자 요청)/PV(인프라 관리)는 **인터페이스** 분리 — PV 솔루션이 바뀌어도 파드는 그대로.
- 🔄 **Deployment 업데이트**: template 아래가 하나라도 바뀌면 업데이트(다운그레이드 포함). 새 ReplicaSet 생성, 기존 RS는 replicas 0으로 남아 **롤백용**. Recreate(중단 있음) vs RollingUpdate(중단 없음, 자원 150%, 두 버전 동시 호출) vs Blue/Green(200%, 별도 솔루션). **maxUnavailable 0 + maxSurge 100%** = 서비스 파드 수 유지 + 시간 단축.
- 🌐 **Service 4역할**: ①퍼블리싱(NodePort) ②디스커버리(내부 DNS, `서비스명[.네임스페이스]`) ③레지스트리(파드 IP 자동 등록/제거) ④로드밸런싱. targetPort에 **포트 이름**(name 매핑)을 쓰면 컨테이너 포트가 바뀌어도 서비스는 무수정.
- 📈 **HPA**: requests가 100% 기준, `ceil(파드수 × 평균/목표)` 로 계산, **limits가 임계치 도달 가능성을 좌우.** 메모리는 부적합(GC), CPU만으론 부하 판단에 한계. **예상 못한 트래픽엔 장사 없다** — 보조 수단으로 보고 피크 분석·대기열 아키텍처가 확실. behavior 안정화 윈도우로 잦은 스케일링 방지.
- ⚙️ **컴포넌트 동작**: apiserver→etcd 저장 → controller-manager(RS/Pod 데이터 생성) → scheduler(노드 결정) → kubelet(containerd에 요청, Probe 호출). Service는 kube-proxy가 iptables 룰 추가, 전달은 CNI. Secret은 **노드 메모리**에 마운트. HPA는 metrics-server 필수, 최악 85초.
- 🔧 **트러블슈팅**: kubelet 상태→재시작→로그 구글링→재부팅→재설치("시스템은 거짓말하지 않는다")→끝까지 구글링 → 파드 상태→**이벤트 로그**→앱 로그(tail/-f/since). 이것만 알면 90% 커버.
