## AI 요약

### 1. 왜 EBS 성능 진단은 어려운가?

EBS(Elastic Block Store)는 이름처럼 "디스크"처럼 보이지만, 실제로는 **네트워크 너머에 있는 원격 스토리지(networked storage)** 다. EC2 인스턴스에 로컬로 붙은 디스크가 아니라, 네트워크를 통해 연결된 분산 스토리지 시스템이다. 이 구조적 사실이 진단을 어렵게 만드는 모든 원인의 출발점이다.

```
  [EC2 인스턴스]
       │  애플리케이션이 발생시킨 I/O
       ▼
  [OS 블록 레이어]  ← iostat이 보는 지점 (await, queue)
       │
       ▼
  [EBS-Optimized 네트워크 대역폭]  ← 인스턴스 레벨 한도(병목 후보 #1)
       │  (다른 EBS 볼륨들과 공유)
       ▼
  ===== 네트워크 =====  ← 지연(latency)이 더해지는 구간
       │
       ▼
  [EBS 볼륨]  ← 볼륨 레벨 한도(병목 후보 #2)
       │  (물리 인프라를 다른 고객과 공유)
       ▼
  [실제 물리 스토리지]
```

| 특성 | 로컬 디스크 (Instance Store) | EBS (네트워크 스토리지) |
|------|------------------------------|--------------------------|
| 연결 방식 | 물리적으로 직결 | 네트워크 경유 |
| 지연 요소 | 디스크 자체 | 디스크 + 네트워크 + 공유 경합 |
| 인프라 공유 | 없음 | 다른 고객/볼륨과 공유 |
| 영속성 | 인스턴스 종료 시 소실 | 독립적으로 영속 |
| 진단 포인트 | 디스크 하나 | 볼륨 한도 + 인스턴스 한도 둘 다 |

핵심: **"느려졌다"는 증상은 OS에서 보이지만, 원인은 볼륨 한도일 수도, 인스턴스 한도일 수도, 공유 경합일 수도 있다.** 진단의 목표는 이 셋 중 무엇인지 가려내는 것이다.

### 2. 진단의 4대 핵심 지표

성능 이슈를 진단할 때 봐야 할 지표는 결국 "**얼마나 요청했나(수요)**"와 "**얼마나 받아냈나(공급)**", 그리고 "**얼마나 기다렸나(지연)**" 세 축이다.

| 지표 (CloudWatch) | 의미 | 진단 역할 |
|-------------------|------|-----------|
| `VolumeReadOps` / `VolumeWriteOps` → IOPS | 초당 I/O 연산 수 | 실제 처리량(공급) |
| `VolumeThroughput` (MiB/s) | 초당 전송 데이터량 | 대역폭 사용량(공급) |
| `VolumeAvgReadLatency` / `VolumeAvgWriteLatency` (ms) | OS가 측정한 I/O 완료 시간 | 증상(지연) |
| `VolumeQueueLength` | 처리 대기 중인 I/O 요청 수 | 적체 정도 |

여기에 AWS가 진단을 단순화하기 위해 제공하는 **결정적 두 지표**가 있다:

| 지표 | 값의 의미 |
|------|-----------|
| `VolumeIOPSExceededCheck` | 애플리케이션이 **프로비저닝된 IOPS 한도를 초과**해서 요청했는가? (1 = 초과) |
| `VolumeThroughputExceededCheck` | 애플리케이션이 **프로비저닝된 처리량 한도를 초과**해서 요청했는가? (1 = 초과) |

이 두 지표가 진단의 게임 체인저다. 예전에는 "지연이 왜 올라갔는지" 추측해야 했지만, 이제 이 값이 `1`이면 곧바로 "볼륨 한도에 부딪혔다"고 단정할 수 있다.

### 3. 진단 결정 흐름 (Decision Flow)

```
지연(latency)이 높다는 알람 발생
        │
        ▼
VolumeIOPSExceededCheck == 1  또는
VolumeThroughputExceededCheck == 1 ?
        │
   ┌────┴─────────────────────┐
   │ YES                      │ NO
   ▼                          ▼
볼륨 한도에 부딪힘           볼륨은 여유 있음
→ 프로비저닝 IOPS/처리량      → 다른 병목 의심:
  상향 또는 볼륨 타입 변경       1) 인스턴스(EBS-Optimized) 한도
  (gp2→gp3, gp3→io2 등)         2) 마이크로버스팅
                                3) 네트워크/공유 경합
```

**정상 상태의 기준선**:
- `io2 Block Express`: 1ms 미만(sub-millisecond) 지연 목표
- `gp3`: 한 자릿수 ms 평균 지연 목표
- 두 Exceeded 지표 모두 `0`

### 4. 함정 ①: 인스턴스 한도 vs 볼륨 한도

가장 흔한 오진은 **"볼륨은 여유 있는데 느린"** 경우다. EBS-Optimized EC2 인스턴스는 인스턴스에 붙은 **모든 EBS 볼륨의 합산 IOPS/처리량에 대한 상한**을 가진다.

```
예시) 인스턴스 한도: 합산 20,000 IOPS
      ├─ 볼륨 A: 8,000 IOPS 요청  (볼륨 한도 16,000 → 여유)
      ├─ 볼륨 B: 8,000 IOPS 요청  (볼륨 한도 16,000 → 여유)
      └─ 볼륨 C: 8,000 IOPS 요청  (볼륨 한도 16,000 → 여유)
         합계 24,000 > 20,000 → 인스턴스 레벨에서 스로틀!
```

이때 각 볼륨의 `VolumeIOPSExceededCheck`는 `0`인데도 `iostat`의 `await`(지연)와 큐가 치솟는다. **볼륨만 보면 영원히 원인을 못 찾는다.** 반드시 인스턴스 레벨 한도(인스턴스 타입 스펙)와 합산 사용량을 함께 확인해야 한다.

### 5. 함정 ②: 마이크로버스팅 (Micro-bursting)

CloudWatch 기본 지표는 **1분 평균**이다. 그런데 애플리케이션이 1초 동안 폭발적으로 I/O를 쏟아내고 59초를 쉰다면, 1분 평균으로는 "한도 이내"로 보인다. 하지만 그 1초 동안엔 한도를 초과해 스로틀이 걸린다 — 이것이 마이크로버스팅이다.

CloudWatch만으로는 안 보이므로, **OS 레벨의 `iostat`로 더 짧은 간격을 봐야** 한다:

```bash
# 1초 간격으로 확장 통계 관찰
iostat -xdmt 1
```

| iostat 컬럼 | 판단 |
|-------------|------|
| `r/s + w/s` (rps+wps) | 볼륨의 최대 IOPS를 초과하면 → IOPS 마이크로버스팅 |
| `rMB/s + wMB/s` | 볼륨의 최대 처리량을 초과하면 → 처리량 마이크로버스팅 |
| `aqu-sz`(큐 길이) ↑ + `await`(지연) ↑ | 스로틀 발생 중이라는 신호 |

### 6. 함정 ③: 큐 깊이와 버스트 크레딧

**큐 깊이(Queue Depth) 규칙**: 최대 일관성을 위해 볼륨은 *프로비저닝 IOPS 1,000당 평균 큐 깊이 1*을 유지해야 한다.

```
프로비저닝 3,000 IOPS → 적정 평균 큐 깊이 ≈ 3
큐 깊이가 과하게 높다(예: gp3에서 10 이상 지속) →
   애플리케이션이 볼륨이 처리할 수 있는 속도보다 빠르게 I/O를 던지는 중
```

**버스트 크레딧(`BurstBalance`)**: `gp2`, `st1`, `sc1` 볼륨은 버스트 버킷으로 동작한다. `BurstBalance`(%)가 0에 가까워지면 기준선(baseline) 성능으로 스로틀된다.
- 피크 시간마다 `BurstBalance`가 0으로 떨어진다면 → 크레딧 고갈이 지연 스파이크의 원인
- 해결: 버스트 크레딧 개념이 없는 **`gp3`로 전환**(일정한 기준 성능 제공)

### 7. 종합 진단 체크리스트

```
1. [지연 확인]   VolumeAvg(Read|Write)Latency 가 기준선 초과?
2. [볼륨 한도]   VolumeIOPS/ThroughputExceededCheck == 1?
                   YES → 볼륨 프로비저닝 상향 / 타입 변경
3. [인스턴스]    볼륨 합산 사용량이 인스턴스 EBS 한도 근처?
                   YES → 인스턴스 타입 업그레이드 / 볼륨 분산
4. [마이크로버스팅] iostat -xdmt 1 로 초당 r/s+w/s, rMB/s+wMB/s 가
                   순간적으로 한도 초과? → 워크로드 평탄화
5. [버스트 크레딧] gp2/st1/sc1 의 BurstBalance 0 접근? → gp3 전환
6. [큐 깊이]     aqu-sz 가 (IOPS/1000) 보다 과도? → 동시성/배치 튜닝
```

## 내가 얻은 인사이트

### 진단 방법론 관점

1. **"증상의 레이어"와 "원인의 레이어"를 분리하라**
   - 지연(latency)은 OS에서 측정되는 *증상*이지만, 원인은 볼륨/인스턴스/네트워크/공유 경합 등 더 아래 레이어에 있다. 네트워크 스토리지 진단의 본질은 "어느 레이어가 한도에 부딪혔나"를 좁혀가는 것이다. EBS만 그런 게 아니라 GCP Persistent Disk, Azure Managed Disk도 동일한 구조(원격 + 인스턴스 한도 + 디스크 한도)라서 이 멘탈 모델이 그대로 이식된다.

2. **Exceeded Check 지표는 "추측을 단정으로 바꾼다"**
   - 전통적으로 스토리지 진단은 "지연이 왜 높지?"라는 추측 게임이었다. `VolumeIOPSExceededCheck`/`ThroughputExceededCheck`라는 불리언 신호는 "한도 초과 여부"라는 가장 흔한 원인을 0/1로 즉답해준다. 진단에서 가장 비싼 건 시간인데, 이 지표는 가설 공간을 절반으로 잘라준다.

3. **평균값의 함정 — 시간 해상도를 의심하라**
   - CloudWatch 1분 평균이 "정상"인데 앱은 느린 마이크로버스팅 사례는, 모든 모니터링에 적용되는 교훈이다. **평균은 피크를 숨긴다.** p99 지연이나 짧은 간격 샘플링(iostat 1초)이 없으면 "지표상 멀쩡한데 사용자는 느린" 미스터리에 빠진다.

### 설계/운영 트레이드오프 관점

4. **gp2 → gp3 전환은 단순 비용 절감이 아니라 "진단 가능성"의 향상이다**
   - 버스트 크레딧 모델(gp2)은 성능이 시간/누적 사용량에 따라 변하므로 진단 변수가 하나 더 늘어난다. gp3는 기준 성능이 일정해서 *예측 가능*하고, 그 자체로 진단을 단순화한다. 일관성은 성능 숫자만이 아니라 운영 인지 부하의 문제다.

5. **인스턴스 한도라는 "보이지 않는 천장"을 항상 용량 산정에 포함하라**
   - 볼륨 IOPS를 아무리 올려도 인스턴스 EBS-Optimized 대역폭이 막혀 있으면 무의미하다. 신규 워크로드 사이징 시 "볼륨 스펙 × 개수"가 아니라 "인스턴스 합산 한도"부터 봐야 한다. 이건 비용 최적화 관점에서도 중요한데, 큰 인스턴스의 진짜 가치 중 하나가 EBS 대역폭이기 때문이다.

6. **큐 깊이 1/1000 IOPS 규칙은 "동시성 튜닝의 나침반"이다**
   - 큐가 적정선보다 낮으면 볼륨 성능을 다 못 쓰는 것이고(동시성을 더 줘도 됨), 과하게 높으면 볼륨이 못 따라오는 것이다(스로틀 임박). 이 규칙은 DB 커넥션 풀, fio 벤치마크의 `iodepth` 설정, 비동기 I/O 동시성을 정할 때 실전 기준이 된다.

### 실무 적용 관점

7. **진단은 "Top-down(클라우드 지표) + Bottom-up(OS 지표)"를 교차 검증해야 한다**
   - CloudWatch(상위, 1분 평균)와 iostat(하위, 1초)는 서로의 사각지대를 메운다. CloudWatch가 깨끗한데 iostat에서 큐가 튄다 → 마이크로버스팅. iostat은 멀쩡한데 앱이 느리다 → 스토리지가 아니라 애플리케이션/네트워크 문제. 두 시점의 교차가 곧 분리 진단(differential diagnosis)이다.
