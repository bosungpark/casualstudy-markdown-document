## AI 요약

### 1. Proxmox VE란?

Proxmox VE(Virtual Environment)는 **Debian 기반 오픈소스 가상화 플랫폼**으로, KVM(VM)과 LXC(컨테이너)를 하나의 웹 UI / REST API로 통합 관리할 수 있게 해준다. AWS·Azure 같은 퍼블릭 클라우드를 "내 서버에서 직접 굴리는 것"에 가까운 경험을 제공한다.

| 특성 | 설명 |
|---|---|
| 하이퍼바이저 | KVM (Type-1, 커널 내장) + QEMU |
| 컨테이너 | LXC (OS-level 가상화) |
| 관리 | Web UI + REST API + CLI |
| 라이선스 | 오픈소스 (AGPLv3), 유료 서브스크립션은 안정 채널/지원 제공 |
| 사용처 | 홈랩, 중소기업, 엣지, 온프레미스 사설 클라우드 |

---

### 2. 가상화 계층 - KVM vs LXC

Proxmox는 **두 가지 워크로드 모델**을 동시에 지원한다. 워크로드 특성에 따라 선택한다.

```
┌─────────────────────────────────────────────────────────────┐
│                     Proxmox VE Node                         │
│                                                             │
│  ┌──────────────────────┐    ┌────────────────────────────┐ │
│  │   KVM Virtual Machine│    │   LXC Container            │ │
│  │   (Full Virtualization)│  │   (OS-level Virtualization)│ │
│  │                      │    │                            │ │
│  │  Guest OS (Win/Linux)│    │  App + Userland Only       │ │
│  │  Virtual Hardware    │    │  (호스트 커널 공유)         │ │
│  └──────────┬───────────┘    └────────────┬───────────────┘ │
│             │                              │                │
│  ┌──────────▼──────────────────────────────▼──────────────┐ │
│  │         KVM + QEMU         |        LXC Runtime         │ │
│  └──────────┬──────────────────────────────┬──────────────┘ │
│             └───────────┬──────────────────┘                │
│                ┌────────▼────────┐                          │
│                │  Linux Kernel   │  (Debian Host)           │
│                └────────┬────────┘                          │
└─────────────────────────┼───────────────────────────────────┘
                          │
                    Physical Hardware
```

| 구분 | KVM (VM) | LXC (Container) |
|---|---|---|
| 격리 수준 | 강함 (별도 커널) | 약함 (커널 공유) |
| 부팅 속도 | 수십 초 | 1~2초 |
| 리소스 오버헤드 | 큼 | 거의 없음 |
| OS 선택 | Windows/Linux/BSD 자유 | Linux only |
| 적합 워크로드 | DB, Windows AD, 강한 격리 필요 서비스 | Web, microservice, 개발 환경 |

### 3. 클러스터(Cluster) 구조 - 여러 노드를 하나처럼

여러 Proxmox 노드를 묶어 **단일 데이터센터처럼** 관리할 수 있다. 클러스터를 구성하면 라이브 마이그레이션과 HA가 가능해진다.

```
        ┌─────────────────────────────────────────────┐
        │              Datacenter (Web UI)            │
        │   - 권한/유저/스토리지/네트워크 통합 관리      │
        └─────────────────────────────────────────────┘
                            │
        ┌───────────────────┼───────────────────┐
        ▼                   ▼                   ▼
   ┌─────────┐         ┌─────────┐         ┌─────────┐
   │ Node A  │◀───────▶│ Node B  │◀───────▶│ Node C  │
   │ pve-01  │ Corosync│ pve-02  │ Corosync│ pve-03  │
   │         │  (실시간) │         │         │         │
   │ VM/CT   │         │ VM/CT   │         │ VM/CT   │
   └────┬────┘         └────┬────┘         └────┬────┘
        │                   │                   │
        └─────── Shared Storage (Ceph/NFS/iSCSI) ─────┘
```

**핵심 용어**

| 용어 | 의미 |
|---|---|
| Node | Proxmox VE가 설치된 물리 서버 1대 |
| Cluster | 여러 노드를 묶은 논리적 그룹 (Corosync로 상태 동기화) |
| Quorum | 클러스터 의사결정용 다수결. **최소 3노드** 권장 (Split-brain 방지) |
| QDevice | 2노드 환경에서 외부 투표자 역할을 하는 경량 노드 |
| Datacenter | 클러스터 전체를 관리하는 최상위 논리 단위 |

> 2노드 클러스터는 한 노드가 죽으면 quorum(과반)을 잃어 클러스터 전체가 멈춘다. 그래서 3노드 또는 2노드 + QDevice 구성을 한다.

### 4. 스토리지(Storage) 계층

Proxmox는 다양한 스토리지 백엔드를 **추상화된 형태로 동일하게** 사용할 수 있게 해준다.

| 스토리지 | 특징 | 적합 시나리오 |
|---|---|---|
| **ZFS** | 파일시스템 + 볼륨매니저. 스냅샷/압축/체크섬/복제 | 단일 노드, 강한 데이터 무결성 |
| **LVM / LVM-thin** | 전통적 블록 스토리지, 씬 프로비저닝 | 로컬 단순 구성 |
| **Ceph (RBD)** | 분산 블록 스토리지, 자가 복구 | 3노드 이상 클러스터 + HA |
| **NFS / CIFS** | 외부 NAS 활용 | 백업, ISO 이미지 공유 |
| **iSCSI** | 외부 SAN 연결 | 엔터프라이즈 스토리지 통합 |

```
┌──────────────────────────────────────────────────────┐
│                  Proxmox Storage View                │
├──────────────────────────────────────────────────────┤
│  Content Types                                       │
│  - Disk image  (VM 디스크)                            │
│  - Container   (LXC rootfs)                          │
│  - ISO         (설치 이미지)                          │
│  - Template    (CT 템플릿)                            │
│  - Backup      (vzdump)                              │
│  - Snippets    (cloud-init 등)                       │
└──────────────────────────────────────────────────────┘
```

> HA를 쓰려면 **모든 노드에서 접근 가능한 공유 스토리지(Ceph, NFS, iSCSI)** 가 필수. 로컬 ZFS만 있으면 마이그레이션은 가능해도 진정한 HA는 불가능.

### 5. 네트워킹 - 가상 스위치와 분리 설계

Proxmox 네트워킹의 핵심은 **vmbr(Linux Bridge)** 라는 가상 스위치다.

```
       Physical NIC (eno1)
              │
       ┌──────▼───────┐
       │  vmbr0       │  ← Linux Bridge (가상 스위치)
       │  (가상 스위치) │
       └─┬──┬───┬───┬─┘
         │  │   │   │
       VM1 VM2 CT1 Host
```

**프로덕션 네트워크 분리 권장 구성**

| 네트워크 | 목적 | 권장 대역폭 |
|---|---|---|
| Management | Web UI, SSH, API | 1GbE |
| Cluster (Corosync) | 노드 간 상태 동기화 (낮은 지연 중요) | 1GbE 전용 |
| Storage (Ceph) | 스토리지 복제 트래픽 | 10~25GbE |
| VM Public/Workload | VM 외부 통신 | 10GbE + VLAN |

**고급 옵션**
- **VLAN 태깅**: 트래픽 분리(테넌트, 환경별)
- **Bonding (LACP)**: 링크 이중화 + 대역폭 증가
- **SDN(Software Defined Networking)**: 멀티 테넌트용 가상 네트워크, VXLAN 오버레이

### 6. 고가용성(HA) 동작 흐름

```
[정상 상태]
   Node A (VM100 실행 중)   Node B (대기)   Node C (대기)
        │                        │              │
        └────── Corosync로 상태 공유 ───────────┘

[Node A 장애 발생]
   Node A ✗   ──→  Quorum 손실 감지
                       │
                       ▼
                Node B/C가 다수결로 살아있음 확인
                       │
                       ▼
                HA Manager: "VM100을 Node B에서 재시작"
                       │
                       ▼
        Shared Storage(Ceph)에서 VM100 디스크 마운트
                       │
                       ▼
                Node B에서 VM100 부팅 완료 (1~2분 내)
```

**HA의 두 가지 모드**

| 종류 | 동작 | 다운타임 |
|---|---|---|
| Live Migration | 운영 중 VM을 다른 노드로 이동 (계획된 작업) | 거의 0 |
| HA Failover | 노드가 죽으면 다른 노드에서 자동 재시작 | 수십 초 ~ 수 분 |

### 7. 백업과 자동화

| 도구 | 역할 |
|---|---|
| **Proxmox Backup Server (PBS)** | 증분 백업, 중복 제거, 압축, 암호화 전용 서버 |
| **vzdump** | 노드 내장 백업 도구 (스케줄 가능) |
| **Templates** | "VM의 골든 이미지" - 클론으로 빠른 배포 |
| **Cloud-Init** | 부팅 시 IP, SSH 키, hostname 자동 주입 |
| **REST API** | Terraform/Ansible 연동, IaC 가능 |

```
Template (Ubuntu 22.04 + Cloud-Init)
    │
    ├── Clone → VM101 (web-01)  cloud-init: IP=10.0.0.11
    ├── Clone → VM102 (web-02)  cloud-init: IP=10.0.0.12
    └── Clone → VM103 (web-03)  cloud-init: IP=10.0.0.13
                    ↑
            Terraform / Ansible로 선언적 관리
```

### 8. 입문자용 최소 구성 vs 프로덕션 구성

| 항목 | 최소 (홈랩) | 프로덕션 |
|---|---|---|
| 노드 수 | 1대 | 3대 이상 |
| 스토리지 | 로컬 ZFS | Ceph 또는 외부 SAN |
| 네트워크 | 1개 NIC | 4개 분리 (Mgmt/Cluster/Storage/VM) |
| 백업 | 외장 디스크 | PBS + 오프사이트 |
| HA | 불가 | 가능 (Quorum + 공유 스토리지) |
| 비용 | 기존 PC 1대 | 서버 3대 + 10GbE 스위치 |

## 내가 얻은 인사이트

### 아키텍처 관점

1. **"Linux 커널 위에 모든 게 다 있다"는 단순함의 힘**
   - KVM은 별도 하이퍼바이저가 아니라 **리눅스 커널 모듈**, LXC도 커널의 cgroups/namespaces, 클러스터링은 Corosync, 스토리지는 ZFS/Ceph 다 리눅스 생태계 기술이다.
   - Proxmox는 새 기술을 만든 게 아니라 **검증된 리눅스 컴포넌트를 잘 묶어서 UI를 입힌 것**. 그래서 안정적이고 디버깅 가능하다 (VMware처럼 블랙박스가 아님).

2. **Quorum이라는 분산 합의의 본질**
   - 3노드 권장의 진짜 이유는 "장애 허용"이 아니라 **Split-brain 방지**다. 노드 A, B가 서로 안 보이는 네트워크 파티션이 생겼을 때 양쪽 다 "내가 살아있으니 VM을 띄우자" 하면 같은 VM이 2번 실행되어 데이터가 망가진다.
   - Quorum(과반) 룰은 **"수가 적은 쪽은 무조건 멈춘다"** 는 보수적 안전장치. 분산 시스템의 CAP에서 C(일관성)를 택한 결정.

3. **HA는 공짜가 아니다 - 공유 스토리지의 비용**
   - VM이 다른 노드에서 살아나려면 **그 노드도 디스크에 접근 가능해야** 한다. 즉 HA = 무조건 공유 스토리지 필요.
   - Ceph를 쓰면 별도 SAN 없이 노드들의 디스크를 모아 분산 스토리지로 만들 수 있지만, **3중 복제로 실제 사용 가능 용량은 1/3** 로 줄고 10GbE 이상 네트워크가 필수가 된다. "HA"라는 단어 뒤에 숨은 인프라 비용을 인지해야 한다.

### 실무 적용 관점

1. **Public Cloud 대안으로서의 현실적 위치**
   - 아티클이 말하는 "3년 기준 40~60% 절감"은 사실이지만, 그건 **워크로드가 안정적이고 인력이 있을 때만** 성립한다. 트래픽 스파이크가 큰 서비스나 운영 인력 없는 팀은 여전히 퍼블릭이 답.
   - Proxmox의 진짜 강점은 비용보다 **"데이터 주권"** 과 **"예측 가능성"** - 의료/금융/공공처럼 데이터 위치가 규제되는 도메인에서 강력하다.

2. **컨테이너 시대에 LXC가 잊혀진 이유와 재발견 가치**
   - Docker가 이미지 포맷과 빌드 워크플로우 표준을 가져가면서 LXC는 잊혔지만, **"가벼운 풀-OS 환경이 필요한 경우"** (systemd, ssh, cron 다 살아있는) 에는 여전히 LXC가 더 자연스럽다.
   - Proxmox 환경에서 "DB나 Redis 같은 stateful 워크로드는 LXC로, 앱은 LXC 안에서 Docker로" 같은 2계층 전략이 의외로 잘 동작한다.

3. **IaC와의 궁합 - Terraform Provider의 의미**
   - Proxmox는 REST API가 완전해서 Terraform Provider로 **퍼블릭 클라우드와 동일한 IaC 워크플로우**가 가능하다.
   - 이는 단순 편의가 아니라 **"온프레미스도 GitOps화 가능"** 하다는 뜻. AWS에서 운영하던 팀이 Proxmox로 옮길 때 학습 곡선을 크게 낮춰준다.

### 트레이드오프 관점

1. **VMware vSphere와의 비교 - "기능 90%, 가격 5%"의 함정**
   - Proxmox는 vSphere의 핵심 기능 대부분을 무료로 제공하지만, **운영 자동화 도구의 성숙도**(DRS, vMotion 최적화, NSX 같은 SDN, vSAN의 안정성)에서는 차이가 있다.
   - 즉 "엔지니어 1명이 1000대 관리"는 vSphere, "엔지니어 1명이 50대 관리"는 Proxmox가 적합. **운영 규모와 자동화 성숙도의 트레이드오프**.

2. **홈랩에서 프로덕션으로 가는 경계선**
   - 단일 노드 Proxmox는 정말 쉽다. 하지만 3노드 + Ceph + 네트워크 분리로 가는 순간 **운영 난이도가 10배** 뛴다. Ceph 튜닝, Corosync 지연 관리, NIC 본딩 등 진짜 분산 시스템 운영 지식이 필요해진다.
   - "쉽게 시작할 수 있다"는 강점이 "어려운 부분을 숨겨준다"는 의미는 아님을 인지하는 게 중요.

3. **OpenStack/CloudStack과의 포지셔닝**
   - OpenStack은 "퍼블릭 클라우드 같은 multi-tenant IaaS"를 만드는 도구라면, Proxmox는 **"내부 팀이 쓰는 가상화 인프라"** 에 가깝다.
   - 멀티 테넌시, 사용량 과금, 사용자 셀프 서비스 포털이 필요하면 OpenStack/CloudStack이, 인프라 운영팀이 자원을 관리·제공하는 모델이면 Proxmox가 적합. **"클라우드"라는 단어 안에서도 추상화 계층이 다르다.**
