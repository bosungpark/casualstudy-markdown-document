## AI 요약

### 1. eBPF란?

eBPF(extended Berkeley Packet Filter)는 **커널 소스 코드를 수정하거나 커널 모듈을 로드하지 않고도, 커널 안에서 샌드박스 프로그램을 안전하게 실행할 수 있게 해주는 리눅스 커널 기술**이다.

| 특성 | 설명 |
|------|------|
| 안전성 | 커널에 로드되기 전 verifier가 모든 실행 경로를 검증 |
| 성능 | JIT 컴파일로 네이티브 커널 코드 수준의 실행 속도 |
| 이벤트 기반 | hook에 부착되어 이벤트 발생 시에만 실행 |
| 호환성 | helper 함수라는 안정적인 API로 커널 버전 간 호환 유지 |

용어 참고: 원래는 Berkeley Packet Filter의 확장판이라는 뜻이었지만, 지금의 eBPF는 패킷 필터링을 훨씬 넘어서는 범용 기술이 되어 사실상 독립적인 고유명사로 쓰인다. 옛날 버전은 cBPF(classic BPF)라고 부른다.

### 2. 왜 혁신적인가 - JavaScript 비유

아티클은 eBPF를 **"커널의 JavaScript"**에 비유한다.

```
웹의 진화                          커널의 진화
─────────────────                 ─────────────────
정적 HTML                          고정된 커널 기능
    ↓ JavaScript 등장                  ↓ eBPF 등장
브라우저에서 안전하게                 커널에서 안전하게
임의 코드 실행 가능                   임의 코드 실행 가능
→ 웹 앱 혁신 폭발                    → 네트워킹/관측성/보안 혁신 폭발
```

기존에 커널 기능을 바꾸려면 두 가지 길뿐이었다:

| 방법 | 문제점 |
|------|--------|
| 커널 소스 수정 후 upstream | 커뮤니티 설득 + 배포판 반영까지 수년 소요 |
| 커널 모듈 작성 | 릴리스마다 깨질 수 있고, 버그가 커널 전체를 죽일 수 있음 (보안 경계 없음) |

eBPF는 세 번째 길을 연다: **런타임에, 안전성이 검증된 프로그램을, 재부팅 없이** 커널에 주입한다.

### 3. 실행 모델 - Hook과 이벤트

eBPF 프로그램은 이벤트 기반(event-driven)으로, 커널의 특정 지점(hook)에 부착되어 해당 이벤트가 발생할 때 실행된다.

| Hook 종류 | 예시 |
|-----------|------|
| 시스템 콜 | execve 호출 감지 |
| 커널 tracepoint | 스케줄러 이벤트 추적 |
| 네트워크 이벤트 | 패킷 수신(XDP), 소켓 연산 |
| kprobe / uprobe | 사전 정의된 hook이 없는 임의의 커널/유저 함수에 동적 부착 |

kprobe/uprobe 덕분에 커널이 공식 hook을 표준화해줄 때까지 기다릴 필요 없이 거의 모든 지점을 계측할 수 있다.

### 4. 프로그램의 일생 - 로딩, 검증, JIT

```
┌────────────────────────────────────────────┐
│ 1. C 등으로 작성 → Clang/LLVM 컴파일        │
│         ↓ eBPF 바이트코드                   │
│ 2. bpf() 시스템 콜로 커널에 제출            │
│         ↓                                  │
│ 3. Verifier 검증                           │
│    - 반드시 종료되는가? (무한 루프 금지)     │
│    - 초기화 안 된 변수/범위 밖 메모리 접근?  │
│    - 모든 실행 경로가 분석 가능한 복잡도?    │
│         ↓ 통과 시                           │
│ 4. JIT 컴파일 (바이트코드 → 네이티브 코드)   │
│         ↓                                  │
│ 5. Hardening (읽기 전용 메모리, Spectre     │
│    완화, constant blinding)                │
│         ↓                                  │
│ 6. Hook에 부착 → 이벤트 발생 시 실행        │
└────────────────────────────────────────────┘
```

핵심 포인트:

- **Verifier**: 프로그램의 모든 실행 경로를 시뮬레이션해서 "안전한 경우에만" 로드를 허용한다. 무한 루프, 잘못된 메모리 접근, 분석 불가능한 복잡도는 전부 거부된다. 로드하려면 기본적으로 root 또는 `CAP_BPF` 권한이 필요하다.
- **JIT 컴파일**: 검증을 통과한 바이트코드는 해당 CPU의 네이티브 명령어로 변환된다. 그래서 eBPF는 "커널에 짜넣은 코드만큼 빠르다".
- **Hardening**: 로드 후에도 프로그램이 위치한 커널 메모리를 읽기 전용으로 만들고, Spectre 대응 마스킹, JIT 스프레이 공격을 막는 constant blinding 등을 적용한다.

### 5. 구성 요소 - Maps, Helper, Tail Call

**Maps**: eBPF 프로그램은 상태를 저장하고 유저 공간과 데이터를 공유하기 위해 map을 쓴다. 커널 쪽 프로그램과 유저 공간 애플리케이션 양쪽에서 접근 가능하다.

| Map 타입 | 용도 |
|----------|------|
| Hash table, Array | 일반적인 key-value 저장 |
| LRU | 캐시성 데이터 |
| Ring Buffer | 커널 → 유저 공간 이벤트 스트리밍 |
| Stack Trace | 프로파일링용 스택 수집 |
| LPM(Longest Prefix Match) | 라우팅 테이블류 매칭 |

**Helper Calls**: eBPF 프로그램은 임의의 커널 함수를 직접 호출할 수 없다. 대신 커널이 제공하는 안정적인 helper 함수 API만 호출한다 (난수, 시간 조회, map 접근, 패킷 조작 등). 이 계약 덕분에 커널이 내부 구현을 바꿔도 eBPF 프로그램은 계속 동작한다.

**Tail Calls / Function Calls**: 프로그램 안에서 함수를 정의해 재사용하거나(function call), 실행 컨텍스트를 통째로 다른 eBPF 프로그램으로 넘길 수 있다(tail call — `execve()`와 비슷한 개념). 작은 프로그램들을 조합해 복잡한 로직을 구성할 수 있게 해준다.

### 6. 주요 사용 사례

| 영역 | 활용 | 대표 프로젝트 |
|------|------|---------------|
| 네트워킹 | 고성능 패킷 처리, 로드 밸런싱, kube-proxy 대체 | Cilium, Katran |
| 관측성 | 낮은 오버헤드로 커널/앱 이벤트 수집, 프로파일링 | Hubble, Pixie, bpftrace |
| 보안 | 런타임 위협 탐지, 시스템 콜 수준 정책 강제 | Falco, Tetragon |

세 영역 모두 공통 원리는 같다: **커널이 모든 것(패킷, 시스템 콜, 함수 호출)이 지나가는 길목이므로, 그 길목에 안전하게 코드를 꽂을 수 있으면 낮은 비용으로 관찰하고 개입할 수 있다.**

### 7. 개발 툴체인

대부분의 경우 eBPF 바이트코드를 직접 쓰지 않고 상위 레벨 도구를 쓴다.

| 도구 | 언어/스타일 | 적합한 용도 |
|------|-------------|-------------|
| bcc | Python 안에 C 코드를 내장 | 프로파일링, 트레이싱 스크립트 |
| bpftrace | awk/DTrace풍 원라이너 DSL | 빠른 임시(ad-hoc) 트레이싱 |
| libbpf | C/C++ 라이브러리 | 프로덕션급 eBPF 애플리케이션 |
| ebpf-go | Go 라이브러리 | Go 서비스에 eBPF 통합 |

공통 패턴: C로 커널 쪽 프로그램 작성 → Clang/LLVM으로 바이트코드 컴파일 → 라이브러리가 `bpf()` 시스템 콜로 로드 및 부착 → map/ring buffer로 결과 수신.

## 내가 얻은 인사이트

### 플랫폼 설계 관점

1. **"안전한 확장 지점"이 플랫폼을 만든다**
   - JavaScript가 브라우저를, eBPF가 커널을 플랫폼으로 바꾼 공통 요인은 "제3자의 코드를 안전하게 실행하는 메커니즘"이다.
   - 확장성을 열 때 관건은 기능이 아니라 안전성 보장(verifier, 샌드박스)이라는 점이 인상적이다. 안전 보장이 있어야 생태계가 폭발한다.

2. **안정적인 계약(helper API)이 호환성을 만든다**
   - eBPF 프로그램이 커널 내부 함수를 직접 못 부르게 막은 것은 제약이 아니라 설계다. helper라는 좁은 계약 덕에 커널 내부가 바뀌어도 프로그램이 살아남는다.
   - 내부 구현과 확장 API를 분리하는 원칙은 어떤 플랫폼/라이브러리 설계에도 그대로 적용된다.

### 안전성/검증 관점

3. **정적 검증으로 "실행 전에" 안전을 증명한다**
   - verifier는 테스트가 아니라 모든 실행 경로의 정적 분석으로 종료성과 메모리 안전성을 보장한다. 대신 "분석 가능한 복잡도"라는 표현력 제한을 받아들였다.
   - 표현력을 일부 포기하고 검증 가능성을 얻는 트레이드오프 — 커널처럼 실패 비용이 극단적으로 큰 환경에서는 합리적인 선택이다.

4. **다층 방어(defense in depth)의 실전 사례**
   - 권한 체크(CAP_BPF) → verifier → JIT 후 hardening(읽기 전용 메모리, Spectre 완화, constant blinding)까지, 한 층이 뚫려도 다음 층이 막는 구조다.
   - "verifier는 안전성 도구이지 보안 감사 도구가 아니다"라는 구분도 유용하다. 안전성(크래시 안 남)과 보안(악의적 행위 차단)은 별개의 층에서 다룬다.

### 관측성 실무 관점

5. **관측성 도구들의 공통 기반 기술**
   - SigNoz/OpenTelemetry가 앱 레벨 계측이라면, eBPF는 코드 수정 없는(zero-instrumentation) 커널 레벨 계측이다. Pixie, Cilium Hubble, Datadog 등 최신 관측성 도구가 eBPF 위에 서 있다.
   - 이벤트 기반 + 커널 내 실행이라 항상 켜두어도 오버헤드가 낮다는 점이 "프로덕션에서 상시 프로파일링"을 가능하게 만든 핵심이다.

6. **쿠버네티스 네트워킹과의 연결**
   - Cilium이 kube-proxy(iptables 기반)를 대체할 수 있는 이유가 바로 eBPF다. 패킷이 iptables 체인을 순회하는 대신, XDP/소켓 레벨 hook에서 바로 처리된다.
   - CRI, pause 컨테이너에 이어 쿠버네티스 데이터 플레인을 이해하려면 eBPF가 다음 필수 조각이라는 것을 확인했다.
