# 1부. DevOps 한방 정리

## 왜 역사 얘기는 안 하나

- 컨테이너 한방 정리 때는 히스토리가 의미 있었다 — Kubernetes가 표준이 되기까지 짧은 시간에 변경이 너무 많아서, 인터넷을 중구난방으로 보면 뭐가 맞는 정보인지 헷갈리기 때문.
- 반면 DevOps(2009년 등장)는 이미 안정기에서 **광범위한 영역으로 퍼졌고**, 산업/개발 언어마다 파이프라인 구성이 너무 다양하다. 전체 스토리를 다 설명하면 결국 원론적인 얘기밖에 못 하고, 그건 실무에 도움이 안 된다.
- 그래서 이 강의는 **강사가 경험해 본 실체 있는 내용**만 다룬다. 목표: DevOps가 언제 왜 만들어졌는지는 몰라도, **내가 DevOps를 구성해야 할 때 뭘 해야 할지 전체 그림을 그릴 줄 아는 사람**이 되는 것.

## 시작점 — 지상편 전체 빌드·배포 흐름 (반드시 눈에 익힐 것)

```
[CI/CD 환경]
 빌드 버튼 클릭
   ① GitHub에서 최신 소스 다운로드
   ② 소스 빌드 (Gradle) ─ Maven 저장소에서 라이브러리 다운 → 실행 가능한 jar 생성
   ③ 컨테이너 빌드 (Docker)
        - Docker Hub에서 OpenJDK 베이스 이미지 다운
        - 그 위에 내 jar를 넣어 컨테이너 이미지 생성
        - Docker Hub에 이미지 업로드
   ④ 배포: kubectl 로 파드 생성 명령
[인프라 환경(K8s)]
   ⑤ Kubernetes가 이미지를 Docker Hub에서 다운 → containerd에 컨테이너 생성 요청 → 앱 실행
```

## DevOps에서 가장 중요한 것 — 개발, 빌드, 실행 파일

> 아무리 복잡한 CI/CD 흐름이 있어도, DevOps의 핵심은 **개발하고 → 빌드해서 → 실행 가능한 파일을 만드는 것**이다.

- 개발 환경 = 내 PC. Mac/Windows 위에 IntelliJ, 툴킷은 OpenJDK, 맨바닥 대신 Spring Framework로 금방 앱을 띄운다.
- 코드를 실행 파일로 만들려면 **빌드**: Gradle로 빌드하되 **컴파일은 OpenJDK**가 한다. 만들어진 **jar는 OpenJDK의 JVM 위에서 실행**된다. → **빌드와 실행에는 항상 OpenJDK가 필요하다**는 것만 기억.
- 이 세 가지 동작이 너무 흔해서 중요성을 간과하기 쉽지만, **DevOps 구성의 알맹이**다. 환경별로 필요한 요소가 갈린다:

| 환경 | 필요한 것 |
|------|-----------|
| 개발 환경 | 개발+빌드+실행 전부 (IntelliJ, OpenJDK, Gradle, Spring) |
| 인프라 환경 | 실행 파일 + JVM → **OpenJDK만** 있으면 됨 |
| CI/CD 환경 | 빌드 → **OpenJDK + Gradle** (+ 실행 파일을 인프라로 넘기는 **배포** 과정 추가) |

### 인프라 환경 3종 — 각각의 존재 이유

| 환경 | 목적 | 핵심 포인트 |
|------|------|------------|
| **개발 환경** | 여러 개발자가 같이 쓰며, 내 코드가 **다른 앱과 문제없는지** 편하게 테스트 | 보안 해제, 임시 저장소 사용 등이 흔함 |
| **QA(검증) 환경** | 개발자가 아닌 **전문 테스트 담당자**용 | **최대한 운영과 똑같이** 구성해야 운영 배포 시 문제를 사전에 파악 (개발 환경처럼 보안 해제/임시 저장소 쓰면 안 됨) |
| **운영 환경** | 실제 서비스 | 핵심은 **이중화** |

- 운영 중 서버가 다운될 이유는 너무 많고, 가장 흔하지만 효율적인 대비가 **이중화**다.
- 그래서 운영까지 들어가는 오픈소스를 고를 땐 **이중화 구성이 문제없는지 확인이 필수** — 스펙만 보고 넘기지 말고 **실제로 이중화시켜 앱을 띄우고 테스트**해 볼 것.
- 운영 환경 구성은 프로젝트 일정의 **가장 마지막**인데, 이때 이중화가 안 되면 그동안 수없이 테스트해 온 오픈소스를 이제 와서 바꿔야 하는 상황이 온다 → 구축 일정이 무너진다.

### CI/CD 환경이 필요한 이유

- 혼자 개발하지 않기 때문. 여러 개발자의 코드를 **통합**해야 하고(GitHub로 최신 소스 유지), **Jenkins**가 GitHub에서 소스를 받아 빌드·배포한다. 이런 툴이 설치된 서버가 CI/CD 환경.

### 기존 구성 vs 컨테이너 구성

- **기존(컨테이너 이전)**: Jenkins에 OpenJDK+Gradle 세팅 → 빌드로 jar 생성. 인프라 환경(개발/운영 — OS가 다를 수 있음, 실제로 Red Hat 라이선스 비용 때문에 개발 환경은 무료 OS를 쓰는 경우 많음)에 OS+OpenJDK를 미리 구성 → 배포 = **jar 복사 + 실행 명령** (Jenkins 배포 스크립트의 두 명령).
- **컨테이너 구성**: 소스 빌드 → 컨테이너 빌드 → Docker Hub 업로드 → kubectl 배포 → K8s가 이미지 받아 실행. 컨테이너 안에 OpenJDK와 실행 파일이 들어 있다.
- 빌드·배포가 복잡해졌어도 **"개발하고 빌드해서 실행 파일을 만든다"는 핵심은 항상 존재**한다.

> 💡 처음 보는 기술을 쉽게 쓰는 사람들의 특징: **내가 필요한 게 뭔지 안다.** Docker를 처음 써도 기능을 하나하나 훑지 않고 "OpenJDK 베이스 이미지를 어떻게 얻지? 그 위에 내 jar를 어떻게 올리지?"에만 집중해 원하는 결과물을 만든다. 전체 시스템 흐름을 알고 필요한 것만 잘 알면 새 툴도 빠르게 파이프라인에 연결할 수 있다.

## DevOps 8단계와 단계별 오픈소스

> CI = 통합된 소스로 **빌드·테스트를 자동화**, CD = **배포를 자동화.** → 커밋하는 순간 운영 환경에 자동 배포되는 파이프라인.
> 8단계: **계획 → 개발 → 빌드 → 테스트 → 릴리즈 → 배포 → 운영 → 모니터링.** DevOps계의 OSI 7 레이어 같은 것 — 교과서 같지만, OSI도 "이 트래픽은 헤더를 봐야 하니 L7 로드밸런서가 필요해"처럼 실무에서 쓰인다. 이 8단계도 **대표 오픈소스로** 실무적으로 이해하자.

| 단계 | 대표 도구 | 메모 |
|------|-----------|------|
| **계획** | Jira, Notion (예전엔 Redmine) + 협업 메신저 **Slack** | 일정 공유·이슈 기록. Slack을 쓰는 진짜 이유는 아래에 |
| **개발** | IntelliJ, OpenJDK, Spring + **JUnit**, 코드 분석(FindBugs, PMD) | JUnit: 복잡한 로직은 하루만 지나도 이해가 안 됨 → 입력/결과로 확인하는 **테스트 코드**를 만들어 두면 남이 수정해도 안심. 코드 분석 툴은 코드 패턴의 버그 체크 |
| **빌드** | Gradle(소스), Docker(컨테이너) | 라이브러리 저장소는 Maven 저장소(퍼블릭) / **Nexus**(프라이빗) |
| **테스트** | JUnit(기능), JMeter(성능), Jacoco(커버리지) | 개발 단계에서 돌렸어도 **병합 후 결과가 달라질 수 있어** 빌드 단계에서 JUnit을 한 번 더 자동 실행. Jacoco는 테스트가 전체 로직의 어느 범위를 커버했는지 → 커버리지가 높을수록 신뢰도↑. JMeter는 자동화가 아니라 전문 인력이 날 잡아 수동 진행 |
| **릴리즈** | Dockerfile, K8s YAML → GitHub에 보관 | **배포 가능한 패키지를 만드는 과정.** 이 파일들도 변경 관리가 필요해 GitHub에 올린다. 릴리즈+배포가 Sprint 2의 주요 범위 |
| **배포** | **Kustomize, Helm, Argo CD** | 이전엔 kubectl로 배포했지만 이제 이런 도구로. 모두 이번 강의에서 배움 |
| **운영** | NGINX, Istio(트래픽 관리) 등 | 인프라 환경 안에 설치. 운영자는 이 툴들이 정상 동작하는지 확인 |
| **모니터링** | Grafana, Loki, Prometheus(자원·로그) + Jaeger, Zipkin(트래픽 추적) | MSA가 많아져 **서비스 간 트래픽 흐름 추적**이 중요해짐. 역시 인프라 환경(파드)에 설치 |

- 🔒 기업에서는 GitHub/Docker Hub 같은 **외부(퍼블릭)** 서비스 대신, 내부 데이터의 인터넷 업로드가 제한되는 경우가 많아 **GitLab, Docker Registry** 같은 제품을 **사내에 설치**해 쓴다 (비용이 들어도 많이 씀).
- 💬 **Slack을 쓰는 이유**: DevOps의 중요 포인트를 차지하는 툴들과 **연동**이 되기 때문. 파이프라인에서 발생하는 알람을 개발자/DevOps 엔지니어/운영자 방에 각각 울리도록 설정할 수 있다.
- 이렇게 DevOps가 복잡해졌기 때문에 기업은 **DevOps 엔지니어를 따로 뽑는다.** 단, 한 번에 다 만들 필요는 없다 — kubectl 배포부터 연결해 놓고 **장비 업그레이드하듯 하나씩** 바꿔나가면 된다.

## DevOps에 엮여 있는 IT 직군들 — 연차별 로드맵 (강사 주관)

각 영역엔 담당자가 있지만, **연차가 올라갈수록 영역을 넓혀야 한다.**

### 개발자

| 연차 | 목표 |
|------|------|
| ~2년 | 팀의 개발 환경에 빨리 적응, 주어진 개발 범위를 쳐내는 퍼포먼스 |
| ~4년 | IntelliJ 설정부터 GitHub까지 장애/변경을 **스스로 처리** |
| ~6년 | 개발 환경 몇몇 툴의 **초기 세팅을 주도**, CI의 빌드·테스트 자동화까지 |
| ~8년 | **새 트렌드를 리드** — 컨테이너 전환 프로젝트에서 내 앱을 컨테이너로 만들어 운영 배포·자동화까지 누구보다 먼저 |

- 9년차 이상이 되면 PM/PL/아키텍트/컨설턴트 같은 직책의 기회가 온다. 기회는 누구에게나 오지만 **전체를 모르면 유지를 못 한다.** ("제 상사는 몰라도 잘만 계신데요?" → 여기서 말하는 건 **기술적으로 인정받는** PM/PL 수준. 프리랜서로 치면 월 1,000 이상 계약이 되는 실력.)
- 이 연차쯤 되면 해보고 싶은 아키텍처/방법론을 시도할 **영향력**이 생긴다 — 그때 승승장구하려면 실력을 미리 갖춰야 한다. 영역 확장은 한 번에 안 되므로 **미리 경계를 풀어 둘 것.**

### DevOps 엔지니어

- 예전엔 전체 시스템을 알아야 해서 신입이 힘든 분야였지만, 지금은 고연차 엔지니어들이 기반을 잡아놔서 신입을 키울 수 있는 분위기가 됐다.
- ~2년: 팀의 CI/CD 흐름에 빨리 적응, 비슷한 파이프라인 추가는 스스로. ~4년: 운영 환경 설치 툴 + 개발~GitHub 구간 담당. ~6년: **모니터링 시스템 설치·구성**까지 — 모니터링은 **개발을 알아야** 제대로 구성되는 요소가 많으므로, DevOps로 시작했어도 개발을 꾸준히 알려고 노력할 것.

### 운영자

- ~4년: 만들어진 운영 환경을 모니터링하며 구성 익히기 (기술 스택보다 **담당 시스템 종류가 많아서** 시간을 길게 잡음). ~6년: 릴리즈·배포 구성이 운영에 문제를 일으키지 않는지 파악 + 그걸 모니터링하는 화면 추가. ~8년: **빌드부터 운영 배포까지 다 알고** 운영 편의를 위해 주도적으로 구성을 업데이트 → 이 정도를 **SRE(Site Reliability Engineer)** 라고 부른다 (CI/CD까지 관여하는 운영자).

> 강사 경험: 개발을 오래 하다 보니 어느새 배포까지 다 할 수 있었고, Kubernetes에 관심이 커지며 DevOps 엔지니어로 전직. 운영·모니터링까지 넓히다 보니 아키텍트 자리가 났을 때 쉽게 도전. **내 영역의 경계에만 머물지 말고 조금씩 다른 영역을 시도**하면 누구나 가능하다.

## DevOps 말고 다른 Ops들

> 각각 제대로 알려면 1시간 이상 들어야 하는 주인공들. 직접 해보지 않은 Ops는 강사도 잘 모른다(예: Kubeflow를 질문받고 찾아보니 MLOps의 유명 툴이었던 일화). 다만 **어떤 주제와 목적인지**는 알아두면 유익하다.

| Ops | 목적 | 메모 |
|-----|------|------|
| **GitOps** | 파이프라인을 **Git 하나로 통일** | DevOps 툴이 너무 많아져 선택조차 힘들어짐 → Git에는 이슈/협업/빌드/테스트/배포 기능이 다 있다(예: GitHub Actions는 Jenkins 대체 가능). 도입·유지 비용이 문제일 뿐, 도입하면 GitOps 구성을 많이 한다 |
| **DevSecOps** | **빠른 배포 + 보안**을 동시에 | 배포가 빨라졌는데 보안은 안전한가? → 파이프라인 중간중간 보안 검사를 자동화. 예: 빌드 전 소스 검사(SonarQube), 이미지 업로드 시 취약 라이브러리 검사, K8s 클러스터 보안 툴. 보안 솔루션 업체들이 홍보용으로 많이 쓰는 용어 |
| **MLOps** | 머신러닝/AI 분야의 DevOps | 핵심 인물은 **데이터 분석가**(교수/박사/연구원 — 개발은 잘 모름). 분석→검증→학습→평가로 모델을 만들고 시스템에 반영(**서빙**). DevOps가 개발자↔운영자 소통을 위한 파이프라인이라면, MLOps는 **데이터 분석가↔개발자** 소통을 위한 파이프라인. 전문 툴이 많아 ML 개발 경험자가 MLOps를 하는 게 유리 |
| **LLMOps** | 대규모 언어모델 특화 | 같은 AI라도 다루는 데이터 규모가 방대 → 특화 파이프라인. 현재 MLOps의 끝판왕 위치 |
| **FinOps** | **클라우드 비용 절감** | 클라우드는 좋은데 비용이 생각보다 많이 든다는 문제의식에서 출발. 가장 최근에 나왔고 아직 툴이 별로 없다 — 세미나 등으로 공감대를 만드는 초기 단계 (공감대 이후 툴이 나오며 파이프라인이 만들어지는 게 보통의 순서) |

# 2부. 손쉽게 DevOps 환경 구축 — CI/CD 서버 만들기

## Sprint 2 실습 환경 개요

- Sprint 1에서는 CI/CD 환경이 없어 인프라 환경만 구성했다. 이제 **CI/CD 환경(VM)** 을 추가한다.
- VirtualBox + Vagrant로 게스트 OS 생성 → 스크립트로 필요한 프로그램 일괄 설치 → 내 PC 브라우저에서 **Jenkins 대시보드** 접속.
- 흐름: 빌드 실행 → 강사 GitHub에서 앱 소스(Sprint 1 실습 앱) 다운 → 소스 빌드(jar) → 컨테이너 빌드 → **내 Docker Hub**에 업로드 → 배포 시 **내 GitHub**의 릴리즈 파일(Dockerfile/YAML) 사용.
- ⚠️ **본인 GitHub·Docker Hub 가입이 필수.** 지금까지는 강사 계정(1pro)의 이미지를 썼지만, 이제 이미지 이름에 **본인의 Docker Hub 유저네임**이 들어간다. 배포되는 YAML의 이미지명에 본인 유저네임이 있어야 내 저장소에서 이미지를 가져올 수 있다.

## Vagrant 설치 스크립트에서 일어나는 일

- **자원**: CPU 2코어 / 메모리 2GB / 디스크 30GB (디스크 확장 설정을 안 하면 기본 10GB로 만들어짐). Sprint 2부터는 **인프라 + CI/CD 두 VM을 동시에** 띄워야 하니 PC 자원 확인.
- **네트워크**: host-only 네트워크(VM 간 통신·호스트에서 접속용, IP 중복 불가 — 인프라 30번 / **CI/CD 20번**) + NAT(외부 인터넷용, IP가 같아도 인터넷 전용이라 문제없음).
- **설치 내용**:

| 설치 | 용도 |
|------|------|
| 리눅스 기본 설정 | 인프라 환경 구성 때와 동일 |
| kubectl | Jenkins에서 **배포**할 때 사용 |
| Docker | Jenkins에서 **컨테이너 빌드 + Docker Hub 업로드** |
| OpenJDK **17** + Gradle | **소스 빌드**용 (강사 소스가 17 기반) |
| Git | 소스 코드·릴리즈 파일 가져오기 |
| Jenkins + OpenJDK **11** | Jenkins 자체가 11에서 돌아가서 **설치용 11**을 하나 더 설치 |

- 운영 팁: `vagrant up` 은 최초 생성 때만 쓰고, 이후 켜고 끌 땐 VirtualBox UI 사용 권장 (Vagrant로 올리고 내릴 때 네트워크 인터페이스가 안 내려가 문제가 되는 경우가 있음). 설치 에러 시 `vagrant destroy` 로 전부 지우고 다시 `vagrant up`.

## 설치·설정 절차 (요약 체크리스트)

1. **Vagrant로 CI/CD 서버 생성** — 폴더 생성 → 스크립트 다운 → 플러그인 설치 → `vagrant up`. (PC마다 추가 설정이 필요한 경우는 정리된 자주 발생 문제 참고.)
2. **원격 접속 툴 세팅** — IP `192.168.56.20`, root/vagrant 자격 증명 저장.
3. **Jenkins 초기 세팅** — 초기 어드민 패스워드 확인 → 대시보드 접속 → 권장 플러그인 설치 → 관리자 계정 생성.
4. **전역 툴 설정** — Jenkins 관리 > Tools에서 직접 설치한 **JDK 경로**와 **Gradle 홈** 등록. (Gradle은 자동 설치 목록에 원하는 버전이 없을 수 있어 **수동 경로** 지정 — 이래서 자동 설치 체크를 해제.)
5. **Docker Hub 가입** — 유저네임이 곧 **컨테이너 이미지 이름**에 들어가니 잘 정할 것.
6. **Docker 사용 설정** — ① Jenkins 사용자에게 Docker 사용 **권한 부여**, ② CI/CD 서버에서 내 Docker Hub로 **docker login** (로그인 정보가 저장돼 VM을 껐다 켜도 유지. 단 중요 정보가 디스크에 남는 방식이라, 나중에 파이프라인 플러그인으로 해결 — 실무에서도 초반에 급히 구성할 때만 이렇게).
7. **인증서 복사** — 마스터 노드의 K8s config(인증서)를 SCP로 CI/CD 서버에 복사 → Jenkins가 kubectl로 K8s에 API를 날릴 수 있게 됨. `kubectl get pods` 로 확인.
   - 이 화면이 안 나오면 마스터 노드가 잘 떠 있는지 확인. (강사 팁: 마스터 노드 CPU 렉이 나면 재시작. 이후로는 **CI/CD 서버 먼저 → 마스터 노드** 순으로 기동.)
8. **GitHub 가입 + Fork** — 강사의 릴리즈 파일 리포지토리를 **Fork** → 실습 폴더(일련번호)에 Dockerfile·배포 YAML이 있음 → `deployment.yaml` 의 이미지 부분 `1pro` 를 **본인 Docker Hub 유저네임으로 수정 후 커밋.**
   - 강사가 소스를 수정하면 **Sync fork** 버튼으로 업데이트 (강의 중 파일이 다른 것 같으면 이걸 누르기).

## Jenkins 프로젝트 3종 만들기

### ① 소스 빌드 (Freestyle)

- GitHub project URL = **강사 GitHub**(앱 소스는 그대로 강사 것 사용) → 소스 관리 Git, 브랜치는 `master` → `main` 으로 변경(강사가 브랜치명을 main으로 바꿈).
- Build Step: **Invoke Gradle script** → 전역 설정에서 등록한 Gradle 선택 → Task에 빌드 전 디렉토리를 비우는 명령 포함(clean 후 build).
- 빌드 실행 → Console Output에서 소스 다운로드/Gradle 빌드 로그 확인 → 결과 위치에 **jar 파일** 생성 확인.

### ② 컨테이너 빌드 (Freestyle)

- GitHub project URL·소스 관리 Git 경로에 **본인 GitHub 유저네임**이 들어가야 한다. 브랜치도 main.
- Additional Behaviours > **Sparse Checkout paths**: 리포지토리 전체가 아니라 **해당 실습 폴더만** 다운로드.
- Build Step: **Execute shell** —
  1. 소스 빌드에서 만들어진 **jar를 Dockerfile이 있는 위치로 복사**
  2. `docker build -t <본인유저네임>/...` (⚠️ 태그의 유저네임 수정)
  3. `docker push <본인유저네임>/...` (⚠️ 여기도 수정)
- 실행 후 본인 Docker Hub에 이미지가 올라갔는지 확인.

### Dockerfile 뜯어보기 (별거 없다)

```
FROM  openjdk:17 ...      # Docker Hub에서 OpenJDK 17 기반 이미지 다운
COPY  *.jar  /.../app.jar # 내 jar를 컨테이너 안 지정 위치에 app.jar로 복사
ENTRYPOINT [java, ...]    # 실행 명령 — ConfigMap으로 주입받은 환경변수 3개를
                          #   스프링 시작 시 argument로 전달, app.jar 실행
EXPOSE 8080               # 컨테이너 포트 오픈 (트래픽 수신용)
WORKDIR ...               # 컨테이너 접속 시 홈 디렉토리
```

> 💬 **"쿠버네티스 전에 도커부터 공부해야 하나요?"** (자주 받는 질문) → **전혀 그럴 필요 없다.** 도커가 기초, 쿠버네티스가 심화인 관계가 아니다. K8s를 공부하다 도커가 나오면 **필요한 만큼만** 공부하면 된다 — K8s에서 알아야 할 여러 종목 중 하나가 도커인 것. 주로 컨테이너 빌드에서 쓰이니 개념 정도만 이해하고, 나중에 Dockerfile을 제대로 만들어야 할 상황이 올 때 그때 제대로 공부하는 게 효율적이다.

### ③ 배포 (Freestyle)

- 컨테이너 빌드 프로젝트와 거의 같으므로 새 아이템 생성 시 **Copy from**으로 복사.
- Sparse Checkout 경로만 배포 파일 위치로 변경 + Execute shell을 **다운받은 YAML을 kubectl로 그대로 배포**하는 내용으로 교체.
- 실행하면 K8s 대시보드에서 네임스페이스(anotherclass-212) 아래 Deployment/Service/ConfigMap/Secret/PV 등 리소스 생성 확인.

> 🏋️ **마무리 숙제**: 배포까지 성공해도 **파드는 정상 기동되지 않는 상태**다. 여기까지 왔으면 파이프라인은 잘 만든 것 — 이전 강의(트러블슈팅)를 잘 들었다면 **왜 기동이 안 되는지** 스스로 찾을 수 있다. (힌트: 파드 이벤트 로그부터.)

- 이번 구성은 CI/CD가 흘러가기 위한 **최소한의 기능**이며, 배포 강의가 진행되며 점점 고도화된다. 빨리 완성하려 서두르지 말고 **지금 어느 부분을 세팅 중이고 왜 필요한지** 생각하며 진행할 것.

## 한 장 요약

- 🎯 **DevOps의 핵심 = 개발 → 빌드 → 실행 파일.** 아무리 파이프라인이 복잡해져도 이 알맹이는 항상 존재. 빌드/실행엔 늘 OpenJDK. 전체 흐름과 "내가 필요한 것"을 알면 새 툴도 빨리 붙일 수 있다.
- 🏗 **인프라 환경 3종**: 개발(같이 테스트) · QA(운영과 동일하게) · 운영(**이중화가 핵심** — 오픈소스는 실제 이중화 테스트 필수, 일정 마지막에 터지면 답이 없다).
- 🧰 **8단계와 오픈소스**: 계획(Jira/Notion/Slack) · 개발(JUnit/FindBugs/PMD) · 빌드(Gradle/Docker, Nexus) · 테스트(JUnit/Jacoco/JMeter) · 릴리즈(Dockerfile·YAML을 GitHub에) · 배포(Kustomize/Helm/ArgoCD) · 운영(NGINX/Istio) · 모니터링(Grafana/Loki/Prometheus + Jaeger/Zipkin). 기업은 내부용(GitLab/Docker Registry)을 설치해 쓰고, Slack은 파이프라인 **알람 연동** 때문에 쓴다.
- 🧑‍💻 **직군 로드맵**: 개발자/DevOps 엔지니어/운영자 모두 연차에 따라 **영역을 넓혀야** 한다. 빌드~운영 배포까지 아는 운영자 = **SRE**. 9년차+ 직책(PM/PL/아키텍트)은 전체를 알아야 유지된다.
- 🔤 **다른 Ops들**: GitOps(Git으로 통일) · DevSecOps(배포 속도+보안 자동화) · MLOps(데이터 분석가↔개발자 파이프라인) · LLMOps(대규모 특화) · FinOps(클라우드 비용, 아직 초기).
- 🛠 **CI/CD 서버 구축**: Vagrant(2코어/2GB/30GB, IP .20) → kubectl/Docker/OpenJDK 17+Gradle/Git/Jenkins(+JDK 11). 본인 **GitHub Fork + Docker Hub** 필수, deployment.yaml 이미지명을 내 유저네임으로. Jenkins 프로젝트 3종(소스 빌드 → 컨테이너 빌드 → 배포), 마지막 파드 에러는 스스로 트러블슈팅.
