## AI 요약

### 1. 헥사고날 아키텍처란?

2005년 Alistair Cockburn이 제안한 아키텍처 패턴으로,
**핵심 비즈니스 로직(도메인)을 입출력 인프라(DB, 웹, 외부 API 등)로부터 명확히 분리**하는 것이 목적이다.
시스템을 느슨하게 결합된, 서로 교체 가능한 컴포넌트로 나눈다.

> ⚠️ "육각형"은 그냥 그림 그릴 때 편한 관례일 뿐이다.
> 사각형이든 팔각형이든 상관없다. 핵심은 **모양이 아니라 "분리(separation)" 원칙**이다.

| 특성 | 설명 |
|------|------|
| 별칭 | Ports & Adapters (포트와 어댑터) |
| 핵심 목표 | 도메인 로직을 기술 세부사항으로부터 격리 |
| 의존성 방향 | 기술이 도메인에 의존 (그 반대 아님) |
| 대표 효과 | 인프라 교체 자유, 테스트 용이, 도메인 의도 명확화 |

### 2. 전체 구조 한눈에 보기

```
        Driving (주도하는) 측                         Driven (주도되는) 측
        "애플리케이션에게 시킨다"                       "애플리케이션이 필요로 한다"

   ┌──────────────┐                                      ┌──────────────┐
   │ REST 컨트롤러 │                                      │   SQL DB      │
   │ CLI          │── Primary ──┐          ┌── Secondary ──│  파일시스템   │
   │ 테스트 코드   │   Adapter   │          │    Adapter    │  메시지 큐     │
   └──────────────┘             ▼          ▼               └──────────────┘
                          ┌─────────────────────┐
                          │   Primary Port (in)  │
                          │  ┌───────────────┐   │
                          │  │    도메인       │   │
                          │  │ (비즈니스 로직) │   │
                          │  └───────────────┘   │
                          │  Secondary Port (out)│
                          └─────────────────────┘
                                    ▲
                          기술이 도메인 인터페이스에
                              "맞춰" 들어온다
```

- **왼쪽(Driving / Primary)**: 도메인을 *사용해서* 무언가를 하려는 쪽. (사용자, 테스트, 다른 앱)
- **오른쪽(Driven / Secondary)**: 도메인이 일을 하기 위해 *필요로 하는* 쪽. (DB, 외부 서비스)

---

### 3. 핵심 4요소: Port vs Adapter

가장 헷갈리는 부분이다. **포트는 "인터페이스(약속)", 어댑터는 "구현(번역기)"** 라고 외우면 쉽다.

| 구분 | Port (포트) | Adapter (어댑터) |
|------|-------------|-----------------|
| 정체 | 인터페이스 / 계약 | 구현체 / 번역기 |
| 위치 | 도메인 **안쪽** | 도메인 **바깥쪽** |
| 역할 | "이렇게 쓸 수 있다 / 이게 필요하다" 선언 | 실제 기술을 포트 모양에 맞게 변환 |

#### Primary(Driving) 쪽
- **Primary Port**: 도메인을 *어떻게 사용할 수 있는지* 정의 (예: `OrderService` 인터페이스)
- **Primary Adapter**: 특정 기술의 요청을 도메인이 이해하는 형태로 변환
  - 예: REST 컨트롤러가 HTTP 요청 → 도메인 메서드 호출로 번역

#### Secondary(Driven) 쪽
- **Secondary Port**: 도메인이 *무엇을 필요로 하는지* 정의 (예: `OrderRepository` 인터페이스)
- **Secondary Adapter**: 도메인의 호출을 실제 기술 호출로 변환
  - 예: `OrderRepository`를 구현한 JPA/SQL 어댑터

> Driven 액터는 두 종류로 나뉜다.
> - **Recipient(수신자)**: 정보를 받기만 함 (프린터, 메시지 큐)
> - **Repository(저장소)**: 정보를 주고받음 (DB, 파일시스템)

### 4. 의존성 역전(Dependency Inversion)이 핵심

헥사고날의 마법은 **의존성 방향**에 있다.

```
  ❌ 전통적 계층형:   도메인 ──→ DB 라이브러리 (도메인이 기술에 의존)

  ✅ 헥사고날:        도메인  ←── DB 어댑터    (기술이 도메인에 의존)
                    (Secondary Port 인터페이스를 어댑터가 구현)
```

도메인은 **인터페이스(포트)만 알 뿐**, 그 뒤에 MySQL이 있는지 MongoDB가 있는지 모른다.
기술이 도메인의 인터페이스에 *맞춰서* 들어오기 때문에, 도메인은 절대 기술에 오염되지 않는다.

### 5. 권장 패키지 구조 (기능별 구성)

```
feature/
├── domain/                  # ← 기술에 독립적인 순수 영역
│   ├── model/               #   도메인 엔티티
│   ├── services/            #   비즈니스 로직
│   ├── repositories/        #   Secondary Port 인터페이스
│   └── exceptions/
├── adapters/                # ← 기술 의존 영역
│   ├── http-api/            #   Primary Adapter
│   │   ├── controllers/
│   │   ├── model/           #   DTO
│   │   └── mappers/
│   └── database/            #   Secondary Adapter
└── config/                  # ← 모든 걸 와이어링(조립)하는 곳
```

핵심: `domain/`은 `adapters/`를 **import 하지 않는다.** 의존은 항상 바깥 → 안쪽으로만.

### 6. 개발 순서 (테스트 주도와 잘 맞음)

각 포트는 **최소 2개의 어댑터**(가짜 1 + 진짜 1)를 갖는 것이 이상적이다.

1. 테스트 하네스 + 가짜(mock) Secondary 어댑터
2. 진짜 Primary 어댑터 + 가짜 Secondary 어댑터
3. 테스트 하네스 + 진짜 Secondary 어댑터
4. 진짜 Primary + 진짜 Secondary 어댑터 통합

→ DB나 UI 없이도 도메인 동작을 먼저 검증할 수 있다.

## 내가 얻은 인사이트

### 설계 관점
1. **"포트=인터페이스, 어댑터=구현"이 전부의 80%다**
   - 용어가 추상적이라 어려워 보이지만, 결국 우리가 늘 쓰는 인터페이스/DI 패턴을 *방향성 있게 강제*한 것이다.
   - Spring을 써봤다면 `Repository` 인터페이스(포트)와 JPA 구현(어댑터)이 이미 헥사고날의 절반이다.

2. **육각형 모양에 집착하지 말 것**
   - 면이 6개인 이유는 "여러 종류의 어댑터가 붙을 수 있다"는 것을 그림으로 보여주려던 것뿐.
   - 진짜 메시지는 단 하나: *도메인을 기술로부터 격리하라.*

### 실무 관점
3. **테스트 속도가 곧 보상이다**
   - 도메인이 DB/네트워크에 의존하지 않으니, 핵심 로직을 in-memory 가짜 어댑터로 밀리초 단위로 테스트할 수 있다.
   - "DB 띄우고 통합 테스트 30초" → "순수 단위 테스트 0.01초"의 차이.

4. **기술 교체 비용을 미리 0에 가깝게 만든다**
   - 초기엔 빠른 개발을 위해 document DB를 쓰다가, 나중에 SQL로 갈아탈 때 *도메인은 한 줄도 안 바뀐다.* 새 어댑터만 작성.
   - 외부 결제사 교체, 메시지 브로커 교체에도 동일하게 적용.

### 트레이드오프 관점
5. **작은 CRUD 앱엔 과한 오버엔지니어링일 수 있다**
   - 포트/어댑터/매퍼/DTO가 늘어나며 보일러플레이트가 증가한다.
   - 도메인 로직이 복잡하고 오래 살아남을 시스템, 인프라가 바뀔 가능성이 큰 프로젝트에서 진가를 발휘한다.
   - 클린 아키텍처/DDD와 자연스럽게 결합되며, 사실상 같은 철학("의존성은 안쪽으로")의 다른 표현이다.
