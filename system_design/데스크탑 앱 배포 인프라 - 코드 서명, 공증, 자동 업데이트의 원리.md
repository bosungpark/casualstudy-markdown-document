## AI 요약

### 1. 배포 인프라란 왜 별도의 학습 주제인가?

앱을 "만드는 것"과 "사용자 PC에서 신뢰받으며 실행되고, 스스로 갱신되게 하는 것"은 완전히 다른 문제다. 현대 OS는 **출처 불명의 바이너리를 위협으로 간주**하기 때문에, 코드를 짜는 일만큼이나 **서명 → 패키징 → 공증 → 배포 → 자동 업데이트**라는 파이프라인이 중요하다.

| 단계 | 하는 일 | 안 하면? |
|---|---|---|
| **코드 서명** | "이 앱은 내가 만들었다" 증명 | 실행 차단/경고 |
| **공증(notarization)** | Apple이 악성코드 스캔 후 발급 | macOS가 실행 거부 |
| **스테이플링** | 공증 티켓을 앱에 박음 | 오프라인 첫 실행 실패 |
| **자동 업데이트** | 사용자 개입 없이 신버전 적용 | 보안 패치 전파 불가 |

---

### 2. macOS 배포 파이프라인 - 건너뛸 수 없는 4단계

```
   소스 빌드
      │
      ▼
┌───────────────┐   Developer ID Application 인증서 ($99/년)
│ ① 코드 서명   │   + Hardened Runtime (--options runtime)
└───────┬───────┘
        ▼
┌───────────────┐
│ ② 패키징      │   .dmg / .zip / .pkg
└───────┬───────┘
        ▼
┌───────────────┐   xcrun notarytool 로 Apple 서버에 제출
│ ③ 공증        │   → 악성코드 스캔 (보통 2~15분)
└───────┬───────┘
        ▼
┌───────────────┐   공증 티켓을 아티팩트에 embed
│ ④ 스테이플링  │   → Gatekeeper가 오프라인에서도 검증 가능
└───────┬───────┘
        ▼
   배포 (사용자 PC에서 경고 없이 실행)
```

**순서를 건너뛰면 앱 실행 자체가 깨진다.** 각 단계는 다음 단계의 전제 조건이다.

---

### 3. Developer ID 인증서와 Hardened Runtime

- **Developer ID Application 인증서**: Apple Developer Program($99/년)에서 발급. Mac App Store 밖 배포에 필수. 저작자를 증명한다.
- **Hardened Runtime**: 서명 시 `--options runtime` 플래그로 강제. 런타임 보호(코드 인젝션·메모리 변조 방어)를 켠다.
- **Entitlements(권한)**: Electron 앱이 반드시 필요로 하는 단 하나는 **`com.apple.security.cs.allow-jit`** — V8 자바스크립트 엔진이 바이트코드를 JIT 컴파일해야 하기 때문.

> "특정 기능(마이크·카메라·네트워크 서버·파일 접근)이 요구할 때만 권한을 추가하라 — **권한 하나하나가 또 다른 공격면**이다."

---

### 4. Gatekeeper - macOS의 3중 방어선

```
사용자가 앱 첫 실행
      │
      ▼
┌─────────────────────────────────────────────┐
│  ① Gatekeeper : 서명·공증 여부 검사          │
│  ② XProtect   : 알려진 악성코드 시그니처 스캔 │
│  ③ Notary     : Apple 공증 서비스 검증        │
└─────────────────────────────────────────────┘
      │                          │
  서명+공증+스테이플 OK       미서명/미공증
      ▼                          ▼
  경고 없이 실행            "확인되지 않은 개발자" → 차단/경고
```

- 2026년 표준 도구는 **`xcrun notarytool`** (구버전 `altool`은 2023년 11월 1일 폐기).
- 공증은 보통 **2~15분** 소요.
- **스테이플링**이 핵심: 공증 티켓을 아티팩트에 박아두면 Gatekeeper가 **오프라인에서도** 검증한다. 스테이플링을 안 하면 첫 실행 시 인터넷으로 Apple 서버에서 티켓을 받아와야 한다.

### 5. 자동 업데이트의 동작 원리

macOS의 Electron `autoUpdater`는 **Squirrel.Mac** 위에 구축돼 별도 설정이 거의 없다. Windows는 패키징 방식에 따라 자동 선택된다.

| 플랫폼 | 업데이트 엔진 | 서명 요구 |
|---|---|---|
| **macOS** | Squirrel.Mac | **필수** (서명 안 하면 자동 업데이트 불가) |
| **Windows (전통 인스톨러)** | Squirrel.Windows | 권장 |
| **Windows (MSIX)** | MSIX updater | — |
| **크로스플랫폼(Linux 포함)** | electron-updater | Win/Mac 서명 검증 |

#### 업데이트 생명주기 (이벤트 시퀀스)
```
setFeedURL(server)            ← 업데이트 피드 서버 지정
      │
      ▼
checkForUpdates()
      │
      ├─► 'checking-for-update'
      │
      ├─► 'update-available'  ──► (자동 다운로드)  ──► 'update-downloaded'
      │                                                      │
      └─► 'update-not-available'                             ▼
                                                     quitAndInstall()
                                                  (앱 재시작하며 적용)
```

> ⚠️ "업데이트가 있으면 자동으로 다운로드된다. `checkForUpdates()`를 두 번 부르면 두 번 다운로드된다." Windows Squirrel 앱은 첫 실행 시 `--squirrel-firstrun` 인자로 잠깐 락이 걸리므로, 첫 실행 직후의 업데이트 체크는 지연시켜야 한다.

#### electron-updater의 추가 이점
- **Linux 지원** (내장 autoUpdater는 Win/Mac만)
- **코드 서명 검증을 Windows에서도** 수행
- GitHub Releases / S3 / DigitalOcean Spaces / 일반 HTTP(S) 서버 등 **다양한 provider** 기본 지원
- 서버는 릴리스 메타데이터(예: `latest.yml`)를 호스팅하고, 클라이언트가 현재 버전과 비교해 갱신 트리거

### 6. 점진적 배포(Staged Rollout)

```
신버전 릴리스
   │
   ▼  10% 사용자에게만 노출
   │   → 크래시/회귀 모니터링
   ▼  이상 없으면 50%
   │
   ▼  100%
```

저자는 **10% → 50% → 100%** 단계적 롤아웃으로 회귀 위험을 줄일 것을 권한다. macOS 네이티브 앱은 Sparkle(1Password·Sketch 사용), Electron 앱은 electron-updater + Squirrel.Mac이 표준 조합이다.

## 내가 얻은 인사이트

### 보안/신뢰 관점

1. **현대 배포는 "코드"가 아니라 "신뢰 체인"을 배포하는 일이다**
   - 인증서(저작 증명) → 공증(Apple의 보증) → 스테이플(오프라인 검증)로 이어지는 사슬은 **"이 바이너리를 믿어도 되는가"를 사용자가 아닌 OS가 판단**하게 만든다. 앱 개발의 마지막 1마일은 기능이 아니라 신뢰의 문제다.

2. **권한(entitlement)은 부채다**
   - "기능이 요구할 때만 추가하라"는 조언은 최소 권한 원칙의 데스크탑 버전. Electron이 JIT 권한 하나만 강제하는 것도 같은 맥락. **켜둔 권한 하나가 공격면 하나**이므로, 배포 설정에서도 보안은 "뺄셈"으로 한다.

### 인프라 설계 관점

1. **자동 업데이트는 "보안 패치 전파 채널"이다**
   - autoUpdater를 단순 편의 기능으로 보면 안 된다. 취약점이 터졌을 때 **사용자 개입 없이 전 사용자에게 패치를 밀어 넣는 유일한 통로**다. 그래서 macOS가 "서명 없으면 자동 업데이트 불가"로 강제하는 것 — 갱신 채널 자체가 공격 표적이 되면 안 되기 때문.

2. **"두 번 부르면 두 번 다운로드"가 알려주는 것**
   - autoUpdater는 멱등(idempotent)하지 않다. 체크 호출을 앱 곳곳에서 중복 트리거하면 대역폭·중복 설치 문제가 생긴다. **업데이트 체크는 단일 스케줄러로 일원화**해야 한다는 운영 교훈.

### 실무 적용 관점

1. **CI에 서명·공증을 자동화 파이프라인으로 박아라**
   - 공증이 2~15분 걸리고 도구(notarytool)·인증서·스테이플 순서가 까다롭다. 수동으로 하면 매 릴리스가 고통이다. **인증서를 CI 시크릿에 넣고 빌드→서명→공증→스테이플을 한 잡으로** 묶어야 지속 가능하다.

2. **점진적 롤아웃 없이 100% 배포하지 마라**
   - 자동 업데이트의 위력은 곧 위험이다. 버그 있는 버전을 100%에 밀면 전 사용자가 동시에 깨진다. **10%→50%→100% 게이트와 롤백 계획**은 자동 업데이트를 켜는 순간 함께 갖춰야 할 안전장치다.

3. **첫 실행 타이밍의 함정을 기억하라**
   - Windows `--squirrel-firstrun` 락처럼, 설치 직후는 특수 상태다. 첫 실행에 곧장 업데이트 체크를 걸면 충돌한다. **설치 라이프사이클의 엣지 케이스**(첫 실행, 업데이트 직후 재시작)를 코드에서 명시적으로 다뤄야 한다.
