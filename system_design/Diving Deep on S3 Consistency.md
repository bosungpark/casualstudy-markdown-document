## AI 요약

### 1. S3의 진화

S3는 2006년 "인터넷을 위한 스토리지"로 출시되었다. 당시 설계 우선순위는 보안, 탄력성, 신뢰성, 내구성, 성능, 비용이었으며, 이는 고객들이 e-commerce 웹사이트 같은 애플리케이션에 가장 중요하다고 말한 것이었다.

**15년 후의 변화:**
- S3는 100조 개 이상의 객체를 보유하고 초당 수천만 건의 요청을 처리한다.
- 수만 명의 고객이 S3를 데이터 레이크에 사용하여 몇 년 전에는 불가능했던 규모로 분석을 수행하고 비즈니스에 새로운 인사이트(와 경쟁 우위)를 창출한다.
- 새로운 용도: 위성 이미지 분석, 백신 연구, 자율주행 트럭/자동차 개발

### 2. Eventual Consistency의 기원과 한계

#### 2.1 왜 Eventual Consistency였나

S3의 경우, 고가용성을 위해 최적화된 캐싱 기술을 메타데이터 서브시스템에 구축했지만, 그 설계 결정의 부작용 중 하나는 극히 드문 상황에서 쓰기에 대해 eventual consistency를 나타낼 수 있다는 것이었다. 즉, 시스템은 거의 항상 가용 상태였지만, 가끔 API 호출이 아직 시스템의 모든 노드에 완전히 전파되지 않은 이전 버전의 객체를 반환할 수 있었다.

Eventual consistency는 2006년에 웹사이트 이미지나 백업 복사본을 제공할 때는 적합했다.

#### 2.2 고객들의 자체 해결책

데이터 처리 애플리케이션은 종종 strong consistency를 요구한다—객체가 병렬로 모든 노드에서 동일해야 한다—그래서 고객들은 S3 사용을 위해 S3 외부에서 일관성을 추적하는 자체 애플리케이션 코드를 구현했다.

한 예로, Netflix는 s3mper를 오픈소스로 공개했는데, 이는 Amazon DynamoDB를 일관된 저장소로 사용하여 S3가 비일관적인 응답을 제공하는 드문 경우를 식별했다. Cloudera와 Apache Hadoop 커뮤니티는 S3Guard를 개발했는데, 이도 유사하게 드문 비일관성 발생을 완화하기 위해 애플리케이션에 별도의 뷰를 제공했다.

#### 2.3 AWS의 목표

AWS 로드맵의 90%가 고객으로부터 직접 나온다는 것을 기억하라. 고객들은 추가 인프라를 실행할 필요를 피하기 위해 S3를 변경할 수 있는지 물었다.

비용, 일관성 적용 객체 범위, 또는 성능에서 트레이드오프가 필요한 접근법을 고려했다. 우리는 그런 트레이드오프를 원하지 않았다. 그래서 더 높은 기준을 향해 계속 노력했다: **추가 비용 없이, 모든 신규 및 기존 객체에 적용되며, 성능이나 가용성 트레이드오프 없는 strong consistency**를 원했다.

다른 제공자들은 strong consistency를 모든 스토리지가 아닌 버킷이나 계정에 대한 opt-in 설정으로 만들거나, 서비스의 리전 가용성을 저하시키는 리전 간 종속성으로 일관성을 구현하거나, 기타 제한을 두는 등의 타협을 한다.

### 3. S3의 메타데이터 서브시스템

객체별 메타데이터는 별도의 S3 서브시스템 내에 저장된다. 이 시스템은 GET, PUT, DELETE 요청의 데이터 경로에 있으며, LIST와 HEAD 요청을 처리한다. 이 시스템의 핵심에는 메타데이터를 저장하는 persistence tier가 있다. persistence tier는 높은 복원력을 위해 설계된 캐싱 기술을 사용한다.

**Eventual Consistency의 근본 원인:**
S3 요청은 캐시를 지원하는 인프라가 손상되더라도 여전히 성공해야 한다. 이는 드문 경우에 쓰기가 캐시 인프라의 한 부분을 통해 흐르는 반면 읽기는 다른 부분을 쿼리하게 될 수 있음을 의미했다. 이것이 S3의 eventual consistency의 주요 원인이었다.

### 4. Cache Coherence Protocol 구현

#### 4.1 캐시 우회 접근법의 거부

strong consistency를 제공하기 위한 초기 고려사항 중 하나는 캐싱 인프라를 우회하고 요청을 직접 persistence layer로 보내는 것이었다. 그러나 이것은 성능에 대한 트레이드오프 없음 기준을 충족하지 못했다. 캐시를 유지해야 했다. CPU가 코어 간 값을 적절히 동기화하기 위해 cache coherence 프로토콜을 구현하듯이, 여기에도 그것이 필요했다: 모든 요청에 대해 strong consistency를 허용하는 메타데이터 캐시를 위한 cache coherence 프로토콜.

#### 4.2 핵심 빌딩 블록

S3의 at-least-once 이벤트 알림 전달 시스템과 Replication Time Control 기능을 위한 빌딩 블록으로 작용하는 새로운 복제 로직을 persistence tier에 도입했다. 이 새로운 복제 로직은 S3에서 객체별 "연산 순서"에 대해 추론할 수 있게 한다. 이것이 cache coherency 프로토콜의 핵심 부분이다.

#### 4.3 Witness 컴포넌트

캐시의 객체 메타데이터 뷰가 stale한지 이해하기 위해 S3 메타데이터 서브시스템에 새로운 컴포넌트를 도입했다. 이 컴포넌트는 쓰기에 대한 witness 역할을 하며, 객체가 변경될 때마다 알림을 받는다. 이 새로운 컴포넌트는 읽기 작업 중 read barrier 역할을 하여 캐시가 자신의 객체 뷰가 stale한지 알 수 있게 한다. 캐시된 값이 stale하지 않으면 제공할 수 있고, stale하면 무효화하고 persistence tier에서 읽는다.

```
┌─────────────────────────────────────────────────────┐
│                   Client Request                    │
└─────────────────────────┬───────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────┐
│                  Metadata Cache                     │
│  ┌─────────────────────────────────────────────┐    │
│  │ "Is my view of object X stale?"             │    │
│  │              │                              │    │
│  │              ▼                              │    │
│  │     ┌───────────────┐                       │    │
│  │     │   Witness     │◄── notified on every  │    │
│  │     │  (read barrier)│    object mutation   │    │
│  │     └───────┬───────┘                       │    │
│  │             │                               │    │
│  │    stale?   │  not stale?                   │    │
│  │      │      │      │                        │    │
│  │      ▼      │      ▼                        │    │
│  │  Invalidate │  Serve from cache             │    │
│  │  & read from│                               │    │
│  │  persistence│                               │    │
│  └─────────────┴───────────────────────────────┘    │
└─────────────────────────────────────────────────────┘
```

### 5. 고가용성 설계

Witness는 종종 디스크에 가지 않고 인메모리에서 약간의 상태만 추적하면 되기 때문에 분산 시스템에서 인기가 있다. 이를 통해 매우 낮은 지연시간으로 극도로 높은 요청 처리율을 달성할 수 있다. 그리고 우리가 여기서 한 것이 바로 그것이다. S3가 계속 성장함에 따라 이 fleet을 계속 확장할 수 있다.

**장애 대응:**
내가 오랫동안 말해왔듯이, *모든 것은 항상 실패한다*, 그래서 개별 호스트/서버가 실패할 것이라고 가정하고 시스템을 설계했다. 부하 집중과 개별 서버 장애에 빠르게 대응할 수 있는 자동화를 구축했다. consistency witness가 최소한의 상태만 인메모리에서 추적하기 때문에, 긴 상태 전송을 기다리지 않고 빠르게 교체할 수 있다.

### 6. 정확성 검증

strong consistency가 올바르게 구현되어 일관성을 깨는 엣지 케이스가 없도록 하는 것이 중요하다. S3는 대규모 분산 시스템이다. 이 새로운 cache coherency 프로토콜은 정상적인 경우뿐만 아니라 모든 경우에 올바라야 한다.

**검증해야 할 시나리오들:**
- 동일 객체에 대한 동시 쓰기 진행 중일 때 (값이 신구 사이에서 "깜빡"이면 안 됨)
- 단일 객체에 대해 GET, LIST, PUT, DELETE의 매우 높은 동시성
- 버저닝이 활성화되고 깊은 버전 스택이 있는 경우
- 우리 규모에서는 무언가가 10억 요청 중 한 번만 발생하더라도 S3 내에서 하루에 여러 번 발생한다는 것을 의미한다.

**적용된 검증 기법들:**
unit testing과 integration testing 같은 일반적인 테스팅 기법은 모든 프로덕션 시스템에서 가치 있고 필요한 도구다. 그러나 정확성에 대해 그렇게 높은 기준을 가진 시스템을 구축해야 할 때는 충분하지 않다. 우리는 "아마도 정확한(probably correct)" 시스템이 아닌 "증명 가능하게 정확한(provably correct)" 시스템을 원한다.

그래서 strong consistency를 위해 우리가 구축한 것이 올바르고 시스템이 진화함에 따라 계속 올바르도록 보장하기 위해 다양한 기법을 활용했다. 통합 테스트, 제안된 cache coherence 알고리즘의 연역적 증명, 일관성 설계를 형식화하고 정확성을 입증하기 위한 모델 체킹, 실제 실행 가능한 코드를 검사하는 확장된 모델 체킹을 사용했다.

이러한 검증 기법들은 많은 작업이 필요했다. 사실 실제 구현 자체보다 더 많은 작업이었다. 그러나 우리 고객이 필요로 하는 것이기 때문에 S3의 strong consistency 설계와 구현에 이러한 엄격함을 적용했다.

### 7. 결론

기본적으로, 고객에게 추가 비용 없이, 고성능으로 strong consistency를 제공하는 것은 거대한 도전이었다. S3는 15년 이상 규모에서 클라우드 스토리지를 운영하고 수백만 고객에게 서비스한 경험을 활용하여 다른 곳에서는 사용할 수 없는 기능을 혁신하며, 그 경험을 활용하여 S3 고객이 높이 평가하게 된 고가용성에서 strong consistency를 추가했다. 그리고 다양한 테스팅 및 검증 기법을 활용하여 strongly consistent 시스템에서 고객이 요구하는 정확성을 제공할 수 있었다.
