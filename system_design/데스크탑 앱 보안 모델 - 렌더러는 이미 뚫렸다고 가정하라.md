## AI 요약

### 1. 데스크탑 앱 보안 모델이란?

웹 기술로 만든 데스크탑 앱은 브라우저보다 **훨씬 큰 권한**(파일시스템·셸 접근)을 코드에 부여한다. 문서가 못 박는 핵심: **"Electron is not a web browser."** 따라서 보안의 출발점은 **"렌더러(웹 콘텐츠 영역)는 언제든 뚫릴 수 있다고 가정"**하고, 뚫려도 피해를 격리하는 것이다.

```
┌──────────── 신뢰 영역 (Trusted) ────────────┐
│  Main Process                               │
│  - Node.js 전권 / 파일·셸·시스템 접근        │
└──────────────────┬──────────────────────────┘
                   │  IPC = 유일한 경계
┌──────────────────▼──────────────────────────┐
│  Renderer Process (Untrusted)               │
│  - 웹 콘텐츠 렌더링                          │
│  - "공격자가 여기 들어왔다고 가정한다"       │
└─────────────────────────────────────────────┘
```

| 영역 | 신뢰 수준 | 가진 권한 |
|---|---|---|
| **Main** | 신뢰 | 파일시스템, 셸, 모든 Node API |
| **Renderer** | **비신뢰** | UI 렌더링만 (샌드박스) |
| **IPC** | 경계 | 둘 사이의 유일한 합법 통로 |

### 2. 4대 방어 메커니즘

```
┌─────────────────────────────────────────────────────────────┐
│  ① Context Isolation (v12+ 기본)                            │
│     렌더러 스크립트가 전역 객체 수정 / Node 원시값 사용 차단 │
├─────────────────────────────────────────────────────────────┤
│  ② Process Sandbox (v20+ 기본)                              │
│     OS 수준 제약으로 렌더러의 시스템 자원 접근 제한          │
├─────────────────────────────────────────────────────────────┤
│  ③ nodeIntegration: false (v5+ 기본)                        │
│     렌더러가 Node 모듈 접근 불가 → XSS가 RCE로 못 번짐       │
├─────────────────────────────────────────────────────────────┤
│  ④ IPC sender 검증                                          │
│     비신뢰 프레임이 특권 동작을 트리거하지 못하게 발신자 확인│
└─────────────────────────────────────────────────────────────┘
```

| 메커니즘 | 기본화 시점 | 막는 것 |
|---|---|---|
| **Context Isolation** | v12 | 프리로드/Electron 내부 객체를 웹 코드가 오염 |
| **Sandbox** | v20 | 렌더러의 직접 시스템 호출 (CPU·메모리 외) |
| **nodeIntegration off** | v5 | XSS → 파일시스템 장악(RCE) 연쇄 |
| **IPC 검증** | — | 비신뢰 발신자의 특권 호출 |

> **핵심 연쇄 차단**: 일반 웹은 XSS가 "쿠키 탈취" 정도지만, Node 통합이 켜진 데스크탑 앱은 XSS 한 줄이 **`require('child_process')`로 임의 명령 실행(RCE)**까지 번진다. `nodeIntegration: false`가 이 연쇄의 첫 고리를 끊는다.

### 3. 샌드박스와 Node 통합의 트레이드오프

```
nodeIntegration: true  ──────► 샌드박스 비활성화!
   (렌더러에서 Node 직접 사용)    └─► 보안 모델 붕괴

권장 구조:
   렌더러(샌드박스 ON, Node OFF)
        │ 필요한 일만
        ▼ contextBridge로 노출된 좁은 API
   프리로드 ──IPC──► 메인(Node 전권)
```

샌드박스는 Node 통합과 묶여 있다. **`nodeIntegration: true`로 Node를 켜면 그 렌더러의 샌드박스가 꺼진다.** 따라서 "렌더러에서 편하게 Node를 쓰자"는 유혹은 보안 모델 전체를 무너뜨린다. 올바른 길은 샌드박스를 켠 채, 필요한 기능만 contextBridge+IPC로 메인에 위임하는 것이다.

### 4. 원격 콘텐츠 로딩 - 가장 위험한 영역

| 규칙 | 이유 |
|---|---|
| HTTPS만 로드 | 중간자(MITM)가 코드 주입 방지 |
| 원격 콘텐츠에 Node 통합 금지 | 원격 코드가 RCE로 직행하는 것 차단 |
| `webview`는 엄격한 `webPreferences`로 통제 | 임베드된 콘텐츠의 권한 상승 차단 |
| `shell.openExternal()` 전 URL 검증 | 악성 스킴(예: 임의 실행) 차단 |

**CSP(Content Security Policy)**: `script-src 'self'` 같은 지시어로 로드 가능한 리소스를 제한해 주입 공격을 막는다.

---

### 5. 자격 증명 저장 - safeStorage

비밀번호·토큰 같은 민감 정보는 평문 파일이 아니라 **OS 보안 저장소**에 맡긴다.

| 플랫폼 | 보안 저장소 |
|---|---|
| macOS | Keychain |
| Windows | DPAPI |
| Linux | libsecret |
| 크로스플랫폼 라이브러리 | keytar |

Electron의 `safeStorage` API가 이를 추상화한다. 추가로 **소유자 전용 파일 권한 적용 + 저장 시 암호화(encrypt at rest)**로 노출을 줄인다.

### 6. 보안 체크리스트 (요약)

```
[필수 기본값 유지]
 □ HTTPS 콘텐츠만 로드
 □ 원격 콘텐츠에 nodeIntegration off
 □ contextIsolation 켜기
 □ sandbox 켜기
 □ webSecurity 켜둔 채로
 □ 제한적 CSP 정의

[입력/네비게이션 통제]
 □ 모든 IPC 메시지의 sender 검증
 □ 예상 URL로만 네비게이션 제한
 □ setWindowOpenHandler()로 새 창 통제
 □ shell.openExternal() 전 URL 검증
 □ will-attach-webview로 webview 생성 통제

[표면 최소화]
 □ contextBridge로 raw Electron API 절대 노출 금지
 □ 프리로드 API 표면 최소화
 □ 불필요한 Electron fuse 비활성화
 □ 실험적 기능 / enableBlinkFeatures 사용 금지

[운영]
 □ Electron 최신 stable 유지
 □ 자격증명은 safeStorage로
 □ npm audit 정기 실행
```

## 내가 얻은 인사이트

### 보안 철학 관점

1. **"가정"이 보안 설계를 결정한다 — 렌더러는 이미 뚫렸다**
   - 이 문서의 모든 규칙은 "렌더러는 비신뢰"라는 단 하나의 가정에서 파생된다. Chromium Site Isolation의 "렌더러는 결국 뚫린다"와 정확히 같은 사고방식. **막을 수 있는가가 아니라, 뚫렸을 때 피해를 어디로 한정할 것인가**가 설계의 출발점이다.

2. **신뢰 경계는 단 하나여야 한다 — IPC**
   - 메인(신뢰)과 렌더러(비신뢰) 사이의 합법 통로를 IPC 하나로 좁힌 것은, 검증해야 할 지점을 한 곳으로 모은 설계다. 경계가 여러 개면 그중 하나만 새도 끝난다. **공격면을 줄이는 가장 강력한 방법은 통로의 수를 줄이는 것**이다.

### 연쇄 차단 관점

1. **XSS → RCE 연쇄를 끊는 것이 핵심**
   - 웹에서 "그저 XSS"가 데스크탑에선 임의 코드 실행이 된다. `nodeIntegration: false`, 샌드박스, contextIsolation은 모두 **이 연쇄의 서로 다른 고리를 끊는 다중 방어**다. 하나가 뚫려도 다음 고리가 막는다 — "방어선이 하나면 방어가 아니다."

2. **편의와 보안의 결합 함정**
   - `nodeIntegration: true` 한 줄이 샌드박스를 통째로 끈다. 보안 설정들이 서로 얽혀 있어서, "편하게 하려고" 하나를 켜면 여러 방어가 동시에 무너진다. **보안 기본값을 끄는 결정은 그 한 줄이 끄는 전체 사슬을 이해한 뒤에** 내려야 한다.

### 실무 적용 관점

1. **민감정보는 절대 직접 저장하지 말고 OS에 위임하라**
   - 토큰을 설정 파일에 평문 저장하는 실수는 흔하다. Keychain/DPAPI/libsecret을 safeStorage로 추상화해 쓰면 **OS의 검증된 암호화·접근통제**를 공짜로 얻는다. "내가 암호화를 구현하지 않는다"가 더 안전하다.

2. **체크리스트는 "기본값 유지"가 절반이다**
   - 20개 항목 중 다수가 "이미 켜진 기본값을 끄지 마라"이다. Electron은 안전한 기본값을 제공하지만 개발자가 편의를 위해 끄는 순간 무너진다. **새 기능을 추가할 때마다 어떤 보안 기본값을 건드렸는지 체크**하는 습관이 곧 보안이다.

3. **`shell.openExternal`과 네비게이션은 숨은 공격면이다**
   - 파일 접근만 위험한 게 아니다. 검증 없는 `openExternal`은 악성 스킴 실행으로, 자유로운 네비게이션은 피싱/원격 코드 로딩으로 이어진다. **"외부로 나가는 모든 출구"(새 창, 외부 링크, 네비게이션)에 화이트리스트**를 둬야 한다.
