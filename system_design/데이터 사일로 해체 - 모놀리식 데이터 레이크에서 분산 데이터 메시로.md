# 데이터 사일로 해체 — 모놀리식 데이터 레이크에서 분산 데이터 메시로

## 출처
- **아티클/논문**: How to Move Beyond a Monolithic Data Lake to a Distributed Data Mesh
- **저자/출처**: Zhamak Dehghani (martinfowler.com)
- **링크**: https://martinfowler.com/articles/data-monolith-to-mesh.html

> 데이터 사일로(data silo) 문제를 "조직·아키텍처 차원의 근본 원인"에서부터 다시 설계한 글.
> "데이터 메시(Data Mesh)"라는 용어가 처음 제안된 2019년의 원전(原典)이다.

## AI 요약

### 1. 데이터 사일로란, 그리고 왜 생기는가?

데이터 사일로는 보통 "부서마다 데이터를 따로 쌓아서 안 통한다"는 증상으로 이야기되지만,
이 글의 핵심 주장은 **중앙집중식 데이터 레이크/웨어하우스 자체가 사일로를 만드는 구조**라는 것이다.

전사의 모든 데이터를 한 곳(central lake)으로 끌어모으려는 순간, 다음 단절이 생긴다:

```
[소스 도메인 팀]  ──push──>  [중앙 데이터 플랫폼 팀]  ──serve──>  [소비 도메인 팀]
   데이터 생성자                 도메인 지식 없음               데이터 사용자
        │                          │                            │
   "품질 좋게 줄                "맥락 모른 채               "원하는 데이터
    인센티브 없음"               파이프라인만 운영"           기다리다 지침"
```

즉 사일로의 본질은 *저장소가 나뉘어 있어서*가 아니라,
**데이터를 만드는 사람과 쓰는 사람 사이에 도메인 지식 없는 중앙 팀이 끼어 단절**되기 때문이다.

| 통념 | 이 글의 관점 |
|------|-------------|
| 데이터를 한 곳에 모으면 사일로가 사라진다 | 한 곳에 모으는 행위 자체가 새로운 병목·사일로를 만든다 |
| 사일로 = 저장소 분리 문제 | 사일로 = 소유권(ownership)과 책임의 분리 문제 |
| 전문 데이터 엔지니어 팀에 집중시키면 효율적 | 도메인과 분리된 초전문화 팀이 오히려 확장을 막는다 |

### 2. 중앙집중 데이터 플랫폼의 3가지 실패 모드

#### ① 중앙집중·모놀리식 구조
서로 다른 도메인의 데이터를 하나의 소유권 아래에서 "조화시키려" 한다.
소스가 늘고 소비 요구가 다양해질수록 감당 불가능해진다.
"모든 데이터를 한 곳에 저장해야 한다"는 가정이 새 소스에 대한 대응 속도를 떨어뜨린다.

#### ② 결합된(coupled) 파이프라인 분해
플랫폼을 **비즈니스 도메인이 아니라 기술 기능**으로 쪼갠다:

```
[수집] ─> [정제] ─> [집계] ─> [서빙]
  ingestion  cleansing  aggregation  serving
```

문제: 새 데이터셋/기능 하나를 추가하려면 **모든 단계를 동시에 수정**해야 한다.
아키텍처가 "변화의 축과 직교(orthogonal to the axis of change)"하게 분해되어 있어,
작은 변경 하나가 파이프라인 전체를 흔든다 → 높은 결합도, 느린 변경 속도.

#### ③ 사일로화된 초전문화 소유권
데이터 엔지니어가 소스 도메인·소비 도메인 양쪽에서 격리된다.
비즈니스 맥락 없이 다양한 요구를 처리해야 하고,
소스 팀은 정확한 데이터를 줄 인센티브가 없다 → 품질 악화의 악순환.

### 3. 해결책: 데이터 메시의 4대 원칙

핵심 발상의 전환은 **"데이터를 밀어넣기(push)"에서 "도메인이 서빙(serve)"으로** 모델을 뒤집는 것.

```
        [기존] 모놀리식 데이터 레이크                 [데이터 메시]
                                                도메인A      도메인B
   소스 ─push─> ┌──────────────┐                 (데이터       (데이터
   소스 ─push─> │  중앙 레이크   │ ─serve─> 소비    제품)◀──────▶제품)
   소스 ─push─> └──────────────┘                    ▲           ▲
                  중앙 팀이 소유                      └─연결된 제품의 생태계─┘
                                                  각 도메인이 자기 데이터 소유·서빙
```

#### 원칙 1. 도메인 중심의 분산 데이터 소유권 (Domain-oriented decentralized ownership)
데이터를 중앙으로 흘려보내지 말고, **각 도메인이 자기 데이터셋을 쉽게 소비 가능한 형태로 직접 호스팅·서빙**한다.

| 데이터 유형 | 설명 | 형태 |
|------------|------|------|
| 소스 정렬(source-oriented) | 비즈니스의 "사실과 실제"를 반영 | 불변 이벤트 스트림, 과거 스냅샷 |
| 소비 정렬(consumer-oriented) | 특정 use case에 맞게 가공 | 예: 추천용 그래프 DB |

데이터 파이프라인은 더 이상 중앙 인프라가 아니라 **도메인 내부의 구현 세부사항**이 된다.

#### 원칙 2. 제품으로서의 데이터 (Data as a Product)
도메인은 데이터셋을 "내부 고객(데이터 과학자·분석가·엔지니어)을 위한 제품"으로 취급한다.
좋은 데이터 제품의 6가지 특성:

| 특성 | 의미 |
|------|------|
| Discoverable (발견 가능) | 중앙 카탈로그에서 소유자·계보·샘플과 함께 검색됨 |
| Addressable (주소 지정 가능) | 표준 네이밍으로 프로그램적 접근 가능 |
| Trustworthy (신뢰 가능) | 소유자가 정확도·무결성에 대한 SLO를 보장 |
| Self-describing (자기 기술적) | 스키마·시맨틱·샘플 포함, 별도 설명 거의 불필요 |
| Interoperable (상호운용 가능) | 전역 표준 준수 → 도메인 간 상관관계 분석 가능 (예: 일관된 artist ID) |
| Secure (보안) | 접근 지점에서 중앙 정책으로 접근 제어 |

조직적으로는 **데이터 제품 책임자(product owner) + 도메인에 내장된 데이터 엔지니어**로 구성된 교차기능 팀이 소유.

#### 원칙 3. 셀프서비스 데이터 인프라 플랫폼 (Self-serve data platform)
도메인마다 인프라를 중복 구축하지 않도록, **도메인 비종속(domain-agnostic) 공통 플랫폼**을 별도 팀이 제공:

- 다중형(polyglot) 저장소·암호화
- 파이프라인 오케스트레이션·버전 관리
- 스키마 자동 관리, 데이터 품질 모니터링
- 통합 카탈로그 등록·검색
- 접근 제어·거버넌스 도구
- 컴퓨트·캐싱 인프라

성공의 핵심 지표 = **"새 데이터 제품을 만드는 리드 타임"을 자동화로 얼마나 줄이는가.**

#### 원칙 4. 연합형 거버넌스 (Federated Computational Governance)
도메인 자율성은 지키되, 상호운용을 위한 최소한의 전역 표준은 중앙에서 합의:

- 공통 이벤트 포맷·필드 타입 표준
- 연합 엔티티 식별자 (도메인마다 다른 "artist" 같은 다의어 해소)
- 로컬에 적용되는 전역 접근 제어 정책
- 공유 인프라 표준

### 4. 아키텍처 패러다임의 전환 정리

| 기존 (모놀리식) | 데이터 메시 |
|----------------|------------|
| 수집 중심 (push to center) | 서빙 중심 (도메인이 제품 발행) |
| 추출·적재 (Extract & Load) | 발견·소비 (Discover & Consume) |
| 중앙 데이터 레이크 | 상호 연결된 데이터 제품의 생태계 |
| 플랫폼 팀의 단일 소유 | 연합 거버넌스 하의 분산 도메인 소유 |

> 데이터 레이크·웨어하우스가 사라지는 게 아니다.
> 아키텍처의 "중심"이 아니라 **메시 위의 한 노드(node on the mesh)**로 역할이 바뀐다.
> 여러 도메인의 단일 도메인 레이크들이 모여 협력하는 "워터 가든(water garden)"이 곧 메시다.

**핵심 개념 — 아키텍처 퀀텀(architectural quantum)**
- 기존: 독립 배포 단위 = 파이프라인 단계
- 메시: 독립 배포 단위 = **도메인** → 도메인 단위로 독립적 진화·배포 가능

## 내가 얻은 인사이트

### 문제 정의 관점
1. **"사일로 = 저장소 분리"라는 통념을 깨야 한다**
   - 흔히 사일로 해결책으로 "데이터 레이크/웨어하우스로 다 모으자"를 떠올리지만,
     이 글은 *모으는 행위 자체가 새로운 중앙 병목을 만든다*고 지적한다.
   - 근본 원인은 물리적 분산이 아니라 **소유권과 도메인 지식의 단절**.
     해결의 방향이 "통합"이 아니라 "올바른 분산"이라는 게 역설적이면서 설득력 있다.

### 조직 설계 관점
2. **콘웨이의 법칙을 데이터에 적용한 것**
   - 마이크로서비스가 "도메인 중심 분해 + 팀 자율성"으로 모놀리스를 깬 것을,
     그대로 분석/데이터 영역에 가져온 구조다.
   - 데이터 메시는 기술 솔루션이라기보다 **조직·소유권 재설계**에 가깝다.
     도구만 도입하고 팀 구조를 안 바꾸면 실패한다는 신호.

3. **인센티브 설계가 핵심**
   - "소스 팀이 좋은 데이터를 줄 인센티브가 없다"는 진단이 날카롭다.
     데이터를 *제품*으로 만들고 SLO·책임자를 붙이는 순간, 품질이 비로소 누군가의 KPI가 된다.

### 트레이드오프 관점
4. **분산의 비용 — 거버넌스 없는 메시는 또 다른 카오스**
   - 도메인 자율성을 극단으로 밀면 "상호운용 불가능한 데이터 제품들의 새 사일로"가 된다.
   - 그래서 원칙 4(연합 거버넌스)와 원칙 3(셀프서비스 플랫폼)이 안전장치다.
     **자율성과 표준화 사이의 균형**이 데이터 메시 성패를 가른다.
   - 작은 조직에는 과한 모델일 수 있다. 도메인이 몇 개 안 되면 중앙 레이크가 더 단순하고 싸다.
     메시는 "도메인 수와 소비 다양성이 폭발해 중앙 팀이 병목이 될 때" 정당화된다.

### 실무 적용 관점
5. **"리드 타임"을 단일 성공 지표로 삼는 발상**
   - 셀프서비스 플랫폼의 성공을 "새 데이터 제품 생성까지 걸리는 시간"으로 측정하라는 제안은,
     추상적인 아키텍처 논의를 즉시 검증 가능한 지표로 환원해준다. 어떤 데이터 플랫폼 투자든 이 질문으로 점검 가능: *"새 데이터 제품을 만드는 데 며칠 걸리나?"*
