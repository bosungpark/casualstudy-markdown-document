## AI 요약

### 1. Obsidian이란?

Obsidian은 **로컬에 저장된 평문 마크다운(.md) 파일**을 노드로 삼아, 노트 간 링크로 지식의 네트워크를 구성하는 개인 지식 관리(PKM, Personal Knowledge Management) 도구다.

핵심은 "노트 앱"이 아니라 **로컬 폴더 위에 얹힌 뷰어/에디터 + 인덱싱 엔진**이라는 점이다. 데이터는 회사 서버가 아니라 사용자 디스크에 있고, Obsidian은 그 위에서 동작하는 레이어일 뿐이다.

| 특성 | 내용 |
|------|------|
| 저장 형식 | 평문 마크다운(.md) + YAML frontmatter |
| 저장 위치 | 로컬 폴더(= Vault), 클라우드 강제 없음 |
| 데이터 소유권 | 100% 사용자 (앱을 지워도 노트는 그대로 남음) |
| 데스크톱 런타임 | Electron (v37.x 기준) |
| 모바일 런타임 | Capacitor (v5.x) |
| 에디터 엔진 | CodeMirror 6 |
| 확장 방식 | 코어 플러그인 + 커뮤니티 플러그인 (TypeScript API) |
| 핵심 차별점 | Local-first + 양방향 링크 + 그래프 + 플러그인 |

### 2. Vault — "폴더가 곧 데이터베이스"

대부분의 노트 앱은 독자적인 바이너리 DB(SQLite 등)에 노트를 욱여넣는다. Obsidian은 반대로, **그냥 OS 위의 폴더 하나가 Vault**다.

```
my-vault/                      ← Vault = 그냥 폴더
├── 인박스/
│   └── 오늘 떠오른 생각.md     ← 노트 = 그냥 .md 파일
├── 프로젝트/
│   ├── Obsidian 정리.md
│   └── attachments/
│       └── diagram.png
└── .obsidian/                 ← 앱 설정/상태 (노트와 분리)
    ├── config.json            (전역 설정)
    ├── workspace.json         (열린 탭/레이아웃)
    ├── plugins/               (커뮤니티 플러그인)
    ├── themes/
    └── snippets/              (CSS)
```

설계상의 의미:

- **이식성(Portability)**: 노트를 다른 앱으로 옮길 때 "export"가 필요 없다. 이미 표준 마크다운이기 때문이다.
- **관심사 분리**: 사용자 콘텐츠(`.md`)와 앱 상태(`.obsidian/`)가 물리적으로 분리되어 있다. `.obsidian` 폴더를 지워도 노트는 1바이트도 손상되지 않는다.
- **동기화를 앱이 강제하지 않음**: 폴더이기 때문에 Git, Dropbox, iCloud, rsync 등 **어떤 동기화 수단과도 직교(orthogonal)**한다. (공식 Sync는 선택지일 뿐)

> 한 줄 요약: Obsidian의 가장 큰 아키텍처 결정은 "저장소를 소유하지 않겠다"는 것이다.

### 3. 링크와 백링크 — "그냥 텍스트"를 양방향 그래프로

Obsidian의 연결은 두 문법으로 표현된다.

```markdown
[[Atomic Habits]]              ← 위키링크 (파일명 기준)
[[Atomic Habits#핵심정리]]      ← 특정 헤딩으로
[[Atomic Habits#^block-id]]    ← 특정 블록으로
[표준 마크다운 링크](note.md)   ← 일반 마크다운 링크도 지원
```

여기서 핵심 구현 원리는 **링크 해석(Link Resolution)과 인덱싱**이다.

```
   파일 저장/변경
        │
        ▼
  ┌──────────────┐   각 .md 파일을 파싱해
  │  메타데이터    │   - 링크([[...]])
  │  캐시 인덱스   │   - 태그(#tag)
  │ (in-memory)  │   - 헤딩 / 블록 ID
  └──────┬───────┘   - frontmatter 를 추출
        │
        ▼
  링크 그래프 구성 (A → B)
        │
        ├─ Forward link:  A가 B를 가리킴 (파일에 실제로 적힌 것)
        └─ Backlink:      B 입장에서 "나를 가리키는 A들" (역방향으로 계산)
```

**백링크는 파일에 저장되지 않는다.** `[[Atomic Habits]]`라고 쓴 쪽(forward link)만 실제 텍스트에 존재하고, 대상 노트의 "이 노트를 링크한 문서들" 목록은 **인덱스를 역방향으로 조회해 런타임에 만들어낸다.**

이것이 Roam Research가 대중화한 양방향 링크를, **데이터를 오염시키지 않고** 구현하는 방법이다. 마크다운 파일 자체는 순수하게 유지되고, 양방향성은 메모리 인덱스의 책임이 된다.

추가로 **Unlinked mentions(언급되었지만 링크되지 않은 곳)**도 제공한다 — 본문에 "Atomic Habits"라는 텍스트는 있지만 `[[ ]]`로 감싸지 않은 위치를 전문 검색으로 찾아 보여준다.

### 4. Graph View — 인덱스의 시각화

그래프 뷰는 새로운 데이터가 아니라, **위에서 만든 링크 인덱스를 그대로 그림으로 옮긴 것**이다.

```
        ┌─────────┐
        │ 노트 A  │
        └────┬────┘
       링크   │   링크
     ┌───────┴───────┐
     ▼               ▼
 ┌───────┐       ┌───────┐
 │ 노트 B │──────│ 노트 C │   ● 노드 = 노트(.md 파일)
 └───────┘  링크  └───────┘   ─ 엣지 = 링크
     │
     ▼
 ┌───────┐   ← 들어오는 링크가 많을수록
 │ 노트 D │      노드 크기가 커짐 (in-degree에 비례)
 └───────┘
```

- **Global Graph**: Vault 전체 노드/엣지
- **Local Graph**: 현재 노트에서 N단계 떨어진 이웃만 (서브그래프)
- 노드 크기 = 들어오는 링크 수(in-degree) → 어떤 노트가 지식의 "허브"인지 한눈에 드러남
- force-directed layout(노드 간 척력 + 엣지의 인력)으로 자동 배치

그래프 뷰의 가치는 "예쁜 그림"이 아니라, **링크를 충실히 달면 지식 구조가 창발적으로 드러난다**는 데 있다. 폴더 계층(트리)으로는 표현 못 하는 다대다 관계를 보여준다.

### 5. 에디터 — CodeMirror 6와 Live Preview

편집/렌더링은 **CodeMirror 6** 위에 구현된다. (코드 에디터로 유명한 그 라이브러리)

| 모드 | 동작 |
|------|------|
| Source mode | 마크다운 원문(`**굵게**`)을 그대로 표시 |
| Live Preview | 커서가 없는 부분은 렌더링(**굵게**), 커서가 닿으면 원문 노출 |
| Reading view | 전체를 HTML로 렌더링 (읽기 전용) |

Live Preview는 "에디터 = 텍스트, 프리뷰 = 별도 패널"이라는 전통적 마크다운 앱의 분리를 없앴다. CodeMirror 6의 **decoration(텍스트 위에 시각 효과/위젯을 덧씌우는 메커니즘)**으로, 원문을 유지한 채 보이는 모습만 렌더링한다. 그래서 WYSIWYG처럼 보이지만 내부 데이터는 끝까지 평문 마크다운이다.

### 6. 플러그인 아키텍처 — 같은 런타임 안에서 돌아가는 확장

Obsidian의 확장성은 다음 구조에서 나온다.

```
┌──────────────────────────────────────────────┐
│            Electron (Chromium + Node)          │
│                                                │
│   ┌──────────────────────────────────────┐    │
│   │           Obsidian Core App           │    │
│   │   App / Vault / Workspace / MetaCache │    │
│   └───────────────┬──────────────────────┘    │
│                   │  require('obsidian') API   │
│         ┌─────────┴─────────┐                  │
│         ▼                   ▼                  │
│  ┌────────────┐      ┌────────────┐            │
│  │ Core Plugin│      │ Community  │            │
│  │ (Graph,    │      │ Plugin     │            │
│  │  Search,   │      │ (Dataview, │            │
│  │  Canvas)   │      │  Kanban…)  │            │
│  └────────────┘      └────────────┘            │
│      플러그인도 "같은 프로세스" 안에서 실행      │
└──────────────────────────────────────────────┘
```

구현상의 특징:

1. **단일 번들 CommonJS 모듈**: 플러그인은 esbuild/rollup으로 모든 의존성을 `main.js` 하나로 묶어 배포한다. Obsidian이 런타임에 이를 로드한다.
2. **`Plugin` 클래스 + 생명주기**:
   ```typescript
   import { Plugin } from "obsidian";

   export default class MyPlugin extends Plugin {
     async onload()   { /* 명령 등록, 이벤트 구독, UI 추가 */ }
     async onunload() { /* 정리: 리스너 해제 등 */ }
   }
   ```
3. **같은 런타임 공유**: 플러그인은 별도 샌드박스가 아니라 **앱과 동일한 Electron 환경**에서 돈다. 그래서 `require('fs')`, `require('electron')`로 파일시스템·OS 자원에 직접 접근할 수 있다 → 매우 강력하지만, **신뢰 기반 모델**(설치 = 권한 부여)이라는 트레이드오프가 있다.
4. **이벤트 기반 API**: Vault 변경, 파일 열림, 메타데이터 갱신 등에 훅을 걸 수 있다.

이 "코어도 플러그인, 확장도 플러그인" 구조 덕분에, Graph View·Canvas·검색 같은 핵심 기능조차 끄고 켤 수 있는 모듈이 된다.

### 7. 왜 가벼운가? — React 없는 커스텀 UI

Obsidian은 Electron 앱이면서도 "가볍다"는 평을 받는데, 그 이유 중 하나는 **React/Angular 같은 프레임워크를 쓰지 않고 UI를 직접 구현**했기 때문이다. 가상 DOM diffing 등의 오버헤드 없이, CodeMirror와 직접 만든 DOM 컴포넌트로 구성된다.

## 내가 얻은 인사이트

### 아키텍처 관점

1. **"저장소를 소유하지 않는다"는 결정의 위력**
   - 대부분의 SaaS 노트 앱은 데이터를 자사 DB에 가둬 락인(lock-in)을 만든다. Obsidian은 반대로 **로컬 폴더라는 가장 보편적인 인터페이스**를 채택해, 동기화·백업·버전관리를 전부 외부 도구에 위임했다.
   - 이건 "기능을 덜 만들어서" 오히려 강해진 케이스다. Git/Dropbox/iCloud와 직교하므로 회사가 모든 동기화 시나리오를 떠안을 필요가 없다.

2. **데이터와 파생 인덱스의 분리**
   - 백링크/그래프/태그는 전부 **파일을 파싱해 만든 in-memory 파생물**이고, 원본 .md에는 forward link만 존재한다.
   - "원천 데이터(source of truth)는 단순하게, 부가 기능은 인덱스로"라는 원칙은 DB 설계의 정석과 똑같다. 마크다운 파일이 곧 WAL이고, 메타데이터 캐시가 곧 인덱스인 셈.

### 설계 트레이드오프 관점

3. **플러그인이 같은 프로세스에서 도는 것의 양면성**
   - 장점: API가 극도로 강력하다(파일시스템·OS 직접 접근). 그래서 Dataview 같은 "노트 위의 쿼리 엔진"까지 가능했다.
   - 단점: 샌드박스가 없어 악성 플러그인이 디스크 전체를 읽을 수 있다. **보안을 격리가 아니라 신뢰로 푼 선택**이며, 이는 VSCode 확장과 같은 모델이다. 편의성과 보안을 맞바꾼 전형적 사례.

4. **평문을 유지한 채 WYSIWYG처럼 보이게 하기**
   - Live Preview는 "보이는 것은 렌더링, 데이터는 평문"이라는 까다로운 요구를 CodeMirror 6의 decoration으로 풀었다.
   - Notion(블록 = DB 레코드)과 정반대 철학이다. Notion은 리치함을 위해 평문을 버렸고, Obsidian은 평문 이식성을 지키려고 에디터 레이어에서 고생을 감수했다. **"데이터 포맷의 단순함을 어디까지 사수할 것인가"**가 두 제품을 가른다.

### 실무 적용 관점

5. **"폴더 위의 레이어" 패턴은 재사용 가능한 설계다**
   - 내 데이터를 표준 포맷(마크다운, JSON, CSV)으로 평범한 폴더에 두고, 그 위에 인덱싱/뷰 레이어를 얹는 구조는 도구 종속을 피하는 강력한 패턴이다.
   - 사내 문서 도구나 지식 베이스를 만들 때도, "DB에 가두기"보다 "파일 + 파생 인덱스"를 먼저 고민할 가치가 있다. 마이그레이션 비용이 0에 수렴하기 때문.

6. **연결을 강제하지 않되 보상하는 UX**
   - Obsidian은 폴더 정리를 강요하지 않는다. 대신 링크를 달면 백링크·그래프·unlinked mention으로 **즉시 보상**을 준다.
   - "구조를 위에서 강제(top-down 분류)"하는 대신 "아래에서 창발(bottom-up 링크)"하게 하는 설계는, 사람이 실제로 생각하는 방식(연관 기억)에 더 가깝다.
