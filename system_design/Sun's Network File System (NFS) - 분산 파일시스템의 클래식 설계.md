## AI 요약

### 1. 분산 파일시스템은 왜 필요한가

여러 머신이 네트워크 너머의 파일을 **마치 로컬처럼** 다루게 하면 두 가지가 가능해진다.

- **데이터 공유**: 누가 어느 워크스테이션에 앉든 같은 홈 디렉터리를 본다.
- **중앙 관리**: 백업·용량·보안 정책을 서버 한 곳에 모은다.

기본 구도는 단순하다.

```
   ┌──────────┐    RPC over network    ┌──────────────┐
   │ Client A │ ─────────────────────▶ │              │
   ├──────────┤                        │   NFS Server │
   │ Client B │ ─────────────────────▶ │  (real disk) │
   ├──────────┤                        │              │
   │ Client C │ ─────────────────────▶ │              │
   └──────────┘                        └──────────────┘
   (캐시 보유)                         (영구 저장소)
```

클라이언트는 시스템 콜(`open`, `read`, `write`)을 그대로 사용하고, 커널의 VFS/vnode 계층이 "이 파일은 NFS 마운트야"라는 걸 보고 호출을 **RPC**로 변환해 서버로 보낸다.

| 특성 | 설명 |
|------|------|
| 모델 | Client-Server, RPC 기반 |
| 투명성 | 애플리케이션은 로컬/원격 차이를 모름 |
| 도입 시기 | NFSv2가 1980년대 중반 Sun에서 등장 |
| 핵심 철학 | 단순함, 크래시 복구, "적당히 좋은 성능" |

### 2. NFSv2의 가장 큰 결단 — Stateless Server

NFSv2의 모든 설계는 한 줄의 결정에서 파생된다.

> **서버는 클라이언트에 대한 어떠한 상태도 보관하지 않는다.**

"누가 어떤 파일을 열었는지", "파일 포인터가 어디에 있는지", "어떤 락을 쥐고 있는지" — 서버는 **모른다**. 모든 요청은 그 자체로 자기완결적이다.

왜 이렇게 했는가? **크래시 복구가 압도적으로 단순해지기 때문**이다.

```
[Stateful 서버라면]
   서버 다운 → 모든 클라이언트의 open file/lock 상태 재구성 필요
            → 복잡한 reconnect 프로토콜, 타임아웃 협상...
            → 클라이언트도 서버 다운을 "알아야" 함

[Stateless 서버라면]
   서버 다운 → 그냥 재부팅
            → 클라이언트는 RPC 응답이 없으면 그냥 재시도
            → 서버 입장에선 "방금 들어온 새 요청"과 구분 불가능 (그래도 OK)
```

대신 이 결정이 뒤에 나올 모든 설계(**파일 핸들**, **idempotency**, **flush-on-close** 등)의 제약이 된다.

### 3. 파일 핸들 (File Handle) — 경로 대신 쓰는 식별자

서버가 상태를 안 가지므로, 클라이언트는 "내가 지금 다루는 파일이 뭔지" 매 요청마다 서버에 알려야 한다. 경로 문자열(`/home/u/a.txt`)을 매번 보내면 비싸므로, **파일 핸들**이라는 불투명한(opaque) 토큰을 쓴다.

```
 File Handle 구조
 ┌────────────────┬───────────────┬──────────────────┐
 │  Volume ID     │  Inode Number │  Generation #    │
 │ (어느 파일시스템) │ (그 안의 inode)  │ (재사용 감지용)     │
 └────────────────┴───────────────┴──────────────────┘
```

- **Volume ID**: 서버가 여러 파일시스템을 export할 때 어느 것인지
- **Inode Number**: 해당 파일시스템 내 inode
- **Generation Number**: inode가 삭제 후 재사용됐을 때 옛 핸들이 새 파일을 가리키지 않도록 막는 단조 증가 카운터

핸들 자체에 모든 식별 정보가 들어가므로 서버는 캐시·테이블 없이도 매 요청을 처리할 수 있다.

### 4. NFS 프로토콜 — 주요 RPC

| RPC | 역할 |
|-----|------|
| `LOOKUP(dir_fh, name)` | 디렉터리 안에서 이름 하나를 풀어 자식의 file handle 반환 |
| `READ(fh, offset, count)` | 특정 오프셋에서 데이터 읽기 |
| `WRITE(fh, offset, data)` | 특정 오프셋에 데이터 쓰기 |
| `GETATTR(fh)` | 메타데이터(size, mtime, mode) 조회 |
| `SETATTR(fh, attrs)` | 메타데이터 갱신 |
| `CREATE`, `REMOVE`, `MKDIR`, `RMDIR` | 생성/삭제 계열 |

핵심 관찰: **`open`/`close` 같은 상태성 호출이 없다.** 클라이언트가 path를 풀려면 컴포넌트별로 `LOOKUP`을 반복한다.

```
 /home/user/file.txt 읽기:
   LOOKUP(root_fh, "home")     → H1
   LOOKUP(H1,      "user")     → H2
   LOOKUP(H2,      "file.txt") → H3
   READ(H3, offset=0, count=4096) → data
```

매 호출에서 서버가 받는 정보만으로 답이 결정되므로, 서버는 호출 이력을 기억할 필요가 없다.

### 5. Idempotency — 네트워크 손실을 무지성으로 견디기

Stateless 서버 + "응답 없으면 그냥 재전송" 정책이 동작하려면, 같은 요청이 두 번 처리돼도 결과가 같아야 한다.

```
 정상:                       응답 손실:                  서버 응답 지연:
  C → REQ → S                C → REQ → S                C → REQ → S
  C ← OK  ← S                C ← XX  ← S (드롭)         (타임아웃)
                             C → REQ → S (재시도)        C → REQ → S (재시도)
                             C ← OK  ← S                C ← OK  ← S
                                                        C ← OK  ← S (지연된 원본)
```

대부분의 NFS RPC는 본질적으로 idempotent하다.

| 연산 | Idempotent? | 이유 |
|------|-------------|------|
| `READ(fh, off, n)` | ✅ | 같은 위치 읽기는 같은 결과 |
| `WRITE(fh, off, data)` | ✅ | 같은 자리에 같은 데이터 다시 써도 동일 |
| `LOOKUP`, `GETATTR` | ✅ | 순수 조회 |
| `CREATE` | ⚠️ | 두 번째 호출은 "이미 존재" 에러 — 별도 처리 필요 |
| `REMOVE` | ⚠️ | 두 번째 호출은 "없음" 에러 |

비-멱등 연산은 서버측 **duplicate request cache** (트랜잭션 ID로 인덱싱)로 막는다.

### 6. 클라이언트 캐시와 일관성 문제

성능을 위해 클라이언트는 데이터 블록과 메타데이터를 캐시한다. 그 순간 두 가지 문제가 생긴다.

**문제 1: Update Visibility** — 한 클라이언트가 쓴 내용을 다른 클라이언트가 언제 보는가?

```
  C1: write("hello")  → C1의 캐시에만 있음
  C2: read()          → 옛날 내용 본다 ❌
```

**문제 2: Stale Cache** — 서버에서 파일이 바뀌었는데 캐시는 옛 버전을 들고 있다.

NFS의 답은 **약한 일관성(weak consistency) + 두 가지 규칙**이다.

#### 규칙 A — Flush-on-close

클라이언트가 파일을 `close()`할 때, 더티 블록을 모두 서버로 flush한다. 즉 다른 클라이언트는 **close 이후에야** 새 내용을 볼 가능성이 생긴다 (close-to-open consistency).

#### 규칙 B — Attribute cache + GETATTR 검증

```
  파일 열 때 / 타임아웃(3~60초) 만료 시:
      GETATTR(fh) 호출
      ├─ mtime이 캐시된 값과 같다 → 캐시 그대로 사용
      └─ mtime이 달라졌다        → 캐시 무효화 + 재요청
```

타임아웃이 짧으면 GETATTR 폭주, 길면 stale 데이터가 오래 보인다. 이 트레이드오프가 NFS 캐시 튜닝의 핵심.

```
 t=0   C1: open & read → 데이터+attr 캐시
 t=10s C2: write & close → 서버 mtime 갱신
 t=15s C1: read → 캐시 hit (아직 타임아웃 안 됨, stale!)
 t=60s C1: read → 타임아웃, GETATTR 발사 → mtime 변경 감지
              → 캐시 invalidate → 서버에서 새로 fetch
```

전통적인 UNIX의 "한 줄 쓰면 다음 줄 read가 즉시 본다" 같은 strict consistency는 **포기**한다. 대부분의 워크로드(한 사용자가 한 파일을 다룸)에서는 문제가 없고, 다중 작성자 시나리오는 애플리케이션이 알아서 락이나 임시 파일로 해결하라는 입장.

### 7. 서버측 쓰기 — Stable Storage 강제

NFSv2의 가장 비싼 규칙: **WRITE의 응답을 보내기 전에 데이터는 안정 저장소(stable storage)에 도달해야 한다.**

이유는 단순하다.

```
 [버퍼링만 한다면]
   C → WRITE → S (RAM 버퍼만)
   C ← OK   ← S
   C: "내 데이터는 안전해"
   S: 💥 크래시
   C: 재시도? 못 한다 — 이미 OK 받았으니 application은 다음 일로 갔다
   → 데이터 손실 (silent!)
```

따라서 서버는 모든 `WRITE`에 대해:

| 옵션 | 비용 | 안전성 |
|------|------|--------|
| 디스크 fsync | 매우 비쌈 (수 ms) | ✅ |
| 배터리 백업 NVRAM에 쓰기 | 빠름 | ✅ |
| RAM 버퍼만 | 매우 빠름 | ❌ (프로토콜 위반) |

이 제약이 1990년대에 **NVRAM 캐시를 탑재한 NFS 서버 어플라이언스**(NetApp 등)가 비싸게 팔린 핵심 이유다 — 정합성을 깨지 않으면서 fsync 비용을 우회할 수 있는 유일한 방법이었기 때문.

---

### 8. 한계와 그 이후

NFSv2의 약점은 곧 그 단순함의 대가다.

| 한계 | 영향 |
|------|------|
| Weak consistency | 다중 클라이언트 동시 갱신 시 stale 가능 |
| 동기 WRITE | 쓰기 지연이 크다 — NVRAM으로 우회 |
| 락 없음 | NFSv2엔 파일 락 부재 (별도 NLM 프로토콜로 보강) |
| 약한 인증 | IP 기반 신뢰 — `root_squash` 정도 |
| 단일 서버 병목 | 복제·샤딩 없음 |
| RENAME 원자성 약함 | 네트워크 레벨에서 보장 어려움 |

그럼에도 NFS가 사실상 표준이 된 이유:

- 프로토콜이 미친 듯 단순해서 누구나 구현 가능
- close-to-open 일관성이 전통적 UNIX 워크플로(편집기, 컴파일러)와 잘 맞음
- Stateless 덕에 운영이 압도적으로 쉬움 — 서버 그냥 껐다 켜도 됨

## 내가 얻은 인사이트

### 설계 철학 관점

1. **"상태를 어디에 둘 것인가"가 분산 시스템의 첫 질문**
   - NFS는 "서버에 상태를 두지 않는다"로 시작해서 거기서 파생되는 모든 비용(idempotency, 캐시 일관성 약화, 동기 쓰기)을 감수했다.
   - 반대로 AFS·SMB 계열은 상태를 두는 대가로 강한 일관성·콜백 무효화를 얻었다.
   - 정답이 없는 축이고, **어떤 실패 모드를 가장 두려워하는가**가 결정한다. NFS는 "서버 크래시 후 복잡한 복구"를 가장 두려워했다.

2. **Idempotency는 분산 시스템의 안전벨트**
   - "응답이 없으면 그냥 재시도"라는 어처구니없이 단순한 정책이 동작하려면, 모든 연산이 멱등이어야 한다.
   - 현대 REST API에서 `PUT`을 권장하고 `POST`에 idempotency key를 붙이는 관행도 같은 뿌리. 1985년 NFS가 이미 답을 알고 있었다.

3. **"적당히 좋은 일관성"의 가치**
   - Linearizability는 비싸다. NFS는 "어차피 한 파일은 한 사람이 쓴다"는 워크로드 관찰로 close-to-open이라는 약한 모델을 정당화했다.
   - 시스템을 설계할 때 "내 워크로드는 정말 strict consistency가 필요한가?"를 먼저 묻는 습관이 중요하다.

### 실무 운영 관점

1. **NFS 클라이언트 캐시 타임아웃은 워크로드에 맞춰야 한다**
   - 빌드 머신처럼 파일이 자주 바뀌는 환경에선 `actimeo`를 짧게 (수초)
   - 정적 콘텐츠 서빙처럼 거의 안 바뀌는 환경에선 길게 (수십 초 이상)
   - 잘못 두면 "왜 새 파일이 안 보이지?" 또는 "왜 이리 GETATTR이 많지?"가 된다.

2. **`sync` vs `async` export 옵션의 진짜 의미**
   - 서버 측 `async`는 "WRITE를 stable storage 가기 전에 OK 보낸다" — **프로토콜을 위반**하는 옵션이다.
   - 성능은 극적으로 올라가지만, 서버 크래시 시 데이터 손실 가능. NVRAM 어플라이언스가 없는 일반 리눅스 NFS 서버에서 함부로 켜면 안 되는 이유.

3. **NFS hang은 stateless 설계의 그늘**
   - 클라이언트가 응답 없으면 "그냥 재시도" — 서버가 영영 안 돌아오면 프로세스가 D 상태로 영원히 멈춘다.
   - `soft` 마운트로 타임아웃을 줄 수 있지만 데이터 무결성 깨질 위험. 그래서 대부분 `hard,intr` 조합을 쓴다.

### 시스템 설계 일반 관점

1. **"단순한 프로토콜이 표준이 된다"의 또 다른 사례**
   - NFS는 더 정교한 경쟁자(AFS, Coda, DFS)를 다 이겼다. 이유는 한 가지: **구현이 쉬워서**.
   - HTTP가 더 정교한 분산 객체 프로토콜(CORBA, DCOM)을 이긴 것과 같은 패턴.

2. **불완전한 추상화는 결국 새는 법**
   - "로컬 파일처럼 보이게 하자"는 NFS의 약속은 네트워크 분할·서버 크래시·캐시 stale 앞에서 깨진다.
   - Waldo의 "A Note on Distributed Computing"이 지적한 그 문제 — 로컬 호출과 원격 호출은 결코 같지 않다 — 의 가장 유명한 실증.

3. **하드웨어가 소프트웨어 설계를 구원한다**
   - "WRITE는 stable storage까지 가야 한다"는 끔찍한 제약을 NVRAM 한 줄로 우회한 NetApp의 사례는, 설계 제약을 새 하드웨어 계층으로 풀어내는 좋은 본보기.
   - 오늘날 NVMe·PMEM이 분산 시스템 설계를 어떻게 다시 쓰고 있는지 떠올리게 한다.
