# 아이스버그 시크릿 - 보이지 않는 90%와 데스크탑 앱 제품화의 함정

## 출처
- **아티클**: The Iceberg Secret, Revealed
- **저자/출처**: Joel Spolsky (Joel on Software)
- **링크**: https://www.joelonsoftware.com/2002/02/13/the-iceberg-secret-revealed/

> 2002년 글이지만, "데모로 도는 것"과 "출시 가능한 제품" 사이의 간극을 가장 정확히 짚은 클래식.
> 데스크탑 앱 제품화에서 반복적으로 부딪히는 인식의 함정을 다룬다.

## AI 요약

### 1. 아이스버그 시크릿이란?

소프트웨어는 빙산과 같다. 수면 위로 드러난 UI는 전체 작업의 **약 10%**, 수면 아래
"보이지 않는 곳(under the covers)"의 엔지니어링이 **약 90%**를 차지한다.

> "Most software is like that too — there's a pretty user interface that takes about 10%
> of the work, and then 90% of the programming work is under the covers."

핵심 비밀은 기술 자체가 아니라 **인식**에 있다.

> "People Who Aren't Programmers Do Not Understand This."

비개발자(경영진·고객·PM)는 빙산의 수면 아래를 보지 못한다. 그래서 **UI가 곧 제품**이라고
착각하고, 진척도를 화면의 완성도로 판단한다.

```
        보이는 것 (비개발자가 인식하는 "제품")
   ┌─────────────────────────────┐
   │   UI / 화면 / 픽셀  ~ 10%     │  ← 데모에서 평가받는 전부
~~~│~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~│~~~ 수면선
   │                             │
   │   상태 관리 / 에러 처리        │
   │   엣지케이스 / 데이터 정합성     │   보이지 않는 것 ~ 90%
   │   성능 / 동시성 / 보안          │   = 제품화의 실체
   │   설치 / 업데이트 / 배포         │
   │   크래시 복구 / 호환성          │
   └─────────────────────────────┘
```

| 구분 | 비개발자의 인식 | 실제 |
|------|----------------|------|
| 작업 비중 | UI = 제품의 거의 전부 | UI ≈ 10%, 내부 ≈ 90% |
| 진척 판단 기준 | "화면이 슬릭해 보이는가" | 기능·안정성·엣지케이스 |
| "거의 다 됐다" 신호 | 예쁜 화면 = 완성 임박 | 예쁜 화면 ≠ 완성 |

### 2. 다섯 가지 따름정리(Corollaries)

Joel이 컨설팅 데모에서 겪은 일화를 토대로 정리한 따름정리들.
(팀은 기능을 거의 완성했지만, 임원들은 오로지 "비주얼이 덜 슬릭하다"에만 꽂혔다.
그래픽 보정은 하루면 끝날 일이었는데 데모 논의 전체를 잡아먹었다.)

| # | 따름정리 | 의미 |
|---|---------|------|
| 1 | UI가 90% 나빠 보이면 제품이 90% 나쁘다고 여긴다 | 시각적 미완성 = 제품 실패로 오인 |
| 2 | 매끈한 UI를 보면 "거의 완성됐다"고 여긴다 | 픽셀이 예쁘면 내부도 끝난 줄 안다 |
| 3 | 미적 완성도가 기능보다 "가치 인식"을 좌우한다 | 보기 좋은 것 > 잘 돌아가는 것 (인식상) |
| 4 | 비개발자에겐 **사소한 결정**을 맡겨라 | 폰트·색·로고 사인오프로 만족시키고, 아키텍처는 사수 |
| 5 | 발표에선 스크린샷이 인식을 지배한다 | Steve Jobs가 시각 최적화에 집착한 이유 |

### 3. 실무 처방

**고객/이해관계자 인식 관리**
- 고객은 자신이 뭘 원하는지 정확히 모른다고 가정하라.
- 도메인 전문성을 가진 **당신이** 설계하라. 세부 설계를 고객 투표에 맡기지 마라.
- 단, 비개발자에게 "사인오프할 거리"를 줘라 — **폰트·색상·로고** 같은 무해한 결정.
  중요한 의사결정처럼 느끼게 하면서, 진짜 아키텍처는 보호한다.

**진척 인식 관리 (가장 실용적인 부분)**
- 미완성 UI는 **명백히 미완성으로 보이게** 하라.
  매끈한 가짜 화면을 보여주면 "다 됐네?"라는 오해를 산다.
- 진척은 **주관적 인상**이 아니라 **객관적 지표**로 보고하라
  — 주간 업데이트, 완료율(%), 일정표.
- 데모에서 무엇을 노출할지 통제해, "프로젝트 건강도"가 화면 광택으로
  오판되지 않게 하라.

```
[안티패턴]  매끈한 더미 UI 데모 → "거의 끝났네!" → 90% 작업이 비가시화
                                            → 일정 압박 / 신뢰 붕괴

[처방]      미완성은 미완성처럼 + 완료율 지표 보고
                → 수면 아래 90%를 "보이게" 만들어 기대치 정렬
```

## 내가 얻은 인사이트

### 데스크탑 앱 제품화 관점

1. **"데모가 도는 것"은 빙산의 수면선일 뿐이다**
   - Electron/Tauri로 창 띄우고 버튼 누르면 동작하는 데모는 10%다.
   - 수면 아래 90%는 데스크탑 특유의 제품화 작업 — 코드 사이닝·notarization,
     자동 업데이트 채널, 인스톨러(MSI/DMG/AppImage), 크래시 리포팅,
     OS별 권한·파일시스템 차이, 오프라인 상태 처리. 이게 전부 "안 보이는" 영역이다.
   - 프레임워크 선택(네이티브 vs 크로스플랫폼)은 결국 **이 90%를 누가, 얼마나
     떠안느냐**의 문제다. 데모 화면만 보면 둘이 똑같아 보이는 게 함정의 본질.

2. **비개발자가 못 보는 곳에 데스크탑 비용이 숨는다**
   - 웹앱과 달리 데스크탑은 "배포 후"가 더 어렵다. 서버처럼 핫픽스를 즉시 밀 수 없고,
     사용자 머신에 깔린 구버전과 영원히 공존해야 한다.
   - 이 비가시 비용을 이해관계자에게 **지표로 번역**하지 않으면, "버튼은 다 되는데
     왜 출시가 안 되냐"는 압박으로 돌아온다. Joel의 처방(완료율·일정표 보고)이
     데스크탑에서 특히 유효한 이유.

### 설계·관리 원칙 관점

3. **사소한 결정권을 의도적으로 양보하라**
   - 따름정리 4는 정치적 기술이다. 색·폰트 사인오프를 내주고 아키텍처 결정권을
     지키는 것 — 데스크탑에선 "어떤 자동 업데이트 전략, 어떤 보안 경계"가
     양보 불가 영역에 해당한다.

4. **시각적 완성도와 진척을 의도적으로 분리(decouple)하라**
   - 미완성을 미완성처럼 보이게 하는 건 정직함이자 기대치 관리 도구다.
   - 반대로 출시 직전 마지막 10%(폴리시)가 인식상 가치를 좌우하므로,
     "90%는 안 보이지만 마지막 10%는 과하게 보인다"는 비대칭을 역이용할 수 있다.

### 트레이드오프 관점

5. **이 글의 한계도 인식하기**
   - 2002년 기준이라 "고객은 설계에 끼우지 말고 당신이 결정하라"는 부분은
     현대의 사용자 리서치·지속적 디스커버리 문화와는 결이 다르다.
   - 다만 **"보이는 10% / 안 보이는 90%"의 비대칭**이라는 핵심 통찰은
     데스크탑 제품화에서 여전히 — 오히려 더 — 유효하다.
