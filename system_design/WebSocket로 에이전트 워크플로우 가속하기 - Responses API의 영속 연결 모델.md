# WebSocket로 에이전트 워크플로우 가속하기 — Responses API의 영속 연결 모델

## 출처

- **저자/출처**: OpenAI (공식 블로그)

## AI 요약

### 1. 무엇을 해결하려는가?

에이전트(agentic) 워크플로우는 한 번의 요청-응답으로 끝나지 않는다. 모델이 추론하고, 도구(tool)를 호출하고, 그 결과를 받아 다시 추론하는 **루프(loop)**가 수십 번 반복된다. 코딩 에이전트가 파일을 읽고-수정하고-테스트를 돌리는 과정을 떠올리면 된다.

기존 방식에서는 이 루프의 **매 스텝마다 별도의 HTTP 요청**을 새로 보냈다. 모델 추론 속도 자체가 빨라지면서, 역설적으로 **네트워크 왕복 시간(round-trip time)과 매 요청마다의 핸드셰이크 오버헤드**가 전체 지연(latency)의 지배적인 원인이 되어버렸다.

> "각 스텝(도구 호출, 중간 추론, 후속 질의)이 별도의 HTTP 요청을 필요로 했다. 추론 속도가 빨라지면서, 이 반복되는 네트워크 왕복 시간이 지연과 운영 복잡성의 지배적 원인이 되었다."

```
[기존 HTTP 방식 — 매 턴마다 새 연결]

Turn 1:  TCP+TLS 핸드셰이크 ──> 요청 ──> 응답 ──> 연결 종료
Turn 2:  TCP+TLS 핸드셰이크 ──> 요청 ──> 응답 ──> 연결 종료
Turn 3:  TCP+TLS 핸드셰이크 ──> 요청 ──> 응답 ──> 연결 종료
   ...                ▲
                      └─ 매번 반복되는 이 오버헤드가 누적된다
                         (20+ 턴이면 치명적)
```

### 2. WebSocket 모드란?

매 턴마다 연결을 새로 맺는 대신, **하나의 영속적(persistent)·양방향 연결**을 `/v1/responses`에 열어두고 그 위에서 모든 턴을 주고받는다.

```
[WebSocket 방식 — 연결을 한 번만 맺고 재사용]

  핸드셰이크 (단 1회)
        │
        ▼
  ┌─────────────────────────────────────────┐
  │  영속 연결 (wss://api.openai.com/v1/responses)
  │                                           │
  │  Turn 1:  response.create ──> 응답 스트림 │
  │  Turn 2:  response.create ──> 응답 스트림 │  ← 핸드셰이크 없음
  │  Turn 3:  response.create ──> 응답 스트림 │  ← 핸드셰이크 없음
  │     ...                                   │
  └─────────────────────────────────────────┘
        │
        ▼
  60분 후 만료 → 재연결 필요
```

핵심은 두 가지다.
1. **핸드셰이크 제거**: TCP+TLS 셋업을 매 턴 반복하지 않는다.
2. **증분 입력(incremental input)**: 매 턴 전체 컨텍스트를 다시 보내지 않고, **새로 생긴 항목**(새 도구 출력, 사용자 메시지)만 보낸다.

### 3. 턴(turn)은 어떻게 동작하는가?

연결은 `wss://api.openai.com/v1/responses`로 맺고, Authorization 헤더에 API 키를 담는다. 각 턴은 `response.create` 이벤트로 시작한다.

| 필드 | 설명 |
|------|------|
| `model` | 모델 식별자 (예: `gpt-5.5`) |
| `input` | 메시지 + 도구 출력 항목 배열 |
| `tools` | 도구 정의 |
| `store` | 응답 영속화 여부 |
| `previous_response_id` | 이전 응답과 체이닝 (첫 턴은 생략 가능) |

> 페이로드는 HTTP Responses API의 create 바디와 거의 동일하지만, `stream`·`background` 같은 **전송(transport) 관련 필드는 생략**한다. (연결 자체가 이미 스트리밍이므로)

**첫 턴**: 전체 입력 컨텍스트를 담아 `response.create`를 보낸다.
**이후 턴**: `previous_response_id`로 직전 응답을 가리키고, **증분 항목만** 보낸다.

```
First turn:   response.create { input: [전체 컨텍스트], tools: [...] }
                                       │
                                       ▼ response_id = resp_A
Continuation: response.create { previous_response_id: "resp_A",
                                input: [새 도구 출력만] }   ← 페이로드가 작다
```

### 4. `previous_response_id`와 연결-로컬 캐시

WebSocket 모드의 속도 비결은 **연결별 인메모리 캐시(connection-local cache)**다. 서비스가 각 연결에서 **가장 최근 응답을 메모리에 들고 있기** 때문에, 이어가기 할 때 디스크에서 상태를 복원(hydration)할 필요가 없다.

```
[캐시 동작 흐름]

previous_response_id 가 캐시에 있나?
        │
   ┌────┴────┐
  YES        NO
   │          │
빠른 이어가기  store=true  → 디스크에서 복원 (느림)
(디스크 접근   store=false → previous_response_not_found 에러
 없음)         / ZDR
```

이 설계 덕분에 `store=false`와 **Zero Data Retention(ZDR)** 모드에서도 동작한다. 데이터를 영구 저장하지 않아도, 연결이 살아있는 동안은 메모리 캐시로 빠른 이어가기가 가능하기 때문이다.

### 5. HTTP/SSE와의 차이 정리

| 항목 | HTTP / SSE | WebSocket 모드 |
|------|-----------|----------------|
| 연결 | 매 턴 새로 맺음 | 영속 (최대 60분) |
| 핸드셰이크 오버헤드 | 매 턴 반복 | 1회 |
| 동시 처리 | 멀티플렉싱 가능 | **한 번에 응답 1개**(no multiplexing) |
| 상태 복원 | 디스크에서 hydration | 연결-로컬 캐시 |
| 병렬 워크플로우 | 요청 여러 개 | **연결을 여러 개** 열어야 함 |

### 6. 성능 결과

OpenAI가 보고한 프로덕션 초기 결과:

| 지표 | 수치 |
|------|------|
| 종단간(end-to-end) 지연 감소 | **최대 40%** (20+ 도구 호출 워크플로우 기준) |
| 지속 처리량(throughput) | **~1,000 TPS** |
| 버스트 처리량 | **최대 4,000 TPS** |

얼리 어답터 사례:

| 회사 | 개선 |
|------|------|
| Vercel | 지연 40% 감소 |
| Cline | 멀티 파일 워크플로우 **39% 더 빠름** |
| Cursor | OpenAI 모델 사용 시 **최대 30% 더 빠름** |

> 알파 사용자들은 자신의 에이전트 워크플로우에서 **최대 40%의 개선**을 보고했다.

### 7. 제약과 베스트 프랙티스

- **연결당 한 번에 하나의 응답만** 처리 → 병렬 작업은 연결을 여러 개 사용.
- 연결은 **60분 후 만료** → 만료 시 ① 영속화된 응답 ID로 이어가거나 ② 전체 컨텍스트로 새로 시작.
- **연결 워밍업**: 시스템 프롬프트와 도구 정의를 먼저 보내두면, 에이전트가 셋업 반복 없이 곧바로 작업을 시작할 수 있다.
- **긴 컨텍스트**: `/responses/compact` 엔드포인트로 컨텍스트를 압축한 뒤, 압축된 윈도우를 입력으로 새 응답을 시작.
- **가장 효과적인 경우**: 모델-도구 왕복이 많은 워크플로우(에이전틱 코딩, 반복 도구 호출이 있는 오케스트레이션 루프).

## 내가 얻은 인사이트

### 아키텍처 관점

1. **"빨라진 추론이 네트워크를 병목으로 만든다"는 역설**
   - 컴퓨팅 한 축이 빨라지면 다음 병목이 드러난다는 고전적 패턴이다. 모델 추론이 충분히 느렸을 때는 핸드셰이크 오버헤드가 노이즈였지만, 추론이 빨라지자 같은 절대 시간이 갑자기 지배적 비용이 됐다. **최적화 대상은 시스템 전체의 균형에 따라 이동한다**는 점을 잘 보여준다.

2. **상태(state)를 어디에 두느냐가 지연을 결정한다**
   - WebSocket 모드의 본질은 "WebSocket"이라는 프로토콜이 아니라 **연결-로컬 인메모리 캐시**다. 디스크 hydration을 메모리 조회로 바꾼 것이 진짜 속도 향상의 원천이다. 영속 연결은 그 캐시를 안전하게 보장할 수 있는 "수명 단위(lifetime scope)" 역할을 한다. 프로토콜보다 **데이터 지역성(locality)** 설계가 먼저다.

3. **증분 전송 = 클라이언트 상태와 서버 상태의 계약**
   - `previous_response_id`로 "직전까지는 네가 기억하고 있지?"를 전제하고 새 항목만 보낸다. 이는 분산 시스템에서의 전형적인 **상태 동기화 트레이드오프**다. 빠르지만, 캐시가 날아가면(`previous_response_not_found`) 클라이언트가 전체 컨텍스트로 폴백할 수 있어야 한다.

### 트레이드오프 관점

4. **멀티플렉싱을 포기하고 지연을 얻었다**
   - HTTP/2의 강점인 한 연결 내 동시 다중 요청을 버리고, "연결당 한 번에 하나"라는 단순한 직렬 모델을 택했다. 대신 캐시 일관성이 단순해지고 이어가기가 빨라진다. **병렬성이 필요하면 연결을 늘려라**는 설계는, 복잡성을 프로토콜이 아닌 운영(연결 풀 관리) 쪽으로 밀어낸 선택이다.

5. **60분 만료와 폴백 설계는 클라이언트 몫**
   - 영속 연결은 공짜가 아니다. 만료·재연결·압축(compact)을 클라이언트가 직접 다뤄야 한다. 즉 이 기능을 제대로 쓰려면 SDK/클라이언트 레이어에 **연결 수명 관리 + 컨텍스트 폴백 로직**이 필수다. "빠르지만 상태를 책임져야 하는" 전형적 성능 최적화의 모습.

### 실무 적용 관점

6. **언제 쓰고 언제 쓰지 말아야 하는가**
   - 단발성 요청(챗봇 1턴 응답)에는 의미가 없다. 효과는 **왕복 횟수에 비례**한다. 20+ 도구 호출이 도는 코딩 에이전트·오케스트레이션 루프에서만 40% 같은 숫자가 나온다. 도입 전에 "내 워크플로우의 평균 턴 수가 몇인가?"를 먼저 측정하는 게 합리적이다.

7. **ZDR과 속도를 동시에 잡은 점**
   - 보안/규제상 데이터 영속화가 금지된(`store=false`/ZDR) 환경에서는 보통 성능을 포기해야 했다. 연결-로컬 캐시는 "저장은 안 하지만 연결 수명 동안은 빠르다"는 절충안을 제공한다. **컴플라이언스와 성능이 반드시 상충하는 건 아니다**라는 좋은 예시.
