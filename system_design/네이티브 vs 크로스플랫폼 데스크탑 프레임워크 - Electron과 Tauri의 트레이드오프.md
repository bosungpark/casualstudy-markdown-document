## AI 요약

### 1. 데스크탑 앱 프레임워크의 근본 선택이란?

데스크탑 앱을 만들 때 가장 먼저 부딪히는 결정은 **"UI를 무엇으로 그릴 것인가"**이다. 이 선택이 번들 크기, 메모리, 보안 모델, 팀 구성, 개발 속도까지 연쇄적으로 결정한다. 크게 세 갈래가 있다.

| 갈래 | 대표 기술 | UI 렌더링 | 백엔드 언어 |
|---|---|---|---|
| **네이티브** | Qt, WinUI, AppKit/Cocoa | OS 네이티브 위젯 (C++ 컴파일) | C++, Swift, C# |
| **번들 웹뷰** | Electron | 앱에 동봉한 Chromium | Node.js (JS/TS) |
| **시스템 웹뷰** | Tauri | OS가 제공하는 웹뷰 (WRY) | Rust |

**핵심 질문**: "렌더링 일관성"과 "리소스 효율" 중 무엇을 살 것인가. Electron은 Chromium을 통째로 동봉해 일관성을 사고, Tauri는 OS 웹뷰를 빌려 효율을 산다.

---

### 2. 아키텍처 - 무엇을 동봉하는가가 모든 것을 가른다

```
┌──────────────────────── Electron ────────────────────────┐
│  앱 설치 파일 (~150MB+)                                    │
│  ┌────────────────────┐  ┌──────────────────┐            │
│  │  Chromium 엔진 통째 │  │  Node.js 런타임   │            │
│  │  (수십 MB)          │  │  (수십 MB)        │            │
│  └────────────────────┘  └──────────────────┘            │
│  → 모든 OS에서 "같은 브라우저"로 렌더 → 픽셀 단위 동일     │
└───────────────────────────────────────────────────────────┘

┌──────────────────────── Tauri ───────────────────────────┐
│  앱 설치 파일 (~3~10MB)                                    │
│  ┌─────────────────┐                                      │
│  │  Rust 바이너리   │ ──── WRY ────► OS 웹뷰 호출           │
│  └─────────────────┘                  │                   │
│                    Windows: WebView2 (Chromium 기반)       │
│                    macOS:   WKWebView (WebKit)             │
│                    Linux:   WebKitGTK (WebKit)             │
│  → OS마다 다른 엔진 → 미세한 렌더링 차이 가능              │
└───────────────────────────────────────────────────────────┘
```

Electron은 Chromium 브라우저 엔진을 통째로 번들링해 **모든 플랫폼에서 동일한 UI 렌더링을 보장**한다. Tauri는 WRY 라이브러리를 통해 각 OS의 네이티브 웹뷰를 활용해 이 중복을 제거한다.

---

### 3. 번들 크기 - 가장 극적인 차이

| 항목 | Electron | Tauri |
|---|---|---|
| "Hello World" 앱 | **최대 150MB** | 수 MB |
| 원인 | Chromium 엔진 + Node.js 런타임 동봉 | OS 웹뷰 재사용 |

> 기본 to-do 앱 벤치마크(타 출처): Electron 1.5초 기동 / 250MB RAM vs Tauri 0.4초 기동 / 28MB RAM. 메모리·기동 속도에서 한 자릿수 배의 차이가 난다.

차이가 "극적(dramatic)"인 이유는 단순하다. **Electron은 모든 앱이 자기만의 브라우저를 들고 다니고, Tauri는 OS에 이미 있는 브라우저를 빌린다.**

---

### 4. 웹뷰 일관성 - Tauri가 지불하는 대가

```
같은 CSS 코드
     │
     ├── Electron ──► 어디서나 Chromium ──► 동일 렌더링 ✓
     │
     └── Tauri ──┬── Windows: WebView2  (Chromium) ──► OK
                 ├── macOS:   WKWebView (WebKit)    ──► 미세 차이
                 └── Linux:   WebKitGTK (WebKit)    ──► 프리픽스/폰트 차이 가능
```

Electron은 어디서나 표준화된 렌더링을 보장한다. Tauri는 이 확실성을 효율과 맞바꾼다. 다만 저자는 **"주요 OS에서 시스템 웹뷰 간 호환성 문제는 매우 드물다(exceedingly rare)"**고 평가한다. Linux WebKitGTK가 프리픽스나 폰트 렌더링에서 뒤처지는 경우가 실무에서 가장 자주 보고되는 엣지 케이스다.

---

### 5. 언어와 개발자 경험 - JS냐 Rust냐

| 측면 | Electron | Tauri |
|---|---|---|
| 메인 프로세스 언어 | Node.js (JS/TS만으로 완성) | **Rust** (단, JS API도 "꽤 견고") |
| 적합한 팀 | 웹 개발자 중심 | Rust를 다룰 수 있는 팀 |
| IPC 추상화 | 직접 다뤄야 함 (장황) | 더 자연스럽게 맞아떨어짐 |
| Next.js 통합 | 유지보수 끊긴 Nextron 우회 필요 | `output: 'export'` 한 줄 |
| 사이드카 프로세스 | 동봉된 Node로 간단 실행 | `pkg` 등으로 바이너리 컴파일 필요 |

저자는 Tauri의 추상화가 Electron의 IPC 요구사항보다 **"더 자연스럽게 맞아떨어졌다(fit more naturally)"**고 평한다. 반대로 Node 생태계에 의존하는 사이드카(별도 Node 앱 동봉 실행)는 Electron이 압도적으로 간편하다.

---

### 6. Tauri의 현재 한계 (2025 기준)

| 한계 | 내용 |
|---|---|
| Windows 패키지 | `.appx` / `.msix` 번들 미지원 |
| macOS 서명 | universal binary 코드사이닝 이슈 |

저자는 **"두 한계 모두 대부분의 앱에서 하드 블로커는 아니다"**라고 본다. 한편 Tauri 생태계는 빠르게 성장 중이다(타 출처: 공식·커뮤니티 플러그인이 2025년 1월 47개 → 2026년 4월 120개+).

### 7. 선택 가이드 - 한눈에

```
                ┌─────────────────────────────────────────┐
                │  팀이 JS/TS 중심이고 빠르게 출시해야 한다 │
                │  복잡한 웹 자산을 그대로 재사용한다       │
                │  Node 생태계/사이드카에 깊이 의존한다     │──► Electron
                └─────────────────────────────────────────┘
                ┌─────────────────────────────────────────┐
                │  번들 크기·메모리·기동 속도가 중요하다    │
                │  보안 공격면을 최소화하고 싶다            │
                │  Rust를 다룰 수 있다                      │──► Tauri
                └─────────────────────────────────────────┘
                ┌─────────────────────────────────────────┐
                │  네이티브 룩앤필·최고 성능·임베디드/IoT   │
                │  OS 위젯 그대로의 접근성이 필요하다       │──► Qt 등 네이티브
                └─────────────────────────────────────────┘
```

## 내가 얻은 인사이트

### 아키텍처 관점

1. **"무엇을 동봉하는가"가 프레임워크의 정체성이다**
   - Electron = 브라우저를 동봉(번들 폭발, 일관성 획득), Tauri = 브라우저를 빌림(경량, 일관성 포기). 같은 "웹 기술로 데스크탑"이라는 목표를 정반대 전략으로 푼다. **프레임워크 선택은 기능 비교가 아니라 "어떤 리스크를 내가 떠안을 것인가"의 선택**이다.

2. **일관성은 비용을 치르고 사는 상품이다**
   - Chromium을 동봉하는 150MB는 "어디서나 똑같이 보인다"는 보장의 가격표다. Tauri는 이 보장을 포기하는 대신 수 MB로 내려간다. **"공짜 일관성"은 없고, OS 웹뷰의 파편화를 누가 감당하느냐의 문제일 뿐**이다.

### 트레이드오프 관점

1. **번들 크기 vs 렌더링 확실성**
   - 50배 작은 바이너리(3MB vs 150MB)를 얻는 대가가 "Linux에서 폰트가 살짝 다를 수 있음"이라면, 다운로드 이탈률이 중요한 소비자 앱에서는 충분히 합리적인 거래다. 반대로 픽셀 단위 디자인이 생명인 제품은 그 확실성에 150MB를 지불할 만하다.

2. **언어 선택이 곧 채용 시장 선택이다**
   - Tauri의 Rust 요구는 기술적 장벽이자 **채용 풀의 제약**이다. JS 개발자는 흔하고 Rust 개발자는 귀하다. 프레임워크 결정에 "우리 팀이 6개월 뒤 이걸 유지보수할 수 있는가"를 반드시 넣어야 한다.

### 실무 적용 관점

1. **사이드카 의존성을 먼저 점검하라**
   - 기존 Node 백엔드를 데스크탑에 동봉해야 한다면 Electron이 압도적으로 쉽다. Tauri는 `pkg`로 바이너리화하는 추가 공정이 든다. **레거시 자산의 형태가 프레임워크를 사실상 결정**하는 경우가 많다.

2. **"하드 블로커가 아니다"를 곧이곧대로 믿지 마라**
   - Tauri의 `.msix` 미지원, universal binary 서명 이슈는 "대부분 괜찮다"지만, 엔터프라이즈 배포(MSIX 강제)나 Apple Silicon+Intel 동시 지원이 필요하면 바로 블로커가 된다. **내 배포 채널의 제약을 프레임워크 한계와 먼저 대조**해야 한다.

3. **벤치마크 숫자는 "기본 앱" 기준임을 기억하라**
   - 28MB vs 250MB는 to-do 앱 기준이다. 실제 복잡한 앱에서는 프론트엔드 자산이 메모리를 지배하기 시작하면서 격차가 줄 수 있다. **자기 앱 프로토타입으로 직접 측정**하는 것이 모든 비교표보다 정확하다.
