## AI 요약

### 1. 한 줄 요약

`A`를 누르면 화면에 `A`가 찍히는 "즉각적인" 동작은, 사실 **물리 스위치 → 스캔코드 → 인터럽트 → 커널 → 키코드 → 이벤트 → 디스플레이 서버 → 문자**로 이어지는 여러 계층의 정교한 릴레이다. 각 계층은 바로 위/아래 계층만 알고, 그 덕분에 키보드 제조사와 애플리케이션이 서로를 몰라도 된다.

### 2. 전체 흐름 한눈에 보기

```
 [물리 스위치]  A 키를 누른다
      │  키 매트릭스(행×열) 회로가 닫힘
      ▼
 [키보드 MCU]  스캔코드 생성  (make code 0x1E / break code 0x9E)
      │
      ▼
 [키보드 컨트롤러]  하드웨어 버퍼에 적재 (레거시: I/O 포트 0x60, 8042 칩)
      │  IRQ 1 라인을 올림
      ▼
 [인터럽트 컨트롤러(PIC/APIC)]  IRQ → 인터럽트 벡터로 변환
      │
      ▼
 [CPU]  현재 작업 중단 → 컨텍스트 저장 → Ring 3에서 Ring 0로 강제 전환
      │  IDT(Interrupt Descriptor Table)에서 벡터로 핸들러 주소 조회
      ▼
 [ISR: 키보드 인터럽트 핸들러]
      │  포트 0x60에서 스캔코드 읽기 → EOI 전송 → iret
      ▼
 ┌── Linux Input Subsystem ─────────────────────────┐
 │  ① 디바이스 드라이버 (atkbd/usbhid)              │
 │       스캔코드 0x1E  →  키코드 KEY_A (30)        │
 │  ② 인풋 코어                                     │
 │       디바이스 등록 관리, 핸들러로 라우팅        │
 │  ③ 이벤트 핸들러 (evdev)                         │
 │       struct input_event 로 표준화               │
 │       → /dev/input/eventX 캐릭터 디바이스 파일   │
 └──────────────────────────────────────────────────┘
      │
      ▼
 [디스플레이 서버 (X.org / Wayland) 또는 커널 TTY 드라이버]
      │  키맵 적용(QWERTY/AZERTY) + Shift 등 모디파이어 상태 확인 + 포커스 창 판단
      │  KEY_A + Shift  →  문자 'A'
      ▼
 [애플리케이션]  read(0, buf, 1024) 가 블록 해제되고 'A' 를 수신
      │
      ▼
 [화면]  'A' 렌더링
```

### 3. 하드웨어 계층: 키보드는 "문자"를 보내지 않는다

키보드 안에는 마이크로컨트롤러와 **키 매트릭스**(행/열 격자 회로)가 들어 있다. 키를 누르면 특정 행과 열이 접점으로 연결되고, MCU는 어느 교차점이 닫혔는지로 "몇 번 자리의 키인가"를 안다.

여기서 나오는 것이 **스캔코드(scancode)** — **문자가 아니라 키의 물리적 위치를 나타내는 하드웨어 수준 식별자**다.

| 구분 | 이름 | 예시 (A 키) | 의미 |
|---|---|---|---|
| 누를 때 | make code | `0x1E` | 키 활성화 |
| 뗄 때 | break code | `0x9E` | 키 비활성화 |

> 누름과 뗌이 별도 코드로 나뉘기 때문에 "Shift를 누른 채로 A를 누른다" 같은 **동시 입력·조합키·키 반복**이 성립한다. 만약 키보드가 문자 하나만 보냈다면 모디파이어라는 개념 자체가 불가능하다.

스캔코드는 **키보드 컨트롤러**(레거시 Intel 8042, 현대에는 칩셋에 통합)의 하드웨어 버퍼에 담기고, CPU는 I/O 포트 `0x60`을 통해 읽어간다.

### 4. 인터럽트: CPU를 어떻게 부르는가

키보드는 폴링이 아니라 **인터럽트**로 CPU를 부른다. 관례상 키보드는 **IRQ 1**을 사용한다.

```
키보드 컨트롤러 ──IRQ 1──> PIC/APIC ──인터럽트 벡터──> CPU
```

CPU가 인터럽트를 받으면 벌어지는 일:

1. 실행 중이던 명령 흐름을 중단하고 **컨텍스트(instruction pointer 등)를 저장**한다.
2. **강제 컨텍스트 스위치**가 일어난다. 유저 공간(Ring 3, 비특권) → 커널 공간(Ring 0, 특권).
3. **IDT(Interrupt Descriptor Table)** 를 참조한다. IDT는 부팅 시 커널이 세팅해 둔 배열로, "인터럽트 벡터 → 핸들러 주소"의 **교환대(switchboard)** 역할을 한다.
4. 해당 **ISR(Interrupt Service Routine)** 로 점프한다.

키보드 ISR이 하는 일:

```c
// 개념적 흐름
save_registers();                 // ① CPU 레지스터 상태 저장
scancode = inb(0x60);             // ② 포트 0x60에서 스캔코드 읽기
send_eoi();                       // ③ 인터럽트 컨트롤러에 EOI(End Of Interrupt)
input_subsystem_push(scancode);   // ④ 상위 소프트웨어 스택으로 전달
restore_registers();              // ⑤ 컨텍스트 복원
iret;                             // ⑥ 원래 하던 일로 복귀
```

> **인터럽트는 하드웨어가 커널에 진입할 수 있는 "통제된 관문"이다.** 유저 프로세스가 마음대로 Ring 0에 들어갈 수는 없지만, 하드웨어는 CPU가 보장하는 이 경로로만 커널을 깨울 수 있다.

### 5. 리눅스 인풋 서브시스템: 3계층 구조

리눅스는 입력 처리를 세 층으로 나눈다.

| 계층 | 역할 | 대표 구현 | 입력 → 출력 |
|---|---|---|---|
| ① 디바이스 드라이버 | 하드웨어 의존 코드를 흡수 | `atkbd`, `usbhid` | 스캔코드 `0x1E` → 키코드 `KEY_A`(30) |
| ② 인풋 코어 | 디바이스 등록/라우팅 | `input.ko` | 키코드를 적절한 핸들러로 |
| ③ 이벤트 핸들러 | 표준 이벤트로 포장 + 유저 공간 노출 | `evdev` | `struct input_event` → `/dev/input/eventX` |

**키코드(keycode)** 는 커널이 정의한 표준 번호다. `KEY_A`는 **어떤 키보드에서 왔든 항상 30**이다. Dell PS/2 키보드든 Lenovo USB 키보드든 상관없다. 즉 스캔코드는 "이 키보드의 이 자리", 키코드는 "추상적인 A 키"다.

evdev가 만드는 표준 구조체:

```c
struct input_event {
    struct timeval time;  // 이벤트 발생 시각
    __u16 type;           // EV_KEY  (이벤트 범주: 키/상대좌표/절대좌표…)
    __u16 code;           // KEY_A   (구체적 키코드)
    __s32 value;          // 1=press, 0=release, 2=autorepeat
};
```

이것이 `/dev/input/eventX` 캐릭터 디바이스 파일로 노출되고, **어떤 프로그램이든 이 파일을 read()해서 원시 입력 이벤트를 받을 수 있는 통일된 API**가 된다.

#### 데이터 변형 요약표

| 단계 | 데이터 형태 | 'a' 키 예시 | 위치 |
|---|---|---|---|
| 하드웨어 | 스캔코드 | `0x1E` | 키보드 컨트롤러 |
| 드라이버 | 키코드 | `KEY_A` (30) | atkbd 드라이버 |
| 이벤트 핸들러 | input_event 구조체 | `{EV_KEY, KEY_A, 1}` | evdev |
| 애플리케이션 | 문자 | `'a'` 또는 `'A'` | 디스플레이 서버 / 콘솔 |

### 6. 마지막 번역: 키코드는 아직 문자가 아니다

여기까지 와도 우리가 가진 건 `KEY_A`라는 **위치의 추상화**일 뿐, `'a'`도 `'A'`도 아니다. 마지막 번역은 GUI 환경이냐 텍스트 콘솔이냐에 따라 담당자가 다르다.

- **GUI (X.org / Wayland)**: 디스플레이 서버가 `/dev/input/*`를 열어 읽고,
  1. **키맵(레이아웃)** 적용 — QWERTY/AZERTY/QWERTZ에 따라 같은 물리 위치가 다른 글자가 된다
  2. **모디파이어 상태** 확인 — Shift/Ctrl/Alt가 눌려 있는가
  3. **포커스 창** 판단 — 지금 이 키를 받아야 할 애플리케이션은 누구인가
  4. 최종 문자를 그 애플리케이션에 디스패치
- **텍스트 콘솔**: 커널의 **TTY(터미널 드라이버) 서브시스템**이 같은 역할을 한다.

즉 `KEY_A` + `Shift` → `'A'`가 되는 지점은 커널이 아니라 **훨씬 위쪽**이다. 그래서 레이아웃을 AZERTY로 바꿔도 커널을 다시 컴파일할 필요가 없다.

### 7. 애플리케이션까지: read()가 깨어나는 순간

```c
n = read(0, buffer, 1024);   // stdin(fd 0)에서 읽기 — 데이터 없으면 블록
```

1. 애플리케이션이 `read()`를 호출하고 **블록되어 잠든다** (CPU를 쓰지 않는다).
2. 디스플레이 서버가 문자를 터미널로 디스패치한다.
3. 커널이 그 문자를 터미널의 stdin 버퍼에 넣는다.
4. 커널이 잠든 프로세스를 **깨우고**, `read()`가 블록 해제된다.
5. 커널이 문자를 프로세스 메모리로 복사하고, 읽은 바이트 수를 반환한다.

`strace`로 실제로 볼 수 있다:

```
read(0, "h\n", 1024) = 2
      │    │     │      └─ 반환값: 2바이트 전송됨
      │    │     └─ 버퍼 크기
      │    └─ 실제 읽힌 데이터 ('h' + 개행)
      └─ 파일 디스크립터 0 = stdin
```

### 8. 이 설계가 말하는 세 가지 원칙

| 원칙 | 내용 |
|---|---|
| **추상화 (Abstraction)** | 스캔코드 → 키코드 → 문자로 단계마다 하드웨어 색깔을 벗긴다. 덕분에 키보드 제조사와 애플리케이션이 서로 독립적으로 진화한다. |
| **보안 경계 (Security Boundary)** | 유저/커널 분리(Ring 3 / Ring 0)와 IDT를 통한 통제된 진입점 덕분에, 잘못된 애플리케이션이 시스템 전체를 무너뜨릴 수 없다. |
| **이벤트 기반 효율 (Event-Driven)** | 인터럽트가 폴링을 대체한다. 키를 누르지 않는 동안 CPU는 낭비 없이 다른 유용한 일을 한다. |

## 내가 얻은 인사이트

### 시스템 설계 관점

1. **"위치"와 "의미"의 분리가 이 파이프라인의 핵심 발명이다.**
   - 키보드는 끝까지 `'A'`를 보내지 않는다. 보내는 건 "이 자리의 키가 눌렸다"뿐이다.
   - 만약 키보드가 문자를 직접 보냈다면, 프랑스어 AZERTY 레이아웃을 지원하려고 키보드 펌웨어를 갈아엎어야 했을 것이다. **의미 부여를 최대한 늦게(=가장 위 계층에서) 하는 것**이 유연성의 원천이다.
   - 이건 도메인 설계에도 그대로 적용된다. 하위 계층은 raw fact("클릭됨", "row 3 col 5")를 전달하고, 정책·해석은 상위 계층에 둔다.

2. **계층마다 "가장 늦게 결정해도 되는 것"을 미룬다.**
   - 스캔코드 시점에는 문자를 모른다 → 키코드 시점에도 여전히 대소문자를 모른다 → 디스플레이 서버에 와서야 Shift 상태와 레이아웃을 합쳐 문자가 된다.
   - **정보를 잃지 않고 위로 올리기만 하면**, 상위 계층은 언제든 다른 해석을 선택할 수 있다. 게임이 raw scancode를 직접 받아 "AZERTY에서도 ZQSD가 아니라 WASD 자리"를 쓰는 게 가능한 이유다.

3. **`/dev/input/eventX` 라는 파일 추상화의 힘.**
   - "모든 것은 파일이다"라는 유닉스 철학 덕분에, 입력 장치를 다루는 특별한 API를 배울 필요 없이 `open`/`read`만 알면 된다.
   - 그 결과 키 로거, 매크로 도구, 리맵퍼(kmonad, evremap 등)가 커널 수정 없이 유저 공간에서 만들어질 수 있다. **좋은 추상화는 설계자가 의도하지 않은 확장을 허용한다.**

### 성능 / 운영 관점

4. **인터럽트 vs 폴링은 지금도 살아 있는 트레이드오프다.**
   - 키보드처럼 **저빈도·저지연**이 필요한 장치는 인터럽트가 압도적으로 유리하다 (안 눌리는 동안 CPU 0%).
   - 반대로 NVMe·고속 NIC처럼 **초당 수십만 이벤트**가 오는 장치는 인터럽트 폭풍(interrupt storm)이 문제가 되어 오히려 폴링(NAPI, io_uring polled mode, DPDK)으로 되돌아간다.
   - 즉 "인터럽트가 항상 옳다"가 아니라 **이벤트 빈도가 컨텍스트 스위치 비용을 정당화하는가**의 문제다.

5. **블로킹 `read()`가 "느린" 게 아니다.**
   - 잠든 프로세스는 CPU를 쓰지 않는다. 커널이 데이터 도착 시점에 정확히 깨워준다.
   - 바쁜 대기(busy-wait)로 입력을 기다리는 코드가 더 빠를 것 같지만, 실제로는 CPU만 태우고 지연은 나아지지 않는다. **커널의 스케줄러를 신뢰하는 편이 대개 낫다.**

### 디버깅 관점

6. **키 입력이 이상할 때, 어느 계층에서 깨졌는지 이분 탐색할 수 있다.**

   | 증상 | 확인할 계층 | 도구 |
   |---|---|---|
   | 키를 눌러도 아무 반응 없음 | 하드웨어 / 드라이버 | `evtest`, `dmesg`, `/dev/input/` 존재 여부 |
   | 이벤트는 오는데 엉뚱한 키 | 스캔코드 → 키코드 매핑 | `evtest`, `showkey`, `setkeycodes` |
   | 키코드는 맞는데 문자가 틀림 | 레이아웃 / 키맵 | `xev`, `setxkbmap`, `localectl` |
   | 특정 앱에서만 안 먹힘 | 포커스 / 앱 단 키 바인딩 | 앱 로그, 창 관리자 설정 |

   - **계층 구조를 안다는 건 곧 문제를 절반씩 잘라낼 수 있다는 뜻이다.** "키가 안 눌려요"가 4가지 완전히 다른 문제로 갈라진다.

7. **`strace`는 유저/커널 경계를 관찰하는 창이다.**
   - `read(0, "h\n", 1024) = 2` 한 줄이 "커널이 내 프로세스를 깨워서 2바이트를 복사해줬다"는 사실을 증명한다.
   - 추상적으로만 알던 시스템 콜 경계를 **눈으로 확인할 수 있는 것**이 이해의 질을 바꾼다.

### 남은 질문 (이 아티클이 다루지 않은 부분)

8. 이 글은 **문자가 애플리케이션에 도달하는 지점에서 끝난다.** 실제로 화면에 `A` 픽셀이 찍히기까지는 아직:
   - 폰트 파일에서 글리프 조회 → **래스터라이즈**(FreeType 등, 힌팅/서브픽셀 안티에일리어싱)
   - 텍스트 셰이핑(HarfBuzz), 레이아웃 계산
   - GPU 텍스처 업로드 → 컴포지터가 창들을 합성 → 프레임버퍼
   - 디스플레이로 스캔아웃 (VSync, 리프레시 주기)

   가 남아 있다. **"키 to 픽셀 지연시간(input latency)"** 논쟁에서 실제 병목은 대부분 이 후반부(컴포지팅 + 디스플레이 파이프라인)에 있다는 점은 별도로 파볼 만하다.
