## AI 요약

### 1. 클린 아키텍처란?

클린 아키텍처(Clean Architecture)는 Robert C. Martin이 헥사고날(Hexagonal), 어니언(Onion), DCI, BCE 등 그동안 나온 여러 아키텍처들을 관통하는 **공통점 하나**를 뽑아 정리한 설계 지침이다. 그 공통점은 다음과 같다.

> **관심사를 계층으로 분리하고, 소스 코드의 의존성은 오직 안쪽(고수준 정책)을 향하게 한다.**

이렇게 만든 시스템은 다음과 같은 특성을 갖는다.

| 특성 | 의미 |
|------|------|
| **프레임워크 독립성** | 스프링/장고 같은 프레임워크를 도구로 쓸 뿐, 거기에 시스템을 끼워 맞추지 않는다 |
| **테스트 용이성** | 비즈니스 규칙을 UI·DB·웹서버 없이 단독으로 테스트할 수 있다 |
| **UI 독립성** | UI를 웹 → CLI로 바꿔도 비즈니스 규칙은 그대로다 |
| **DB 독립성** | Oracle을 Mongo로 갈아끼워도 비즈니스 규칙은 영향받지 않는다 |
| **외부 세계 독립성** | 핵심 비즈니스 규칙은 바깥세상에 대해 아무것도 모른다 |

핵심 아이디어 자체는 의외로 단순하다. **UI와 비즈니스 규칙 사이에 단순한 데이터 구조만 주고받게 해서 둘을 떼어놓는 것**이다.

### 2. 동심원 4계층 구조

클린 아키텍처는 흔히 양파 같은 동심원으로 그려진다. 안쪽일수록 추상적이고 변하지 않는 정책, 바깥쪽일수록 구체적인 도구(detail)다.

```
            ┌─────────────────────────────────────────────┐
            │  Frameworks & Drivers (가장 바깥)             │
            │  Web / DB / UI / Devices / External Interface │
            │   ┌─────────────────────────────────────┐    │
            │   │  Interface Adapters                  │    │
            │   │  Controllers / Presenters / Gateways │    │
            │   │   ┌─────────────────────────────┐    │    │
            │   │   │  Use Cases                   │    │    │
            │   │   │  (애플리케이션 비즈니스 규칙)  │    │    │
            │   │   │   ┌─────────────────────┐    │    │    │
            │   │   │   │   Entities          │    │    │    │
            │   │   │   │ (전사적 비즈니스 규칙)│    │    │    │
            │   │   │   └─────────────────────┘    │    │    │
            │   │   └─────────────────────────────┘    │    │
            │   └─────────────────────────────────────┘    │
            └─────────────────────────────────────────────┘

       의존성 방향:  바깥 ───────────────────▶ 안쪽 (한 방향만!)
```

> 계층 개수가 꼭 4개일 필요는 없다. 규모에 따라 더 많거나 적을 수 있다. **변하지 않는 규칙은 "의존성은 안쪽을 향한다"는 것 하나뿐**이다.

| 계층 | 무엇이 들어가나 | 변경 빈도 |
|------|----------------|-----------|
| **Entities** | 전사적(enterprise-wide) 핵심 비즈니스 규칙. 가장 일반적이고 고수준 | 가장 적게 변함 |
| **Use Cases** | 애플리케이션에 특화된 비즈니스 규칙. 엔티티로 드나드는 데이터 흐름을 조율 | 애플리케이션 동작이 바뀔 때 |
| **Interface Adapters** | Use Case ↔ 외부 세계 간 데이터 형식 변환. MVC의 Controller/Presenter, SQL 코드 등 | 외부 형식이 바뀔 때 |
| **Frameworks & Drivers** | DB, 웹 프레임워크, 디바이스 등 "세부사항(detail)" | 가장 자주 바뀜 |

### 3. 핵심 중의 핵심 — 의존성 규칙 (The Dependency Rule)

> **소스 코드의 의존성은 오직 안쪽을 향해야 한다. 안쪽 원은 바깥쪽 원에 대해 아무것도 알면 안 된다.**

- 바깥 원에 선언된 이름(클래스, 함수, 변수, 데이터 포맷 등)이 안쪽 원 코드에 등장해서는 **절대 안 된다.**
- 특히 바깥쪽에서 변하기 쉬운 것(예: DB 테이블 구조)이 안쪽 코드에 영향을 주면 안 된다.
- 즉, **고수준 정책(High-level policy)은 절대 저수준 정책(Low-level detail)에 의존하지 않는다.** 항상 그 반대.

이 규칙 덕분에 바깥쪽(DB, UI, 프레임워크)을 통째로 갈아끼워도 안쪽(비즈니스 규칙)은 멀쩡하다.

### 4. 경계를 넘는 법 — 의존성 역전 (Dependency Inversion)

여기서 자연스러운 의문이 생긴다.

> "Use Case가 DB에서 데이터를 가져와야 하는데, Use Case(안쪽)가 DB(바깥쪽)를 호출하면 의존성이 바깥을 향하는 것 아닌가?"

해결책은 **의존성 역전 원칙(DIP)** 이다. 호출의 흐름과 의존성의 방향을 분리한다.

```
   [잘못된 방향처럼 보이는 것]            [인터페이스로 역전시킨 것]

   Use Case ──호출?──▶ Database          Use Case
   (안쪽)              (바깥쪽)            │ 정의
                                          ▼
   ✗ 의존성이 바깥을 향함              ┌──────────────┐
                                       │ <interface>  │  ← 안쪽이 소유
                                       │ Repository   │
                                       └──────────────┘
                                          ▲ 구현
                                          │
                                       Database  (바깥쪽이 인터페이스를 구현)
                                       ✓ 의존성이 안쪽을 향함
```

- 안쪽 계층(Use Case)이 **인터페이스를 정의**하고, 바깥쪽 계층(DB 구현체)이 그 인터페이스를 **구현**한다.
- 실행 시점의 제어 흐름(control flow)은 안쪽 → 바깥으로 흐르지만, **소스 코드 의존성은 인터페이스를 통해 바깥 → 안쪽으로 역전**된다.
- 이 트릭으로 어떤 경계든 의존성 규칙을 지키면서 넘을 수 있다.

### 5. 경계를 넘는 데이터는 "단순"해야 한다

계층 경계를 가로질러 데이터를 주고받을 때는 **안쪽 계층에 편한, 단순하고 격리된 데이터 구조(DTO)** 만 넘긴다.

- ❌ 엔티티 객체나 DB의 Row 구조체를 그대로 넘기지 않는다.
- ❌ 의존성 규칙을 위반하는 형태(바깥쪽 타입에 대한 참조를 품은 데이터)도 안 된다.
- ✅ 그냥 단순한 데이터 구조 — 기본형 인자, DTO, 혹은 단순 맵 — 를 넘긴다.

> "우리는 항상 데이터가 안쪽 원에서 다루기 편한 형태로 경계를 넘게 하고 싶다."

## 내가 얻은 인사이트

### 설계 관점

1. **외울 건 단 하나, "의존성은 안쪽으로"**
   - 계층 4개, 동심원, 화살표 다 잊어도 된다. "변하기 쉬운 것이 변하기 어려운 것에 의존한다"는 한 문장만 지키면 나머지는 따라온다. 클린 아키텍처가 "쉬우면서도 쉽게 잊힌다"고 느껴지는 이유는, 다이어그램을 외우려 하기 때문이다. 규칙 하나만 잡으면 된다.

2. **"세부사항(detail)을 바깥으로 미루기"가 본질**
   - DB, 프레임워크, 웹은 전부 "detail"이다. 프로젝트 초반에 "DB를 뭘 쓸까"부터 정하는 건 가장 변하기 쉬운 것에 시스템을 묶어버리는 일. 비즈니스 규칙을 먼저 세우고 DB 결정을 최대한 미룰 수 있다면 설계가 건강하다는 신호다.

### 실무 적용 관점

3. **인터페이스 위치가 의존성 방향을 결정한다**
   - DIP의 실전 포인트는 "인터페이스를 누가 소유하는가"다. `Repository` 인터페이스를 도메인 계층(안쪽)에 두고, 구현체를 인프라 계층(바깥쪽)에 두는 패키지 구조만 잡아도 의존성 역전의 80%는 달성된다. 헥사고날 아키텍처의 Port & Adapter가 정확히 이걸 말한다.

4. **경계에서 DTO로 갈아끼우는 비용은 "보험료"**
   - 엔티티를 컨트롤러까지 그대로 흘려보내면 당장은 편하지만, DB 스키마 변경이 UI까지 전파되는 결합이 생긴다. 경계마다 단순 데이터 구조로 변환하는 약간의 보일러플레이트는, 나중에 한 계층을 통째로 교체할 자유를 사는 보험료다.

### 트레이드오프 관점

5. **모든 프로젝트에 4계층이 정답은 아니다**
   - Uncle Bob 본인도 "계층 수는 상황에 따라 다르다"고 명시했다. 작은 CRUD 앱에 4계층 + DTO 변환을 전부 도입하면 오버엔지니어링이다. 핵심은 "비즈니스 복잡도가 높고 오래 살아남을 코어가 있는가". 그럴 때만 격리 비용이 회수된다. 규칙(의존성 방향)은 항상 지키되, 계층 수와 격리 강도는 도메인 복잡도에 맞춰 조절하는 게 실용적이다.
