## AI 요약

### 1. "그래프 기반 데이터 정제"란?

데이터를 정제(cleaning)할 때 가장 흔한 문제 중 하나가 **중복 레코드**입니다.
그런데 현실의 중복은 깔끔하게 똑같지 않습니다.

```
ID   이름            이메일                 가입일
1    김보성          bs@coxwave.com         2024-01-10
2    김 보성         bs@coxwave.com         2024-03-22   ← 띄어쓰기만 다름
3    Kim Bosung      bs@coxwave.com         2024-05-01   ← 표기법이 다름
```

세 레코드는 사실 **같은 사람**이지만, 단순히 "이름이 정확히 같은 행을 지워라"는
규칙으로는 절대 못 잡습니다. 이런 걸 **퍼지 중복(fuzzy duplicate)** 이라고 합니다.

그래프 기반 정제의 핵심 아이디어는 이렇습니다:

> **레코드를 점(node)으로 보고, "비슷하다"는 관계를 선(edge)으로 연결한 뒤,
> 서로 이어진 덩어리(connected component)를 "같은 실체"로 본다.**

| 그래프 요소 | 데이터 정제에서의 의미 |
|------------|----------------------|
| 노드(Node / Vertex) | 개별 레코드 (사용자 1명, 상품 1개 …) |
| 엣지(Edge) | 두 레코드가 "충분히 비슷하다"는 판정 |
| 연결 요소(Connected Component) | 서로 이어진 레코드 묶음 = **같은 실체로 추정되는 그룹** |

### 2. 왜 "그래프"여야 하는가? — 이행적 관계 문제

가장 직관적인 이유는 **이행성(transitivity)** 입니다.

```
A ─(비슷함)─ B ─(비슷함)─ C
```

- A와 B는 비슷하다고 판정됨
- B와 C는 비슷하다고 판정됨
- 그런데 A와 C는 직접 비교하면 "안 비슷하다"고 나올 수 있음
  (예: 이름 표기가 너무 달라서)

쌍(pair) 단위로만 비교하면 A와 C를 다른 사람으로 처리해버립니다.
하지만 그래프로 보면 **A-B-C가 하나의 연결 덩어리**이므로
"이 셋은 전부 같은 사람"이라고 한 번에 묶을 수 있습니다.

이것이 그래프 방식이 단순 `GROUP BY`나 정확 일치 매칭보다 강력한 이유입니다.

### 3. 파이프라인 4단계

아티클은 Apache Spark(+ GraphFrames)로 다음 4단계 파이프라인을 보여줍니다.

```
 [원본 레코드]
      │
      ▼
┌─────────────────────────────┐
│ ① 유사도 계산 (Similarity)   │  레코드 쌍이 얼마나 비슷한지 점수화
│   - Levenshtein 거리 < 3     │  (이름 글자 차이 3 미만이면 "비슷")
└─────────────────────────────┘
      │
      ▼
┌─────────────────────────────┐
│ ② 그래프 구성 (Graph Build)  │  레코드 = 노드,  비슷한 쌍 = 엣지
│   vertices = 고유 ID들       │
│   edges    = 유사한 쌍       │
└─────────────────────────────┘
      │
      ▼
┌─────────────────────────────┐
│ ③ 연결 요소 탐색             │  서로 이어진 레코드를 한 그룹으로
│   connectedComponents()      │  (같은 component = 같은 실체)
└─────────────────────────────┘
      │
      ▼
┌─────────────────────────────┐
│ ④ 대표 레코드 선택 (Survivor)│  그룹마다 1개만 남기기
│   - 가장 최근 가입일 등 기준  │  (Window 함수로 순위 매겨 1등만)
└─────────────────────────────┘
      │
      ▼
 [정제된 레코드]
```

#### ① 유사도 계산 — Levenshtein 거리

문자열이 얼마나 다른지를 "몇 글자를 고쳐야 같아지는가"로 잰 것이 **Levenshtein 거리**입니다.

```
"김보성" → "김 보성"   : 1글자 삽입 → 거리 1   → 비슷함 ✅
"apple"  → "appla"      : 1글자 치환 → 거리 1   → 비슷함 ✅
"banana" → "orange"     : 거리 큼               → 안 비슷함 ❌
```

거리가 임계값(예: 3) 미만이면 "이 둘은 같은 실체 후보"로 보고 엣지를 만듭니다.

#### ② 그래프 구성 (Spark / GraphFrames 예시)

```python
from graphframes import GraphFrame

# 노드: 모든 레코드의 고유 ID
vertices = df.select("id").distinct()

# 엣지: 유사도 임계값을 넘은 레코드 쌍 (src, dst)
edges = similar_pairs.select("id_a", "id_b") \
                     .toDF("src", "dst")

g = GraphFrame(vertices, edges)
```

#### ③ 연결 요소 탐색

```python
components = g.connectedComponents()
# 결과: 각 레코드 id 옆에 'component' 라벨이 붙음
#   같은 component 값 = 같은 그룹(같은 실체)
```

```
id   component
1    100        ┐
2    100        ├─ 모두 같은 사람으로 묶임
3    100        ┘
4    205        ← 다른 사람
```

#### ④ 대표 레코드 선택 (survivorship)

그룹을 묶었으면 **하나만 남기고** 나머지는 버립니다.
어떤 걸 남길지(생존 규칙, survivorship rule)는 비즈니스가 정합니다.
아티클 예시는 "가장 최근 가입 레코드"를 남깁니다.

```python
from pyspark.sql import Window
from pyspark.sql.functions import row_number, col

w = Window.partitionBy("component").orderBy(col("signup_ts").desc())

deduped = components.withColumn("rn", row_number().over(w)) \
                    .filter(col("rn") == 1) \
                    .drop("rn")
```

### 4. 현실의 장벽: 비교 폭발과 Blocking

레코드 N개를 전부 쌍으로 비교하면 비교 횟수가 **N²** 으로 폭발합니다.
100만 건이면 약 5천억 번 비교 → 비현실적입니다.

그래서 실무에서는 **블로킹(Blocking)** 으로 먼저 후보를 좁힙니다.

```
[전체 100만 건]
   │  같은 '이메일 도메인' 또는 '이름 첫 2글자'끼리만 같은 블록에 넣기
   ▼
[작은 블록 여러 개]  ← 블록 안에서만 쌍 비교 → 비교량 급감
   │
   ▼
[유사 쌍 = 엣지] → 그래프 구성 → connected components …
```

블로킹은 "정확도(다 비교하면 좋음)"와 "속도(다 비교하면 느림)"의 트레이드오프를
조절하는 손잡이입니다.

### 5. 전체 그림 한 장 요약

```
원본 → [블로킹으로 후보 좁히기]
      → [유사도 계산: Levenshtein 등]
      → [그래프 구성: 노드=레코드, 엣지=유사쌍]
      → [connected components: 같은 실체 그룹핑]
      → [survivorship: 그룹당 대표 1개 선택]
      → 정제된 데이터
```

## 내가 얻은 인사이트

### 모델링 관점

1. **"같다"가 아니라 "이어져 있다"로 사고를 전환한 것이 핵심**
   - 중복 제거를 `DISTINCT` / `GROUP BY`로만 생각하면 정확 일치의 함정에 빠진다.
   - 그래프는 "직접 비슷하지 않아도 다리(B)를 거쳐 연결되면 같은 그룹"이라는
     이행적 관계를 자연스럽게 표현한다. 이것이 단순 매칭으로는 도달 못 하는 영역.

2. **정제 파이프라인은 결국 "후보 생성 → 매칭 → 그룹핑 → 생존자 선택"의 4박자**
   - 도구가 Spark든 pandas든 dedupe 라이브러리든, 이 골격은 동일하다.
   - 각 단계는 독립적으로 교체 가능하다 (유사도 함수만 바꿔도 되고, 그룹핑만 바꿔도 됨).

### 엔지니어링 관점

1. **진짜 난제는 알고리즘이 아니라 N² 비교 폭발이다**
   - connected components 자체는 잘 알려진 알고리즘. 실무 병목은 "엣지를 어떻게
     싸게 만드느냐"이고, 그 답이 블로킹/LSH(MinHash)다.
   - 즉, 그래프 방식의 성패는 **그래프를 만들기 전 단계(blocking)** 에서 갈린다.

2. **survivorship 규칙은 기술이 아니라 도메인 결정이다**
   - "최신 레코드를 남길까, 정보가 가장 많은 레코드를 남길까, 필드별로 골라 합칠까"는
     코드가 정해주지 않는다. 데이터 정제는 비즈니스 합의가 절반이다.

3. **임계값(threshold)은 정밀도/재현율 손잡이**
   - Levenshtein 임계값을 키우면 더 많이 묶이지만(재현율↑) 다른 실체를 잘못 합칠
     위험(정밀도↓)이 커진다. 정제 품질 = 이 손잡이를 데이터에 맞게 튜닝하는 일.

### 적용 관점

1. **이 패턴은 사용자/고객 마스터 데이터(MDM), 상품 카탈로그 통합, 그리고
     LLM 학습용 코퍼스 중복 제거까지 그대로 재사용된다.**
   - 최근 GraphRAG·LLM 데이터 큐레이션에서도 "비슷한 문서를 엣지로 잇고 덩어리를
     줄인다"는 동일한 그래프 사고가 쓰인다. 도메인만 다를 뿐 구조는 같다.
