## AI 요약

### 1. Amazon Redshift란?

Amazon Redshift는 **페타바이트 규모의 클라우드 데이터 웨어하우스**입니다. Massively Parallel Processing(MPP)과 Columnar Storage를 기반으로 대규모 분석 쿼리를 빠르게 처리합니다.

| 특성 | 설명 |
|-----|------|
| **유형** | 완전 관리형 클라우드 데이터 웨어하우스 |
| **아키텍처** | MPP (Massively Parallel Processing) |
| **스토리지** | Columnar Storage (열 기반 저장) |
| **확장성** | 페타바이트 규모까지 확장 가능 |
| **기반 기술** | PostgreSQL 파생 (SQL 호환) |

### 2. 클러스터 아키텍처

Redshift 클러스터는 **Leader Node**와 **Compute Node**로 구성됩니다.

```
                    ┌─────────────────────┐
                    │    Client/BI Tool   │
                    └──────────┬──────────┘
                               │
                               ▼
                    ┌─────────────────────┐
                    │    Leader Node      │
                    │  - Query Parsing    │
                    │  - Execution Plan   │
                    │  - Result Caching   │
                    └──────────┬──────────┘
                               │
          ┌────────────────────┼────────────────────┐
          │                    │                    │
          ▼                    ▼                    ▼
   ┌────────────┐       ┌────────────┐       ┌────────────┐
   │ Compute    │       │ Compute    │       │ Compute    │
   │ Node 1     │       │ Node 2     │       │ Node N     │
   │ ┌────┬────┐│       │ ┌────┬────┐│       │ ┌────┬────┐│
   │ │Slice│Slice││     │ │Slice│Slice││     │ │Slice│Slice││
   │ └────┴────┘│       │ └────┴────┘│       │ └────┴────┘│
   └────────────┘       └────────────┘       └────────────┘
```

#### Leader Node
| 기능 | 설명 |
|-----|------|
| **쿼리 파싱** | SQL 쿼리를 분석하고 쿼리 트리 생성 |
| **실행 계획** | 최적화된 실행 계획 수립 및 배포 |
| **결과 집계** | Compute Node 결과를 모아 클라이언트에 반환 |
| **캐싱** | 쿼리 결과 캐싱으로 반복 쿼리 가속화 |

#### Compute Node
| 기능 | 설명 |
|-----|------|
| **데이터 저장** | 테이블 데이터 저장 및 관리 |
| **쿼리 실행** | 할당된 쿼리 세그먼트 병렬 처리 |
| **슬라이스 구성** | 각 노드는 여러 슬라이스로 분할되어 병렬성 극대화 |

#### Node 유형
```
┌─────────────────────────────────────────────────────────────┐
│                      Node Types                              │
├──────────────────┬──────────────────┬───────────────────────┤
│   Dense Compute  │   Dense Storage  │         RA3           │
│     (DC2)        │      (DS2)       │   (최신, 권장)         │
├──────────────────┼──────────────────┼───────────────────────┤
│ SSD 기반         │ HDD 기반         │ Managed Storage       │
│ 속도 우선        │ 용량 우선        │ 컴퓨팅/스토리지 분리   │
│ ~500GB 이하      │ ~500GB 이상      │ 독립적 확장 가능       │
└──────────────────┴──────────────────┴───────────────────────┘
```

### 3. RA3 노드와 Managed Storage

RA3는 Redshift의 최신 노드 유형으로, **컴퓨팅과 스토리지를 분리**합니다.

```
┌─────────────────────────────────────────────────────────────┐
│                    RA3 Architecture                          │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│   ┌──────────────────┐                                       │
│   │  Compute Layer   │  ◀── 독립적 확장 가능                  │
│   │  (RA3 Nodes)     │                                       │
│   └────────┬─────────┘                                       │
│            │                                                 │
│            ▼                                                 │
│   ┌──────────────────────────────────────┐                   │
│   │      Redshift Managed Storage (RMS)  │                   │
│   │  ┌──────────────┐  ┌──────────────┐  │                   │
│   │  │  Local SSD   │  │   Amazon S3  │  │                   │
│   │  │  (Hot Data)  │  │  (Cold Data) │  │                   │
│   │  └──────────────┘  └──────────────┘  │                   │
│   └──────────────────────────────────────┘                   │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

#### RA3 노드 장점
| 장점 | 설명 |
|-----|------|
| **독립적 스케일링** | 컴퓨팅과 스토리지를 별도로 확장 |
| **자동 계층화** | Hot/Cold 데이터 자동 분류 |
| **Data Sharing** | 클러스터 간 데이터 공유 |
| **Cross-AZ 복구** | AZ 장애 시 자동 재배치 |

### 4. AQUA (Advanced Query Accelerator)

AQUA는 RA3 노드에서 사용 가능한 **하드웨어 가속 캐시**입니다.

```
              기존 아키텍처                    AQUA 아키텍처

┌────────────┐                        ┌────────────┐
│  Compute   │                        │  Compute   │
│   Node     │                        │   Node     │
└─────┬──────┘                        └─────┬──────┘
      │ 전체 데이터                          │ 필터링된 결과만
      │ 이동                                │ 이동
      ▼                                    ▼
┌────────────┐                        ┌────────────┐
│  Storage   │                        │   AQUA     │ ◀── 스토리지 계층에서
│            │                        │  (필터링,   │     스캔/집계 처리
└────────────┘                        │   집계)    │
                                      └─────┬──────┘
                                            │
                                      ┌─────▼──────┐
                                      │  Storage   │
                                      └────────────┘
```

- **성능**: 특정 쿼리에서 최대 10배 성능 향상
- **비용**: RA3 노드에 추가 비용 없이 포함
- **코드 변경**: 불필요 (자동 적용)

### 5. Columnar Storage (열 기반 저장)

Redshift는 데이터를 **행(row)** 대신 **열(column)** 단위로 저장합니다.

```
        Row-based Storage                 Columnar Storage
        (전통적 RDBMS)                    (Redshift)

     id  name   age  salary           id    name    age   salary
    ┌───┬──────┬───┬───────┐         ┌───┐ ┌──────┐ ┌───┐ ┌───────┐
    │ 1 │ Kim  │ 30│ 50000 │         │ 1 │ │ Kim  │ │ 30│ │ 50000 │
    ├───┼──────┼───┼───────┤         │ 2 │ │ Lee  │ │ 25│ │ 45000 │
    │ 2 │ Lee  │ 25│ 45000 │   →     │ 3 │ │ Park │ │ 35│ │ 60000 │
    ├───┼──────┼───┼───────┤         │...│ │ ... │ │...│ │  ...  │
    │ 3 │ Park │ 35│ 60000 │         └───┘ └──────┘ └───┘ └───────┘
    └───┴──────┴───┴───────┘

   SELECT SUM(salary)              → salary 열만 읽으면 됨
   WHERE age > 30                  → age, salary 열만 스캔
```

#### 장점
| 장점 | 설명 |
|-----|------|
| **I/O 감소** | 필요한 열만 읽어 디스크 I/O 최소화 |
| **압축 효율** | 동일 타입 데이터 압축률 3배 이상 |
| **벡터 처리** | CPU 벡터 연산 최적화 가능 |

### 6. Data Distribution Styles

데이터가 Compute Node에 분배되는 방식을 정의합니다.

```
┌─────────────────────────────────────────────────────────────┐
│                   Distribution Styles                        │
├─────────┬──────────────────────────────────────────────────┤
│  AUTO   │ Redshift가 자동으로 최적 스타일 선택 (기본값)       │
├─────────┼──────────────────────────────────────────────────┤
│  EVEN   │ 모든 슬라이스에 균등 분배 (라운드 로빈)             │
├─────────┼──────────────────────────────────────────────────┤
│  KEY    │ DISTKEY 컬럼 값 기준으로 동일 슬라이스에 배치        │
├─────────┼──────────────────────────────────────────────────┤
│  ALL    │ 모든 노드에 전체 테이블 복제                        │
└─────────┴──────────────────────────────────────────────────┘
```

#### KEY Distribution 예시
```sql
-- user_id가 같은 행들이 같은 슬라이스에 배치됨
CREATE TABLE orders (
    order_id INT,
    user_id INT,
    amount DECIMAL(10,2)
)
DISTKEY(user_id);
```

```
                    DISTKEY(user_id) 분배

     user_id=1,3,5                  user_id=2,4,6
          │                              │
          ▼                              ▼
    ┌──────────┐                   ┌──────────┐
    │  Slice 1 │                   │  Slice 2 │
    │ ┌──────┐ │                   │ ┌──────┐ │
    │ │uid=1 │ │                   │ │uid=2 │ │
    │ │uid=3 │ │                   │ │uid=4 │ │
    │ │uid=5 │ │                   │ │uid=6 │ │
    │ └──────┘ │                   │ └──────┘ │
    └──────────┘                   └──────────┘
```

**장점**: 같은 user_id 기준 JOIN 시 네트워크 이동 없이 로컬 처리

### 7. Sort Keys

데이터의 **물리적 정렬 순서**를 정의하여 Zone Map 최적화를 가능하게 합니다.

#### Sort Key 유형
```
┌───────────────────────────────────────────────────────────────┐
│                      Sort Key Types                            │
├───────────────┬───────────────────────────────────────────────┤
│   Compound    │ 첫 번째 컬럼 기준으로 계층적 정렬                │
│               │ ORDER BY col1, col2, col3                      │
│               │ 첫 번째 컬럼 필터 시 효과적                      │
├───────────────┼───────────────────────────────────────────────┤
│  Interleaved  │ 모든 컬럼에 동등한 가중치                        │
│               │ 다양한 컬럼 조합 필터 시 효과적                  │
├───────────────┼───────────────────────────────────────────────┤
│     AUTO      │ ML 기반 자동 최적화                             │
│               │ 워크로드 패턴 학습 후 적용                       │
└───────────────┴───────────────────────────────────────────────┘
```

#### Zone Map 동작 원리
```
                       Zone Map

     Block 1: date 2024-01-01 ~ 2024-01-31
     Block 2: date 2024-02-01 ~ 2024-02-28
     Block 3: date 2024-03-01 ~ 2024-03-31
     ...

     SELECT * FROM events WHERE date = '2024-02-15'
                    │
                    ▼
     Block 1 → SKIP (min/max 범위 밖)
     Block 2 → SCAN (범위 내)
     Block 3 → SKIP (min/max 범위 밖)
```

### 8. MPP (Massively Parallel Processing)

```
                    쿼리 분산 처리

                 ┌─────────────────┐
                 │   Leader Node   │
                 │  Query Plan:    │
                 │  SUM(amount)    │
                 │  GROUP BY region│
                 └────────┬────────┘
                          │ 분할
         ┌────────────────┼────────────────┐
         │                │                │
         ▼                ▼                ▼
    ┌─────────┐      ┌─────────┐      ┌─────────┐
    │ Node 1  │      │ Node 2  │      │ Node 3  │
    │ Region: │      │ Region: │      │ Region: │
    │ Asia    │      │ Europe  │      │ America │
    │ SUM=100 │      │ SUM=200 │      │ SUM=150 │
    └────┬────┘      └────┬────┘      └────┬────┘
         │                │                │
         └────────────────┼────────────────┘
                          │ 집계
                 ┌────────▼────────┐
                 │   Final Result  │
                 │  Asia: 100      │
                 │  Europe: 200    │
                 │  America: 150   │
                 └─────────────────┘
```

**Shared Nothing Architecture**: 각 노드가 독립적인 CPU, 메모리, 디스크를 보유하며 병렬 처리

### 14. Serverless vs Provisioned

| 항목 | Provisioned | Serverless |
|-----|-------------|------------|
| **노드 관리** | 직접 선택 및 관리 | 자동 |
| **스케일링** | 수동 또는 반자동 | 완전 자동 |
| **비용 모델** | 시간당 과금 | RPU(쿼리 처리량) 기반 |

## 내가 얻은 인사이트

### 아키텍처 설계 관점

1. **컴퓨팅-스토리지 분리의 진화**
   - 초기: DC2/DS2에서 컴퓨팅과 스토리지가 결합
   - 현재: RA3에서 완전 분리 → 독립적 스케일링
   - 이는 클라우드 네이티브 데이터 플랫폼의 필수 트렌드

2. **Columnar Storage의 힘**
   - OLAP 워크로드에서 행 기반 대비 압도적 성능
   - 압축률 향상 + I/O 감소 + 벡터 처리 최적화
   - 열 선택도(selectivity)가 높을수록 효과 극대화

3. **Zone Map은 저비용 최적화**
   - 메타데이터만으로 블록 스킵 가능
   - Sort Key 설정이 Zone Map 효과를 결정
   - 시계열 데이터에서 특히 효과적

### 실무 적용 관점

4. **Distribution Style 선택 전략**
   ```
   테이블 크기 작음 (<1M rows) → ALL
   JOIN 없는 테이블           → EVEN
   자주 JOIN되는 대형 테이블   → KEY (JOIN 키 기준)
   불확실할 때               → AUTO
   ```

5. **Sort Key는 쿼리 패턴 기반으로**
   - 단일 컬럼 필터 위주 → Compound (해당 컬럼 첫 번째)
   - 다양한 컬럼 조합 → Interleaved
   - 시계열 데이터 → 날짜/시간 컬럼을 Sort Key로

6. **RA3 마이그레이션 고려사항**
   - DC2/DS2에서 RA3로 마이그레이션 시 스토리지 비용 재계산 필요
   - Data Sharing이 필요하면 RA3 필수
   - AQUA 활용으로 추가 비용 없이 성능 향상

### 성능 최적화 포인트

7. **네트워크 트래픽 최소화**
   - DISTKEY를 JOIN 키와 일치시켜 로컬 JOIN 유도
   - 작은 dimension 테이블은 ALL 분배로 브로드캐스트 JOIN 회피
   - 슬라이스 간 데이터 이동(redistribution)이 가장 비용이 큼

8. **압축과 VACUUM**
   - ENCODE AUTO 사용으로 자동 최적화 위임
   - DELETE/UPDATE 후 VACUUM 필수 (고스트 레코드 정리)
   - ANALYZE로 통계 정보 최신화

9. **Concurrency Scaling 비용 주의**
   - 무료 크레딧(24시간/일) 초과 시 추가 과금
   - 예측 가능한 피크는 클러스터 확장이 더 경제적일 수 있음

### PostgreSQL과의 차이점

10. **Redshift ≠ PostgreSQL**
    - VACUUM, ANALYZE 수동 실행 필요
    - 인덱스 없음 (Sort Key + Zone Map으로 대체)
    - UPSERT 미지원 → Staging + DELETE + INSERT 패턴
    - 단일 행 연산 성능 낮음 (OLAP 최적화)

### 비용 최적화

11. **비용 절감 전략**
    - Reserved Instance로 60-75% 절감 가능
    - Pause/Resume으로 개발/테스트 환경 비용 절감
    - Spectrum으로 콜드 데이터 S3에 저장 (스토리지 비용 감소)
    - Serverless는 간헐적 워크로드에 유리
