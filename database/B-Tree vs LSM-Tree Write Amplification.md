## AI 요약

### 3가지 핵심 성능 지표

데이터 구조의 성능을 평가하는 세 가지 증폭(Amplification) 지표:

| 지표 | 정의 | 예시 |
|------|------|------|
| **Write Amplification** | DB에 쓴 데이터 대비 실제 디스크에 쓴 양 | 10MB 쓰기 → 30MB 기록 = 3x |
| **Read Amplification** | 쿼리당 필요한 디스크 읽기 횟수 | 1개 키 조회에 5번 읽기 = 5x |
| **Space Amplification** | 실제 데이터 대비 디스크 사용량 | 10MB 데이터 → 100MB 사용 = 10x |

### 핵심 트레이드오프 원칙

```
┌─────────────────────────────────────────────────────┐
│  "하나의 자료구조는 세 가지 증폭 중                    │
│   최대 두 가지만 최적화할 수 있다"                     │
└─────────────────────────────────────────────────────┘

Read ◄─────────► Write
       \     /
        \   /
         \ /
        Space
```

| 구조 | 최적화 대상 | 희생 |
|------|------------|------|
| **B-Tree** | Read, Space | Write |
| **LSM-Tree (Leveled)** | Write, Space | Read |
| **LSM-Tree (Size-Tiered)** | Write | Read, Space |

### Write Amplification 상세 비교

#### B+ Tree

```
쓰기 증폭 = Θ(B)    [B = 블록/페이지 크기, 보통 4~16KB]
```

**왜 이런가?**
```
1개 키 업데이트
    ↓
해당 키가 있는 리프 페이지 전체를 다시 써야 함
    ↓
1바이트 수정해도 4KB~16KB 페이지 전체 쓰기

추가 비용:
- WAL (Write-Ahead Log) 기록
- 페이지 분할(split) 시 추가 쓰기
```

**특징**: 데이터셋 크기와 거의 무관 (페이지 크기에 의존)

---

#### LSM-Tree (Leveled Compaction)

```
쓰기 증폭 = Θ(k × log_k(N/B))

k = 레벨 간 크기 비율 (보통 10)
N = 전체 데이터 크기
B = 블록 크기
```

**왜 이런가?**
```
Level 0:  [  10MB  ]
Level 1:  [  100MB  ]         ← Level 0의 10배
Level 2:  [  1GB    ]         ← Level 1의 10배
Level 3:  [  10GB   ]         ← Level 2의 10배

데이터가 각 레벨을 통과할 때마다:
- 다음 레벨과 병합(Compaction)
- 평균적으로 k/2번 재병합됨

총 쓰기 = 레벨 수 × 레벨당 재병합 횟수
        = log_k(N/B) × k
        ≈ 10 × 4 = 40x (1TB 데이터 기준)
```

**특징**: 데이터셋 크기가 커지면 레벨이 늘어나 증폭도 증가

---

### 수치 비교 예시

**가정**: 페이지 크기 4KB, 데이터 1TB, k=10

| 구조 | 쓰기 증폭 계산 | 결과 |
|------|---------------|------|
| **B+ Tree** | 4KB 페이지 + WAL | ~10x |
| **LSM-Tree (Leveled)** | 10 × log₁₀(1TB/4KB) ≈ 10 × 4 | ~40x |
| **LSM-Tree (Size-Tiered)** | 더 적은 Compaction | ~10x |

---

### Read Amplification 비교

| 구조 | 읽기 증폭 | 설명 |
|------|----------|------|
| **B+ Tree** | O(log_B(N/B)) | 트리 깊이만큼 (보통 3-4회) |
| **LSM-Tree** | O(레벨 수 × 레벨당 탐색) | 각 레벨 SSTable 검색 필요 |

```
B+ Tree:      1개 키 조회 → 3~4회 디스크 읽기
LSM-Tree:     1개 키 조회 → Bloom Filter + 여러 SSTable 검색
              (Bloom Filter가 대부분 걸러주지만 최악의 경우 많음)
```

---

### Space Amplification 비교

| 구조 | 공간 증폭 | 원인 |
|------|----------|------|
| **B+ Tree** | ~1.5x | 페이지 단편화, 여유 공간 |
| **LSM-Tree (Leveled)** | ~1.1x | Compaction이 정리 |
| **LSM-Tree (Size-Tiered)** | ~2x+ | 같은 키의 여러 버전 공존 |

---

### 왜 LSM이 쓰기에 유리한가? (진짜 이유)

쓰기 증폭 숫자만 보면 LSM이 더 클 수 있습니다. 하지만:

```
실제 쓰기 처리량 = 1 / (쓰기 증폭 × I/O 시간)

B+ Tree:   10x 증폭 × 랜덤 I/O (10ms)  = 느림
LSM-Tree:  40x 증폭 × 순차 I/O (0.1ms) = 빠름
```

**순차 I/O가 랜덤 I/O보다 100배 빠르기 때문에**, 증폭이 4배 커도 처리량은 LSM이 25배 높을 수 있습니다.

```
┌────────────────────────────────────────┐
│  LSM의 진짜 장점:                       │
│  "높은 쓰기 증폭을 순차 I/O로 상쇄"      │
└────────────────────────────────────────┘
```

### 언제 무엇을 선택할까?

| 워크로드 | 추천 | 이유 |
|----------|------|------|
| **쓰기 중심** | LSM-Tree | 순차 I/O로 높은 처리량 |
| **읽기 중심** | B+ Tree | 예측 가능한 낮은 읽기 지연 |
| **혼합 (OLTP)** | 상황에 따라 | 캐시 전략이 중요 |
| **SSD 수명 중요** | B+ Tree | 쓰기 증폭이 낮음 |

## 내가 얻은 인사이트

"쓰기 증폭이 낮다 = 쓰기가 빠르다"는 **틀린 공식**입니다.

LSM-Tree는 쓰기 증폭이 B-Tree보다 높을 수 있지만, **순차 I/O**라는 무기로 이를 상쇄합니다. 반면 B-Tree는 쓰기 증폭은 낮지만 **랜덤 I/O**로 인해 실제 처리량이 낮습니다.

결국 중요한 것은:
- 쓰기 처리량이 중요하면 → LSM (순차 I/O)
- 읽기 지연이 중요하면 → B-Tree (단일 트리 탐색)
- SSD 수명이 중요하면 → B-Tree (낮은 쓰기 증폭)

TiKV가 LSM을 선택한 이유도 명확합니다: **"읽기 성능은 캐시로 개선할 수 있지만, 쓰기 성능은 구조적 한계가 있다"**. 캐시로 읽기를 커버하고, 구조적으로 쓰기를 최적화하는 전략입니다.
