## AI 요약

### 1. 왜 패스워드 해싱인가?

패스워드를 DB에 저장할 때 **암호화(Encryption)가 아닌 해싱(Hashing)** 을 사용해야 한다. 암호화는 복호화 키가 있으면 원문 복원이 가능하지만, 해싱은 단방향 함수(one-way function)이므로 원래 패스워드를 역으로 알아낼 수 없다.

```
[평문 비교 - 위험]
  DB: password = "mysecret123"
  → DB 유출 시 모든 패스워드 노출

[단순 해싱 - 취약]
  DB: hash = SHA256("mysecret123")
  → Rainbow Table 공격에 취약

[Salted Slow Hashing - 권장]
  DB: hash = Argon2id(salt + "mysecret123", params)
  → 사전 계산 공격 불가 + 무차별 대입 비용 극대화
```

| 저장 방식 | 역추적 가능 | Rainbow Table | 무차별 대입 내성 |
|-----------|:----------:|:------------:|:--------------:|
| 평문 저장 | O | - | 없음 |
| AES 암호화 | O (키 탈취 시) | - | 없음 |
| MD5/SHA-256 | X | **취약** | 없음 |
| bcrypt/scrypt | X | X | **있음** |
| **Argon2id** | X | X | **최고** |

### 2. 핵심 방어 메커니즘

#### Salt (솔트)
패스워드마다 고유한 랜덤 문자열을 추가하여 동일한 패스워드라도 서로 다른 해시를 생성한다.

```
사용자 A: Argon2id(salt_A + "password123") → hash_A
사용자 B: Argon2id(salt_B + "password123") → hash_B

hash_A ≠ hash_B  (같은 패스워드여도 해시가 다름)
```

- 최소 128비트 크기의 CSPRNG(암호학적 난수 생성기) 사용
- Argon2id, bcrypt, PBKDF2는 **자동으로 솔트를 처리**

#### Pepper (페퍼)
솔트와 달리 **모든 패스워드에 공유**되는 비밀 값으로, DB와 **분리 보관** (HSM, Vault 등)한다.

```
                ┌──────────────┐
                │  Secrets Vault │
                │  pepper = "X"  │
                └──────┬───────┘
                       │
  ┌────────────────────▼─────────────────────┐
  │  hash = Argon2id(salt + password + pepper) │
  └────────────────────┬─────────────────────┘
                       │
                ┌──────▼───────┐
                │   Database    │
                │  salt + hash  │
                └──────────────┘
```

**전략 두 가지:**
1. **Pre-hashing**: 해싱 전에 페퍼를 패스워드에 추가
2. **Post-hashing (HMAC)**: 해싱 후 HMAC 적용 → 더 안전 (권장)

#### Work Factor (작업 인수)
해싱 연산 비용을 의도적으로 높여 공격자의 무차별 대입 비용을 극대화한다. 로그인 시 **1초 이내** 완료를 목표로 튜닝한다.

### 3. 알고리즘 비교

```
┌─────────────────────────────────────────────────────┐
│              패스워드 해싱 알고리즘 선택 흐름도         │
├─────────────────────────────────────────────────────┤
│                                                     │
│  새 시스템인가? ──Yes──→ Argon2id 사용 (최우선)       │
│       │                                             │
│      No                                             │
│       │                                             │
│  FIPS-140 필수? ──Yes──→ PBKDF2 사용                │
│       │                                             │
│      No                                             │
│       │                                             │
│  Argon2id 지원? ──Yes──→ Argon2id로 마이그레이션     │
│       │                                             │
│      No                                             │
│       │                                             │
│  scrypt 지원? ──Yes──→ scrypt 사용                   │
│       │                                             │
│      No                                             │
│       │                                             │
│  bcrypt 유지 (work factor 12 이상)                   │
│                                                     │
└─────────────────────────────────────────────────────┘
```

#### Argon2id (2015, Password Hashing Competition 우승)

| 파라미터 | 설명 | 권장 값 |
|---------|------|--------|
| memory (m) | 메모리 사용량 | 최소 19 MiB, 프로덕션 64-128 MiB |
| iterations (t) | 반복 횟수 | 1~3 |
| parallelism (p) | 병렬 스레드 | 1 |
| salt | 솔트 크기 | 128비트 이상 |

- **Argon2d**: 데이터 의존적 메모리 접근 → GPU 공격에 강하나 사이드 채널 취약
- **Argon2i**: 데이터 독립적 접근 → 사이드 채널 내성
- **Argon2id**: 하이브리드 → **두 장점을 결합 (권장)**
- 내부 압축 함수 G: 1024바이트 입력 2개 → 1024바이트 출력, Blake2b 기반

#### bcrypt (1999, Blowfish 기반)

| 파라미터 | 설명 | 권장 값 |
|---------|------|--------|
| cost | 2^cost 반복 | 12 이상 (매년 증가) |
| password | 입력 제한 | **최대 72바이트** |
| salt | 솔트 크기 | 128비트 |

```
⚠️ bcrypt 72바이트 제한 주의:
  긴 패스워드 → SHA-384로 Pre-hash 후 Base64 인코딩 → bcrypt 적용
  (단순 해싱 시 Password Shucking 공격 가능)
```

- Eksblowfish 키 스케줄을 사용하여 의도적으로 느린 키 파생
- 메모리 사용량은 약 4KB로 매우 적음 → GPU 공격에 상대적으로 취약

#### scrypt (2009, Tarsnap에서 개발)

| 파라미터 | 설명 | 권장 값 |
|---------|------|--------|
| N | CPU/메모리 비용 | 2^17 (131,072) |
| r | 블록 크기 | 8 |
| p | 병렬화 | 1 |
| 메모리 | N × r × 128 bytes | ~128 MB |

- 순차적 메모리 하드 알고리즘: 연산 중 전체 메모리를 유지해야 함
- 메모리-시간 트레이드오프(TMTO) 공격에 일부 취약성 존재

#### PBKDF2 (2000, NIST 권장)

| 해시 함수 | 최소 반복 횟수 (OWASP 2025) |
|-----------|:-------------------------:|
| HMAC-SHA-1 | 1,300,000 |
| HMAC-SHA-256 | 600,000 |
| HMAC-SHA-512 | 210,000 |

- FIPS-140 인증 환경에서 유일한 선택지
- 메모리 하드가 아님 → GPU 병렬화에 취약

### 5. 마이그레이션 전략

#### 레거시 해시 업그레이드

**Lazy Upgrade (점진적 업그레이드):**
사용자가 로그인에 성공할 때 현재 해시가 구버전이면 새 알고리즘으로 재해싱한다.

```python
def login(username, password):
    user = db.get_user(username)

    if verify_hash(password, user.hash):
        # 로그인 성공 → 해시 버전 확인
        if is_outdated(user.hash):
            user.hash = argon2id.hash(password, new_params)
            db.update(user)
        return True
    return False
```

**Layered Hashing (래핑):**
기존 해시를 새 알고리즘으로 감싸서 패스워드 리셋 없이 보안을 강화한다.

```
기존: md5(password)
개선: bcrypt(md5(password))
  → 사용자 재인증 전까지 이중 해시로 보호
```

#### 마이그레이션 긴급도

| 현재 상태 | 긴급도 | 조치 기한 |
|-----------|--------|----------|
| 평문 / MD5 / SHA-1 | **CRITICAL** | 즉시 |
| SHA-256 (단순 해싱) | HIGH | 90일 이내 |
| PBKDF2 <100k 반복 | HIGH | 90일 이내 |
| bcrypt cost <10 | HIGH | 90일 이내 |
| bcrypt cost 12+ | LOW | 계획적 업그레이드 |
| Argon2id | GOOD | 연간 파라미터 리뷰 |

### 6. 구현 시 주의사항

**반드시 지켜야 할 사항:**
1. **상수 시간 비교 (Constant-Time Comparison)**: 라이브러리의 검증 함수 사용, 직접 문자열 비교 금지 → 타이밍 공격 방지
2. **패스워드 길이 제한**: 64~128자 상한선 → DoS 방지 (긴 입력의 해싱 비용)
3. **Unicode 전체 범위 수용**: 이모지, NULL 바이트 포함 → 해싱 전 엔트로피 축소 금지
4. **PHC 문자열 포맷 사용**: 알고리즘·파라미터·솔트·해시를 하나의 문자열에 저장

```
PHC 문자열 형식 예시:
$argon2id$v=19$m=65536,t=3,p=1$c29tZXNhbHQ$hash_output_here
 │         │    │              │             │
 알고리즘  버전  파라미터       솔트(Base64)   해시(Base64)
```

**흔한 실수:**
- MD5/SHA-256 같은 빠른 해시를 패스워드에 사용
- 여러 패스워드에 동일 솔트 재사용
- 해싱이 아닌 암호화로 패스워드 저장
- bcrypt 72바이트 제한을 무시하고 긴 패스워드 입력
- 현재 하드웨어 대비 너무 낮은 work factor 설정

## 내가 얻은 인사이트

### 보안 설계 관점
1. **"느린 것이 안전하다"는 역설적 설계 원칙**
   - 일반적으로 소프트웨어에서 빠른 것이 좋지만, 패스워드 해싱은 의도적으로 느려야 한다. 정상 사용자는 로그인을 1회 수행하지만, 공격자는 수십억 번 시도해야 하므로 200~500ms의 해싱 시간이 공격 비용을 기하급수적으로 증가시킨다.

2. **Memory-Hard의 본질은 "경제학적 방어"**
   - Argon2id가 64~128MB 메모리를 요구하는 이유는, GPU/ASIC에서 연산 코어는 저렴하게 병렬화할 수 있지만 각 코어에 독립적인 대용량 메모리를 할당하는 것은 비용이 급격히 증가하기 때문이다. 즉, 공격을 기술적으로 불가능하게 만드는 것이 아니라 **경제적으로 불가능하게** 만드는 전략이다.

3. **Pepper는 "심층 방어(Defense-in-Depth)"의 좋은 사례**
   - DB가 유출되어도 페퍼가 별도 저장소에 있으면 해시를 크래킹할 수 없다. 단, 페퍼가 유출되면 모든 패스워드 리셋이 필요하므로 HSM/Vault 관리가 중요하다.

### 운영/마이그레이션 관점
1. **Lazy Upgrade 패턴의 실용성**
   - 사용자에게 강제 패스워드 리셋을 요구하지 않으면서도 로그인 시점에 점진적으로 해시를 업그레이드할 수 있다. 다만 오랫동안 로그인하지 않는 사용자의 해시는 구버전으로 남으므로, 일정 기간 후 미업그레이드 계정에 대해 리셋을 강제하는 정책이 필요하다.

2. **Work Factor는 "연간 리뷰" 대상**
   - 하드웨어 성능은 매년 향상되므로, bcrypt cost나 Argon2id 파라미터를 최소 연 1회 재평가해야 한다. 이를 위해 해시에 알고리즘과 파라미터 메타데이터를 함께 저장하는 PHC 포맷이 중요하다.

### 아키텍처 관점
1. **동시 접속 부하에서의 메모리 트레이드오프**
   - Argon2id가 64MB를 사용하면 1,000명 동시 로그인 시 64GB 메모리가 필요하다. 서비스 규모에 따라 메모리 파라미터와 동시성 제한(rate limiting)을 함께 설계해야 한다. 이는 보안과 가용성 사이의 트레이드오프다.

2. **FIPS-140 규정 준수 환경의 현실적 선택**
   - 금융·정부 시스템에서는 FIPS-140 인증이 필수이므로 PBKDF2를 사용해야 할 수 있다. 이 경우 Dual-Hash 전략(PBKDF2 + Argon2id 이중 저장)이 규정 준수와 보안을 동시에 만족시키는 방법이다.
