# Classless Inter-Domain Routing (CIDR)

## 출처

* **링크**: [https://en.wikipedia.org/wiki/Classless_Inter-Domain_Routing](https://en.wikipedia.org/wiki/Classless_Inter-Domain_Routing)

---

## AI 요약

**CIDR (Classless Inter-Domain Routing)** 는
기존의 **클래스 기반 IP 주소 체계(Class A/B/C)** 를 대체하기 위해 도입된
**IP 주소 할당 및 라우팅 표준**이다.

기존 방식의 문제는 두 가지였다.

1. **IP 주소 낭비**

   * Class A/B/C는 네트워크 크기가 고정
   * 실제 필요한 주소 수와 맞지 않아 대량의 주소가 낭비됨
2. **라우팅 테이블 폭증**

   * 네트워크 단위가 잘게 쪼개지면서
   * 인터넷 백본 라우터의 라우팅 테이블이 급격히 커짐

CIDR는 이를 해결하기 위해
👉 **IP 주소를 “클래스”가 아닌 “프리픽스 길이(prefix length)”로 표현**한다.

### 핵심 표기법

```
IP주소/프리픽스길이
예: 192.168.0.0/24
```

* `/24` → 앞 24비트가 네트워크 부분
* 나머지 8비트 → 호스트 주소
* 총 주소 개수 = 2^(32 - 프리픽스 길이)

### CIDR의 핵심 개념

* **가변 길이 네트워크 마스크 (VLSM)**
* **주소 블록 단위 할당**
* **라우트 집계(Route Aggregation, Supernetting)**

이를 통해

* IP 주소 공간 사용 효율 ↑
* 인터넷 라우팅 테이블 크기 ↓
* 대규모 네트워크 확장 가능

CIDR는 현재 IPv4, IPv6 모두에서
**사실상의 표준 주소 표현 방식**으로 사용되고 있다.

---

### 1. CIDR는 “주소 표기법”이 아니라 **인터넷 확장 전략**

CIDR는 단순히 `/24` 같은 표기법이 아니라,

> **“인터넷이 계속 커지도록 만들기 위한 구조적 타협”**

이었다.

* 주소 부족 문제 (IPv4 exhaustion)
* 라우터 성능 한계
* 관리 복잡성

이 세 가지를 **동시에 완화**한 설계라는 점이 중요하다.

---

### 2. 네트워크 설계 단위가 “호스트” → “블록”으로 바뀌었다

CIDR 이전:

* 사고 단위: “이 IP는 어느 클래스인가?”

CIDR 이후:

* 사고 단위: **“이 주소 블록을 어디까지 묶을 수 있는가?”**

이건

* 클라우드 VPC 설계
* 서브넷 분할
* 보안 그룹 / 방화벽 정책

전부에 그대로 이어진다.

👉 **CIDR 감각이 없으면 네트워크를 ‘조합’하지 못한다.**

---

### 3. Route Aggregation은 분산 시스템 사고와 닮았다

CIDR의 핵심 미덕 중 하나인 **라우트 집계**는

> 여러 개의 세부 경로를
> 하나의 상위 경로로 “요약”하는 것

이건 개념적으로:

* 분산 DB의 shard 요약
* 로그 aggregation
* 메트릭 roll-up

과 거의 같은 사고방식이다.

👉 **CIDR는 네트워크판 “압축 & 추상화”다.**

---

### 4. CIDR를 이해하면 클라우드 네트워크가 읽힌다

AWS / GCP / Azure에서 나오는:

* VPC CIDR
* Subnet CIDR
* Peering 충돌
* IP overlap 문제

이건 전부

> “CIDR 블록 간의 포함/겹침 관계”

문제다.

CIDR가 손에 잡히면:

* 왜 이 VPC는 Peering이 안 되는지
* 왜 이 서브넷은 못 만드는지

설명 없이도 **직관적으로 보인다**.

---

### 한 줄 정리

> **CIDR는 “IP 주소를 효율적으로 쓰기 위한 기술”이 아니라
> “인터넷이 무너지지 않게 만든 설계적 진화”다.**
