## AI 요약

### 1. IPv6란 무엇이었어야 했나?

IPv6는 단순히 "주소 공간이 더 큰 IPv4"가 아니라, 1990년대 설계자들이 꿈꿨던 **네트워크 계층의 근본적 재설계**였다.

| 구분 | 초기 이상 (Vision) | 현실 배포 (Reality) |
|------|-----|-----|
| Layer 2 | 완전 제거 | 브리징·SDN으로 더 비대해짐 |
| MAC 주소 | 폐기 | 영구 존속 |
| 브로드캐스트 | 멀티캐스트로 대체 | ARP/Neighbor Discovery 공존 |
| DHCP | 불필요 | 계속 사용 |
| NAT | 소멸 | CGNAT로 강화 |
| 라우팅 | 순수 L3 | L2/L3 혼합 |

핵심 명제는 하나다.

> **"Layers are only ever added, never removed."**
> 계층은 추가될 뿐, 제거되지 않는다.

### 2. 왜 MAC 주소가 만들어졌나 - 버스 네트워크의 죄

```
[초기: Point-to-Point]
  Host A ─── Host B
  (주소 불필요, 선이 곧 주소)

[비용 절감: Bus Network]
  ┌────────┬────────┬────────┐
  │ Host A │ Host B │ Host C │
  └────────┴────────┴────────┘
  (공유 매체 → 누가 누구에게? → MAC 주소 필요)
```

- **Arcnet**: 8비트 주소, 수동 설정 (중복 시 장애)
- **Ethernet**: 48비트 MAC, 제조사가 자동 할당 → **계층 구조 없음**
- 계층이 없으니 **브리지 테이블은 모든 MAC을 flat하게 저장**해야 함

### 3. IPv4 + Ethernet의 뒤엉킴

운영자가 실제로 원하는 라우팅 표현:

```
send packets to 10.1.1.1 via the router at MAC 11:22:33:44:55:66
```

하지만 현실의 라우팅 테이블:

```
send packets to 10.1.1.1 via the router at IP 192.168.1.1
  → (그런데 192.168.1.1의 MAC이 뭔지 ARP로 물어봐야 함)
```

IP로 라우터를 지정하는 척하지만, 결국 **MAC을 우회적으로 지정**하는 이중 구조. 이를 메꾸기 위해:
- **ARP** (브로드캐스트 폭풍 유발)
- **DHCP** (IP 할당)
- **Bridging** (가상 버스 재창조)

### 4. IPv6의 원래 설계도

```
┌─────────────────────────────────────────────┐
│  IPv6 Vision (1990s)                         │
├─────────────────────────────────────────────┤
│                                              │
│   모든 장치 = IPv6 라우터                      │
│   모든 링크 = Point-to-Point                  │
│                                              │
│   [Host] ──/64──[Router]──/64──[Host]        │
│                                              │
│   • Layer 2 인터네트워킹 없음                  │
│   • MAC 주소 없음                             │
│   • ARP 없음                                 │
│   • 브로드캐스트 없음                          │
│   • 헤더 단순화 → HW 가속 가능                 │
│                                              │
└─────────────────────────────────────────────┘
```

이 세상이었다면 라우팅은 투명하고, 모든 서브넷이 `/64`로 충분하며, 하드웨어 가속도 쉬웠을 것이다.

### 5. 모바일 IP - 설계 실패의 원점

IPv6 설계 당시 **"이동하면서 연결이 유지되는 디바이스"** 시나리오가 고려되지 않았다.

#### 4-tuple의 계층 위반

TCP/UDP 세션 식별자:
```
(source IP, source port, destination IP, destination port)
```

이 한 줄에 **Layer 3 정보(IP)**와 **Layer 4 정보(port)**가 섞여 있다.

```
[시나리오: 와이파이 → LTE 전환]

1. X:1111 ──────────► Y:80   (세션 수립)
                              4-tuple = (X, 1111, Y, 80)

2. X의 IP가 Q로 변경 (로밍)

3. Q:1111 ──────────► Y:80
                              4-tuple = (Q, 1111, Y, 80)
                              ↓
                              Y는 이것을 "새 연결"로 인식
                              → 기존 세션 파괴
```

#### 현재의 우회책

- LTE/WiFi 이동성: **거대한 Layer 2 브리징 + 중앙 터널링**
- 즉, "IPv6로 Layer 2를 없애자"가 정반대로 "Layer 2를 지구 규모로 확장"하게 됨
- 대가: **심각한 지연 시간**

### 6. 저자가 제안하는 근본 해법

포트 번호를 확장하여 **UUID 기반 세션 식별자**로 대체:

```
기존:
  (source IP, source port: 16bit, dest IP, dest port: 16bit)

제안:
  (source IP, session UUID: 128~256bit, dest IP, dest port)
                   ↑
  Layer 4 식별은 IP와 독립
  IP가 바뀌어도 UUID가 같으면 같은 세션
  destination port는 "새 세션 시작" 때만 필요
```

**보안 고려**: IP 변경 시 연결 탈취 방지를 위해 TCP SYN-ACK 유사한 **쿠키 교환**으로 MITM 여부를 검증해야 함.

### 7. QUIC - 우회적 해법의 가능성

TCP/UDP를 수정하는 것은 수십 년 걸리는 배포 문제로 현실적으로 불가능.

```
┌──────────────────────────────────────┐
│  Application                          │
├──────────────────────────────────────┤
│  QUIC (Connection ID 기반 세션 식별) │ ← IP 변경에 무관
├──────────────────────────────────────┤
│  UDP                                  │
├──────────────────────────────────────┤
│  IP                                   │
└──────────────────────────────────────┘
```

- QUIC은 이미 **Connection ID**라는 고유 세션 키 보유
- IP가 바뀌어도 같은 Connection ID면 같은 세션
- 이론상 **투명한 로밍** 가능
- **MinimaLT** 역시 같은 문제를 우아하게 해결한 프로토콜로 언급됨

### 8. WiFi 주소 지정의 과잉 복잡성

이더넷에서 넘어가면 더 심해진다.

| 모드 | 주소 필드 수 | 용도 |
|------|---|---|
| 3-address | 3개 | 일반 AP 통신 (to-AP, from-AP) |
| 4-address | 4개 | AP 간 리피터/중계 |
| 6-address | 6개 | Mesh 네트워크 |

저자의 솔직한 고백:

> "and that's about where I gave up trying to understand"

### 9. 현대 데이터센터의 아이러니

SDN(Software-Defined Networking)은 데이터센터를 **거대한 가상 버스 네트워크**로 되돌렸다.

```
물리적 설계:      스타 토폴로지 (Point-to-Point)
논리적 운영:      가상 버스 (Layer 2 over Layer 3)
                 ↑
             IPv6가 없애려 했던 바로 그것
```

저자의 평가:

> **"It is, in short, a mess."**

### 10. 최종 조건 - "진짜" 우아해지려면

저자가 가정하는 이상향 조건:

1. 모든 UDP/TCP 트래픽 제거
2. QUIC 혹은 유사 프로토콜로 완전 이주
3. Layer 2 브리징 제거

> "And then the Internet would be elegant again."

하지만 "the first half was the easy part; the long tail is much harder."

## 내가 얻은 인사이트

### 아키텍처 설계자 관점

1. **"계층은 추가될 뿐 제거되지 않는다"는 보편 법칙**
   - 이 명제는 네트워크에만 적용되지 않는다. 인증(OAuth → SAML → OIDC), DB 접근 계층(ORM → DAO → Repository), 프론트엔드 상태관리 등 모든 장기 운영 시스템에서 반복된다.
   - **시스템 설계 시 "이걸 제거할 수 있는 조건"을 문서화해 두지 않으면, 그 계층은 영원히 남는다.**
   - Deprecation 전략은 기능 추가만큼 중요하다.

2. **"레거시 호환성"은 기술이 아니라 경제학**
   - IPv6가 30년이 지나도 IPv4를 대체하지 못한 이유는 기술적 한계가 아닌 **배포 관성(deployment inertia)**.
   - 신기술이 100% 이주하지 못하면 **구기술은 영원히 1급 시민**이다.
   - 이 관점에서 "점진적 마이그레이션" 전략은 대부분 실패한다는 걸 인정하고 설계해야 한다.

### 프로토콜 설계 관점

3. **계층 간 정보 혼입(Layer Violation)의 장기 비용**
   - TCP의 4-tuple이 L3와 L4를 섞은 탓에 **모바일 시대 전체의 이동성 문제**가 파생되었다.
   - 초기 설계에서 "이 식별자가 어느 계층에 속하는가?"를 엄격히 분리하는 것이 수십 년의 부채를 막는다.
   - **세션 식별은 네트워크 주소와 독립되어야 한다** - 이것이 QUIC Connection ID의 본질적 우월성.

4. **Mobile-First 설계의 역사적 공백**
   - 1990년대 네트워크 설계자들에게 "내가 길을 걸으며 영상통화를 한다"는 시나리오는 SF였다.
   - **현재 설계하는 시스템도 10년 뒤 명백해질 시나리오를 놓치고 있을 것**이다.
   - AI 에이전트 간 통신, 엣지 디바이스 메시 등이 그 후보일지도 모른다.

### 실무 적용 관점

5. **QUIC을 쓰는 진짜 이유**
   - QUIC을 "HTTP/3를 위한 빠른 프로토콜"로만 이해하는 것은 수박 겉핥기다.
   - QUIC의 본질은 **IP 주소에서 세션 식별을 분리**한 것 - 이것이 모바일, 멀티패스, 제로-RTT 로밍의 토대다.
   - 서비스를 글로벌·모바일 환경에서 안정화하려면 **L7 라운드로빈이 아니라 Connection ID 기반 라우팅**을 고려해야 한다.

6. **SDN이 Layer 2를 되살린 역설**
   - IPv6가 꿈꿨던 "Layer 2 없는 세상"과 정반대로, 현대 클라우드는 VXLAN·Geneve·Tailscale·ZeroTier로 **지구 규모의 가상 L2**를 만들어냈다.
   - 이상적인 설계가 현실의 제약(기존 앱 호환성)을 만나면 **정반대의 배포 패턴**이 등장한다.
   - 설계자는 "나의 이상"이 운영 현장에서 어떻게 비틀릴지 상상할 수 있어야 한다.

### 엔지니어링 철학

7. **"우아한 재설계"의 한계**
   - IPv6 사례는 **"greenfield 수준의 재설계는 brownfield 환경에 배포되는 순간 미학을 잃는다"**는 냉혹한 교훈이다.
   - 완벽한 설계보다, **기존 시스템의 옆에 공존 가능한 점진적 설계**가 살아남는다.
   - TCP의 후계자가 QUIC인 이유 - IP 위가 아닌 **UDP 위에 얹어** 미들박스 호환성을 확보했기 때문.

8. **부채를 받아들이는 성숙함**
   - 저자는 IPv6를 비판하면서도 "그래서 IPv4로 돌아가자"고 말하지 않는다.
   - **현실은 항상 이상보다 지저분하며**, 그 지저분함을 이해하고 그 위에서 작동하는 코드를 짜는 것이 엔지니어의 일이다.
   - "It is, in short, a mess" - 이 한 문장이 모든 레거시 시스템 운영자의 위로다.
