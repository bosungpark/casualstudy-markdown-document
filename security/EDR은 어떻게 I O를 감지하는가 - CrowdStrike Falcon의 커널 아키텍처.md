# EDR은 어떻게 I/O를 감지하는가 — CrowdStrike Falcon의 커널 아키텍처

- **저자/출처**: CrowdStrike Engineering & Technology Blog
- **링크**: https://www.crowdstrike.com/en-us/blog/tech-analysis-kernel-access-security-architecture/

## AI 요약

### 1. 출발점 — EDR은 I/O를 "훔쳐보지" 않는다

EDR(Endpoint Detection & Response)이 파일·프로세스·네트워크·레지스트리 활동을 어떻게 아느냐는 질문의 답은 의외로 단순하다. **몰래 엿보는 게 아니라, OS가 어차피 지나가야 하는 경로 위에 콜백을 등록해 두고 통보받는다.**

흔한 오해와 실제를 대조하면 이렇다.

| 흔한 오해 | 실제 |
|-----------|------|
| 네트워크 패킷을 스니핑한다 | 커널 네트워크 스택의 필터 계층에 붙어 연결 이벤트를 통보받는다 |
| 디스크를 주기적으로 스캔한다 | 파일시스템 필터가 모든 create/read/write를 **발생 시점에** 가로챈다 |
| 프로세스 목록을 폴링한다 | 프로세스 생성 콜백이 **생성되는 순간** 커널에서 호출된다 |
| 시그니처 DB로 파일을 대조한다 | 행위(behavior) 연쇄를 보고 판단, 판정의 무게중심이 클라우드에 있다 |

즉 EDR의 관찰은 **폴링이 아니라 인터포지션(interposition)**이다. 이벤트를 놓칠 수 없고, 지연이 0에 가깝고, 원한다면 그 자리에서 **차단**까지 할 수 있는 이유가 여기서 나온다.

그리고 이 인터포지션을 하려면 커널에 있어야 한다 — 이것이 아티클 전체의 논지다.

### 2. 왜 하필 커널인가 — 세 가지 이유

아티클은 "보안 제품, 특히 엔드포인트 보호 제품은 **최고 수준의 가시성(visibility)·강제력(enforcement)·변조 저항성(tamper-resistance)**을 제공하기 위해 커널 접근을 필요로 한다"고 못박는다. 근거는 셋이다.

```
┌──────────────────────────────────────────────────────────────┐
│ ① 부팅 초기 보호 (Early Boot)                                │
│    부트킷/루트킷은 유저모드 프로세스보다 먼저 뜬다.           │
│    → 나중에 뜨는 감시자는 이미 속은 상태에서 시작한다.        │
│    → Microsoft가 ELAM(Early Launch Anti-Malware)을           │
│      Windows 8.1에 "바로 이런 감시를 위해" 만들어 둔 이유.    │
├──────────────────────────────────────────────────────────────┤
│ ② 성능 (Performance)                                         │
│    "네트워킹·파일시스템 스택의 극도로 고성능인 경로에서는,    │
│     현재 고객 워크로드로 확장 가능한 유일한 메커니즘이        │
│     커널 드라이버뿐이다."                                     │
│    예: paging I/O, DirectStorage의 BypassIO                   │
├──────────────────────────────────────────────────────────────┤
│ ③ 변조 저항성 (Tamper Resistance)                            │
│    PPL 이전엔 "권한 상승된 사용자가 보안 제품의 동작을        │
│    손쉽게 방해할 full permission을 가졌다."                   │
│    유저모드에만 있으면 → ETW 소스 조작, 서비스 강제 종료.     │
└──────────────────────────────────────────────────────────────┘
```

②가 특히 중요하다. 파일 I/O 훅은 **모든 파일 접근마다** 호출되는 경로다. 여기서 유저모드로 컨텍스트 스위치를 하면 그 비용이 시스템 전체 I/O에 곱해진다. "커널이 아니면 안 된다"의 상당 부분은 이념이 아니라 산술이다.

### 3. 실제로 어떤 훅을 쓰는가 (Windows)

여기가 이 질문의 핵심이다. 아티클은 CrowdStrike가 **침습적인 커널 후킹(SSDT 패치 같은)이 아니라 Microsoft가 문서화한 공식 확장점**을 쓴다고 명시한다. 그리고 "이 메커니즘들이 **모든 보안 벤더가 따르는 표준화·통일된 프로그래밍 모델의 기반**을 세웠다"고 덧붙인다.

원문이 나열한 확장점과, 그것이 잡아내는 I/O를 매핑하면:

| 아티클이 명시한 메커니즘 | 감지하는 I/O | 대응 API (보충) |
|---|---|---|
| **Filter Manager** | 파일 create/read/write/close 전체 | 미니필터 드라이버, `FltRegisterFilter` |
| **Registry filtering** | 레지스트리 키/값 생성·수정·삭제 | `CmRegisterCallbackEx` |
| **Process manager callbacks** | 프로세스 생성/종료 | `PsSetCreateProcessNotifyRoutineEx` |
| **Thread notification callbacks** (2세대) | 스레드 생성/종료 (인젝션 단서) | `PsSetCreateThreadNotifyRoutineEx` |
| **Image signature verification callback** | 모듈/DLL 로드 + 서명 검증 | `PsSetLoadImageNotifyRoutine` |
| **Object callbacks** | **핸들 오픈 시도** | `ObRegisterCallbacks` |
| **Named pipe filtering** | 파이프를 통한 프로세스 간 통신 | (NPFS 필터) |
| **Kernel mode file copy** (Win11 22H2) | 커널 내 파일 복사 경로 | — |

`ObRegisterCallbacks`(object callbacks)가 왜 EDR에서 유독 중요한지 짚어둘 만하다. **파일을 읽는 것도, 프로세스 메모리를 읽는 것도 결국 "핸들을 여는" 행위**다. LSASS 프로세스에 `PROCESS_VM_READ` 핸들을 여는 시도 — 즉 자격증명 덤프의 첫 동작 — 는 파일 I/O도 네트워크 I/O도 아니지만, 오브젝트 콜백에는 걸린다. I/O를 "파일/네트워크"로만 생각하면 놓치는 축이다.

**흐름으로 보면:**

```
  악성 프로세스가 파일을 연다
            │
            ▼
   ┌────────────────┐
   │   유저모드      │   CreateFile(...)
   └────────┬───────┘
            │  syscall 경계
════════════▼═══════════════════════════════════════
   ┌────────────────┐
   │  커널: I/O Mgr │
   └────────┬───────┘
            ▼
   ┌────────────────────────────────┐
   │  Filter Manager                 │
   │   ├─ [Falcon 미니필터] ◄── pre-op 훅에서 통보/차단
   │   └─ 다른 필터들...             │
   └────────┬────────────────────────┘
            ▼
   ┌────────────────┐
   │  파일시스템     │  NTFS
   └────────────────┘
```

미니필터는 **pre-operation / post-operation** 두 지점에 붙는다. pre에서 이벤트를 보고, 필요하면 **STATUS_ACCESS_DENIED를 반환해 요청 자체를 죽인다.** "감지"와 "차단"이 같은 훅에서 나오는 게 EDR과 단순 로거의 차이다.

### 4. 커널에 다 넣지는 않는다 — 하이브리드 분할

아티클이 강조하는 반전은, CrowdStrike가 **커널 근본주의자가 아니라는** 점이다. 위험하고 복잡한 로직은 적극적으로 유저모드로 밀어낸다.

```
┌─────────────────── 커널모드 ───────────────────┐
│  · 파일/프로세스/네트워크/레지스트리 감시       │
│  · 부팅 초기 보호 (ELAM)                        │
│  · 변조 저항 강제                               │
│    → "얇고, 빠르고, 놓치지 않는" 역할만          │
└────────────────────────┬───────────────────────┘
                         │ 이벤트 스트림
┌────────────────────────▼───────────────────────┐
│                   유저모드                      │
│  · NGAV 머신러닝 엔진 (2017 Spring~)            │
│    → PPL + AppContainer 샌드박스 안에서 실행     │
│    → 업계 최초의 서드파티 PPL 샌드박스           │
│  · AMSI: 스크립트/문서 분석                     │
│  · Secure ETW 채널 (Windows 10~)                │
└─────────────────────────────────────────────────┘
```

머신러닝 모델 추론처럼 **복잡하고 공격 표면이 넓은 코드를 커널에 두는 것은 미친 짓**이다. 그래서 그건 PPL + AppContainer 샌드박스로 격리해 유저모드에 두고, **최소 권한 원칙(principle of least privilege)**을 적용해 "많은 Falcon 샌드박스가 full SYSTEM 권한을 가질 필요를 없앴다"고 아티클은 말한다.

정리하면 설계 원칙은 이렇다: **커널에는 "관찰 지점"만 두고, "판단"은 최대한 밖으로 뺀다.**

### 5. Rapid Response Content — 드라이버가 아니라 데이터다

2024년 7월 사태 때문에 이 대목이 아티클에 들어가 있다. 핵심 주장 한 줄:

> "rapid response 채널 파일은 **실제 드라이버 같은 실행 가능한 기계어가 아니며, 설정 데이터와 이미 하드코딩된 규칙에 대한 입력만 담는다.**"

```
  커널 드라이버 (실행 코드)          채널 파일 (데이터)
  ─────────────────────────         ──────────────────
  · WHQL 서명                        · 설정/입력값
  · 자주 안 바뀜                      · 하루에도 여러 번 갱신
  · 배포 무겁다                       · 배포 가볍다
           │                                  │
           └────────► [Content Interpreter] ◄─┘
                       드라이버 안의 인터프리터가
                       채널 파일을 "해석"해 규칙을 적용
```

이 구조의 동기는 명확하다. **새 위협 하나 대응할 때마다 커널 드라이버를 재서명·재배포할 수는 없다.** 그래서 코드(느리게, 엄격하게 배포)와 데이터(빠르게 배포)를 분리한다.

동시에 이것이 2024년 사고의 구조적 원인이기도 하다. **"데이터"라는 이유로 검증 기준이 코드보다 느슨했는데, 그 데이터를 해석하는 인터프리터는 커널에 있었다.** 커널에서 도는 인터프리터에게 잘못된 입력을 주면, 그건 데이터가 아니라 코드다. 신뢰 경계는 "코드냐 데이터냐"가 아니라 **"누가 그걸 파싱하고, 어디서 파싱하느냐"**로 그어져야 했다.

### 6. 다른 OS에서는 이미 커널을 떠났다

아티클은 Windows 외의 플랫폼에서는 CrowdStrike가 **이미 유저모드 메커니즘을 채택했다**고 밝힌다.

| OS | 메커니즘 | 커널 모듈 필요? | 안전장치 |
|---|---|---|---|
| **Windows** | 커널모드 드라이버 + 유저모드 서비스 | 예 | WHQL, HVCI, MVI |
| **Linux** | **BPF** (eBPF) | **아니오** | eBPF verifier가 로드 전 검증 |
| **macOS** | **Endpoint Security Framework** | 아니오 (kext 불필요) | Apple이 제공하는 공식 API |

Linux의 eBPF가 특히 우아한 답이다. **유저 공간에서 eBPF 프로그램을 커널에 로드**하지만, 커널의 verifier가 로드 시점에 그 프로그램이 무한루프에 빠지지 않고 메모리를 벗어나지 않음을 **증명**한 뒤에만 실행을 허용한다. 커널의 관찰력을 얻으면서 커널을 패닉시킬 여지는 크게 줄인다. — 4장의 하이브리드 분할과 같은 사상을, 이번엔 훅 지점 자체에 적용한 셈이다.

Windows에 대해서도 방향은 같다고 아티클은 말한다:

> "Microsoft가 유저모드에서 변조 증거를 남기는(tamper-evident) 소비·가시성·강제 기술을 계속 만들어 감에 따라, Falcon 센서도 그런 능력을 계속 활용해 나갈 것이다."

즉 커널은 **목적이 아니라, 현재 Windows에서 필요한 가시성을 얻는 유일한 수단**이라는 입장이다. 대안이 생기면 나간다.

### 7. 커널에 있으려면 치러야 하는 값

커널 드라이버를 배포한다는 건 검증 부담을 진다는 뜻이다.

| 요구사항 | 내용 |
|---|---|
| **WHQL** | Windows Hardware Quality Labs 인증. 단순 attestation 서명보다 "훨씬 엄격(far more rigorous)" |
| **HLK / HCK** | Hardware Lab Kit / Hardware Certification Kit 테스트 통과 후 결과 제출 |
| **HVCI 호환** | 하이퍼바이저 강제 코드 무결성. "커널 메모리 페이지는 코드 무결성 검사를 통과한 뒤에만 실행 가능해지고, **실행 가능한 페이지는 결코 쓰기 가능하지 않다**" (W^X) |
| **MVI 정회원** | Microsoft Virus Initiative. WHQL 위에 추가 요구사항이 얹힌다 |

여기서 미묘한 지점이 하나 있다. **이 모든 검증은 "드라이버 바이너리"에 걸린다.** 5장의 채널 파일은 이 관문을 통과하지 않는다 — 그게 빠른 배포의 이유이자, 리스크가 새어 나온 구멍이다.

## 내가 얻은 인사이트

### 아키텍처 관점

1. **관찰의 위치가 관찰의 성질을 결정한다**
   - 애플리케이션 계층 관찰(프록시, SDK, 에이전트 주입)은 **대상의 협조**를 전제한다. 대상이 설정을 안 따르거나, 우회 경로로 나가면 그냥 안 보인다.
   - 커널 훅은 협조가 필요 없다. 프로세스가 파일을 열려면 **반드시** 커널을 통과해야 하기 때문이다. "우회할 수 없는 좁은 길목"에 서는 것 — 이게 커널 계측의 본질이고, 애플리케이션 계층 계측이 구조적으로 가질 수 없는 성질이다.
   - 일반화하면: **어떤 계층에서 관찰하느냐가 "완전성(completeness)"을 결정한다.** 관찰 지점이 대상보다 아래에 있어야 완전해진다.

2. **얇은 커널, 두꺼운 유저모드 — 특권 코드는 최소 표면으로**
   - 4장의 분할이 이 문서에서 가장 실무적으로 재사용 가능한 원칙이다. 특권 위치(커널, root, DB superuser, 프로덕션 자격증명을 쥔 서비스)에는 **"놓치면 안 되는 최소한의 것"만** 두고, 복잡한 판단은 권한을 벗겨낸 곳으로 밀어낸다.
   - ML 추론을 커널에서 돌리지 않고 PPL+AppContainer 샌드박스로 뺀 결정이 정확히 그것이다. **복잡도와 특권은 반비례하게 배치해야 한다.**

3. **인터포지션은 감지와 차단을 통합한다**
   - 폴링 기반 모니터링은 본질적으로 사후적이다(이미 벌어진 일을 나중에 본다). pre-operation 훅은 **아직 벌어지지 않은 일**을 본다. 그래서 같은 코드가 탐지도 하고 차단도 한다.
   - 설계할 때 "로깅이냐 차단이냐"를 별도 시스템으로 나누기 전에, **훅 지점을 옳게 잡으면 둘이 같은 지점에서 나온다**는 걸 기억할 것.

### 배포/운영 관점

4. **"데이터"라는 이름표는 신뢰 경계를 만들어주지 않는다** ★
   - 이 문서에서 가장 값비싼 교훈. 채널 파일은 "실행 코드가 아니라 설정 데이터"였고, 그래서 코드보다 가벼운 검증 경로를 탔다. 그런데 그 데이터를 해석하는 인터프리터는 **링 0에 있었다.**
   - **커널 인터프리터에 입력되는 데이터는, 실질적으로 커널 코드다.** 튜링 완전하지 않더라도 크래시를 유발할 수 있으면 신뢰 경계 안쪽이다.
   - 실무 일반화: 설정 파일, 피처 플래그, 규칙 DSL, 정규식 — "코드가 아니니까"라며 배포 게이트를 낮춘 모든 것에 같은 질문을 해야 한다. **"이걸 파싱하는 코드가 죽으면 무엇이 같이 죽는가?"** 그 답이 커널/프로세스 전체면, 그건 데이터가 아니라 코드로 취급해야 한다.

5. **빠른 배포 경로는 그 자체로 위험 표면이다**
   - 코드/데이터 분리는 옳은 최적화다. 하루에도 여러 번 위협 대응을 내보내야 하는데 드라이버를 재서명할 수는 없으니까. 문제는 분리 자체가 아니라, **빠른 경로에 느린 경로만큼의 검증을 붙이지 않은 것**이다.
   - 원칙: **배포 속도를 높이려고 만든 경로에는, 속도에 걸맞은 자동 검증(스키마 검증, 카나리, 단계적 롤아웃)이 반드시 따라붙어야 한다.** 사람이 리뷰하지 않을 것이므로 기계가 리뷰해야 한다.

### 트레이드오프 관점

6. **eBPF는 "커널의 힘 vs 커널의 위험" 딜레마의 실질적 해답이다**
   - Linux 쪽 접근이 구조적으로 더 낫다는 걸 인정해야 한다. 커널 안에서 실행하되, **로드 시점에 안전성을 증명**받는다. verifier가 "이 프로그램은 종료하고, 메모리를 벗어나지 않는다"를 검증한 뒤에만 실행된다.
   - 이건 "신뢰하되 검증하라"가 아니라 **"검증되기 전엔 실행조차 시키지 않는다"**다. 커널 확장이라는 위험한 힘을, 표현력을 의도적으로 제한함으로써 안전하게 만들었다.
   - 확장성(extensibility)을 제공하는 모든 시스템 — 플러그인, 웹훅, 사용자 정의 규칙 — 이 참고할 패턴이다. **완전한 표현력을 주고 격리로 막을 것인가, 표현력을 깎고 정적 검증을 통과시킬 것인가.** eBPF는 후자를 택했고, 그래서 커널 안에 있어도 된다.

7. **커널 접근은 "능력"이 아니라 "부채"로 회계해야 한다**
   - 아티클의 어조 자체가 시사적이다. CrowdStrike는 커널을 자랑하지 않는다. **"현재 Windows에서 이 가시성을 얻는 유일한 방법이라서 쓴다"**고 방어적으로 설명하고, macOS/Linux에서는 이미 나갔으며, Windows에서도 대안이 생기면 나가겠다고 말한다.
   - 특권 접근은 얻는 순간부터 **유지 비용(WHQL, HVCI, HLK, MVI)과 사고 시 폭발 반경**을 지불하기 시작한다. 그래서 질문은 "커널 접근으로 뭘 할 수 있나"가 아니라 **"이걸 커널 밖에서 할 방법이 정말 없나"**가 되어야 한다. 없을 때만 들어가고, 생기면 나온다.
