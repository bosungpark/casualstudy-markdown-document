## AI 요약

### 1. 난독화(Obfuscation)란?

난독화는 **프로그램의 동작(semantics)은 그대로 보존하면서, 사람이 읽고 이해하기는 어렵게 만드는 의미-보존 변환(semantics-preserving transformation)**이다.

핵심은 "암호화"와 다르다는 점이다. 암호화는 키 없이는 실행조차 불가능하지만, 난독화된 코드는 **그대로 실행 가능**하다. 단지 그 코드를 정적/동적으로 분석해 원본 로직을 복원하려는 비용(reverse engineering cost)을 끌어올릴 뿐이다.

| 구분 | 목적 | 실행 가능성 | 되돌릴 수 있나? |
|------|------|-------------|------------------|
| 암호화 | 기밀성 (키 없으면 못 봄) | 복호화 전엔 실행 불가 | 키 있으면 100% 복원 |
| 난독화 | 분석 비용 증가 | 항상 실행 가능 | 이론상 가능, 비용만 비쌈 |
| 압축/패킹 | 크기 축소 + 부수적 은닉 | 언패킹 후 실행 | 언패킹하면 복원 |

데스크탑 앱이 난독화를 특히 필요로 하는 이유:
- 바이너리/바이트코드가 **사용자 기기에 통째로 배포**된다 (서버처럼 숨길 수 없음).
- .NET(IL), Java(bytecode), Electron(JS)처럼 **디컴파일이 쉬운 런타임**이 많다.
- 라이선스 검증, 크립토 키, 안티치트, 영업 비밀 로직이 클라이언트에 노출된다.

**난독화는 은탄환이 아니다.** 충분한 시간과 동기를 가진 공격자는 결국 뚫는다. 목표는 "불가능"이 아니라 "충분히 비싸게" 만드는 것이다.

### 2. 난독화의 4개 계층

난독화 기법은 무엇을 변형하느냐에 따라 네 계층으로 나뉜다.

```
┌─────────────────────────────────────────────┐
│ ① Layout    이름·메타데이터  →  무의미하게   │  비용 ↓ / 효과 ↓
├─────────────────────────────────────────────┤
│ ② Data      문자열·상수·자료구조 → 인코딩    │
├─────────────────────────────────────────────┤
│ ③ Control   실행 흐름(CFG)    →  뒤섞기       │
├─────────────────────────────────────────────┤
│ ④ Preventive 분석 도구 자체를 방해(anti-*)   │  비용 ↑ / 효과 ↑
└─────────────────────────────────────────────┘
```

---

### 3. ① Layout 난독화 — 이름 지우기

가장 싸고 가장 흔한 기법. **식별자(클래스/메서드/변수명)를 의미 없는 토큰으로 치환**하고, 디버그 심볼·주석 같은 부가 정보를 제거한다.

```
// 원본
class CreditCardValidator {
    boolean validateCreditCard(String number) { ... }
}

// 난독화 후
class a {
    boolean m1(String s) { ... }
}
```

- **원리**: 코드의 의미는 "이름"이 많이 담고 있다. `validateCreditCard` → `m1`이 되는 순간, 공격자는 이 함수가 무슨 일을 하는지 함수 본문 전체를 읽어야만 알 수 있다.
- **데스크탑 맥락**: .NET/Java 디컴파일러는 식별자를 그대로 복원하므로 효과가 크다. C/C++ 네이티브는 컴파일 시 심볼이 어차피 사라져 효과가 제한적(stripped binary).
- **트레이드오프**: 런타임 오버헤드 거의 0. 단, 크래시 스택트레이스가 `a.m1`처럼 나와 **디버깅이 어려워짐** → 원본↔난독화 매핑 파일을 따로 보관해야 함.

---

### 4. ② Data 난독화 — 상수와 문자열 숨기기

평문 문자열(`"https://api.example.com/license"`, `"INSERT INTO ..."`, 에러 메시지)은 리버서가 가장 먼저 찾는 단서다. `strings` 명령 한 번이면 코드의 지도가 드러난다.

**문자열 암호화(string encryption)**의 원리:
1. 빌드 타임에 문자열을 암호화/인코딩해 바이너리에 저장.
2. 런타임에 **실제로 쓰이는 순간** 복호화 루틴을 호출해 평문을 만든다.

```
// 원본
String url = "https://api.example.com/license";

// 난독화 후 (개념)
String url = decrypt(new byte[]{0x3f,0xa1,0x09,...}, key);
```

- **원리**: 정적 분석(파일을 실행 없이 훑기)으로는 평문이 안 보인다. 평문을 얻으려면 복호화 루틴을 찾아 동적으로 실행하거나 메모리를 덤프해야 한다 → 비용 증가.
- **확장**: 숫자 상수도 산술 항등식으로 분해(`100` → `(x*7+30)` 형태), 자료구조를 평탄화/병합하는 변형도 데이터 난독화에 속한다.
- **트레이드오프**: 복호화가 런타임마다 일어나므로 **성능 오버헤드**가 생긴다. 그래서 "모든 문자열"이 아니라 민감한 것만 선택적으로 적용한다.

### 5. ③ Control Flow 난독화 — 흐름을 뒤엉키게

가장 강력하고 가장 비싼 계층. **프로그램의 제어 흐름 그래프(CFG)를 알아보기 힘들게 재구성**한다. 핵심 기법 두 가지를 본다.

#### 5-1. 제어 흐름 평탄화 (Control Flow Flattening)

원본의 `if`/`while`/`for` 같은 **구조적 흐름을 전부 없애고**, 하나의 거대한 디스패처(switch 루프)로 바꾼다. "다음에 실행할 블록 번호"를 담는 상태 변수 `next` 하나가 모든 흐름을 지배한다.

```
// 원본: 흐름이 코드 구조에 그대로 보임
int fac(int n) {
    int r = 1;
    while (n > 1) { r *= n; n--; }
    return r;
}

// 평탄화 후: 모든 블록이 평등하게 나열되고, next가 순서를 지배
int fac(int n) {
    int r, next = 0;
    while (1) {
        switch (next) {
            case 0: r = 1;       next = 1; break;
            case 1: if (n > 1) next = 2; else next = 3; break;
            case 2: r *= n; n--;  next = 1; break;
            case 3: return r;
        }
    }
}
```

```
원본 CFG               평탄화된 CFG
  [A]                    ┌────── dispatch(next) ──────┐
   │                     │   │      │      │      │   │
  [B]<┐                 [A]  [B]   [C]   [D]   ...    │
  /  \ │                 └────┴──────┴──────┴─────────┘
[C]  [D]│                (어느 블록이 어디로 가는지 한눈에 안 보임)
  \__/
```

- **원리**: 사람과 디컴파일러는 "코드의 모양 = 흐름"이라는 가정에 의존한다. 평탄화는 그 모양을 부숴 모든 블록을 한 층(flat)에 늘어놓는다. 블록 A 다음이 B인지 D인지는 오직 런타임 `next` 값을 추적해야만 알 수 있다.
- **디스패처 변형**: switch뿐 아니라 goto 점프 테이블(`goto *jumpTab[next]`), 함수 포인터 호출, 심지어 블록마다 스레드를 띄우는 동시(concurrent) 디스패처까지 가능 — 분석을 더 어렵게.

#### 5-2. 불투명 술어 (Opaque Predicate)

**결과가 사실상 항상 참(또는 항상 거짓)인데, 정적 분석으로는 그것을 증명하기 어려운 조건문**을 끼워 넣는다.

```
// (x^2 + x)는 항상 짝수 → 아래 조건은 항상 거짓.
// 하지만 분석기는 "정말 항상 거짓인가?"를 쉽게 단정 못 함.
if ((x*x + x) % 2 != 0) {
    <죽은 코드 / 가짜 분기 — 절대 실행 안 됨>
}
```

- **원리**: 리버서/심볼릭 실행 엔진에게 **가짜 경로(dead branch)**를 진짜처럼 보이게 해 분석 공간을 폭발시킨다. 평탄화의 `next` 계산을 불투명 술어로 감싸면, 상태 전이를 추론하기가 훨씬 어려워진다.
- **재료**: 수학적 항등식, 포인터 별칭(alias) 관계처럼 정적 분석이 결정하기 어려운(undecidable에 가까운) 성질을 이용한다.

#### 5-3. 보조 기법
- **죽은 코드/정크 코드 삽입**: 실행 결과에 영향 없는 명령을 끼워 노이즈를 키움.
- **명령어 치환(instruction substitution)**: `a + b`를 `a - (-b)`, XOR 조합 등 동등하지만 복잡한 형태로.
- **함수 인라이닝/아웃라이닝**: 함수 경계를 흩어 "이 로직은 한 함수"라는 단서를 지움.

**트레이드오프**: 분기·반복 횟수가 늘어 **성능과 바이너리 크기에 가장 큰 부담**. 핫패스(자주 도는 루프)에 무겁게 걸면 앱이 눈에 띄게 느려진다.

### 6. ④ Preventive(Anti-*) — 분석 도구 자체를 방해

코드를 바꾸는 게 아니라, **분석 환경을 적대적으로** 만든다.

| 기법 | 막으려는 것 | 동작 원리 |
|------|-------------|-----------|
| Anti-debugging | 디버거 부착(динамическ 분석) | `ptrace`/`IsDebuggerPresent` 등으로 디버거 탐지 → 종료·교란 |
| Anti-tampering | 바이너리 변조(크랙) | 자기 코드의 체크섬/해시 검증, 위변조 시 실행 거부 |
| Anti-VM/sandbox | 자동 분석 환경 | 가상화 흔적 탐지 시 행동 변경 |
| RASP | 런타임 공격 전반 | 앱 안에 보호 로직을 심어 실행 중 무결성 감시 |
| White-box crypto | 키 추출 | 키와 암호 알고리즘을 룩업테이블에 "녹여" 메모리에서도 키가 드러나지 않게 |

- **원리**: 리버싱은 보통 "디버거로 멈춰 보기 + 바이너리 패치"의 반복이다. 이 두 행위 자체를 탐지·차단하면 위 모든 정적/동적 분석의 효율이 급락한다.
- **특히 White-box cryptography**는 데스크탑/모바일처럼 **공격자가 실행 환경을 완전히 통제(white-box)하는** 상황을 가정한다. 키를 메모리에 평문으로 두지 않고 연산 테이블에 분산시켜, 메모리 덤프로도 키를 못 뽑게 한다.

---

### 7. 계층은 겹쳐 쓴다 (Defense in Depth)

실무에서는 한 기법만 쓰지 않는다. 싼 것부터 비싼 것까지 **층층이 누적**해 공격자가 한 층을 벗겨도 다음 층이 기다리게 한다.

```
배포 바이너리
  └─ Layout(이름 제거)
       └─ Data(문자열 암호화)
            └─ Control(평탄화 + 불투명 술어)
                 └─ Preventive(anti-debug + 무결성 검사)
```

핵심 설계 원칙: **민감도에 비례해 강도를 배분한다.** 라이선스 검증·크립토 같은 핵심 비밀에는 무거운 control/preventive를, 나머지에는 가벼운 layout만 적용해 성능 예산을 지킨다.

## 내가 얻은 인사이트

### 보안 설계 관점

1. **난독화의 목표 함수는 "불가능"이 아니라 "비용"이다.**
   - 암호화처럼 이진(뚫림/안 뚫림)으로 생각하면 실망한다. 난독화는 *공격자의 시간·전문성·동기*라는 자원을 소진시키는 게임이다. "우리 앱의 가치 < 리버싱 비용"이 되면 승리다.
   - 그래서 위협 모델(누가, 왜, 얼마의 비용까지 들일까)을 먼저 정하지 않으면 과/소 투자하게 된다.

2. **계층별 비용-효과 곡선이 다르다 — 싼 것부터 깔아라.**
   - Layout(이름 제거)은 공짜에 가까운데 디컴파일 환경(.NET/Java)에선 효과가 크다. 반면 control flow 평탄화는 효과는 크지만 성능세(tax)가 비싸다.
   - "전부 최대로"는 앱을 망가뜨린다. **핫패스 프로파일링 → 민감 영역에만 무거운 변환** 이라는 선택적 적용이 정석.

### 데스크탑/런타임 관점

3. **런타임 종류가 전략을 결정한다.**
   - Electron/JS, .NET IL, Java bytecode는 *심볼과 구조가 거의 그대로 살아있어* layout + string 난독화의 ROI가 매우 높다.
   - 반대로 C/C++ 네이티브는 컴파일이 이미 1차 난독화를 해주므로(심볼 제거, 최적화), 추가로는 control/preventive 계층에 투자하는 게 합리적이다.

4. **난독화는 운영 비용을 만든다 — 매핑 파일과 크래시 리포팅.**
   - `a.m1()`으로 찍힌 스택트레이스는 디버깅 지옥이다. 난독화를 도입하는 순간 **deobfuscation 매핑 파일을 빌드 산출물로 관리**하고, 크래시 리포팅 파이프라인(Sentry 등)에 연동하는 운영 설계가 반드시 따라와야 한다.

### 트레이드오프 관점

5. **성능세는 "어디에" 거느냐로 통제한다.**
   - 문자열 복호화·평탄화·불투명 술어는 모두 런타임 비용이다. 초기화 1회 경로엔 마음껏 걸어도 되지만, 60fps 렌더 루프나 핫 데이터패스에 걸면 체감 저하가 온다.
   - "보안 강도 ∝ 위치의 민감도, 반비례 ∝ 실행 빈도" — 이 두 축으로 적용 지점을 결정하는 게 엔지니어링의 핵심.

6. **불투명 술어는 '정적 분석의 결정 불가능성'을 무기로 쓴다.**
   - 가장 우아한 지점은, 평탄화의 `next` 전이를 불투명 술어로 감싸 *공격자의 자동화 도구(심볼릭 실행)* 를 상태 폭발로 무력화하는 부분이다. 사람이 아니라 **도구를 겨냥**한다는 발상의 전환이 인상적이다.
