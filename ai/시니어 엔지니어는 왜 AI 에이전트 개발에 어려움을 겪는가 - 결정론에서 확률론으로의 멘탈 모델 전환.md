## AI 요약

### 1. 핵심 주장: 전통 소프트웨어 vs 에이전트

발표의 핵심은 "왜 경험 많은 시니어 엔지니어일수록 오히려 AI 에이전트 개발에서 헤매는가"이다.
수십 년간 엔지니어링은 **모호함을 제거하고 엄격한 인터페이스를 정의하는 일**이었다. 그런데 에이전트는 확률적(probabilistic)이라, 그동안 몸에 밴 습관이 오히려 발목을 잡는다.

두 패러다임을 개발 루프로 비교하면:

```
[ 전통 소프트웨어 ]                     [ AI 에이전트 ]
spec/PRD 작성                          instruction(목표) 정의
   │                                      │
코드 작성                               실행(run)
   │                                      │
테스트 작성                             관찰(observe)
   │                                      │
배포                                    프롬프트/툴 조정(adjust)
   │                                      │
사용자 사용                             다시 실행 ──┐
                                          ▲          │
                                          └──────────┘
                                      (반복적 개선 루프)
```

가장 좋은 비유: **교통 관제관(traffic controller) → 배차원(dispatcher)** 으로의 역할 변화.

| 구분 | 전통 소프트웨어 (관제관) | AI 에이전트 (배차원) |
|------|--------------------------|----------------------|
| 통제 대상 | 신호등, 속도, 차선 — "차가 어떻게 달릴지" | 목적지(goal)만 지정 |
| 비유 | "이 도로로, 이 속도로 가라" | "런던 가줘" (기차·비행기·자동차 선택은 에이전트가) |
| 경로 | 모든 단계를 코드로 명시 | 단계는 정의하지 않음, 결과(outcome)만 정의 |
| 특징 | 결정론적 | 가끔 이상하게 굴어도 결국 목표 달성 |

> 핵심 통찰: 우리는 더 이상 "에이전트가 밟을 정확한 스텝"을 정의하지 않는다. **목표만 주고 경로는 위임**한다.

### 2. Text is the New State (텍스트가 새로운 상태다)

전통적으로 모든 것은 데이터 구조 — boolean, flag, enum — 로 매핑됐다. 하지만 LLM은 **의미(semantic meaning)** 를 이해할 수 있다.

**예시 — Deep Research 에이전트:**
에이전트가 리서치 계획(plan)을 반환할 때,

```
[ 전통적 방식 ]
plan → [Accept] or [Deny]   ← 의미를 담을 수 없는 이진 선택
  └ Deny 시: 추가 입력 → 새 plan 생성 → 재시도 (멀티스텝)

[ LLM 방식 ]
plan → "승인하되, US 시장에 집중하고 캘리포니아는 제외해줘"
  └ 승인과 추가 맥락 제공을 한 번에 (semantic feedback)
```

**예시 — 메모리/개인화:**
"나는 유럽 사람이라 평소엔 섭씨를 쓰지만, **요리할 땐 화씨가 좋아**" 같은 선호는 `user.unit = Celsius` 같은 플래그로는 표현 불가능하다. 사용자의 동적 선호를 데이터 구조에 가둘 수 없다.

> 결론: 이제 우리는 명확하게 구조화된 데이터(structured data)가 아니라 **텍스트와 맥락(context)** — 이미지·비디오·오디오 포함 — 위에서 동작한다.

### 3. Hand Over Control (통제권을 넘겨라)

**예시 — 고객 지원 (구독 취소):**

```
[ 전통적 방식 ]
"구독 취소할래요" → 의도 분류 모델(intent: churn)
                  → 사전 정의된 워크플로우
                     ├ 리텐션 시도?
                     └ 구독 취소 실행
   (정적, 동적 반응 불가)

[ 에이전트 방식 ]
"구독 취소할래요" → 에이전트가 의미를 이해
                  → 대안 제안 → 사용자 마음 변경
                  → 완전히 다른 의도로 전환 (동적)
```

수많은 분기와 상태(stateful) 워크플로우로 모든 케이스를 모델링하는 것은 매우 어렵다.
→ **순수 결정론적 환경을 벗어나, LLM을 신뢰하고 통제권을 넘겨야 한다.**

### 4. Errors Are Just Inputs (에러는 그냥 입력이다)

에이전트 흐름에서 실패가 발생하면, 그것을 **사용자 입력과 유사한 정상 입력**으로 취급해야 한다.
(Go 언어가 이미 이렇게 한다 — 함수 호출 결과가 error이거나 value이거나, 둘을 동등하게 다룬다.)

| 과거 (HTTP 요청은 저렴) | 지금 (긴 에이전트 실행) |
|--------------------------|--------------------------|
| 검색/조회 실패 → 그냥 재요청 | 5~15분 실행 중 실패 → 처음부터? |
| 작업 재수행 비용 작음 | 막대한 compute 재소모 |
| — | 기존 context 손실 위험 |

```
실패 발생
   │
   ├─ (X) 처음부터 재시작 → 비용·맥락 손실
   └─ (O) 에러를 모델에 input으로 되먹임
          + 추가 검증/우회책 → 앞으로 계속 진행(forward)
```

> 결론: 처음부터 다시 시작하지 말고, 에러를 다르게 다뤄 흐름을 **전진**시켜라. **Design for recovery.**

---

### 5. From Unit Tests to Evals (단위 테스트에서 평가로)

전통 소프트웨어: "입력 A → 코드 B → 항상 출력 C" 를 가정 (unit/integration/smoke test).
에이전트: **비결정론적**. 같은 입력이 같은 스텝·같은 결과를 보장하지 않는다.

| 항목 | 전통 테스트 | 에이전트 Eval |
|------|-------------|----------------|
| 가정 | 동일 입력 → 동일 출력 | 동일 입력 → 다른 경로 가능 |
| 측정 | pass/fail (assert) | 10번 중 몇 번 통과? (신뢰성) |
| 결과 | 객관적 | 주관적 (리서치 리포트 등) |
| 판정 | 코드 단언 | **LLM-as-a-judge**, 인간 전문가 |
| 관찰 | — | 항상 trace, 단 **결과(outcome)** 로 평가 |

> 같은 프롬프트가 10번 중 1번만 동작하는 고객 에이전트는 프로덕션에 못 올린다. 신뢰성(reliability)이 곧 성공이다. 어떤 사용자에겐 4스텝, 다른 사용자에겐 토큰을 더 쓰더라도 — **최종 결과**를 측정 대상으로 삼아라.

---

### 6. Agents Evolve and APIs Don't (에이전트는 진화하지만 API는 안 한다)

백엔드 API를 만들어 봤다면, `deleteItem(id)` 같은 엔드포인트는 너무나 자명(self-explaining)하게 느껴진다.
**하지만 에이전트는 코드를 보지 못한다.** 당신이 수년간 쌓은 맥락·배경을 모른다. 에이전트가 보는 것은 오직:

```
function schema + docstring + tool definition
```

```
[ 개발자가 보는 것 ]              [ 에이전트가 보는 것 ]
deleteItem(id: string)            { "name": "deleteItem",
// 당연히 알지: id는 상품 ID,        "params": { "id": "string" } }
// 실패 시 404 등                   → id가 뭔지, 실패 시 뭐가 되는지 모름
```

> 결론: **에이전트를 위해 쓰인 도구(agent-ready tools)** 가 필요하다. 자기 설명적(self-documenting)이고 의미론적 인터페이스(semantic interface)를 갖춰야 하며, "API를 만든 시니어 개발자의 전문성"을 전제하면 안 된다.

### 7. 요약 — 6가지 원칙

| 원칙 | 한 줄 요약 |
|------|-----------|
| **Trust but verify** | 신뢰하되 검증하라 |
| **Stop fighting the model** | 모델을 특정 워크플로우(step1→step2)에 억지로 끼워 넣지 마라 |
| **Preserve meaning** | 모든 것이 context다. 잘 정의된 데이터 구조에 의존하지 마라 |
| **Design for recovery** | 모델·에이전트는 완벽하지 않다. 장기 실행일수록 이상한 일이 생긴다 |
| **Evaluate, don't only assert** | 100% 신뢰 불가. 사용자에게 제공할 "성공 비율"의 적정선을 찾아라 |
| **Build to delete** | (the bitter lesson) 소프트웨어는 일회용. 더 나은 모델로 같은 것을 계속 다시 만든다 |

## 내가 얻은 인사이트

### 멘탈 모델 관점
1. **"왜 시니어가 더 어려운가"의 역설**
   - 경력이 길수록 "모호함 제거 / 엄격한 인터페이스 정의"가 근육 기억으로 박혀 있다. 그 강점이 확률적 시스템에서는 오히려 저항이 된다. 주니어가 아니라 **언러닝(unlearning)이 필요한 시니어**가 타깃이라는 점이 인상적이다.
   - DevOps/SRE 배경(known risk → unknown uncertainty에 익숙)이 오히려 이 전환에 유리하다는 외부 평가도 같은 맥락이다.

2. **관제관 → 배차원 비유의 실무적 함의**
   - 코드 리뷰/설계 시 "이 단계를 내가 명시하는 게 맞나, 목표만 주는 게 맞나"를 매번 자문하게 만든다. 과도한 분기 하드코딩은 이제 안티패턴 신호다.

### 아키텍처/설계 관점
1. **에러 핸들링의 재정의 = compute 경제학**
   - "에러는 입력"이라는 주장은 철학이 아니라 **비용 논리**다. 15분짜리 에이전트를 처음부터 재시작하는 건 토큰·시간·context를 통째로 버리는 것. 체크포인트/부분 복구/에러 되먹임을 1급 설계 요소로 끌어올려야 한다.
   - Go의 `value or error` 동등 취급 비유는 에이전트 루프에서 에러를 "예외(throw)"가 아니라 "다음 턴의 관찰값"으로 모델링하라는 구체적 설계 지침으로 읽힌다.

2. **Agent-ready API는 곧 새로운 DX 표준**
   - 기존 마이크로서비스 API를 그대로 툴로 노출하면 실패한다. docstring·schema가 곧 에이전트의 UI다. **"툴 정의 = 프롬프트의 일부"** 라는 관점으로 tool description을 작성해야 한다. 내부 API 거버넌스에 "agent-readiness" 항목을 추가할 만하다.

### 검증/품질 관점
1. **Assert에서 Eval로 = QA 조직의 재편**
   - "10번 중 몇 번 통과하느냐"가 합격 기준이 되면, CI 게이트·회귀 테스트의 형태 자체가 바뀐다. LLM-as-a-judge + 인간 전문가 + trace 관찰의 3중 구조를 파이프라인에 어떻게 녹일지가 실무 과제다.
   - 결과가 주관적일수록 "무엇을 outcome으로 정의할 것인가"를 먼저 합의하는 것이 평가 설계의 출발점이다.

### 전략/마인드셋 관점
1. **Build to delete & the bitter lesson**
   - 정교하게 짠 스캐폴딩/프롬프트 엔지니어링이 다음 모델 세대에서 통째로 폐기될 수 있다는 전제. 과도한 조기 최적화를 경계하고, **버릴 수 있게 짜는 것**이 미덕이 된다. 이는 코드 자산화에 익숙한 엔지니어에게 가장 받아들이기 힘든 전환점이다.
