## AI 요약

### 1. 한 줄 요약

> **에이전트 기반 E2E 테스트는 "정해진 UI 경로(journey)"가 아니라 "목표(goal) 달성 여부"를 검증한다.** Slack이 200건 이상 실행해 본 결과, 회당 $15–30·10분 이상으로 비싸고 느려서 결정론적 테스트를 *대체하지는 못하지만*, 테스트 피라미드 **최상단**에서 탐색·디버깅·복잡 동작 검증 계층으로 가치가 있다. 그리고 결과를 가르는 건 모델이 아니라 **실행 환경(MCP vs CLI)**이었다.

| 항목 | 내용 |
|------|------|
| **핵심 전환** | "테스트는 여정을 강제, 에이전트는 목표를 검증" |
| **가장 안정적인 실행 모델** | Agent + Playwright **MCP** (실패율 0–12%) |
| **비용** | 회당 약 $15–30, 10분 이상 |
| **비용을 결정하는 것** | 모델 출력이 아니라 **턴 수 × 컨텍스트 누적량** |
| **결론** | 결정론적 테스트(CI 토대) + 에이전틱 테스팅(피라미드 최상단) 결합 |

---

### 2. 여정(Journey)에서 목표(Goal)로

전통적 E2E 테스트와 에이전트 테스트의 근본적 차이는 "무엇을 검증하느냐"다.

```
[전통적 E2E 테스트]  — 여정(journey)을 강제
  클릭 → 클릭 → 입력 → assert
  (정해진 UI 경로를 그대로 따라가야 통과)

[에이전트 기반 테스트] — 목표(goal)를 검증
  목표 부여 → 에이전트가 UI 관찰·적응 → 결과(목표 상태) 검증
  예: "스레드에 메시지를 보내라" → 경로는 매번 달라도 됨
```

같은 워크플로(로그인 → 검색 → 결과 → 초기화)라도 **실제 동작 순서는 매번 달라졌다.**
- 서로 다른 입력 방식 (검색 추천 클릭 vs Enter 입력)
- 서로 다른 탐색 패턴 (검색 재실행 vs 기존 상태 재사용)
- 단계의 추가/생략 (중간 스냅샷, 추가 클릭 등)

> 유연성에는 **신뢰성·비용·실행 시간**의 트레이드오프가 따라온다.

### 3. 실험 구성 — 세 가지 실행 모델

| 실행 모델 | 동작 방식 | 특징 |
|-----------|-----------|------|
| **Agent + Playwright MCP** | 에이전트가 사전 정의된 브라우저 동작(클릭·입력·DOM 읽기)으로 상호작용 | DOM 스냅샷·로그를 **지속적 컨텍스트로 유지** |
| **Agent + Playwright CLI** | 셸에서 CLI 명령을 한 번에 한 단계씩 실행 | 매 단계 UI 상태를 다시 보고 다음 동작 결정 |
| **Generated Playwright Tests** | AI가 자연어 설명 → **결정론적 Playwright 코드 생성** | 표준 E2E로 실행, 통과할 때까지 반복 개선 |

**실험 환경**
- MCP·CLI 에이전트 모델: **Claude Sonnet 4.5**
- 생성형 테스트 모델: **Claude Opus 4.6**
- 실행: 비대화형 Claude Code (`claude -p`)
- 도구: Playwright MCP / Playwright CLI, Slack Dev API MCP
- 데이터: 전부 **비프로덕션 테스트 워크스페이스**

**테스트 플로 (복잡도 2단계)**
- **Thread Reply (단순)**: 채널 생성 → 메시지 전송 → 스레드 답글 → 상태 검증 (약 15–20단계)
- **Search Discovery (중간)**: 검색어 입력 → 결과 탐색 → 뷰 이동 → 결과 검증 (약 25–30단계)

**입력 형식 2종**: 자연어(NL) vs 구조화 YAML — 차이는 상세함이 아니라 **표현 방식**(NL은 에이전트가 해석·매핑, YAML은 매핑이 명시적).

→ 각 구성 20회씩, 총 5개 실험(MCP-NL, MCP-YAML, CLI-NL, CLI-YAML, 생성테스트-NL)을 두 플로에 적용.

### 4. 관찰 결과 — 신뢰성·속도·비용

#### 결과 요약표

| 실행 모델 | 실패율 (단순/복잡) | 평균 소요 |
|-----------|:---:|:---:|
| Agent (Playwright **MCP**) | **0% / ~12%** | 5–8분 |
| Agent (Playwright **CLI**) | ~12% / ~20% | 9–11분 |
| **Generated** Playwright Tests | ~8% / **~48%** | 3분 (생성+실행) |

#### 신뢰성 (Reliability)
- **플로가 복잡해질수록 격차가 가장 뚜렷.** MCP는 복잡 플로에서도 0–12% 유지.
- **CLI 실패의 대부분은 모델 추론이 아니라 실행 계층 문제** — 인증 처리, 내비게이션 타이밍, 세션 불안정.
- **생성 테스트**는 단순 플로(8%)는 괜찮았으나 복잡 플로에서 48%로 급락. 완전히 틀린 게 아니라 **플로의 70–80%까지 진행 후 마지막 상호작용·단언에서 실패**.
  - 원인: UI 상태 변동성 + 추상화 불일치(느슨한 NL에서 생성 + 기존 페이지 오브젝트 재사용 → 정밀한 요소 타깃팅 방해).

```
[MCP] 앱의 실시간·안정적 뷰를 세션 내에서 유지
       이전 단계의 성공 상호작용을 재사용 → 불일치 누적 적음
[CLI] 매 단계 스냅샷으로 상태를 처음부터 재구성
       → 플로가 길어질수록 불일치 누적
```

#### 속도 (Speed)
- 생성 테스트가 일관되게 가장 빠름(~3분, 생성+5회 실행 평균).
  - **순수 실행은 훨씬 빠름** — thread reply ~32초, search discovery ~45초.
  - CI처럼 반복 실행되면 **일회성 생성 비용이 무시할 수준** → 결정론적 테스트가 효율적으로 확장.
- 에이전트 워크플로는 **매 실행마다 비용 지불**(매 단계 관찰 → 추론 → 실행 → 검증).

#### 적응성 (Adaptability)
- **동일 동작 순서를 따른 실행은 약 20%뿐.** 나머지는 같은 목표에 도달하는 서로 다른 유효 경로.
- 측정 방법: 실행 간 **액션 시그니처(action signature)** 비교 = 도구 호출·UI 동작의 순서 목록.
  - 비교 전 **정규화**: 파라미터·대기/스냅샷·동등 변형(`fill` vs `type`)을 통합해 의미 있는 차이만 계산.

### 5. 비용은 어디서 오는가 (핵심 통찰)

에이전트 실행은 회당 보통 **$15–30**. 동일 Search Discovery 플로의 토큰 사용량:

| 구성 | 토큰 | 턴 수 |
|------|:---:|:---:|
| MCP (Opus 4.6) | ~3.8M | ~40 |
| MCP (Sonnet 4.5) | ~3.5M | ~40 |
| MCP (Haiku 4.5) | ~5.7M | ~60 |
| **CLI** (Opus 4.6) | ~6M | ~85 |
| **Code Gen** (Opus 4.6) | ~7M | ~70 |

> **어떤 모델을 쓰는지보다 어떻게 실행하는지가 더 중요하다.** Haiku가 토큰을 더 썼어도, *모든 MCP 방식*이 CLI·CodeGen보다 토큰을 적게 썼다.

**왜?** 기반 API가 **무상태(stateless)** → 매 턴마다 전체 시스템 프롬프트 + 전체 대화 이력을 재전송.

```
비용 = 모델 출력(X)  →  턴 수 × 컨텍스트 누적 속도 (O)

[CLI] 한 번의 브라우저 상호작용 = 동작·대기·스냅샷·읽기·요소조회 …
       여러 명령으로 분할 → 평균 85턴
[MCP] 상호작용 + 상태 반환을 단일 왕복으로 결합 → 약 40턴

매 추가 턴마다 → 전체 시스템 프롬프트 + 이전 컨텍스트 재전송 비용
```

**컨텍스트를 채우는 주범**
- MCP·CLI: Playwright가 반환하는 **접근성 트리(accessibility tree) 스냅샷**이 이후 모든 턴에 누적.
- Code Gen: 재시도마다 **전체 오류 트레이스·단언 실패·DOM 상태**가 누적.
- → 비용 대부분은 **이미 본 내용의 재전송**, 턴당 새 정보는 극히 일부.

**아직 최적화하지 않은 영역**: 프롬프트 캐싱, 컨텍스트 압축(compaction), 스냅샷 빈도 축소.
→ 현재로선 고빈도 CI보다 **표적 디버깅·탐색적 테스트**에 적합.

### 6. 인프라가 중요하다 (MCP vs CLI)

| 항목 | Playwright MCP | Playwright CLI |
|------|----------------|----------------|
| 실패율 | 0–12% | 12–20% |
| 실패 원인 | (적음) | 인증·내비게이션·세션 등 **실행 계층** |
| 브라우저 통합 | 구조화된 프리미티브, 에이전트 도구 호출과 긴밀 | 에이전트–브라우저 사이에 추가 계층 |
| 병렬화 | 동시 실행 쉬움 | 어려움 → 대부분 순차 실행 |

> 신뢰성·속도·비용은 모델뿐 아니라 **실행 환경의 안정성과 설계**에 좌우된다.

**경계(미탐색 영역)**: 본 실험은 *단일 세션 UI 워크플로*에 집중. 크로스 워크스페이스/다중 브라우저 창 같은 플로는 MCP(긴 관찰 루프 → 비용)·CLI(다중 세션 조정 복잡성) 모두 다른 트레이드오프를 가짐.

### 7. 테스트 피라미드 속 위치

에이전틱 테스팅은 기존 방식을 **대체하지 않고 그 위에 새 역량을 추가**한다.

```
            ▲
           ╱ ╲   에이전틱 테스팅 (최상단, 신규 계층)
          ╱   ╲   · 복잡한 UI 동작 탐색
         ╱─────╲  · 불안정(flaky) 워크플로 디버깅
        ╱       ╲ · 프로덕션 버그 재현
       ╱  E2E /  ╲
      ╱ Integration╲  결정론적 테스트 (CI의 안정적 토대)
     ╱──────────────╲ · 빠르고 반복 가능, 낮은 비용
    ╱   Unit Tests    ╲· 특정 UI 여정 강제
   ╱───────────────────╲
```

| | 결정론적 E2E | 에이전틱 테스팅 |
|---|---|---|
| 출발점 | 정해진 스크립트 실행 | **목표**에서 출발(관찰·추론·결정) |
| 적합 용도 | CI 회귀 검사 | 탐색·디버깅·버그 재현 |
| 비용 | 낮음 | 높음(회당 $15–30) |
| 강점 | 빠름·반복 가능 | 유연·적응적 |

> 가장 효과적인 미래 전략은 **둘의 결합** — 결정론적 테스트가 CI의 안정적 토대를, 에이전틱 테스팅이 피라미드 최상단의 탐색·디버깅·복잡 동작 검증을 담당.

## 내가 얻은 인사이트

### 테스트 설계 관점

1. **"경로 검증"에서 "목표 검증"으로의 패러다임 분리**
   - 전통 E2E의 취약함(flaky) 상당수는 *불필요하게 경로까지 고정*하기 때문이다. 에이전트는 "결과 상태"만 단언하므로 UI 리팩토링에 강건하다.
   - 단, 이 유연함은 **검증의 모호함**과 맞바꾼 것 — "목표 달성"의 정의가 느슨하면 잘못된 통과(false pass)가 생길 수 있다. 목표·단언을 얼마나 엄밀히 명세하느냐가 핵심.

2. **생성 테스트의 "70–80% 진행 후 마지막에 실패" 패턴**
   - 단순히 실패율 숫자만 보면 안 된다. *어디서* 깨지는지가 더 중요하다. 마지막 단언에서 깨진다는 건 **요소 타깃팅(셀렉터) 안정성**이 병목이라는 뜻 → 테스트 ID(`data-testid`) 정비, 페이지 오브젝트 추상화 재점검이 ROI 높은 투자.

### 시스템/비용 관점

3. **"모델보다 실행 모델(infra)이 비용·신뢰성을 지배한다"**
   - 가장 반직관적이면서 실무적으로 중요한 결론. Haiku(저렴)가 토큰을 더 써도 MCP라는 이유로 CLI/CodeGen보다 효율적이었다. 에이전트 시스템 최적화의 첫 레버는 **모델 교체가 아니라 도구 호출 라운드트립 수(턴 수) 줄이기**.
   - CLI가 "한 상호작용을 여러 명령으로 쪼개" 85턴이 된 건, *무상태 API + 턴마다 전체 컨텍스트 재전송* 구조에서 치명적. **턴 수 = 비용**이라는 멘탈 모델을 가져야 한다.

4. **stateless API의 비용 구조 = "이미 본 내용의 재전송"**
   - [Code Execution with MCP](Code%20Execution%20with%20MCP%20-%20%ED%86%A0%ED%81%B0%2098.7%25%20%EC%A0%88%EA%B0%90%ED%95%98%EB%8A%94%20%EC%B0%A8%EC%84%B8%EB%8C%80%20%EC%97%90%EC%9D%B4%EC%A0%84%ED%8A%B8%20%ED%8C%A8%ED%84%B4.md) 글과 정확히 같은 통찰. 비용 대부분이 새 정보가 아니라 누적된 스냅샷/트레이스의 반복 전송이다. → **프롬프트 캐싱·컨텍스트 압축·스냅샷 빈도 축소**가 곧바로 비용 절감으로 이어진다.

### 도입 전략 관점

5. **"대체"가 아니라 "계층 추가"라는 현실적 포지셔닝**
   - 회당 $15–30·10분은 CI 모든 PR마다 돌릴 수 없다. 에이전틱 테스팅을 *고빈도 회귀*가 아니라 **표적 디버깅·flaky 재현·탐색적 테스트**에 쓰는 것이 현재 기술 수준에서 옳은 선택.
   - 실무 적용 청사진: ① 결정론 테스트로 CI 토대 유지 → ② flaky/재현 어려운 버그가 나오면 에이전트로 탐색 → ③ 에이전트가 발견한 안정적 경로를 *결정론 테스트로 코드화*해 피라미드 하단으로 내림. 두 계층이 서로를 먹여 살리는 선순환.
