## AI 요약

### 1. 2025 DORA 보고서란?

2025년 DORA(DevOps Research and Assessment) 보고서는 **"State of AI-Assisted Software Development"**라는 제목으로, AI가 소프트웨어 엔지니어링 성과에 미치는 영향을 체계적으로 분석한 대규모 연구다.

| 항목 | 내용 |
|------|------|
| 보고서 제목 | State of AI-Assisted Software Development |
| 발행처 | dora.dev (Google) |
| 조사 대상 | 약 5,000명의 기술 전문가 |
| 정성 인터뷰 | 100시간 이상의 심층 인터뷰 |
| 핵심 메시지 | AI는 기존 엔지니어링 조건의 **승수(Multiplier)**로 작용 |

### 2. 핵심 발견: AI는 자동으로 성과를 개선하지 않는다

보고서의 가장 중요한 결론은 다음 한 문장으로 요약된다:

> "AI does not automatically improve software delivery performance. Instead, it acts as a multiplier of existing engineering conditions."

```
┌─────────────────────────────────────────────────────────────┐
│                  AI의 승수 효과 (Multiplier Effect)           │
│                                                             │
│   성숙한 엔지니어링 조직                                      │
│   ┌─────────────────────┐      ┌─────────────────────────┐  │
│   │ - DevOps 성숙도 높음│      │  AI 도입                │  │
│   │ - 명확한 워크플로우  │  ×   │  ────────►              │  │
│   │ - 강력한 플랫폼 역량│      │  생산성 & 배포 성과 향상│  │
│   └─────────────────────┘      └─────────────────────────┘  │
│                                                             │
│   미성숙한 엔지니어링 조직                                    │
│   ┌─────────────────────┐      ┌─────────────────────────┐  │
│   │ - 도구 단편화       │      │  AI 도입                │  │
│   │ - 불명확한 프로세스  │  ×   │  ────────►              │  │
│   │ - 일관성 없는 관행  │      │  기술 부채 & 불안정성 ↑ │  │
│   └─────────────────────┘      └─────────────────────────┘  │
└─────────────────────────────────────────────────────────────┘
```

### 3. AI 도입 현황

| 지표 | 수치 |
|------|------|
| AI 보조 도구 사용 개발자 비율 | ~90% |
| AI에 크게 의존하는 개발자 비율 | ~2/3 |
| 주요 활용 분야 | 코드 작성, 문서화, 디버깅, 프레임워크 학습 |
| 생산성 향상 보고 | 다수 개발자가 측정 가능한 향상 보고 |

### 4. 생산성과 신뢰 사이의 긴장

AI가 개발 속도를 높이지만, 생성된 코드의 **정확성과 신뢰성**에 대한 우려가 동시에 존재한다.

```
┌───────────────────────────────────────────────┐
│         AI 도입의 이중적 효과                    │
│                                               │
│   긍정적 효과              부정적 리스크        │
│   ┌──────────┐            ┌──────────────┐    │
│   │ 코드 생산량 ↑│          │ 코드 품질 불확실│  │
│   │ 개발 속도 ↑ │  ◄───►  │ 검증 오버헤드 ↑│  │
│   │ 학습 가속   │          │ 기술 부채 위험 │  │
│   └──────────┘            └──────────────┘    │
│                                               │
│   핵심: 적절한 엔지니어링 규율 없이는            │
│         속도 향상이 품질 향상으로 이어지지 않음    │
└───────────────────────────────────────────────┘
```

### 5. DORA AI 역량 모델 (AI Capabilities Model)

보고서는 AI를 효과적으로 활용하기 위한 7가지 핵심 역량을 제시한다:

```
┌─────────────────────────────────────────────────────────┐
│              DORA AI 역량 모델 7가지 축                    │
│                                                         │
│  ┌─────────────────────────────────────────────────┐    │
│  │  1. 명확한 조직 전략 (Clear Org Strategy)        │    │
│  │     AI 도구 사용, 거버넌스, 워크플로우 통합 정책  │    │
│  ├─────────────────────────────────────────────────┤    │
│  │  2. 건강한 데이터 생태계 (Healthy Data Ecosystem)│    │
│  │     신뢰할 수 있는 문서, 아키텍처 지식, 개발 데이터│   │
│  ├─────────────────────────────────────────────────┤    │
│  │  3. 내부 지식 접근성 (Internal Knowledge Access) │    │
│  │     검색 가능한 지식 저장소, 구조화된 내부 데이터  │    │
│  ├─────────────────────────────────────────────────┤    │
│  │  4. 기초 엔지니어링 관행 (Foundation Practices)  │    │
│  │     버전 관리, 코드 검토, 일관된 개발 표준        │    │
│  ├─────────────────────────────────────────────────┤    │
│  │  5. 사용자 중심 개발 (User-Centric Development) │    │
│  │     기술 산출물보다 사용자 결과에 초점             │    │
│  ├─────────────────────────────────────────────────┤    │
│  │  6. 플랫폼 엔지니어링 (Platform Engineering)    │    │
│  │     표준화된 환경, 배포 파이프라인, 인프라 서비스  │    │
│  ├─────────────────────────────────────────────────┤    │
│  │  7. 소규모 배치 작업 (Small Batch Work)          │    │
│  │     작고 점진적인 변경으로 검토 품질 ↑ 위험 ↓     │    │
│  └─────────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────────┘
```

| 역량 | 설명 | AI 효과 연결 |
|------|------|-------------|
| 명확한 조직 전략 | AI 도구 사용·거버넌스에 관한 명시적 정책 정의 | 일관된 도입과 혼란 방지 |
| 건강한 데이터 생태계 | 신뢰할 수 있는 문서·아키텍처 지식에 대한 접근 | AI 도구의 출력 품질 향상 |
| 내부 지식 접근성 | 검색 가능한 지식 저장소와 구조화된 데이터 | 조직 표준에 맞는 AI 권장사항 제공 |
| 기초 엔지니어링 관행 | 버전 관리·코드 검토·개발 표준 | AI는 이를 대체하지 않고 의존함 |
| 사용자 중심 개발 | 기술 산출물보다 사용자 결과에 초점 | AI를 더 효과적으로 통합 |
| 플랫폼 엔지니어링 | 표준화된 환경·배포 파이프라인 제공 | 팀 전체의 일관성 유지하며 AI 확장 |
| 소규모 배치 작업 | 작고 점진적인 변경 | 코드 검토 품질 ↑ 배포 위험 ↓ |

### 6. 플랫폼 엔지니어링의 핵심 역할

보고서는 **플랫폼 엔지니어링**이 AI 도입의 성패를 가르는 핵심 요소라고 강조한다:

```
┌──────────────────────────────────────────────────┐
│        플랫폼 엔지니어링과 AI의 시너지              │
│                                                  │
│   내부 플랫폼                                     │
│   ┌────────────────────────┐                     │
│   │ - 표준화된 개발 환경    │                     │
│   │ - 통합 배포 파이프라인  │  ──► AI 도구 통합   │
│   │ - 인프라 서비스 추상화  │      일관성 보장     │
│   │ - 관찰 가능성 도구      │      신뢰성 유지     │
│   └────────────────────────┘                     │
│                    │                             │
│                    ▼                             │
│   ┌────────────────────────────────────┐         │
│   │ 팀 전체에서 AI 확장 가능            │         │
│   │ (개별 팀 수준이 아닌 조직 수준)      │         │
│   └────────────────────────────────────┘         │
└──────────────────────────────────────────────────┘
```

### 7. 시스템 안정성 경고

AI가 개발 속도를 높이면서 동시에 더 큰 변경 세트와 빠른 실험을 조장할 수 있다. 이는 다음과 같은 위험을 수반한다:

| 위험 요소 | 설명 |
|-----------|------|
| 결함 증가 | 더 많은 코드가 더 빠르게 배포되면서 결함 유입 가능성 ↑ |
| 배포 실패 | 검증 없이 빠른 배포로 인한 실패율 증가 |
| 운영 불안정성 | 빠른 변경이 시스템 안정성을 해칠 수 있음 |

**보고서의 경고**: 조직은 AI로 인한 속도 향상에도 불구하고 **엔지니어링 규율을 절대 완화해서는 안 된다**.

## 내가 얻은 인사이트

### METR 연구와의 대비 관점

1. **METR 연구와 DORA 보고서는 같은 동전의 양면이다**
   - METR 연구(2025)는 경험 많은 개발자 16명을 대상으로 AI가 오히려 19% 속도를 저하시켰다는 결과를 보여주었다. DORA 보고서는 이를 보완하는 시각을 제공한다: **AI의 효과는 개인의 역량이 아니라 조직의 엔지니어링 성숙도에 의해 결정된다**. 즉, METR 연구에서 개별 전문 개발자가 AI를 사용해 느려진 것은 AI 자체의 문제가 아니라, AI가 기존 워크플로우와 충분히 통합되지 않은 조건에서의 결과일 수 있다.

2. **"AI 승수 효과"는 조직의 자기 진단 도구다**
   - AI가 잘 작동하지 않는 조직은 AI를 탓할 것이 아니라, 자신의 엔지니어링 기반을 점검해야 한다. 도구 단편화, 불명확한 프로세스, 일관성 없는 관행이 있다면 AI는 이 문제들을 **증폭**시킬 뿐이다. 이는 AI 도입이 역설적으로 조직의 엔지니어링 성숙도를 진단하는 리트머스 시험지가 될 수 있음을 의미한다.

### 실무 적용 관점

3. **플랫폼 엔지니어링이 AI 시대의 핵심 투자 영역이다**
   - 보고서가 강조하는 플랫폼 엔지니어링의 역할은 매우 실질적이다. 개별 개발자가 각자의 방식으로 AI를 활용하는 것은 풀뿌리 실험에 불과하다. 표준화된 플랫폼 위에서 AI가 작동할 때 비로소 조직 수준의 성과 향상이 가능하다. 이는 "AI 도구 예산"보다 "플랫폼 엔지니어링 예산"이 더 높은 ROI를 가져올 수 있다는 시사점을 준다.

4. **소규모 배치 작업과 AI는 상호 강화 관계다**
   - AI가 더 많은 코드를 빠르게 생산할 수 있게 해주지만, 이를 큰 배치로 배포하면 위험이 증가한다. 소규모 배치 작업 원칙을 유지하면서 AI를 활용하면, AI가 생산한 코드의 검증 부담을 줄이고 배포 안정성을 유지할 수 있다. 이는 AI 시대에 DORA 메트릭의 "변경 리드 타임"과 "변경 실패율"의 균형을 맞추는 핵심 전략이다.

### 조직 전략 관점

5. **"AI를 도입하라"가 아니라 "엔지니어링 기반을 강화한 후 AI를 도입하라"**
   - 보고서의 최종 결론 — *"AI will not fix broken engineering systems. But for organizations that have already built strong foundations, it may become one of the most powerful accelerators of engineering performance yet."* — 은 AI 도입 순서에 대한 명확한 지침을 제공한다. AI 도구의 정교함보다 주변 시스템의 강점에 먼저 투자해야 한다. 이는 많은 조직이 "어떤 AI 도구를 쓸 것인가"에 집중하는 반면, "AI가 효과를 발휘할 기반이 갖춰져 있는가"를 간과하고 있음을 경고한다.
