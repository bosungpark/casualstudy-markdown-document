## AI 요약

### 1. 왜 "에이전트 친화적" 설계인가?

AI 에이전트가 SaaS 제품의 **새로운 1급 사용자(first-class user)**가 되고 있다. 사람이 쓰던 화면·문서·API를 이제는 에이전트가 직접 호출한다. 문제는 에이전트가 사람과 **근본적으로 다른 방식**으로 제품을 소비한다는 것이다.

| 구분 | 사람 사용자 | 에이전트 사용자 |
|------|------------|----------------|
| 문서 해석 | 맥락·의도로 행간 추론 | **문자 그대로(literal)** 파싱 |
| 화면 | 시각 디자인으로 이해 | 구조화된 데이터만 처리, 디자인 무시 |
| 동작 | 시행착오·직관 | 툴 스키마(tool schema)에 전적으로 의존 |
| 실패 대응 | 알아서 우회 | 일관·예측 가능한 응답이 없으면 깨짐 |
| 속도/반복 | 느림, 가끔 | 빠름, 반복 재시도, 상시 실행 |

핵심 키워드 세 가지: 에이전트는 **literal(문자 그대로), procedural(절차적), fragile(취약)** 한 사용자다.

### 2. API 설계 — 명시적 계약(Contract)이 전부

에이전트는 추론하지 않고 **읽은 그대로 실행**하므로, 모호함이 곧 실패다.

**(1) 명확한 문서**
- 필수/선택 파라미터를 명시
- 일관된 포맷, 최신 정보 유지
- 응답 코드와 실제 바디 예시를 함께 제공

```
201: Order created successfully
{ "order_id": "123", "status": "pending" }
```

**(2) 행동 가능한(actionable) 에러**

에러 응답은 "무엇이 실패했고 / 다음에 뭘 해야 하고 / 재시도해도 되는지"를 담아야 한다.

```json
{
  "error": "invalid_customer_id",
  "message": "Customer ID 'abc123' not found",
  "suggestion": "Check if customer exists before creating order"
}
```

**(3) 구조화된 일관 응답**

필드명을 고정하고, 응답을 일관된 봉투(envelope)로 감싼다.

```json
{
  "success": true,
  "data": { ... },
  "metadata": { ... }
}
```

### 3. UI 설계 — 에이전트도 화면을 "읽는다"

브라우저를 조작하는 에이전트를 위해:

```
┌─────────────────────────────────────┐
│  [Stable Selectors]                  │
│   id/class를 고정 → 자동생성 ID 금지   │
│   (예: #submit-order ✅                │
│        .css-1a2b3c ❌)                 │
├─────────────────────────────────────┤
│  [Visual Hierarchy]                  │
│   시맨틱 HTML, 일관 레이아웃,          │
│   명확한 액션 버튼                     │
├─────────────────────────────────────┤
│  [Visible Feedback]                  │
│   라벨 있는 로딩 상태                  │
│   (단순 스피너 ❌ → "Saving..." ✅)    │
└─────────────────────────────────────┘
```

### 4. 인증(Authentication)

| 방식 | 특징 | 적합한 경우 |
|------|------|------------|
| **API Key** | 단순, 스코프 지정, 회전(rotate) 쉬움 | 서비스 에이전트, 백엔드 자동화 |
| **OAuth 2.0** | Authorization Code + Refresh Token | 사용자 위임 시나리오 |
| **Device Code Flow** | 브라우저 없는 환경 | 헤드리스 에이전트 |
| **Service Account** | 조직 단위 권한 | 엔터프라이즈 |

실무 원칙: 스코프를 명확히 정의 · 구조화된 토큰 응답 · 토큰 갱신 허용 · 코드 예시 제공.

### 5. 레이트 리밋 & 사용량

에이전트는 **빠르고, 반복 재시도하고, 상시 실행**되므로 사람과 트래픽 패턴이 다르다. 리밋 응답에 재시도 메타데이터를 반드시 포함:

```json
{
  "error": "rate_limited",
  "retry_after": 60,
  "limit": 100,
  "remaining": 0,
  "reset_time": "2026-06-08T12:00:00Z"
}
```

- 에이전트 사용량을 사람 사용량과 **분리 추적**
- 호출당 예측 가능한 비용 + 볼륨 할인 + 사용량 상한(cap) 기반 과금 설계

### 6. 에이전트 친화성 테스트

- 문서를 **LLM에 직접 먹여** 이해되는지 검증
- 에러 응답이 실제로 행동 가능한지 확인
- 모든 예시 코드가 동작하는지 검증
- 엔드포인트 간 API 일관성 점검
- 실제 에이전트 프레임워크(**LangChain, CrewAI**)로 테스트
- UI는 비전(vision) 테스트 활용

### 7. 종합 체크리스트

| 영역 | 핵심 |
|------|------|
| 문서 | 문자 그대로·명시적·버전 명시·완전한 응답 예시 |
| API | 일관된 구조, 통일된 응답 포맷, 명확한 에러 코드 |
| 에러 | 실패 원인 + 복구 단계 안내 |
| UI | 시맨틱 HTML, 안정적 식별자, 보이는 로딩 상태 |
| 인증 | 스코프 있는 키, OAuth 2.0, 구조화 토큰 |
| 모니터링 | 에이전트별 로깅, 사용량 추적, 실시간 차단 |
| 안전 | 사람 검토 게이트, 감사 로그, 긴급 정지 |

## 내가 얻은 인사이트

### 설계 원칙 관점
1. **"좋은 DX = 좋은 AX(Agent Experience)"의 연장선**
   - 명확한 계약, 일관된 동작, 훌륭한 문서라는 개발자 경험(DX) 원칙이 그대로 에이전트 경험으로 이어진다. 다만 에이전트는 추론으로 빈틈을 메우지 못하므로 **더 엄격한(rigor) 적용**이 필요하다. 즉 새로운 패러다임이 아니라 기존 좋은 관행의 "관용(tolerance) 제로" 버전이다.

2. **모호함은 사람에겐 비용, 에이전트에겐 실패**
   - 사람은 헷갈리는 문서를 보고 시행착오로 답을 찾지만, 에이전트는 문자 그대로 실행하다 그냥 깨진다. 에러 응답에 `suggestion`/`retry_after` 같은 **다음 행동 정보**를 넣는 것이 단순 디버깅 편의가 아니라 에이전트의 자가 복구 능력을 좌우한다.

### 실무 적용 관점
3. **자동 생성 CSS 클래스는 에이전트의 적**
   - `.css-1a2b3c` 같은 빌드 타임 해시 클래스는 사람에겐 무해하지만 브라우저 조작 에이전트에겐 셀렉터를 매번 깨뜨린다. 안정적 `id`/시맨틱 HTML이 곧 접근성(a11y)과도 겹친다 → **에이전트 친화성과 웹 접근성은 상당 부분 같은 작업**.

4. **에이전트 트래픽을 사람과 분리 계측하라**
   - 상시 실행·반복 재시도 특성상 기존 사람 기준 레이트 리밋/과금 모델이 그대로 적용되면 비용 폭증이나 오탐 차단이 생긴다. 처음부터 에이전트 식별·별도 추적·호출당 과금을 설계에 반영해야 한다.

### 트레이드오프/안전 관점
5. **속도와 안전의 균형 — 긴급 정지(emergency stop)는 선택이 아닌 필수**
   - 에이전트는 잘못된 동작도 빠르게 대량 반복한다. 사람 검토 게이트, 감사 로그, 실시간 차단 같은 가드레일이 없으면 한 번의 버그가 순식간에 확산된다. "빠른 자동화"의 이면에는 반드시 **사람이 개입할 수 있는 브레이크**가 필요하다.
