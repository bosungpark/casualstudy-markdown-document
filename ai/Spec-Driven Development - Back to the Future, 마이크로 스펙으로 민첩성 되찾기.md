## AI 요약

### 1. 이 아티클의 핵심 논점

2025년 7월 AWS가 Kiro 출시와 함께 Spec-Driven Development(SDD)를 소개한 이후, SDD가 AI 코딩의 표준 방법론으로 부상하고 있다. 그러나 저자는 **전체 애플리케이션을 미리 명세하는 SDD 방식이 과거 V모델/워터폴의 함정을 되풀이할 위험**이 있다고 경고하며, 대안으로 **마이크로 스펙(Micro-specs)** 접근법을 제안한다.

| 항목 | 내용 |
|------|------|
| 핵심 주장 | SDD 자체는 유효하나, 규모(scope)가 핵심 — 마이크로 스펙으로 민첩성 유지 |
| 비판 대상 | 전체 애플리케이션을 사전 명세하는 "풀 스펙" 방식 |
| 제안 대안 | 마이크로 스펙 (단일 기능/컴포넌트 단위 명세) |
| 핵심 메시지 | "스펙을 작게 유지하고, 반복하며, 변화를 수용하라: 민첩함을 유지하라!" |
| 저자 배경 | AWS Senior Solution Architect |

### 2. SDD의 현재 생태계

2025년 중반 이후 SDD 도구와 방법론이 폭발적으로 증가했다.

```
┌─────────────────────────────────────────────────────────────┐
│                  SDD 생태계 (2025~2026)                      │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  IDE 네이티브           CLI/스킬 기반          방법론         │
│  ┌─────────────┐      ┌──────────────┐     ┌────────────┐  │
│  │ Kiro (AWS)  │      │ OpenSpec     │     │ BMAD       │  │
│  │ Cursor Plan │      │ Spec Kit     │     │ AIDLC      │  │
│  │ Copilot     │      │ Tessl        │     │            │  │
│  └─────────────┘      └──────────────┘     └────────────┘  │
│                                                             │
│  공통 워크플로우: Spec → Design → Tasks → Implementation     │
└─────────────────────────────────────────────────────────────┘
```

### 3. 풀 스펙 SDD의 위험: V모델의 귀환

저자는 현재 SDD 도구들이 전체 애플리케이션 스펙을 사전에 완성하도록 유도하면서, 30년 전 Agile이 극복한 워터폴/V모델의 문제를 되살리고 있다고 지적한다.

```
       풀 스펙 SDD (V모델 재현)                    마이크로 스펙 SDD (Agile 유지)

  ┌──────────────────────┐              ┌───────┐  ┌───────┐  ┌───────┐
  │ 전체 애플리케이션 명세  │              │ Epic1 │  │ Epic2 │  │ Epic3 │
  │ (수십 개 AI 질문 응답) │              │ Spec  │  │ Spec  │  │ Spec  │
  │ (수천 줄 마크다운)     │              │→Design│  │→Design│  │→Design│
  │ (검토에 막대한 시간)   │              │→Tasks │  │→Tasks │  │→Tasks │
  └──────────┬───────────┘              │→Build │  │→Build │  │→Build │
             │                          │→Verify│  │→Verify│  │→Verify│
             ▼                          └───┬───┘  └───┬───┘  └───┬───┘
  ┌──────────────────────┐                  │          │          │
  │  구현 (한 번에 전부)   │                  ▼          ▼          ▼
  │  요구사항 변경 대응 불가│             피드백 반영   범위 조정   방향 전환
  └──────────────────────┘             (반복적, 민첩)
```

**풀 스펙 방식의 구조적 문제점:**

| 문제 | 설명 |
|------|------|
| 요구사항 동결 | 변경·우선순위 조정·구축 중 발견되는 개선점 반영 불가 |
| 마크다운 과부하 | 수천 줄의 명세 문서를 사람이 검토해야 함 |
| LLM 맥락 오염 | 과도한 문서가 토큰 한계를 초과하여 코드 품질 저하 |
| 미사용 문서 낭비 | 변경된 요구사항으로 인해 이미 작성된 문서가 무용지물 |
| 거짓 속도감 | AI가 빨리 만들어준다고 해서 올바른 것은 아님 |

### 4. 마이크로 스펙: 해결책

마이크로서비스가 모놀리스를 분해한 것처럼, **마이크로 스펙은 전체 명세를 단일 기능/컴포넌트 단위로 분해**한다. Agile의 사용자 스토리가 방대한 요구사항 문서를 대체한 것과 같은 원리다.

```
┌────────────────────────────────────────────────────┐
│           마이크로 스펙 적용 기준 4가지               │
├────────────────────────────────────────────────────┤
│                                                    │
│  1. 작업 복잡도                                     │
│     └─ 높을수록 더 작게 분해, 상호의존성 고려         │
│                                                    │
│  2. 도구/방법론                                     │
│     ├─ AIDLC/BMAD → 에픽/복잡 기능 레벨             │
│     └─ Kiro/Spec-kit → 에픽/사용자 스토리 레벨       │
│                                                    │
│  3. 조직 구조                                       │
│     └─ 팀 수, 검토자 역할에 따라 크기 조정            │
│                                                    │
│  4. 가치/결과                                       │
│     └─ 독립적으로 데모·테스트 가능해야 함             │
│                                                    │
└────────────────────────────────────────────────────┘
```

### 5. 저자의 실전 워크플로우 (Kiro CLI + IDE)

저자가 2개월간 실전에서 검증한 6단계 워크플로우다.

```
Step 1              Step 2           Step 3           Step 4
PRODUCT-BRIEF.md    에픽 분해         스티어링/규칙      스펙/디자인/계획
┌──────────┐       ┌──────────┐     ┌──────────┐     ┌──────────────┐
│ 비전·목표  │──────▶│ 마이크로   │────▶│ 기술 스택  │────▶│ 사용자 스토리  │
│ 페르소나   │       │ 에픽 분해  │     │ 아키텍처   │     │ 수용 기준     │
│ MoSCoW    │       │ (선택적)  │     │ 원칙·규칙  │     │ (EARS 표기법) │
│ 에픽 목록  │       └──────────┘     └──────────┘     └──────┬───────┘
└──────────┘                                                │
                                                            ▼
                    Step 6           Step 5
                    검증              개발
                    ┌──────────┐     ┌──────────────┐
                    │ 코드 품질  │◀────│ 특화 에이전트  │
                    │ 보안 검사  │     │ (Python/Infra │
                    │ 수용기준   │     │  /Frontend)  │
                    │ 적합성    │     │ 현재 에픽만   │
                    └──────────┘     │ 맥락에 포함   │
                                     └──────────────┘
```

**EARS 표기법 수용 기준 예시:**
```
WHEN [사용자가 로그인 버튼을 클릭하면],
THE [인증 시스템] SHALL [OAuth 2.0 플로우를 시작하고 토큰을 반환한다]
```

**핵심 원칙:** 현재 작업 중인 에픽의 명세만 LLM 맥락에 포함하고, 미래 에픽은 제외한다. 이로써 토큰 한계 도달을 방지하고 코드 품질을 유지한다.

### 6. 마이크로 스펙의 측정된 효과

저자가 2개월간 실전 적용 후 보고한 개선 사항이다.

| 측면 | 풀 스펙 | 마이크로 스펙 | 개선 |
|------|---------|-------------|------|
| 스펙 검토 부담 | 수천 줄 마크다운 | 기능 단위 소량 | 지루함·과부하 해소 |
| LLM 맥락 효율 | 토큰 한계 빈번 도달 | 적절한 맥락 유지 | 품질 안정화 |
| 코드 수정 요청 | 많음 | ~75% 감소 | 대폭 개선 |
| 민첩성 | 사전 계획에 고정 | 진행 중 범위·방향 조정 가능 | 반복 개발 복구 |

### 7. 기존 프로젝트(Brownfield)에서의 적용

신규 프로젝트(Greenfield)가 아닌 기존 프로젝트에서는 원본 스펙이나 브리프 대신 **소스 코드와 기존 문서를 입력으로 사용**하여 마이크로 스펙을 생성한다.

### 8. 핵심 결론: 스펙이 아니라 규모가 문제다

```
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  "Agile은 포괄적 문서화보다 작동하는 소프트웨어를 강조했고,   │
│   AI는 둘 다 빠르게 만들 뿐이다.                             │
│   문제는 스펙 여부가 아닌 규모(scope)다."                     │
│                                                             │
│  ┌───────────────────────────────────────────────────────┐  │
│  │  행동 지침:                                            │  │
│  │  1. 스펙을 마이크로로 유지하라                           │  │
│  │  2. 계속 반복하라                                       │  │
│  │  3. 피할 수 없는 변화를 수용하라                          │  │
│  │  4. 민첩함을 유지하라                                    │  │
│  └───────────────────────────────────────────────────────┘  │
│                                                             │

```

## 내가 얻은 인사이트

### 방법론 관점
1. **"도구가 방법론을 결정하게 두지 마라"**
   - Kiro, Spec Kit 등의 도구가 풀 스펙을 기본 워크플로우로 제시한다고 해서 그대로 따를 필요는 없다. 도구의 기능을 마이크로 단위로 분할 적용하는 것이 핵심이다.

2. **30년 전 교훈의 재발견**
   - AI가 속도를 바꿨을 뿐, 소프트웨어 공학의 근본 원리는 변하지 않았다. "빠르게 잘못된 것을 만드는 것"은 느리게 잘못된 것을 만드는 것보다 나을 것이 없다. Agile이 워터폴을 대체한 이유를 잊지 말아야 한다.

3. **마이크로 스펙은 사용자 스토리의 진화형**
   - Agile의 사용자 스토리가 요구사항 문서를 대체했듯, 마이크로 스펙은 AI 시대에 사용자 스토리를 구조화한 형태다. EARS 표기법을 적용한 수용 기준은 사람과 LLM 모두에게 명확한 계약을 제공한다.

### 실무 적용 관점
1. **LLM 맥락 관리가 곧 개발 품질**
   - 과도한 명세 → 토큰 한계 → 품질 저하라는 인과 관계가 실측으로 확인되었다. 현재 작업 에픽만 맥락에 포함하는 전략은 즉시 적용할 수 있는 실천적 가이드라인이다.

2. **코드 수정 요청 75% 감소의 의미**
   - 단순히 명세를 작게 만든 것만으로 AI 생성 코드의 품질이 대폭 향상되었다. 이는 "더 많은 맥락 = 더 좋은 결과"라는 직관에 반하며, "적절한 맥락 = 최적의 결과"라는 원칙을 실증한다.

3. **특화 에이전트 분리 패턴**
   - Python 개발자, 인프라 개발자, 프론트엔드 개발자 등 역할별 에이전트를 분리하고 각각에 맞춤형 MCP 서버와 스티어링을 제공하는 패턴은 대규모 프로젝트에서 에이전트 활용의 현실적 모범 사례다.
