### 2. SDD의 세 가지 구현 수준

저자는 SDD를 단일 개념이 아닌, 점진적으로 심화되는 세 단계로 구분한다.

```
┌─────────────────────────────────────────────────────────────────────┐
│                    SDD 구현 수준 스펙트럼                             │
│                                                                     │
│  Level 1           Level 2              Level 3                     │
│  Spec-first        Spec-anchored        Spec-as-source              │
│  ┌───────────┐    ┌──────────────┐     ┌──────────────────┐        │
│  │ 스펙 작성   │    │ 스펙이 유지    │     │ 스펙이 주 소스 파일 │        │
│  │ → 코드 생성 │    │ 기능 진화와    │     │ 인간은 코드를      │        │
│  │ → 스펙 폐기 │    │ 함께 편집됨   │     │ 직접 편집하지 않음  │        │
│  └───────────┘    └──────────────┘     └──────────────────────┘    │
│                                                                     │
│  Kiro ●            Spec-kit △           Tessl ◎                     │
│  (도달)            (지향하나 미흡)       (명시적 추구, 비결정성 문제)   │
└─────────────────────────────────────────────────────────────────────┘
```

- **Spec-first**: 스펙을 작성하여 즉시 활용하고 폐기. 새 작업마다 새 스펙 작성
- **Spec-anchored**: 스펙이 작업 완료 후에도 유지되어 기능 진화 과정에서 지속적으로 편집
- **Spec-as-source**: "스펙이 시간이 지나도 주요 소스 파일이며, 인간은 스펙만 편집하고 코드를 직접 건드리지 않는다"

핵심 구분: **Memory Bank vs Spec**
| 구분 | Memory Bank | Spec |
|------|-------------|------|
| 범위 | 모든 AI 세션에 적용 | 특정 기능 생성/변경에만 관련 |
| 내용 | 규칙, 아키텍처, 코드베이스 개요 | 행동 지향적 기능 명세 |
| 예시 | Kiro의 steering, Cline의 memory bank | 사용자 스토리, 수용 기준, 데이터 모델 |

### 3. Kiro (AWS) — 가장 경량의 도구

Kiro는 세 도구 중 가장 가볍고 직관적인 워크플로우를 제공한다.

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│ Requirements │────▶│    Design    │────▶│    Tasks     │
│              │     │              │     │              │
│ "As a..."    │     │ 컴포넌트 아키 │     │ TODO 목록    │
│ 사용자 스토리 │     │ 텍처, 데이터  │     │ 요구사항 번호 │
│ GIVEN/WHEN/  │     │ 흐름, 에러    │     │ 역추적 가능   │
│ THEN 수용기준 │     │ 핸들링, 테스트│     │              │
└──────────────┘     └──────────────┘     └──────────────┘
       ↑
  마크다운 문서 3개로 구성
```

**Memory Bank ("Steering" 폴더)**:
- `product.md` — 제품 개요
- `structure.md` — 코드 구조
- `tech.md` — 기술 스택

**Requirements 문서 구조**:
- "As a... I want... so that..." 형식의 사용자 스토리
- GIVEN/WHEN/THEN Gherkin 구문의 수용 기준
- 각 요구사항에 번호 식별자 부여

**Design 문서 섹션**:
- Component Architecture Diagram
- Data Flow (접힘 섹션)
- Data Models / Error Handling / Testing Strategy
- Implementation Approach / Migration Strategy

**관찰된 문제 — 문제 규모 부적합**:

> 작은 버그 수정을 요청했을 때, Kiro는 **4개의 사용자 스토리와 16개의 수용 기준**을 생성했다. "호두를 까기 위해 슬레지해머를 쓴 격이다."

예시로 생성된 불필요한 스토리:
```
"As a developer, I want the transformation function to handle edge
cases gracefully, so that the system remains robust when new category
formats are introduced."
```

**평가**: Spec-first 수준만 구현. 경량이지만 모든 규모의 문제에 동일한 무거운 프로세스를 적용하는 한계.

### 4. Spec-kit (GitHub) — 가장 정교한 구조

GitHub이 배포한 CLI 기반 도구로, 여러 코딩 어시스턴트와 슬래시 커맨드로 통합된다.

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐     ┌──────────┐
│ Constitution │────▶│   Specify    │────▶│     Plan     │────▶│  Tasks   │
│              │     │              │     │              │     │          │
│ 불변의 원칙   │     │ 기능 명세     │     │ 구현 계획     │     │ 작업 목록 │
│ 모든 변경에   │     │ 리서치       │     │ 체크리스트    │     │          │
│ 적용됨       │     │ API/데이터   │     │              │     │          │
└──────────────┘     └──────────────┘     └──────────────┘     └──────────┘
                            ↕ (순환적)
```

**Constitution (헌법) 개념**:
> "모든 변경에 항상 적용되어야 하는 '불변의' 상위 수준 원칙. 강력한 규칙 파일로서 워크플로우 전반에 깊이 통합됨."

**생성되는 파일 구조**:
```
.github/prompts/          ← 커맨드 파일들
.specify/
  ├── memory/
  ├── scripts/
  └── templates/
specs/001-when-a-user/    ← 개별 스펙 폴더
  ├── spec
  ├── plan
  ├── tasks
  ├── data-model
  ├── research
  ├── api
  ├── component
  └── ... (8개 이상의 파일)
```

**체크리스트 기반 추적**:
- Content quality (내용 품질)
- Requirement completeness (요구사항 완전성)
- Execution status (실행 상태)
- Constitution violations (헌법 위반)
- User clarifications (사용자 확인)
- Research tasks (리서치 작업)

**GitHub의 공식 비전**: 스펙이 "프로젝트와 함께 진화하는 살아있는 실행 가능한 아티팩트"가 되어야 한다.

**구현과 비전의 괴리**:
각 스펙에 대해 **별도의 브랜치를 생성**하여, 스펙이 "기능의 수명" 동안이 아닌 "변경 요청의 수명" 동안만 살아있는 아티팩트로 취급됨. Spec-anchored를 지향하지만 실제로는 Spec-first에 가까움.

**관찰된 문제들**:

| 문제 | 상세 |
|------|------|
| 마크다운 과잉 | 기존 코드와 중복되는 반복적이고 장황한 파일 대량 생성 |
| 검토 피로 | "솔직히 이 모든 마크다운 파일보다 코드를 검토하겠다" |
| 에이전트 무시 | 기존 클래스 설명 지시를 무시하고 중복 재생성 |
| 과잉 해석 | Constitution 조항을 과도하게 따라 "필요 이상으로 폭주" |

### 5. Tessl — 가장 야심찬 접근

프라이빗 베타 상태의 Tessl은 Spec-anchored 및 Spec-as-source를 **명시적으로** 추구한다.

```
┌─────────────────────┐          ┌─────────────────────┐
│     Spec 파일        │  1:1     │    Code 파일         │
│                     │  매핑    │                     │
│  @generate          │────────▶│  // GENERATED FROM   │
│  @test              │          │  // SPEC - DO NOT    │
│                     │          │  // EDIT             │
│  API 섹션           │          │                     │
│  인터페이스 정의     │          │  JavaScript 코드     │
└─────────────────────┘          └─────────────────────┘
        │                                │
        │    tessl build                 │
        └────────────────────────────────┘
        │    tessl document --code       │
        └────────────◀──────────────────┘
              (역공학: 코드→스펙)
```

**핵심 특징**:
- **1:1 매핑**: 하나의 스펙이 하나의 코드 파일로 생성 (향후 멀티파일 지원 계획)
- **코드 주석**: `// GENERATED FROM SPEC - DO NOT EDIT`
- **태그 시스템**: `@generate` (코드 생성 지시), `@test` (테스트 요구사항)
- **CLI = MCP 서버**: CLI 커맨드가 Model Context Protocol 서버로 이중 동작
- **역공학**: `tessl document --code ...js`로 기존 JS에서 스펙 추출

**Memory Bank 구조** (`.tessl/framework/`):
- 4개 기본 파일 + KNOWLEDGE.md + AGENTS.md

**비결정성 문제**:
> 동일한 스펙에서 코드를 여러 번 생성했을 때 매번 다른 결과가 나왔다. 스펙을 반복적으로 다듬어 반복성을 높여야 했으며, 이 과정이 "모호하지 않고 완전한 명세를 작성하는 것의 함정과 도전"을 상기시켰다.

### 6. 세 도구 비교 종합

| 차원 | Kiro | Spec-kit | Tessl |
|------|------|----------|-------|
| 배포 | IDE 네이티브 | CLI, VS Code 통합 | CLI + MCP 서버 |
| 복잡도 | 경량 (마크다운 3개) | 복잡 (8개+ 파일) | 중간 (1:1 매핑) |
| SDD 수준 | Spec-first | Spec-first (anchored 지향) | Spec-anchored/as-source 추구 |
| Memory Bank | steering/ (3 파일) | constitution + .specify/ | .tessl/framework/ (6 파일) |
| 고유 개념 | Gherkin 수용 기준 | Constitution (불변 원칙) | @generate/@test 태그, 코드 역공학 |
| 핵심 문제 | 규모 부적합 | 마크다운 과잉/검토 피로 | 비결정성 |

### 7. 통제의 착각 (Illusion of Control)

정교한 파일 구조, 템플릿, 프롬프트, 워크플로우, 체크리스트에도 불구하고 AI 에이전트가 지시를 제대로 따르지 않는 현상이 반복 관찰되었다.

```
┌────────────────────────────────────────────────────────────┐
│              통제의 착각 구조                                │
│                                                            │
│  [정교한 스펙]  ──▶  [AI 에이전트]  ──▶  [예상과 다른 결과]  │
│                                                            │
│  관찰된 실패 패턴:                                          │
│  1. 컨텍스트 윈도우가 커도 모든 지시를 따르지 않음            │
│  2. 기존 코드 설명 지시를 무시하고 중복 생성                  │
│  3. 지시를 과도하게 해석하여 범위 폭주                       │
│                                                            │
│  핵심 교훈:                                                 │
│  "더 많은 문서 ≠ 더 나은 통제"                               │
│  "작고 반복적인 단계가 통제의 최선"                           │
└────────────────────────────────────────────────────────────┘
```

> "과거가 보여준 것은, 우리가 구축하는 것을 통제하는 최선의 방법은 작고 반복적인 단계라는 것이다. 그래서 나는 과도한 사전 스펙 설계가 좋은 아이디어라는 데 매우 회의적이다. 특히 지나치게 장황할 때."

### 8. MDD(Model-Driven Development)와의 역사적 비교

저자는 커리어 초기에 MDD 프로젝트에 참여한 경험을 바탕으로 SDD와 MDD의 유사성을 분석한다.

```
MDD (과거)                              SDD (현재)
┌──────────────┐                       ┌──────────────┐
│ UML/텍스트 DSL │                       │ 자연어 스펙    │
│ 커스텀 코드    │                       │ LLM 기반      │
│ 제너레이터    │                       │ 코드 생성      │
└──────┬───────┘                       └──────┬───────┘
       │                                      │
  제약: 파싱 가능한                        자유: 자연어로
  정의된 언어 필요                         제약 없음
       │                                      │
  결과: 비즈니스 앱에서                    결과: 비결정성
  실패 — 어색한 추상                       문제 도입
  수준과 과도한 오버헤드                         │
       │                                      │
       └──────────────┐  ┌───────────────────┘
                      ▼  ▼
              ┌────────────────────┐
              │ 최악의 결합 위험:    │
              │ MDD의 경직성        │
              │   +               │
              │ LLM의 비결정성     │
              └────────────────────┘
```

**MDD가 실패한 이유**: "비즈니스 애플리케이션에서 어색한 추상 수준에 놓여 과도한 오버헤드와 제약만 만들었다."

**SDD가 MDD와 다른 점**: LLM이 사전 정의된 파싱 가능한 스펙 언어의 제약과 정교한 코드 제너레이터의 필요성을 제거함.

**MDD에서 잃어버린 것**: 구조화된 MDD에서는 "도구 지원으로 유효하고 완전하며 일관된 스펙을 작성"할 수 있었으나, 자연어 스펙에서는 이런 자동 검증이 불가능함.

**Verschlimmbesserung** (독일어: 개선 시도가 오히려 상황을 악화시키는 것) — 정교한 파일 구조와 프로세스가 오히려 개발 워크플로우를 방해할 수 있다는 우려.

### 9. 미해결 질문들

저자는 SDD 도구들이 아직 답하지 못한 핵심 질문들을 제기한다.

```
┌──────────────────────────────────────────────────────┐
│          SDD의 적합 영역은?                            │
│                                                      │
│              문제의 명확성                              │
│          낮음 ◀──────────▶ 높음                       │
│    높  ┌─────────┬─────────┐                         │
│    음  │   ???   │   ???   │                         │
│  문    ├─────────┼─────────┤                         │
│  제    │   ???   │   ???   │                         │
│  규    └─────────┴─────────┘                         │
│  모  낮음                                             │
│                                                      │
│  모든 사분면에 ??? — 아직 누구도 답하지 못했다           │
└──────────────────────────────────────────────────────┘
```

| 질문 | 상세 |
|------|------|
| 문제 규모 적응 | 작은 버그부터 대규모 기능까지 어떻게 워크플로우를 조절할 것인가? |
| 대상 사용자 | 개발자가 제품 분석까지 수행하는 것을 가정하는가? |
| 기능/기술 분리 | 언제 기능 수준을 유지하고 언제 기술 세부사항을 추가할 것인가? |
| 의미적 확산 | "스펙"이 "상세 프롬프트"와 동의어로 변해가는 문제 |
| 장기 효용 | 실제 프로젝트에서 장기간 사용했을 때의 데이터가 없음 |

### 10. 핵심 결론

1. **Spec-first는 가치 있다**: AI 코딩에 구조화된 입력을 제공하는 것은 많은 상황에서 유효함
2. **도구들은 아직 미성숙**: 검토 과부하와 환각을 오히려 증폭시킬 위험
3. **"효과적인 SDD 도구는 매우 좋은 스펙 검토 경험을 제공해야 한다"**
4. **"효과적인 SDD 도구는 최소한 다양한 규모와 유형의 변경에 대해 몇 가지 핵심 워크플로우의 유연성을 제공해야 한다"**
5. **과거의 교훈을 활용하라**: 코드 생성 시도의 역사(MDD 등)에서 배울 것이 많다

## 내가 얻은 인사이트

### 도구 평가 관점
1. **"모든 SDD 도구가 Spec-first 이상을 달성하지 못했다"**
   - Kiro는 명시적으로 Spec-first, Spec-kit은 Spec-anchored를 지향하지만 브랜치 기반 구조로 인해 실질적으로 Spec-first, Tessl만이 Spec-as-source를 시도하지만 비결정성 문제로 실용성이 검증되지 않았다. 이는 SDD의 약속과 현실 사이의 격차가 크다는 것을 의미한다.

2. **Constitution 패턴의 양면성**
   - Spec-kit의 Constitution(불변 원칙) 개념은 이론적으로 강력하지만, 실제로는 에이전트가 이를 과잉 해석하여 범위를 폭주시키는 문제를 일으켰다. 규칙을 만드는 것과 규칙을 제대로 따르게 하는 것은 완전히 다른 문제다.

3. **1:1 스펙-코드 매핑의 한계**
   - Tessl의 하나의 스펙 = 하나의 코드 파일 접근법은 깔끔하지만, 현실의 소프트웨어는 횡단 관심사(cross-cutting concerns), 공유 유틸리티, 복합 컴포넌트로 가득하다. 이 제약이 실제 프로젝트에서 어떻게 작동할지는 미지수다.

### 방법론 관점
1. **"더 많은 문서 ≠ 더 나은 통제"의 실증**
   - 이전 아티클(마이크로 스펙)에서 제기된 "LLM 맥락 과부하 → 품질 저하" 가설이 이 글에서 다른 각도로 확인된다. Spec-kit이 8개 이상의 파일을 생성했지만 에이전트는 여전히 지시를 무시했다. 통제의 수단은 문서의 양이 아니라 문서의 적절한 규모와 반복적 피드백이다.

2. **MDD 실패로부터의 핵심 교훈**
   - MDD는 "도구 지원으로 유효하고 완전한 스펙 작성"이 가능했지만 실패했다. SDD는 자연어의 자유도를 얻는 대신 자동 검증 능력을 잃었다. 즉, MDD보다 더 어려운 문제를 풀어야 하면서 도구 지원은 더 약하다. 이 비대칭이 SDD의 근본적 도전이다.

3. **기능/기술 명세 분리의 비현실성**
   - "기능 스펙과 기술 구현을 분리하면 기술 스택만 교체 가능"이라는 SDD의 이상은, 저자의 지적처럼 소프트웨어 산업이 수십 년간 제대로 해내지 못한 과제다. AI 도구가 이를 갑자기 해결해주리라 기대하기 어렵다.

### 실무 적용 관점
1. **SDD 도구 선택 기준 제안**
   - Kiro: 간단한 신규 기능 개발에 경량 가이드가 필요할 때. 단, 작은 작업에는 과잉
   - Spec-kit: 팀 단위로 일관된 원칙을 적용하고 싶을 때. 단, 마크다운 검토 비용 감수 필요
   - Tessl: 코드 생성을 완전히 자동화하려는 실험적 프로젝트에. 단, 비결정성 리스크 인지 필요
   - **또는**: 도구 없이 자체 마이크로 스펙 패턴을 적용하는 것이 현시점에서 가장 실용적일 수 있다

2. **"마크다운보다 코드를 검토하겠다"의 함의**
   - Distinguished Engineer가 스펙 마크다운 검토를 코드 검토보다 더 고통스럽게 느꼈다는 것은 SDD 도구의 UX 실패를 보여준다. 스펙 검토 경험이 코드 리뷰만큼 성숙하지 않으면 SDD의 현실적 채택은 어렵다.
