## AI 요약

### 1. Spec-Driven Development란?

Spec-Driven Development(SDD)는 명세서를 **"실행 가능한 산출물(Executable Artifact)"**로 취급하는 개발 방법론이다. 기존의 수동적 문서와 달리, SDD의 명세는 빌드 검증 중에 실행되며 구현이 이탈할 경우 **자동으로 빌드를 실패**시킨다.

> "명세서가 기본 개발 산출물로 기능하며, 코드에서 파생되는 것이 아니라 코드가 명세서에서 파생된다."

| 항목 | 내용 |
|------|------|
| 핵심 원칙 | 명세 = 실행 가능한 검증 게이트 |
| 등장 배경 | AI 코드 생성 시대의 아키텍처 일관성 문제 |
| 주요 도구 | GitHub Spec Kit, Augment Code |
| 대상 | 엔터프라이즈 다중팀 환경, 규제 준수 필요 조직 |
| 보안 근거 | AI 생성 코드 문제의 약 2%가 보안 취약점, 그 중 56~93%가 심각 |

### 2. 기존 개발 방법론과의 차이

SDD는 TDD, BDD, Vibe Coding과 본질적으로 다른 접근 방식을 취한다.

```
┌─────────────────────────────────────────────────────────────────────┐
│                     개발 방법론 스펙트럼                               │
│                                                                     │
│  구조 없음                                            구조 최대화     │
│  ◀──────────────────────────────────────────────────────────▶       │
│                                                                     │
│  Vibe Coding    TDD         BDD              SDD                    │
│  (자유 생성)   (단위 검증)  (행위 명세)      (아키텍처 계약)           │
│                                                                     │
│  "일단 만들어"  "함수가     "Given-When-     "명세 위반 시             │
│               맞는지 확인"   Then 시나리오"    빌드 실패"              │
└─────────────────────────────────────────────────────────────────────┘
```

| 비교 항목 | TDD | BDD | SDD |
|-----------|-----|-----|-----|
| **주요 산출물** | 단위 테스트 | Given-When-Then 시나리오 | 실행 가능한 명세 |
| **검증 범위** | 개별 함수 | 사용자 행위 | 시스템 전체 아키텍처 계약 |
| **AI 거버넌스** | 없음 | 없음 | 헌법적 제약 및 체크포인트 |
| **문서 역할** | 테스트 코드가 문서 | 종종 문서화로만 활용 | 빌드 게이트로 실행 |
| **한계** | 아키텍처 위반 미감지 | 실행 강제 메커니즘 부재 | 오버헤드 발생 가능 |

### 3. SDD의 세 가지 패턴

SDD는 명세의 역할과 코드의 관계에 따라 세 가지 패턴으로 분류된다.

```
┌───────────────────────────────────────────────────────────────┐
│                    SDD 패턴 성숙도 모델                         │
│                                                               │
│  Spec-First ──────▶ Spec-Anchored ──────▶ Spec-as-Source      │
│  (입문)              (거버넌스)              (완전 자동화)       │
│                                                               │
│  "명세가 AI를        "명세가 체크포인트로     "명세 자체가         │
│   가이드한다"         거버넌스를 수행한다"     소스코드다"          │
└───────────────────────────────────────────────────────────────┘
```

| 패턴 | 명세 역할 | 코드 역할 | 최적 사용 시기 |
|------|----------|----------|--------------|
| **Spec-First** | AI 출력 가이드 및 제약 | 주요 산출물 | SDD 채택 초기 단계 |
| **Spec-Anchored** | 체크포인트/헌법적 제약으로 거버넌스 | 검증된 산출물 | 규제 감시/다중팀 조율 |
| **Spec-as-Source** | 문자 그대로의 소스코드 | 생성된 산출물 | API-우선 도메인 |

### 4. SDD 5단계 워크플로우

GitHub Spec Kit 기반의 표준 워크플로우이다.

```
Phase 1              Phase 2              Phase 3
/speckit.specify     /speckit.plan        /speckit.tasks
┌──────────────┐    ┌──────────────┐    ┌──────────────┐
│ 실행 가능한    │───▶│ 구현 계획     │───▶│ 테스트 가능한  │
│ 명세 정의     │    │ 수립         │    │ 작업 분해     │
│              │    │              │    │              │
│ - 비즈니스    │    │ - 아키텍처    │    │ - 격리 가능한  │
│   컨텍스트   │    │   결정       │    │   구현 단위    │
│ - 사용자     │    │ - 기술 선택   │    │ - 독립 테스트  │
│   요구사항   │    │ - 요구사항    │    │   가능        │
│ - 성공 기준  │    │   추적       │    │              │
│ - 검증 규칙  │    │              │    │              │
└──────────────┘    └──────────────┘    └──────────────┘
                                              │
                         ┌────────────────────┘
                         ▼
Phase 4              Phase 5
/speckit.implement   명세 기반 디버깅
┌──────────────┐    ┌──────────────┐
│ AI 에이전트    │───▶│ 명세 수정     │
│ 실행         │    │ (코드 수정 X) │
│              │    │              │
│ - 명세 제약   │    │ - 코드가 아닌  │
│   컨텍스트   │    │   명세를 고침  │
│ - 기능+아키   │    │ - 모든 생성   │
│   텍처 검증  │    │   코드에 적용  │
│ - merge 전   │    │              │
│   위반 감지  │    │              │
└──────────────┘    └──────────────┘
```

**Phase 5의 핵심 철학:**

> "AI 생성 코드의 문제는 명세의 격차로 인한 결과이다. AI의 비결정성으로 인해, 그 격차는 코드 재생성 시마다 다른 형태로 재발생한다."

따라서 **코드를 고치는 것이 아니라 명세를 수정**하면, 이후 모든 생성 코드에 일관성 있게 적용된다.

```
결제 엔드포인트 명세 수정 예시:

endpoint: POST /charges
constraints:
  - idempotency_key: required    # 변경 전: optional → 프로덕션 사고 후 수정
  - retry_window: 24h            # 새로 추가된 아키텍처 제약

→ 명세 수정 한 번으로 모든 AI 생성 코드가 자동으로 이 제약 준수
```

### 5. AI 코드 보안과 SDD

AI 생성 코드의 보안 취약점은 SDD가 해결하려는 핵심 문제 중 하나이다.

```
┌─────────────────────────────────────────────────┐
│          AI 생성 코드의 보안 위협 구조             │
│                                                 │
│  AI 코드 생성 ──▶ 단위 테스트 통과 ✅             │
│       │                                         │
│       ├──▶ 아키텍처 패턴 위반 ❌ (TDD 미감지)     │
│       ├──▶ 서비스 간 경계 위반 ❌ (TDD 미감지)     │
│       └──▶ 보안 취약점 ❌ (TDD 미감지)            │
│                                                 │
│  SDD 명세 검증 ──▶ 아키텍처 위반 감지 ✅          │
│                ──▶ 경계 위반 감지 ✅              │
│                ──▶ 보안 제약 강제 ✅              │
└─────────────────────────────────────────────────┘
```

**학술 연구 근거:**
- AI 생성 코드 문제의 약 **2%가 보안 취약점**
- 그 중 **56~93%가 심각한 수준**
- 근본 원인: LLM의 파일 간 데이터 흐름 추적 한계, 의미론적 이해 격차

### 8. 엔터프라이즈 채택 전략

SDD 도입은 단계적으로 진행해야 한다.

**아키텍처 선행조건:**
- 명확한 서비스 경계와 기존 API 계약 (OpenAPI, AsyncAPI, GraphQL SDL)
- 명세 이탈 시 빌드 실패 처리 가능한 CI/CD 파이프라인
- 최소 하나의 명세 형식 숙달

**도구 선행조건:**
- 지원되는 AI 에이전트: Claude Code, GitHub Copilot, Gemini CLI
- GitHub Spec Kit (22+ AI 플랫폼 지원, 72.7k GitHub 스타)

```
채택 단계:

Stage 1: 소규모                Stage 2: 중규모              Stage 3: 대규모
┌────────────────┐           ┌────────────────┐          ┌────────────────┐
│ 단일 서비스     │           │ 다중 서비스     │          │ 에이전트        │
│ + 기존 OpenAPI  │    ──▶   │ + 다중 팀       │   ──▶   │ 오케스트레이션  │
│                │           │                │          │                │
│ Spec-First     │           │ Spec-Anchored  │          │ 자동화          │
│ 패턴 적용      │           │ + 헌법적       │          │ 워크플로우      │
│                │           │   거버넌스 추가 │          │                │
└────────────────┘           └────────────────┘          └────────────────┘
```

### 9. SDD가 부적합한 경우

SDD는 만능이 아니다. 아래 상황에서는 다른 접근이 더 효과적이다.

| 상황 | 부적합 이유 | 대안 |
|------|-----------|------|
| **탐색적 R&D** | 요구사항을 미리 알 수 없음 | 프로토타이핑, Vibe Coding |
| **빠른 프로토타이핑** | 명세 작성 오버헤드 > 개발 시간 | Lean 실험, 대화형 개발 |
| **소규모 팀** | 명세 오버헤드가 생산성 저하 | TDD/BDD로 충분 |
| **레거시 유지보수** | 암묵적 비즈니스 로직 역엔지니어링 비용 높음 | 점진적 리팩토링 |

## 내가 얻은 인사이트

### 아키텍처 거버넌스 관점
1. **"명세를 고치면 코드가 따라온다"는 패러다임 전환**
   - 전통적 디버깅은 "코드를 고친다"이지만, SDD는 "명세를 고친다"로 전환한다. AI의 비결정성 때문에 코드 수정은 재생성 시 다른 형태로 재발생하지만, 명세 수정은 모든 생성 사이클에 일관성 있게 적용된다. 이는 단순한 방법론 차이가 아니라, AI 시대 소프트웨어 유지보수의 근본적 사고방식 변화이다.

2. **TDD가 놓치는 "아키텍처 계약" 영역**
   - TDD는 개별 함수의 정확성을 보장하지만, 서비스 간 경계 위반이나 아키텍처 패턴 위반은 감지하지 못한다. SDD는 이 빈 공간을 "실행 가능한 명세"로 채운다. TDD와 SDD는 경쟁이 아니라 보완 관계이다.

3. **세 가지 패턴(First → Anchored → Source)의 성숙도 모델**
   - 조직이 SDD를 채택할 때 한 번에 Spec-as-Source로 가는 것이 아니라, Spec-First로 시작해 점진적으로 성숙시키는 경로가 현실적이다. 이는 기존의 Marmelab 아티클에서 비판한 "과도한 명세 오버헤드" 문제를 완화하는 실용적 접근이다.

### 엔터프라이즈 현실 관점
1. **EU AI Act 2026이 SDD를 강제할 수 있다**
   - 2026년 8월부터 시행되는 EU AI Act는 고위험 AI 시스템에 대한 의무 준수를 요구하며, 위반 시 전역 매출의 7% 또는 €35M 벌금이다. SDD의 명세 → 구현 추적 체계는 이 규제 준수의 자연스러운 증거가 된다. 규제가 방법론 채택을 가속화하는 외부 동력이 될 수 있다.

2. **다중 저장소 문제가 SDD의 실제 병목**
   - 현재 GitHub Spec Kit은 단일 저장소에 명세를 함께 저장하지만, 실제 엔터프라이즈는 마이크로서비스, 공유 라이브러리, 인프라 저장소에 걸쳐 있다. 이 갭을 해결하지 않으면 SDD는 소규모 프로젝트에만 적용 가능하다. Augment Code의 Context Engine이 이 문제를 해결한다고 주장하지만, 이는 상용 제품에 대한 종속성을 의미한다.

### 기존 SDD 비판 아티클과의 대비 관점
1. **Marmelab의 비판 vs Augment Code의 옹호**
   - Marmelab(François Zaninotto)은 SDD를 "Waterfall의 부활"로 비판하며 Natural Language Development를 대안으로 제시했다. 반면 이 아티클은 SDD를 엔터프라이즈 필수 방법론으로 옹호한다. 두 관점 모두 타당한 맥락이 있다: 소규모/탐색적 개발에는 Marmelab의 비판이 맞고, 대규모/규제 환경에는 Augment Code의 주장이 맞다. 진실은 "맥락에 따라 다르다"이다.

2. **"부적합 사례"를 명시한 것이 신뢰를 높인다**
   - 이 아티클이 R&D, 프로토타이핑, 소규모 팀, 레거시 유지보수에서 SDD가 부적합하다고 인정한 점은 Marmelab 비판의 핵심 논점을 수용한 것이다. 이는 오히려 SDD가 적합한 영역(엔터프라이즈 다중팀, 규제 준수)에서의 설득력을 강화한다.
