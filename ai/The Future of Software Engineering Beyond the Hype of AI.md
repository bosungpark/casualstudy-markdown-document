## AI 요약

### 1. 이 패널은 무엇인가?

#### Tom DeMarco — "기술보다 인간이 먼저다"
- The Atlantic Systems Guild, 구조적 분석(Structured Analysis) 선구자
- **핵심**: 기술의 발전보다 **인간의 충족감과 팀워크의 즐거움**이 소프트웨어 성공의 본질
- 우아한 제품 창조, 동료의 인정, 협업의 기쁨이 빠지면 단순 거래적 활동에 불과
- "인간으로서의 깊은 필요가 존중되어야 진정한 성공이 가능하다"

#### Irene Manotas — "AI는 도구, 인간은 설계자"
- IBM T.J. Watson Research Center, AI4Code 그룹
- **핵심**: LLM이 코드 완성, 생성, 테스트, 추론을 변혁하고 있지만, **인간 엔지니어는 필수**
- 엔지니어의 초점이 이동: 구현 → **아키텍처, 요구사항 분석, 성능 평가, 복잡한 시스템 상호작용 관리**
- Agentic AI가 프로토타이핑, 버그 발견, 설계 개선을 가속화

#### Bertrand Meyer — "근본적 문제는 변하지 않는다"
- Eiffel 언어 창시자, Design by Contract 개념 제안자
- **핵심**: 정확성(Correctness), 확장성(Extendibility), 재사용성(Reusability) 문제는 AI 시대에도 불변
- 체계적 요구공학 + Design by Contract + 지원 표기법 + 고급 도구의 **종합적 방법론** 필요
- AI는 가치 있는 보조 수단이지만, 핵심 난제를 제거하지는 못함

#### Adriana Meza Soria — "AI는 추상화 진화의 연장선"
- MIT-IBM Watson AI Lab, Granite LLM 개발 핵심 멤버
- **핵심**: 어셈블리 → 고급언어, 기본 에디터 → IDE, 워터폴 → 애자일처럼 **지속적 추상화 상승의 연장**
- AI의 두 경로: ① 기존 도구 향상 (IDE, 저장소) ② 혁신적 도구 설계 (AI 에이전트, 선언적 언어)
- 코딩 외 영역(프로젝트 관리, 시스템 설계, 문서화)은 결과물 캡처 부족으로 AI 적용이 제한적

```
┌──────────────────────────────────────────────────────┐
│            소프트웨어 추상화 진화 타임라인              │
│                                                      │
│  1950s      1970s      1990s      2010s     2020s+   │
│    │          │          │          │         │       │
│  Assembly  Structured  OOP/IDE   Agile/    GenAI     │
│    │       Programming   │       Cloud       │       │
│    │          │          │          │         │       │
│    ▼          ▼          ▼          ▼         ▼       │
│  [저수준] ──────────────────────────────► [고수준]    │
│                                                      │
│  구현 중심 ─────────────────────────► 의도 중심       │
└──────────────────────────────────────────────────────┘
```

#### Landon Noll — "SE를 전문 엔지니어링 수준으로 격상하라"
- 전 Cisco Security Architect, 소수 발견 세계 기록 8개 보유
- **핵심**: 의료, 법률, 항공우주와 달리 **SE에는 면허, 인증, 책임 체계가 부재**
- 제품 라이선스가 적합성과 피해에 대한 책임을 면책하는 현실 비판
- **Hype Cycle 패턴**: 비현실적 기대 → 대규모 유입 → 전문성 인플레이션 → 다음 유행으로 이동

| Hype Cycle 사례 | 시기 | 결과 |
|----------------|------|------|
| 구조적 프로그래밍 | 1970s | 유용하나 과대평가됨 |
| 객체지향 설계 | 1980-90s | 보편화되었으나 만능은 아님 |
| 애자일 방법론 | 2000s | 광범위 채택, 교조적 적용 문제 |
| 빅데이터 | 2010s | 가치 있으나 과잉 투자 발생 |
| 양자 컴퓨팅 | 2010-20s | 아직 실용성 미검증 |
| AI/GenAI | 2020s | 현재 진행 중 |

#### Mary Shaw — "진정하라, AI가 SE를 삼키지 않는다"
- Carnegie Mellon, 소프트웨어 아키텍처 분야 개척자, National Medal of Technology 수상
- **핵심**: "Software engineering is alive and well" — SE는 프로그래밍이나 명세 구현 이상의 모든 것을 포괄
- AI가 역사적으로 SE 기법을 배양해왔음 (연결 리스트, 함수형 프로그래밍, 기호 계산 등)
- GenAI는 **통계적(Statistical)** 특성 → 불투명성, 비결정성, 신뢰성 문제
- 아키텍처 통합과 고수준 설계에는 **암묵지(Tacit Knowledge)**가 필요 → 모델 훈련에 포착 불가
- 참고 논문: *"tl;dr: Chill, y'all: AI Will Not Devour SE"* (Onward! Essays, SPLASH 2024)

#### Laurie Williams — "AI가 절약한 시간을 검증에 쓸 것인가?"
- NCSU, Secure Software Supply Chain Center(S3C2) 디렉터, ACM/IEEE Fellow
- **핵심**: 현대 소프트웨어의 **80%가 오픈소스 공급망**에서 유래
- 위협의 진화:

```
┌──────────────────────────────────────────────────────┐
│             공급망 보안 위협 진화                      │
│                                                      │
│  과거: 수동적 공격자                                  │
│  ┌─────────┐    취약점 발견     ┌──────────────┐     │
│  │ 공격자   │ ──────────────► │ 기존 라이브러리 │     │
│  └─────────┘                  └──────────────┘     │
│  (예: Log4j 취약점)                                  │
│                                                      │
│  현재: 능동적 공격자                                  │
│  ┌─────────┐   취약점 심기/침투  ┌──────────────┐    │
│  │ 공격자   │ ──────────────► │ 의존성/빌드/배포│    │
│  └─────────┘                  └──────────────┘    │
│  (예: node-ipc 2022, SolarWinds 2024)               │
└──────────────────────────────────────────────────────┘
```

- **핵심 질문**: GenAI가 구현을 가속화하면, 절약된 시간을 **검증(Validation)과 확인(Verification)**에 실제로 투자할 것인가?

## 내가 얻은 인사이트

### 소프트웨어 엔지니어 역할 변화 관점
1. **"구현자"에서 "감독자"로의 전환은 이미 시작되었다**
   - Irene Manotas와 Adriana Meza Soria가 공통으로 지적한 것처럼, AI가 구현을 담당하면 엔지니어의 가치는 아키텍처 결정, 요구사항 분석, 시스템 간 상호작용 설계로 이동한다. 이는 코딩 능력만으로 엔지니어를 평가하던 기존 패러다임의 근본적 전환을 의미한다.

2. **Mary Shaw의 "암묵지" 개념이 핵심이다**
   - 고수준 설계와 아키텍처 의사결정에 필요한 암묵지(Tacit Knowledge)는 훈련 데이터로 포착할 수 없다. 이는 시니어 엔지니어의 경험적 판단이 AI 시대에 오히려 더 가치 있어질 수 있음을 시사한다.

### 역사적 패턴 인식 관점
3. **Hype Cycle은 반복된다 — 하지만 잔여 가치는 누적된다**
   - Landon Noll이 지적한 hype cycle 패턴은 AI에도 적용될 것이다. 그러나 구조적 프로그래밍, OOP, 애자일 모두 과대광고 이후에도 실질적 가치를 남겼다. GenAI도 마찬가지로, 거품이 걷힌 후에도 코드 완성, 테스트 생성, 문서화 같은 영역에서 영구적 도구로 자리잡을 가능성이 높다.

4. **"tl;dr: Chill, y'all"이 가장 현실적인 태도일 수 있다**
   - Mary Shaw의 "AI가 SE를 삼키지 않는다"는 메시지는 과잉 반응하지 말라는 것이 아니라, SE가 역사적으로 AI 기술을 흡수해온 패턴의 연장선에서 바라보라는 뜻이다.

### 보안 및 신뢰성 관점
5. **Laurie Williams의 질문이 가장 불편하다**
   - "AI가 절약한 시간을 검증에 쓸 것인가?"라는 질문의 현실적 답은 아마도 "아니오"일 것이다. 기업은 절약된 시간을 더 많은 기능 개발에 투입할 가능성이 높고, 이는 보안 부채의 가속화로 이어질 수 있다. 이 딜레마가 AI 시대 SE의 가장 현실적인 위험이다.

6. **공급망 보안은 AI와 무관하게 이미 위기다**
   - 소프트웨어의 80%가 오픈소스 의존성에서 오는 현실에서, 수동적 취약점 발견을 넘어 능동적 공격 심기로 위협이 진화했다. AI 코드 생성이 이 복잡성을 더할 수 있다.
