## AI 요약

### 1. Harness Engineering이란?

OpenAI 팀이 5개월 만에 100만 줄 이상의 코드를 AI 에이전트로 생성·유지보수한 사례를 분석하며, Birgitta Böckeler가 정립한 개념이다. **"수동으로 타이핑한 코드 없이(no manually typed code at all)"** 라는 극단적 제약 조건 하에서 AI 코딩을 프로덕션 수준으로 끌어올리기 위해 필요했던 것은 더 좋은 모델이 아니라 **더 좋은 하네스(harness)** 였다.

> "When the agent struggles, we treat it as a signal: identify what is missing — tools, guardrails, documentation — and feed it back into the repository."

| 개념 | 설명 |
|------|------|
| **Harness** | AI 에이전트를 제어하고 유지할 수 있게 하는 도구와 관행의 집합 |
| **핵심 원리** | 신뢰성 = 유연성 포기 + 제약 투자 |
| **역할** | 비결정론적 LLM을 결정론적 제어 안에 가두기 |
| **비유** | 말(LLM)을 조종하는 마구(Harness) — 힘은 말이 내지만 방향은 하네스가 잡는다 |

### 2. Harness의 3가지 구성 범주

OpenAI 팀의 하네스 구성 요소는 결정론적(deterministic) 접근과 LLM 기반 접근을 혼합하여 3가지 범주로 나뉜다.

```
┌─────────────────────────────────────────────────────────────┐
│                    Harness Engineering                       │
│                                                              │
│  ┌─────────────────────────────────────────────────────────┐ │
│  │  1. Context Engineering (컨텍스트 공학)                   │ │
│  │     • 지속적으로 강화되는 코드베이스 지식 기반             │ │
│  │     • 동적 컨텍스트: 관찰성 데이터, 브라우저 등            │ │
│  │     • 코드 설계 자체가 컨텍스트 역할                      │ │
│  └─────────────────────────────────────────────────────────┘ │
│  ┌─────────────────────────────────────────────────────────┐ │
│  │  2. Architectural Constraints (아키텍처 제약)             │ │
│  │     • LLM 기반 에이전트 모니터링                          │ │
│  │     • 결정론적 커스텀 린터                                │ │
│  │     • 구조적 테스트 (ArchUnit 등)                         │ │
│  │     • 모듈 경계 정의 및 강제                              │ │
│  └─────────────────────────────────────────────────────────┘ │
│  ┌─────────────────────────────────────────────────────────┐ │
│  │  3. Garbage Collection (엔트로피 관리)                    │ │
│  │     • 주기적 에이전트 실행                                │ │
│  │     • 문서 불일치 탐지                                    │ │
│  │     • 아키텍처 제약 위반 발견                              │ │
│  │     • 코드베이스 쇠퇴(decay)에 대항                       │ │
│  └─────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────┘
```

### 3. 해결 공간 제약 (Solution Space Narrowing)

하네스 엔지니어링의 핵심 전략은 **LLM이 생성할 수 있는 코드의 범위를 의도적으로 좁히는 것**이다.

```
자유도 높음 (신뢰도 낮음)          자유도 낮음 (신뢰도 높음)
◀──────────────────────────────────────────────▶

  "아무 코드나             "이 패턴, 이 구조,
   생성해"                  이 경계 안에서만 생성해"
      │                            │
      ▼                            ▼
  ┌──────────┐              ┌──────────────┐
  │ 넓은     │              │ 좁은         │
  │ Solution │              │ Solution     │
  │ Space    │              │ Space        │
  │          │              │ ┌──────────┐ │
  │  무한한   │              │ │ 예측 가능 │ │
  │  가능성   │              │ │ 한 결과   │ │
  │  = 혼돈   │              │ └──────────┘ │
  └──────────┘              └──────────────┘
```

구체적 제약 도구:

| 제약 수단 | 역할 | 결정론적 여부 |
|-----------|------|:------------:|
| Pre-commit hooks | 커밋 전 자동 검증 | ✅ |
| 커스텀 린터 | 코드 패턴/규칙 강제 | ✅ |
| 구조적 테스트 (ArchUnit) | 아키텍처 제약 검증 | ✅ |
| Markdown 규칙 파일 | 에이전트에게 규칙 전달 | ❌ (LLM 해석) |
| LLM 기반 모니터링 에이전트 | 다른 에이전트의 출력 감시 | ❌ (LLM 기반) |
| 모듈 경계 정의 | 코드 의존성 방향 제한 | ✅ |

**Markdown 규칙 파일만으로는 불충분하다.** 결정론적 도구(린터, 테스트, hooks)와 결합해야 실질적 제약이 된다.

### 4. 반복적 개선 루프

에이전트가 실패하면 그것을 **시그널**로 취급하고, 부족한 것(도구, 가드레일, 문서)을 식별하여 레포지토리에 피드백한다.

```
에이전트 실패
    │
    ▼
원인 분석: "이 시점에 에이전트가 뭘 보고 있었나?"
    │
    ├─ 컨텍스트 부족? ──→ 지식 기반 보강
    ├─ 규칙 위반?     ──→ 린터/테스트 추가
    ├─ 도구 부재?     ──→ 새 도구 제공
    └─ 구조 모호?     ──→ 아키텍처 제약 명확화
    │
    ▼
하네스 업데이트 → 레포지토리 반영
    │
    ▼
다음 에이전트 실행에서 개선 확인
```

### 5. Harness의 미래에 대한 저자의 가설

#### 가설 1: 서비스 템플릿을 대체할 것인가?

현재 조직에서 "황금 경로(Golden Path)"로 쓰이는 서비스 템플릿처럼, 하네스가 **새 프로젝트의 시작점** 역할을 할 수 있다. 커스텀 린터, 구조적 테스트, 컨텍스트 제공자를 포함한 패키지로 진화할 가능성이 있다.

#### 가설 2: 기술 스택 수렴

좋은 하네스가 존재하는 기술 스택으로 선택이 수렴될 수 있다. **"AI 친화성"** 이 기술 선택의 새로운 기준이 된다.

#### 가설 3: Pre-AI vs Post-AI 애플리케이션

| 구분 | Post-AI 애플리케이션 | Pre-AI (레거시) |
|------|---------------------|----------------|
| 설계 | 하네스와 함께 설계 | 하네스 없이 존재 |
| 개조 비용 | 낮음 (내장) | 높음 (레트로핏) |
| 투자 가치 | 명확 | 비표준 코드일수록 낮음 |

### 6. "엄격함의 재정위" (Relocation of Rigor)

Chad Fowler의 개념을 인용하며, 저자는 하네스 엔지니어링의 본질을 다음과 같이 정리한다:

```
Traditional Software Engineering:
  엄격함 → 코드 작성 과정에 집중
  (코드 리뷰, TDD, 페어 프로그래밍)

Harness Engineering:
  엄격함 → 환경·피드백 루프·제어 시스템 설계에 집중
  (린터, 구조적 테스트, 컨텍스트 파이프라인, 가드레일)
```

**코드를 잘 쓰는 것에서 → 코드가 잘 쓰이는 환경을 만드는 것으로** 엔지니어링의 중심이 이동한다.

## 내가 얻은 인사이트

### 시스템 설계 관점
1. **Harness = LLM을 위한 운영 환경(Runtime)**
   - 하네스는 사실상 LLM 에이전트의 "운영체제"다. OS가 프로세스에게 시스템 콜, 메모리 보호, 스케줄링을 제공하듯, 하네스는 에이전트에게 도구, 제약, 컨텍스트를 제공한다. 이 관점에서 보면 Context Engineering(이전 정리 문서 참조)은 하네스의 **메모리 관리 서브시스템**에 해당한다.

2. **결정론적 제어 + 비결정론적 생성의 이중 구조**
   - 핵심 통찰은 LLM의 비결정론적 특성을 받아들이되, 그 출력을 결정론적 도구(린터, 테스트, hooks)로 검증하는 이중 구조에 있다. 이것은 마치 컴파일러의 프론트엔드(파싱, 비결정론적 최적화)와 백엔드(코드 생성, 결정론적 검증)의 관계와 유사하다.

3. **"Garbage Collection"이라는 메타포의 정확성**
   - 소프트웨어가 시간이 지나며 엔트로피가 증가하듯, AI 생성 코드도 쇠퇴한다. 주기적으로 불일치를 탐지하고 정리하는 에이전트는 GC와 정확히 같은 역할을 한다. 이는 AI 코딩이 "한 번 생성하고 끝"이 아니라 지속적 유지보수 파이프라인을 요구한다는 뜻이다.

### 엔지니어링 관점
1. **하네스 구축은 5개월짜리 투자다**
   - OpenAI 팀도 5개월이 걸렸다. "하네스를 하루 만에 만들어서 내일부터 AI 코딩"은 환상이다. 이는 CI/CD 파이프라인 구축과 비슷한 — 한번 잘 만들면 장기적으로 보상받는 — 인프라 투자의 성격을 갖는다.

2. **"에이전트 실패 = 하네스 개선 시그널"이라는 마인드셋 전환**
   - 에이전트가 잘못된 코드를 생성하면 모델을 탓하거나 프롬프트를 고치는 것이 아니라, 린터를 추가하거나 구조적 테스트를 작성하는 것이 올바른 대응이다. 이것은 "테스트 실패 = 코드 개선 시그널"이라는 TDD의 마인드셋과 정확히 동형(isomorphic)이다.

3. **Markdown 규칙의 한계를 인정하는 것이 성숙함이다**
   - CLAUDE.md, RULES.md 같은 마크다운 규칙 파일은 필요하지만 충분하지 않다. LLM은 규칙을 "읽지만 항상 따르지는 않는다." 결정론적 검증 도구와 결합해야만 실질적 제약이 된다. 이것이 프롬프트 엔지니어링과 하네스 엔지니어링의 결정적 차이점이다.
