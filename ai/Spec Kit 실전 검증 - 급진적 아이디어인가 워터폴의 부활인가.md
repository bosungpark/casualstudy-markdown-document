## AI 요약

### 1. 이 아티클의 핵심 질문

GitHub Spec Kit을 실제 프로젝트에 적용해 본 결과, Spec-Driven Development(SDD)는 **생산성을 높이는 급진적 아이디어인가, 아니면 워터폴의 재탕인가?** 저자 Colin Eberhardt는 Scott Logic CTO로서 직접 실험을 통해 이 질문에 답한다.

| 항목 | 내용 |
|------|------|
| 실험 도구 | GitHub Spec Kit (2025년 9월 공개, ~50K GitHub Stars) |
| 실험 프로젝트 | KartLog - 고카트 경주 데이터 관리 PWA (SvelteKit + Firestore + SMUI) |
| 실험 방법 | 삭제된 기능(트랙 관리, ~1,000줄)을 SDD와 반복적 프롬프팅으로 각각 재구현 |
| 핵심 결론 | SDD는 반복적 프롬프팅 대비 **~10배 느리며**, 현재 실무 프로세스로는 비효율적 |

### 2. GitHub Spec Kit의 6단계 워크플로우

Spec Kit은 명세를 "진실의 단일 원천(Single Source of Truth)"으로 취급하는 오픈소스 도구다.

```
┌──────────────┐    ┌──────────────┐    ┌──────────────┐
│ Constitution │───▶│   Specify    │───▶│     Plan     │
│ (지배 원칙)   │    │ (기능 명세)   │    │ (구현 계획)   │
│ 189줄 MD     │    │ 230줄 MD     │    │ 2,067줄 MD   │
└──────────────┘    └──────────────┘    └──────────────┘
       │                   │                    │
       ▼                   ▼                    ▼
┌──────────────┐    ┌──────────────┐    ┌──────────────┐
│    Tasks     │───▶│  Implement   │───▶│      PR      │
│ (태스크 분해) │    │ (코드 생성)   │    │ (풀 리퀘스트) │
│ 66단계       │    │ ~700줄 코드   │    │              │
└──────────────┘    └──────────────┘    └──────────────┘
```

**각 단계의 산출물 (첫 번째 증분 기준):**

| 단계 | 산출물 | 분량 |
|------|--------|------|
| Constitution | 프로젝트 지배 원칙 마크다운 | 189줄 |
| Specify | 기능 명세 + 5개 사용자 스토리 | 230줄 |
| Plan | 모듈 계약(444줄) + 데이터 모델(395줄) 등 | 2,067줄 |
| Tasks | 구현 단계별 태스크 목록 | 66단계 |
| Implement | 실행 가능한 코드 | 689줄 |
| PR | 풀 리퀘스트 | - |

### 3. 실험 결과: SDD vs 반복적 프롬프팅

저자는 동일한 기능을 두 가지 방식으로 구현하여 직접 비교했다.

#### SDD 방식 (Spec Kit 사용)

**첫 번째 증분 - 회로(Circuit) 관리 기능:**

| 지표 | 값 |
|------|-----|
| 에이전트 실행 시간 | 33분 30초 |
| 생성 코드 | 689줄 |
| 생성 마크다운 | 2,577줄 |
| 검토 시간 | 3.5시간 |
| 버그 | `circuitsData` 변수 미연결 버그 발견 |

**두 번째 증분 - 지오로케이션 기능:**

| 지표 | 값 |
|------|-----|
| 에이전트 실행 시간 | 23분 30초 |
| 생성 코드 | ~300줄 |
| 생성 마크다운 | 2,262줄 |
| 검토 시간 | ~2시간 |

#### 반복적 프롬프팅 방식 (저자의 대안)

```
┌────────────────┐     ┌────────────────┐     ┌────────────────┐
│ 중간 규모       │────▶│ 즉시           │────▶│ 다음 기능 구현  │
│ 증분 프롬프트   │     │ 기능 검증       │     │ 중 이전 코드   │
│ (10분 검토 가능)│     │               │     │ 검토           │
└────────────────┘     └────────────────┘     └────────────────┘
```

| 지표 | 값 |
|------|-----|
| 에이전트 실행 시간 | 8분 |
| 생성 코드 | 1,000줄 |
| 생성 마크다운 | 0줄 |
| 코드 검토 | 15분 |
| 기능 테스트/수정 | 9분 |
| **총 소요 시간** | **~30분** |
| 버그 | 없음 |

### 4. SDD가 워터폴로 회귀하는 이유

```
┌─────────────────────────────────────────────────────────────┐
│                    개발 방법론 스펙트럼                        │
│                                                             │
│  Waterfall              SDD               Vibe Coding       │
│  ┌─────────┐         ┌─────────┐         ┌─────────┐       │
│  │ 긴 사전  │         │ 중간 규모│         │ 즉흥적   │       │
│  │ 설계     │         │ 사전 설계│         │ 프롬프팅 │       │
│  │ + 순차   │         │ + 순차   │         │ + 반복   │       │
│  │ 실행     │         │ 실행     │         │         │       │
│  └─────────┘         └─────────┘         └─────────┘       │
│  피드백 루프: 매우 길다  피드백 루프: 길다    피드백 루프: 짧다  │
│  검토: 문서 중심        검토: 마크다운 중심   검토: 코드 중심    │
└─────────────────────────────────────────────────────────────┘
```

저자의 핵심 비판:

1. **사전 설계 필수**: Constitution → Specify → Plan의 순차적 진행이 강제됨
2. **과도한 마크다운 생성**: 689줄 코드를 위해 2,577줄의 마크다운을 검토해야 함
3. **허위 컨텍스트 생성**: "사용자가 장갑을 낄 수 있으므로 모바일 UX 개선 필요" 같은 실질적 가치 없는 내용
4. **형식성 부족**: 마크다운 명세는 추론 가능한 공식 언어가 아님

### 5. "코드는 법이다" vs "명세는 법이다"

저자의 가장 근본적인 비판은 **명세의 형식성(Formality)** 문제다.

```
┌────────────────────────────────────┬────────────────────────────────────┐
│        코드 (Code)                  │        명세 (Spec)                  │
├────────────────────────────────────┼────────────────────────────────────┤
│ ✅ 공식 언어 (Formal Language)      │ ❌ 비공식 언어 (Informal Markdown)   │
│ ✅ 추론 가능 (Reasoning)            │ ❌ 추론 불가능                       │
│ ✅ 테스트 가능 (Testable)           │ ❌ 테스트 불가능                     │
│ ✅ 실행 가능 (Executable)           │ ❌ 실행 불가능                       │
│ ✅ LLM 학습 데이터에 풍부            │ ❌ 상대적으로 제한된 학습 데이터      │
└────────────────────────────────────┴────────────────────────────────────┘
```

> "코드는 공식 언어이므로 추론과 테스트가 가능하지만, 마크다운 명세는 이러한 형식성이 부족합니다."

### 6. "할머니에게 달걀 빠는 법을 가르치지 말라"

저자는 Spec Kit의 구조화된 프롬프트가 최신 AI 모델에게 **불필요한 제약**이라고 주장한다.

| 논점 | 설명 |
|------|------|
| 모델의 내재적 능력 | GPT-5, Claude Opus 4.5 등 최신 모델은 이미 내부적으로 분석-계획-구현 능력을 갖추고 있음 |
| 오픈소스 한계 | 오픈소스 프롬프트 세트가 최신 기초 모델의 내부 추론을 초월하기 어려움 |
| 급속한 발전 | 모델이 빠르게 진화하므로, 외부 구조화 도구는 금방 뒤처질 가능성 |

### 7. 저자가 인정하는 SDD의 가능성

저자는 비판적이면서도 공정성을 위해 SDD가 유효할 수 있는 시나리오를 언급한다.

| 시나리오 | 설명 |
|---------|------|
| 도구 미성숙 | Spec Kit 포럼에서 활발한 개선 논의 중이며, 향후 개선 여지가 큼 |
| 문제 규모 불확실 | 저자 본인도 "올바른 문제 규모를 확실히 모른다"고 인정 |
| 비기술 사용자 | Vibe 코더, 비기술 제품 소유자에게는 구조화된 프로세스가 더 적합할 가능성 |
| TDD/XP와 유사한 실험 | 과거 TDD, Extreme Programming도 초기에는 급진적이었으나 일부 관행이 표준으로 정착 |

## 내가 얻은 인사이트

### 실증적 비교 관점

1. **"10배 느리다"는 숫자가 주는 메시지**
   - 총 소요 시간 기준 SDD(~6시간) vs 반복적 프롬프팅(~30분)의 차이는 단순한 효율성 문제가 아니라, **방법론의 오버헤드가 AI 생산성 향상분을 상쇄한다**는 구조적 문제를 보여준다. 프로세스를 위한 프로세스가 되어서는 안 된다.

2. **마크다운 검토의 역설**
   - SDD는 코드 검토 부담을 줄이기 위해 명세를 도입했지만, 결과적으로 2,577줄의 마크다운을 검토해야 하는 새로운 부담이 생겼다. 코드를 읽는 것보다 명세를 읽는 것이 반드시 쉬운 것은 아니며, 오히려 **검토 대상만 증가**시킬 수 있다.

3. **명세가 버그를 막지 못했다는 사실**
   - 2,577줄의 명세를 거쳤음에도 `circuitsData` 변수 미연결 같은 구현 버그가 발생했다. 이는 명세의 형식성 한계를 실증적으로 보여주며, **테스트만이 진정한 검증 수단**이라는 전통적 관점을 강화한다.

### 기존 SDD 아티클과의 교차 분석 관점

1. **Thoughtworks vs Scott Logic - 같은 주제, 다른 결론**
   - Thoughtworks의 SDD 아티클이 "Waterfall과 Vibe Coding 사이의 균형점"으로 긍정적 평가를 한 반면, 이 아티클은 실증 실험을 통해 "현재로서는 비효율적"이라는 결론에 도달한다. 두 관점을 함께 보면 SDD의 현재 성숙도를 보다 균형 잡힌 시각으로 파악할 수 있다.

2. **"명세 = 진실의 원천" 급진적 관점에 대한 강력한 반론**
   - 코드는 형식 언어(Formal Language)이므로 추론과 테스트가 가능하지만, 마크다운 명세는 그렇지 않다는 지적은 Tessl Framework 같은 급진적 SDD 접근법의 근본적 한계를 드러낸다. 명세가 진정한 SoT가 되려면 **실행 가능한 형식 명세**(예: TLA+, Alloy)로의 진화가 필요할 수 있다.

3. **AI 모델의 내재적 능력 vs 외부 구조화의 경쟁**
   - 저자의 "할머니에게 달걀 빠는 법" 비유는 핵심을 찌른다. 모델이 매 6개월마다 급격히 발전하는 상황에서, 외부 도구로 워크플로우를 고정하는 것은 **모델의 잠재력을 과소평가하는 결과**를 낳을 수 있다. 다만 이는 현재 모델 수준에 한정된 판단이며, 모델이 복잡한 엔터프라이즈 도메인을 완전히 이해하기에는 아직 갈 길이 멀다.
