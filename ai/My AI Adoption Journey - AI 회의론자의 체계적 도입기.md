## AI 요약

### 1. 이 글은 무엇인가?

Mitchell Hashimoto가 **"강한 AI 회의론자(heavy AI skeptic)"** 에서 **"이제 돌아갈 수 없다(no way I can go back)"** 로 전환하기까지의 6단계 AI 도입 여정을 기록한 에세이다. 저자는 AI 회사에 투자하거나 근무한 적이 없으며, 순수하게 실용적 관점에서 도구로서의 AI를 평가한다.

> "I really don't care one way or the other if AI is here to stay. I'm a software craftsman that just wants to build stuff for the love of the game."

| 항목 | 내용 |
|------|------|
| **저자 성향** | AI 회의론자, 실용주의 장인(craftsman) |
| **핵심 전제** | 모든 도구 도입은 비효율 → 적정 → 발견의 3단계를 거친다 |
| **대상 프로젝트** | Ghostty (터미널 에뮬레이터, 오픈소스) |
| **핵심 메시지** | 에이전트를 쓰되, 언제 쓰지 말아야 하는지 아는 것이 핵심 |

### 2. 도구 도입의 3단계 법칙

저자가 **모든 도구 도입에 공통적으로** 경험하는 패턴이며, AI도 예외가 아니었다.

```
Phase 1              Phase 2              Phase 3
비효율               적정                 발견
(Inefficiency)       (Adequacy)           (Discovery)
    │                    │                    │
    ▼                    ▼                    ▼
"이거 왜 씀?"        "뭐, 쓸만하네"       "없으면 못 살아"

◀────────────────── 시간 / 노력 ──────────────────▶
```

### 3. 6단계 AI 도입 여정

#### Step 1: 챗봇을 버려라 (Drop the Chatbot)

챗봇(ChatGPT, Gemini 웹)은 코딩에서 **극히 제한적**이다. 초기에 Gemini에 Zed의 커맨드 팔레트 스크린샷을 붙여넣고 SwiftUI로 재현해달라고 했을 때 놀랍도록 잘 작동했지만, **브라운필드(기존) 프로젝트**에서는 형편없는 결과가 반복되었다.

```
챗봇의 한계:
┌──────────────────────────────────┐
│  사전 학습된 지식에만 의존         │
│  프로젝트 컨텍스트 접근 불가       │
│  수동 복사-붙여넣기 루프          │
│  수정 비용 > 직접 작성 비용       │
└──────────────────────────────────┘
         │
         ▼ 전환
┌──────────────────────────────────┐
│  에이전트(Agent) 사용             │
│  • 파일 읽기/쓰기                │
│  • 프로그램 실행                  │
│  • HTTP 요청                     │
│  → LLM + 외부 행동을 루프로 결합  │
└──────────────────────────────────┘
```

> "Their utility in coding is highly limited because you're mostly hoping they come up with the right results based on their prior training."

#### Step 2: 내 작업을 재현하라 (Reproduce Your Own Work)

Claude Code를 처음 써봤지만 인상적이지 않았다. 하지만 포기 대신 **강제 학습법(forcing mechanism)** 을 택했다: **수동으로 작업을 완료한 뒤, 동일한 결과를 에이전트로 재현한다.** 말 그대로 같은 일을 두 번 했다.

```
┌────────────┐    ┌────────────┐    ┌────────────┐
│  수동 작업   │───▶│ 에이전트로  │───▶│ 비교 및    │
│  (직접 코딩) │    │  동일 재현  │    │  학습      │
└────────────┘    └────────────┘    └────────────┘
                        │
                        ▼
              "고통스러웠지만 전문성이 쌓였다"
```

이 과정에서 발견한 **3가지 원칙**:

| # | 원칙 | 설명 |
|---|------|------|
| 1 | **작업을 분해하라** | 하나의 거대 세션이 아닌, 명확하고 실행 가능한 단위로 쪼갠다 |
| 2 | **계획과 실행을 분리하라** | 모호한 요청은 계획 세션과 실행 세션을 나눈다 |
| 3 | **검증 수단을 제공하라** | 에이전트에게 자기 검증 방법을 주면 스스로 실수를 고친다 |

그리고 **부정적 공간(negative space)** 의 발견:

> "Part of the efficiency gains here were understanding when **not** to reach for an agent. Using an agent for something it'll likely fail at is obviously a big waste of time."

이 단계에서의 효율: **직접 하는 것보다 느리지 않지만, 순 효율 이득은 아직 없음.**

#### Step 3: 퇴근 전 에이전트 (End-of-Day Agents)

매일 **마지막 30분**을 에이전트에게 할당한다. 가설: "내가 일할 수 없는 시간에 에이전트가 진척을 만들 수 있다면?"

| 성공적인 작업 유형 | 설명 |
|-------------------|------|
| **딥 리서치** | 특정 언어/라이선스의 라이브러리 조사, 장단점 요약 |
| **아이디어 탐색** | 여러 에이전트를 병렬로 돌려 모호한 아이디어 시작점 확보 |
| **이슈/PR 트리아지** | GitHub CLI(gh)를 활용한 이슈 분류 및 리뷰 |

> 밤새 에이전트를 루프로 돌리지는 않았다. 대부분 30분 이내에 완료되었다.

**결과:** 다음 날 아침 **웜 스타트(warm start)** 를 얻어 더 빠르게 작업에 진입. "이전보다 약간 더 많은 일을 하고 있다는 느낌."

#### Step 4: 확실한 것을 위임하라 (Outsource the Slam Dunks)

에이전트가 **거의 확실히 잘 해낼** 작업을 식별하고, 그 작업을 백그라운드에서 실행하면서 자신은 **다른 작업에 집중**한다.

```
아침 루틴:
┌──────────────────┐
│ 전날 밤 트리아지  │
│ 결과 확인         │
└────────┬─────────┘
         │
         ▼
┌──────────────────┐     ┌──────────────────┐
│ "Slam Dunk" 필터 │────▶│ 에이전트에 위임   │
│ (확실한 작업)     │     │ (백그라운드, 순차) │
└──────────────────┘     └──────────────────┘
         │
         ▼
┌──────────────────┐
│ 나는 좋아하는     │
│ 작업에 집중       │
└──────────────────┘
```

**핵심 규칙:**

| 규칙 | 이유 |
|------|------|
| 알림을 끄라 | 컨텍스트 스위칭 비용이 매우 높다 |
| 병렬이 아닌 순차로 | 한 번에 하나씩, 내가 중단 시점을 통제 |
| 인간이 통제권을 유지 | 에이전트가 아닌 내가 인터럽트 시점을 결정 |

> "Turn off agent desktop notifications. Context switching is very expensive."

**심리적 전환점:**

> "At this point I was firmly in the 'no way I can go back' territory. Even if I wasn't more efficient, the thing I liked the most was that I could now focus my coding and thinking on tasks I really loved."

**스킬 형성(Skill Formation) 우려에 대한 반론:** 에이전트에 위임한 작업의 스킬은 형성되지 않지만, 수동으로 계속 작업하는 영역의 스킬은 자연스럽게 형성된다. 단, **주니어에 대해서는 깊이 우려**한다.

#### Step 5: 하네스를 엔지니어링하라 (Engineer the Harness)

에이전트가 실수할 때마다 **다시는 같은 실수를 하지 않도록** 환경을 개선한다.

| 접근법 | 방법 | 예시 |
|--------|------|------|
| **암묵적 프롬프팅** | AGENTS.md 업데이트 | 잘못된 명령어, 잘못된 API 호출 방지 |
| **프로그래밍된 도구** | 스크립트 작성 | 스크린샷 촬영, 필터된 테스트 실행 등 |

AGENTS.md의 각 줄은 에이전트의 **실제 나쁜 행동에서 비롯된 것**이며, 거의 모든 문제를 해결했다.

```
에이전트 실수 발생
    │
    ├─ 단순한 실수 (잘못된 명령어, API) ──→ AGENTS.md 규칙 추가
    │
    └─ 복잡한 실수 (검증 필요) ──→ 전용 스크립트/도구 작성
                                    + AGENTS.md에 도구 안내 추가
    │
    ▼
다음 실행에서 동일 실수 방지
```

#### Step 6: 항상 에이전트를 돌려라 (Always Have an Agent Running)

**목표:** 에이전트가 돌고 있지 않으면 "지금 에이전트가 해줄 수 있는 게 있나?"라고 스스로 묻는다.

| 항목 | 현재 상태 |
|------|----------|
| 목표 달성도 | 작업 시간의 10~20% |
| 모델 선호 | 느리지만 신중한 모델 (Amp deep mode / GPT-5.2-Codex, 30분+) |
| 동시 에이전트 수 | 1개 (병렬 실행 안 함) |
| 철학 | 에이전트를 돌리기 위해 돌리지 않는다 |

> "I don't want to run agents for the sake of running agents. I only want to run them when there is a task I think would be truly helpful to me."

### 4. 6단계 요약: 효율성 변화 곡선

```
효율성
  ▲
  │                                          ╱ Step 6
  │                                     ╱───╱  (항상 에이전트)
  │                                ╱───╱
  │                           ╱───╱  Step 5
  │                      ╱───╱     (하네스 엔지니어링)
  │                 ╱───╱
  │            ╱───╱  Step 4 (Slam Dunk 위임)
  │       ╱───╱
  │──────╱  Step 3 (퇴근 전 에이전트)

  │
  │ ╲  Step 1 (챗봇 탈출)
  │  ╲___ ← 비효율 구간
  │
  └──────────────────────────────────────────▶ 시간/노력
```

### 5. 사용 도구 및 워크플로우

| 도구 | 용도 | 평가 |
|------|------|------|
| ChatGPT / Gemini (웹) | 초기 실험 | 코딩에서는 한계 명확 |
| Claude Code | 에이전트 코딩 | 초기엔 불만, 학습 후 효과적 |
| Amp deep mode (GPT-5.2-Codex) | 신중한 코드 생성 | 느리지만(30분+) 높은 품질 |
| GitHub CLI (gh) | 이슈/PR 트리아지 | 에이전트와 궁합 좋음 |
| AGENTS.md | 에이전트 행동 규칙 | 하네스 엔지니어링의 핵심 |

## 내가 얻은 인사이트

### 방법론 관점
1. **"같은 일을 두 번 하라"는 역설적 학습법**
   - Step 2의 "수동 작업 후 에이전트로 재현"은 비효율적으로 보이지만, 에이전트의 강점과 한계를 체화하는 가장 빠른 경로다. 이것은 새로운 프로그래밍 언어를 배울 때 "이미 아는 프로젝트를 새 언어로 다시 구현"하는 학습 패턴과 동일하다. 도구의 학습 곡선을 의도적으로 가속하는 메타 전략이다.

2. **부정적 공간(Negative Space)의 가치**
   - "언제 에이전트를 쓰지 말아야 하는지 아는 것"이 "언제 써야 하는지"만큼 중요하다는 통찰은 깊다. 실패할 작업에 에이전트를 투입하는 것은 시간 낭비일 뿐 아니라, 에이전트에 대한 신뢰를 불필요하게 훼손한다. 이 판단력이 곧 **AI 리터러시**의 핵심이다.

3. **알림을 끄라는 조언의 본질**
   - "에이전트 알림을 꺼라"는 단순한 생산성 팁이 아니다. **통제권의 방향**에 대한 선언이다. 에이전트가 나를 인터럽트하는 것이 아니라, 내가 에이전트를 점검하는 시점을 결정해야 한다. 이것은 비동기 코드 리뷰와 같은 원리다 — 리뷰어가 자신의 흐름에서 리뷰 시점을 결정한다.

### 실무 관점
1. **Step 3 "퇴근 전 30분"은 즉시 실천 가능한 전략**
   - 6단계 중 가장 진입 장벽이 낮으면서도 효과가 명확한 단계다. 리서치, 아이디어 탐색, 트리아지 같은 **비핵심 but 필요한 작업**을 매일 30분 에이전트에 위임하는 것만으로도 다음 날의 생산성이 향상된다. CI/CD가 "빌드를 밤에 돌려놓는" 것에서 시작했듯, AI 도입도 "비업무 시간 활용"에서 시작할 수 있다.

2. **하네스 엔지니어링은 TDD의 AI 버전이다**
   - Step 5에서 "에이전트 실수 → 규칙/도구 추가"의 루프는 "테스트 실패 → 코드 수정"의 TDD 루프와 구조적으로 동일하다. AGENTS.md의 각 줄이 실제 실패에서 비롯되었다는 점은, 좋은 테스트가 실제 버그에서 비롯되는 것과 같다. **실패 주도 하네스 개발(Failure-Driven Harness Development)** 이라 부를 수 있다.

3. **"에이전트를 위해 돌리지 않는다"는 절제의 지혜**
   - 도구에 대한 과잉 투자는 그 자체로 비용이다. "항상 에이전트를 돌리되, 의미 있는 작업이 있을 때만"이라는 제약은 에이전트를 생산 도구가 아닌 **과시 도구**로 전락시키는 것을 방지한다. 현재 10~20%라는 솔직한 수치도 신뢰를 준다.
