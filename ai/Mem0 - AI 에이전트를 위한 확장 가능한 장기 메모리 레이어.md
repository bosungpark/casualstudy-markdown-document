## AI 요약

### 1. Mem0란?

**LLM 에이전트에게 "장기 기억"을 부여하는 메모리 레이어**. 세션이 끝나도 사용자 맥락·선호·사실을 지속 저장하고, 다음 대화에서 관련 정보만 골라 프롬프트에 주입한다.

| 특성 | 내용 |
|------|------|
| 카테고리 | AI Agent Memory Framework |
| 핵심 기능 | 대화에서 중요 정보 추출 → 영구 저장 → 관련 맥락만 검색 |
| 구조 | Vector Store (Mem0) / Vector + Graph (Mem0g) |
| 대체 대상 | Full-context 방식, 단순 RAG, MemGPT |
| 성과 | 토큰 비용 -90%, p95 지연시간 -91%, 정확도 +26% (vs OpenAI) |

---

### 2. 왜 필요한가 - Stateless LLM의 한계

기존 LLM 에이전트는 매 대화를 "처음부터" 시작한다.

```
[기존 방식]
User: "내 이름은 보성이야"
Agent: "안녕하세요, 보성님!"
--- 세션 종료 ---
User: "내 이름이 뭐지?"
Agent: "죄송합니다, 모르겠습니다."   ← 전부 잊어버림
```

**"그럼 컨텍스트 윈도우에 전부 넣으면?"** → 토큰 비용이 폭발한다.

| 방식 | 쿼리당 토큰 | 문제 |
|------|-----------|------|
| Full-context | 25,000+ | 비용 폭발, 지연 증가 |
| 단순 RAG | 가변 | 관련성 낮은 청크 포함, 관계 구조 손실 |
| **Mem0** | **< 7,000** | 필요한 사실만 정제 저장 |

### 3. 아키텍처 - 2계층 메모리

```
┌─────────────────────────────────────────────────────┐
│                    User Message                     │
└─────────────────────┬───────────────────────────────┘
                      │
         ┌────────────▼────────────┐
         │   Extraction Phase      │
         │  (salient facts 추출)    │
         └────────────┬────────────┘
                      │
         ┌────────────▼────────────┐
         │  Consolidation Phase    │
         │  (중복 제거, 갱신, 병합) │
         └────────────┬────────────┘
                      │
      ┌───────────────┴────────────────┐
      │                                │
┌─────▼──────┐                  ┌──────▼──────┐
│ Short-term │                  │  Long-term  │
│  Memory    │                  │   Memory    │
│ (active    │                  │ Vector DB + │
│  session)  │                  │ Graph Store │
└─────┬──────┘                  └──────┬──────┘
      │                                │
      └───────────┬────────────────────┘
                  │
       ┌──────────▼──────────┐
       │  Retrieval Phase    │
       │  (의미론적 검색)      │
       └──────────┬──────────┘
                  │
       ┌──────────▼──────────┐
       │   LLM Prompt 주입    │
       └─────────────────────┘
```

#### Mem0 vs Mem0g

| 구분 | Mem0 (Base) | Mem0g (Graph) |
|------|-------------|---------------|
| 저장소 | Vector Store | Vector + **Knowledge Graph** |
| 표현 | 임베딩 기반 사실 | 노드(엔티티) + 엣지(관계) |
| 예시 | "사용자가 Python을 사용함" | "사용자 → [works_with] → Python → [for] → 데이터 파이프라인 → [at] → 회사 → [migrating_from] → Spark" |
| 추가 정확도 | 기준선 | **+2%** |
| 추출 도구 | LLM 기반 fact extractor | Entity Extractor + Relations Generator |

### 4. 세 가지 메모리 타입

인지과학 분류를 LLM 에이전트에 차용한다.

| 타입 | 정의 | 예시 |
|------|------|------|
| **Episodic** (일화적) | 특정 사건·대화 기록 | "지난 화요일에 Kafka 설정을 물어봤음" |
| **Semantic** (의미론적) | 일반 사실·속성 | "사용자 = Python 백엔드 개발자" |
| **Procedural** (절차적) | 선호·패턴·스타일 | "응답은 항상 코드 예시 포함 선호" |

### 5. 벤치마크 결과 (LOCOMO)

LOCOMO(Long Conversation Memory) 벤치마크에서 측정한 핵심 수치:

| 지표 | Mem0 | vs 경쟁자 |
|------|------|----------|
| LLM-as-a-Judge 정확도 | +26% | vs OpenAI Memory |
| p95 지연시간 | **-91%** | vs Full-context |
| 토큰 비용 | **-90%+** | vs Full-context |
| 쿼리당 토큰 | < 7,000 | vs 25,000+ (full-context) |
| Mem0g 추가 개선 | +2% | vs Mem0 base |

**비교 카테고리 6종**: 기존 메모리 시스템 / RAG (다양한 chunk size, k값) / Full-context / 오픈소스 메모리 / 독점 모델 메모리 / 전용 메모리 플랫폼.

### 6. 코드 예시 - 기본 사용법

```python
from mem0 import Memory

m = Memory()

# 저장
m.add(
    "나는 Python 백엔드 개발자이고 FastAPI를 선호해",
    user_id="bosung"
)

# 검색 (의미론적)
results = m.search(
    query="이 사용자의 기술 스택은?",
    user_id="bosung"
)
# → ["Python 백엔드 개발자", "FastAPI 선호"]

# LLM 프롬프트에 주입
context = "\n".join([r["memory"] for r in results])
prompt = f"사용자 배경: {context}\n질문: ..."
```

## 내가 얻은 인사이트

### 시스템 설계 관점

1. **"Context Window ≠ Memory"**
   - 긴 컨텍스트 윈도우는 **단기 메모리 확장**일 뿐, 지속성과는 별개 차원이다.
   - 10M 토큰 모델이 나와도 매 세션 전부 재전송하면 비용·지연·signal-to-noise 비율 문제는 그대로다.
   - **영구 저장 + 선택적 검색** 구조가 본질적 해법이다.

2. **Vector Store 단독의 한계**
   - "사용자가 Python 언급함"과 "사용자가 dbt에서 Spark로 마이그레이션 중"은 정보 밀도가 다르다.
   - **관계 그래프**가 있어야 "왜 그 선택을 했는가"까지 추적 가능.
   - Mem0g의 +2%는 작아 보이지만, multi-hop 질문에서 진가가 드러난다.

3. **메모리는 라이프사이클이 있다**
   - Extract → Consolidate → Retrieve → **Decay/Update**
   - 오래된 사실, 번복된 선호, 일회성 맥락은 **정리하지 않으면 쓰레기 데이터로 누적**된다.
   - "저장"이 아니라 "큐레이션" 문제로 봐야 한다.

### 비용·운영 관점

1. **토큰 비용이 아키텍처를 결정한다**
   - Full-context 대비 **90% 절감**은 "최적화"가 아니라 **모델을 쓸 수 있느냐 없느냐**의 차이.
   - 월간 100만 쿼리 × 25K 토큰 vs 7K 토큰 → 운영 가능성 자체가 갈린다.
   - **메모리 레이어는 비용 구조를 선형화**한다 (대화 길이에 비례하지 않음).

2. **p95 지연 -91%의 의미**
   - 평균이 아니라 **p95**가 91% 줄었다는 건, **최악 사례에서도 SLA 보장**이 가능해진다는 뜻.
   - 긴 대화일수록 full-context는 불안정해진다 → 실시간 에이전트에 치명적.

### AI 에이전트 진화 관점

1. **메모리는 "첫 번째 아키텍처 구성요소"로 승격**
   - 2024년: "LLM + 프롬프트"
   - 2025년: "LLM + RAG + 툴"
   - **2026년: "LLM + Memory + Tools + Orchestration"**
   - 메모리 전용 벤치마크(LOCOMO), 전용 연구 문헌, 전용 플랫폼 생태계 형성.

2. **Stateful Agent가 Stateless Agent를 대체할 것**
   - 고객 지원·개인비서·코딩 에이전트 모두 "사용자를 아는 정도"가 경쟁력.
   - ChatGPT Memory, Claude Projects도 같은 방향 → **메모리 레이어가 플랫폼 기본값**이 되는 중.

3. **설계 트레이드오프**
   - **프라이버시**: 기억할수록 개인정보 노출 위험 → 삭제/편집 API 필수.
   - **정확성**: 잘못된 사실을 기억하면 **영구적 환각**이 된다 → consolidation 단계의 검증 중요.
   - **확장성**: 사용자 수 × 메모리 크기 → Vector DB 비용 폭증 가능 → 라이프사이클 정책이 운영의 핵심.
