## AI 요약

### 1. 이 아티클의 핵심 논점

Spec-Driven Development(SDD)가 AI 코딩 에이전트 시대의 필수 방법론으로 부상하고 있지만, 이것은 본질적으로 **Agile이 죽인 Waterfall을 부활시키는 것**이라는 비판적 관점을 제시한다. 저자는 SDD 대신 **Natural Language Development(자연언어 개발)**를 대안으로 주장한다.

| 항목 | 내용 |
|------|------|
| 핵심 주장 | SDD는 불필요한 관료주의를 초래하며 Waterfall의 반복이다 |
| 비판 대상 도구 | Spec-Kit (GitHub), Kiro (AWS), Tessl, BMad Method |
| 제안하는 대안 | Natural Language Development (Lean Startup + Agile 기반) |
| 철학적 쟁점 | 개발자 "제거" vs 개발자 "권한 강화" |
| 저자 배경 | Marmelab CEO, 오픈소스 기여자 (react-admin 등) |

### 2. SDD의 전형적 워크플로우

저자가 비판하는 SDD의 3단계 문서화 과정이다. 초기 프롬프트로부터 AI가 순차적으로 문서를 생성하고, 각 단계에서 개발자가 검토·수정한다.

```
┌─────────────────┐     ┌─────────────────┐     ┌─────────────────┐
│    Product       │────▶│ Implementation  │────▶│   Detailed      │
│ Specifications   │     │     Plan        │     │   Task List     │
│  (제품 명세서)    │     │  (구현 계획)     │     │  (작업 목록)     │
└─────────────────┘     └─────────────────┘     └─────────────────┘
        │                       │                       │
        ▼                       ▼                       ▼
   개발자 검토/수정         개발자 검토/수정         개발자 검토/수정
        │                       │                       │
        ▼                       ▼                       ▼
   "8개 파일,              "아키텍처 결정,           "66개 태스크,
    1,300줄 텍스트"         모듈 계약 정의"           순차 실행"
```

**실험 사례: Spec-Kit으로 시간 추적 앱에 날짜 표시 기능 추가**

| 산출물 | 분량 |
|--------|------|
| 마크다운 명세 문서 | 8개 파일, 1,300줄 |
| 실제 생성 코드 | 상대적으로 소량 |
| 문서:코드 비율 | 압도적으로 문서가 많음 |

### 3. SDD의 7가지 구조적 문제점

저자가 실험을 통해 도출한 SDD의 핵심 약점들이다.

```
┌──────────────────────────────────────────────────────────────┐
│                    SDD의 7가지 문제점                          │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  1. Context Blindness (맥락 맹목)                             │
│     └─ AI가 기존 함수/관련 코드를 놓침 → 전문가 검토 필수       │
│                                                              │
│  2. Markdown Madness (마크다운 광기)                           │
│     └─ 장황한 문서에 개발자가 독서 시간 낭비                    │
│                                                              │
│  3. Systematic Bureaucracy (체계적 관료주의)                   │
│     └─ 3단계 설계가 대부분의 기능에 과도함                      │
│                                                              │
│  4. Faux Agile (가짜 애자일)                                  │
│     └─ "사용자 스토리"의 오용 (DB 저장은 사용자 스토리 아님)     │
│                                                              │
│  5. Double Code Review (이중 코드 리뷰)                       │
│     └─ 명세 검토 + 코드 검토로 리뷰 시간 2배                   │
│                                                              │
│  6. False Security (거짓 안전감)                               │
│     └─ AI가 명세를 따르지 않는 사례 다수 발생                   │
│                                                              │
│  7. Diminishing Returns (수확 체감)                            │
│     └─ 프로젝트 규모 증가 시 명세 정확성 급격히 저하             │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

> "개발자는 읽는 데 80% 시간을 쓰고 생각하는 데 20%만 사용하게 된다."

### 4. SDD = Waterfall의 부활

저자는 SDD와 Waterfall 모델의 본질적 유사성을 강조한다.

| 비교 항목 | Waterfall | SDD |
|-----------|-----------|-----|
| 핵심 철학 | Big Design Up Front (BDUF) | 코딩 전 상세 명세 작성 |
| 개발자 역할 | 요구사항 → 코드 번역기 | 명세 → AI 입력 관리자 |
| 문서 중심성 | 상세 설계 문서 필수 | 마크다운 명세 필수 |
| 대상 실행자 | 인간 개발자 | AI 에이전트 |
| 변경 대응 | 경직적 | 경직적 (명세 수정 → 재생성) |
| 근본 가정 | 사전 계획으로 불확실성 제거 가능 | 상세 명세로 AI 오류 제거 가능 |

저자는 Fred Brooks의 **"No Silver Bullet"** 논문을 인용하며, 소프트웨어 개발은 근본적으로 비결정론적(Non-deterministic) 프로세스이므로 사전 계획이 불확실성을 제거할 수 없다고 주장한다.

```
Waterfall (1970)                    SDD (2025)
┌──────────────────┐               ┌──────────────────┐
│ 요구사항 분석     │               │ Product Spec      │
│      ↓           │               │      ↓            │
│ 시스템 설계      │               │ Implementation    │
│      ↓           │               │ Plan              │
│ 상세 설계        │               │      ↓            │
│      ↓           │               │ Task List         │
│ 코딩             │               │      ↓            │
│      ↓           │               │ AI 코드 생성      │
│ 테스트           │               │      ↓            │
│      ↓           │               │ 코드 리뷰         │
│ 유지보수         │               │                   │
└──────────────────┘               └──────────────────┘

        "실행 주체만 바뀌었을 뿐, 구조는 동일하다"
```

### 5. SDD가 해결하지 못하는 근본 문제

SDD의 암묵적 목표는 **"개발자를 소프트웨어 개발에서 제거하는 것"**이지만, 실제로는 실패한다.

```
SDD를 효과적으로 사용하려면 필요한 역량:

┌─────────────────────────────┐
│   비즈니스 분석가 역량        │  → 요구사항 검토, 명세 품질 판단
│   +                         │
│   개발자 역량                │  → 설계 오류 포착, 아키텍처 판단
│   =                         │
│   극히 희귀한 인재만 가능     │  → No Code 도구의 실패와 동일한 패턴
└─────────────────────────────┘
```

- 비즈니스 분석가만으로는 기술적 설계 오류를 포착할 수 없다
- 개발자만으로는 비즈니스 요구사항의 적절성을 판단하기 어렵다
- 결국 **두 역량을 모두 갖춘 사람**이 필요하며, 이는 SDD의 "개발자 제거" 전제와 모순

### 6. 대안: Natural Language Development

저자는 SDD 대신 **Lean Startup + Agile 기반의 자연언어 개발**을 제안한다.

```
Natural Language Development 워크플로우:

     ┌─────────────────────────────────┐
     │   1. 가장 위험한 가정 파악        │
     │      "이 기능이 정말 필요한가?"    │
     └──────────────┬──────────────────┘
                    ▼
     ┌─────────────────────────────────┐
     │   2. 최소 실험 설계              │
     │      짧고 모호한 자연어 지시       │
     └──────────────┬──────────────────┘
                    ▼
     ┌─────────────────────────────────┐
     │   3. AI와 반복적 대화로 구현      │
     │      오류 허용, 작은 증분 개발     │
     └──────────────┬──────────────────┘
                    │
            ┌───────┴───────┐
            ▼               ▼
       실패 → #2로      성공 → #1로
       (다시 실험)      (다음 가정 검증)
```

**실증 사례: 3D 조각 도구 개발**

| 항목 | 내용 |
|------|------|
| 도구 | Claude Code |
| 명세 문서 | 없음 (spec-free) |
| 소요 시간 | ~10시간 |
| 결과물 | Adaptive mesh를 갖춘 3D 조각 도구 |
| 방법 | 짧고 모호한 지시 → 오류 → 수정 → 반복 |

> "작은 아이디어 구현이 저렴하면, 작은 증분으로 빌드하는 것이 좋은 제품으로 수렴하는 가장 빠른 방법이다."

### 7. SDD vs Natural Language Development 비교

| 비교 항목 | SDD | Natural Language Development |
|-----------|-----|------------------------------|
| 철학적 기반 | Waterfall 부활 | Agile 강화 |
| 문서화 수준 | 과도함 (1,300줄+) | 최소한 |
| 피드백 루프 | 길고 무거움 | 짧고 가벼움 |
| 개발 효율성 | 감소 (문서 검토 오버헤드) | 증가 (빠른 반복) |
| 확장성 | 낮음 (규모↑ → 명세 정확도↓) | 높음 (증분 개발) |
| 개발자 역할 | 실행자/관리자 | 창의적 협력자 |
| 오류 대응 | 명세 수정 → 재생성 | 즉시 대화로 수정 |
| AI 활용 관점 | AI를 제약 | AI의 능력을 최대한 활용 |

### 8. 코딩 에이전트 도구의 미래

저자는 현재 코딩 에이전트의 한계와 개선 방향을 제시한다.

**현재 한계:**
- 텍스트 기반 상호작용만 가능
- 시각적 지시(특정 UI 영역 포인팅 등) 불가능
- 명세 문서에 의존할 수밖에 없는 구조적 제약

**개선 방향:**
- Playwright MCP Server 같은 브라우저 자동화 도구의 고도화
- 텍스트 기반이 아닌 **리치 비주얼 인터랙션** 개발
- 시각적 협업이 가능한 AI 에이전트

### 9. 저자의 핵심 비유

> "코딩 에이전트는 **내연기관의 발명**과 같다. SDD는 이 엔진을 기관차에 가두는 것이다. 하지만 우리는 자동차, 비행기, 보트 등 다양한 것을 만들 수 있어야 한다."

- SDD는 새로운 기술(AI 에이전트)을 **기존 프레임워크(Waterfall)**에 억지로 끼워 맞추는 것
- 진정한 혁신은 AI 에이전트의 가능성을 열어두고 새로운 패러다임을 탐색하는 것

## 내가 얻은 인사이트

### 방법론 비판 관점
1. **SDD ≠ 만능 해결책이라는 현실적 경고**
   - SDD를 긍정적으로만 바라보던 시각에 균형을 잡아준다. 특히 "8개 파일, 1,300줄의 명세로 간단한 날짜 표시 기능 추가"라는 사례는 SDD의 오버헤드를 체감하게 한다.

2. **"가짜 애자일(Faux Agile)" 경고의 중요성**
   - SDD 도구들이 "사용자 스토리"를 남용하는 문제는 실무에서도 흔하다. "데이터베이스에 저장하고 싶다"는 사용자 스토리가 아닌 기술 요구사항이다. 형식만 빌려온 애자일은 오히려 해롭다.

3. **"이중 코드 리뷰" 문제의 실무적 파급력**
   - 명세를 검토하고 다시 코드를 검토하는 이중 부담은 팀 단위로 확대되면 심각한 병목이 될 수 있다. 이는 SDD 도입을 검토할 때 반드시 고려해야 할 비용이다.

### AI 시대 개발 문화 관점
1. **"개발자 제거" vs "개발자 권한 강화" - 철학적 분기점**
   - 이 아티클의 가장 중요한 통찰이다. AI 도구를 "개발자를 대체하는 것"으로 보느냐, "개발자를 강화하는 것"으로 보느냐에 따라 도구 설계와 워크플로우가 완전히 달라진다.

2. **컨텍스트에 따른 도구 선택의 중요성**
   - 모든 프로젝트에 SDD가 필요한 것은 아니다. 프로토타입이나 탐색적 개발에는 Natural Language Development가 적합하고, 대규모 팀 프로젝트에서는 SDD의 명세가 유용할 수 있다. 도구와 방법론의 맥락적 선택이 핵심이다.

3. **Fred Brooks의 "No Silver Bullet"이 여전히 유효한 교훈**
   - 1986년 논문이 2025년 AI 시대에도 적용된다는 점이 인상적이다. 도구가 아무리 발전해도 소프트웨어 개발의 본질적 복잡성(Essential Complexity)은 사라지지 않는다.
