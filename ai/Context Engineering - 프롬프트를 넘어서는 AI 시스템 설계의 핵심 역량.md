## AI 요약

### 1. Context Engineering이란?

> "Context Engineering is the discipline of designing and building dynamic systems that provides the right information and tools, in the right format, at the right time, to give a LLM everything it needs to accomplish a task."
> — Philipp Schmid

Shopify CEO Tobi Lütke가 "prompt engineering"보다 더 적절한 용어로 "context engineering"을 제안하면서 널리 퍼진 개념이다. 그는 이를 **"LLM이 주어진 과제를 해결할 수 있도록 모든 맥락을 제공하는 기술"** 이라고 정의했다.

| 구분 | Prompt Engineering | Context Engineering |
|------|-------------------|-------------------|
| **초점** | 프롬프트 문구 최적화 | 전체 정보 시스템 설계 |
| **범위** | 정적 문자열 (Static String) | 동적 시스템 (Dynamic System) |
| **핵심 질문** | "어떻게 말할까?" | "무슨 정보를 언제, 어떤 형식으로 줄까?" |
| **실패 원인** | 잘못된 지시 | 부족하거나 잘못된 맥락 |
| **관계** | Context Engineering의 부분집합 | 상위 개념 |

핵심 주장: **"대부분의 에이전트 실패는 모델 실패가 아니라 컨텍스트 실패다."**

### 2. Context의 구성 요소

LLM이 응답을 생성하기 전에 보는 모든 것이 Context다.

```
┌─────────────────────────────────────────────────────┐
│                   Context Window                     │
│                                                      │
│  ┌─────────────────────────────────────────────────┐ │
│  │  Instructions / System Prompt                   │ │
│  │  → 행동 가이드라인, 규칙, 예시                     │ │
│  └─────────────────────────────────────────────────┘ │
│  ┌─────────────────────────────────────────────────┐ │
│  │  User Prompt                                     │ │
│  │  → 사용자의 즉각적인 질문/지시                     │ │
│  └─────────────────────────────────────────────────┘ │
│  ┌──────────────────────┬──────────────────────────┐ │
│  │  Short-term Memory   │  Long-term Memory        │ │
│  │  → 현재 대화 흐름     │  → 과거 대화에서 축적된    │ │
│  │                      │    지속적 지식             │ │
│  └──────────────────────┴──────────────────────────┘ │
│  ┌─────────────────────────────────────────────────┐ │
│  │  Retrieved Information (RAG)                    │ │
│  │  → 문서, DB, API에서 가져온 외부 데이터            │ │
│  └─────────────────────────────────────────────────┘ │
│  ┌──────────────────────┬──────────────────────────┐ │
│  │  Available Tools     │  Structured Output       │ │
│  │  → 호출 가능한 함수   │  → 응답 형식 지정         │ │
│  │    정의              │    (JSON Schema 등)       │ │
│  └──────────────────────┴──────────────────────────┘ │
└─────────────────────────────────────────────────────┘
```

### 3. "싸구려 데모" vs "마법 같은" 에이전트 — 컨텍스트의 차이

동일한 입력 `"Hey, just checking if you're around for a quick sync tomorrow"` 에 대해:

**데모 에이전트 (빈약한 컨텍스트)**
```
Context: [System Prompt] + [User Message]
→ "미팅 시간을 알려주세요" 같은 일반적 응답
```

**프로덕션 에이전트 (풍부한 컨텍스트)**
```
Context:
  + 캘린더 데이터 → 내일 하루 종일 미팅
  + 이메일 히스토리 → Jim과는 캐주얼한 톤
  + 연락처 정보 → Jim은 핵심 파트너
  + 사용 가능한 도구 → 캘린더 초대 전송 가능

→ "Hey Jim! 내일은 하루 종일 빡빡해. 목요일 오전은 어때?
   초대장 보냈어, 확인해봐!"
```

**차이를 만드는 건 모델이 아니라 컨텍스트다.**

### 4. Context Engineering의 4가지 핵심 특성

#### 4.1 시스템이지, 문자열이 아니다 (A System, Not a String)

프롬프트 템플릿에 변수를 끼워넣는 수준이 아니다. LLM 호출 전에 실행되는 **동적 시스템**이다.

```
사용자 요청
    │
    ▼
┌──────────────────────┐
│  Context Engineering │
│  System              │
│  ┌────────────────┐  │
│  │ RAG 검색       │  │
│  │ 메모리 조회    │  │
│  │ 도구 선택      │  │
│  │ 형식 변환      │  │
│  └────────────────┘  │
└──────────┬───────────┘
           │ 조립된 컨텍스트
           ▼
┌──────────────────────┐
│       LLM 호출       │
└──────────────────────┘
```

#### 4.2 동적이다 (Dynamic)

태스크마다 맞춤 제작된다. 코드 리뷰 요청이면 관련 파일, diff, 코딩 컨벤션을 가져오고, 고객 문의면 주문 내역, FAQ, 이전 티켓을 가져온다.

#### 4.3 적시에 적합한 정보 (Right Information at Right Time)

잘못된 정보가 들어가면 잘못된 결과가 나온다 (Garbage In, Garbage Out). 필요한 정보만 필요한 시점에 주입하는 것이 핵심이다.

#### 4.4 형식이 중요하다 (Format Matters)

| Bad Practice | Good Practice |
|-------------|--------------|
| 원시 데이터 덤프 | 간결한 요약 |
| 모호한 지시 | 명확한 스키마/구조 |
| 전체 문서 | 관련 섹션만 추출 |

### 5. 왜 Prompt Engineering이 아니라 Context Engineering인가?

Andrej Karpathy는 이를 **"다음 단계에 필요한 정보로 컨텍스트 윈도우를 채우는 섬세한 예술이자 과학"** 이라고 표현했다.

```
┌──────────────────────────────────────────┐
│         Context Engineering              │
│                                          │
│   ┌──────────────────────────────┐       │
│   │    Prompt Engineering        │       │
│   │    (지시 문구 최적화)         │       │
│   └──────────────────────────────┘       │
│                                          │
│   + RAG Pipeline 설계                    │
│   + Memory Architecture                  │
│   + Tool Integration                     │
│   + Output Format 설계                   │
│   + 실시간 데이터 주입                    │
│   + 에이전트 간 상태 전달                 │
│   + 토큰 예산 관리                        │
│   + ...                                  │
└──────────────────────────────────────────┘
```

프로토타입과 프로덕션 에이전트의 차이는 **코드 복잡도나 모델 성능이 아니라 컨텍스트 엔지니어링 수준**에서 결정된다.

## 내가 얻은 인사이트

### 설계 관점
1. **Context = 런타임에 조립되는 정보 파이프라인**
   - 프롬프트를 "잘 쓰는 것"은 전체 퍼즐의 일부일 뿐이다. 진짜 역량은 어떤 정보를 어디서 가져와서, 어떤 형식으로, 어떤 타이밍에 LLM에게 전달하느냐를 설계하는 데 있다. 이는 결국 데이터 파이프라인 설계 문제다.

2. **"모델 실패가 아니라 컨텍스트 실패다"는 디버깅 관점의 전환**
   - LLM 에이전트가 이상한 답을 내놓으면 모델을 바꾸거나 프롬프트를 고치려 하기 쉽다. 하지만 대부분의 경우 모델이 받은 컨텍스트를 들여다보면 원인이 보인다. 디버깅의 첫 단추는 "이 시점에 모델이 뭘 보고 있었나?"여야 한다.

3. **토큰 예산은 유한한 자원이다**
   - 컨텍스트 윈도우가 아무리 커져도 "전부 다 넣기"는 답이 아니다. 원시 데이터 덤프보다 간결한 요약이, 전체 문서보다 관련 섹션 추출이 더 나은 결과를 만든다. 이것은 결국 정보 압축과 우선순위 결정의 문제다.

### 엔지니어링 관점
1. **Context Engineering은 풀스택 역량을 요구한다**
   - RAG 파이프라인 설계, 메모리 아키텍처, API 통합, 출력 포맷 설계 등이 모두 필요하다. 이는 단순히 프롬프트를 잘 쓰는 "프롬프트 엔지니어"의 범위를 넘어선, 시스템 엔지니어링 영역이다.

2. **MCP(Model Context Protocol)의 등장은 필연적이었다**
   - 다양한 외부 시스템에서 정보를 가져와 LLM 컨텍스트를 조립해야 한다면, 그 인터페이스를 표준화하려는 시도는 자연스럽다. Anthropic이 MCP를 만들고 Linux Foundation에 기부한 흐름은 컨텍스트 엔지니어링이 개인 스킬이 아닌 인프라 문제로 격상되었음을 보여준다.
