## AI 요약

### 1. OST(Opportunity Solution Tree)란?

Opportunity Solution Tree(기회 해결책 나무)는 Teresa Torres가 2016년에 제안한, **하나의 원하는 성과(Outcome)에서 출발해 그것을 이루는 경로를 시각적으로 그려내는 발견(Discovery) 프레임워크**다.

핵심 문제의식은 단순하다. 대부분의 팀은 **목표 → 곧바로 기능 목록**으로 점프한다. 검증한 적도 없는 문제에 대한 해결책을 브레인스토밍하는 것이다. OST는 그 사이에 **"기회 공간(Opportunity Space)"** 이라는 층을 강제로 끼워 넣어, 해결책을 떠올리기 전에 **고객의 문제 공간을 먼저 지도로 그리게** 만든다.

Torres는 이를 "고객 가치 vs 비즈니스 생존성"의 긴장을 푸는 도구로 설명한다.

> "우리가 비즈니스 가치를 창출할 때, 시간이 지나도 고객을 계속 섬길 권리를 얻는다."
> — Teresa Torres

| 구분 | 흔한 접근 (Feature Factory) | OST 접근 |
|------|---------------------------|----------|
| 출발점 | 아이디어/기능 요청 | 하나의 측정 가능한 **성과** |
| 문제 이해 | 검증 없이 가정 | 고객 인터뷰로 발굴한 **기회** |
| 해결책 | 바로 백로그로 | 기회에 연결된 것만, 여러 개 비교 |
| 의사결정 | 노력/견적으로 사전 컷 | **가정 테스트(실험)** 로 검증 후 결정 |
| 산출물 | 고정된 로드맵 | 매주 갱신되는 **살아있는 문서** |

### 2. 네 개의 층 (The Four Layers)

OST는 위에서 아래로 네 개의 층이 나무처럼 뻗어 내려간다.

```
                    ┌─────────────────────┐
                    │      OUTCOME        │   ← 뿌리(root): 원하는 성과
                    │  (제품 성과 지표)     │      고객 행동/감정을 측정
                    └──────────┬──────────┘
              ┌────────────────┼────────────────┐
        ┌─────┴─────┐    ┌─────┴─────┐    ┌─────┴─────┐
        │ 기회 A     │    │ 기회 B     │    │ 기회 C     │  ← OPPORTUNITY
        │(고객 니즈)  │    │(pain point)│    │  (desire) │     상위 → 하위로 분해
        └─────┬─────┘    └───────────┘    └───────────┘
        ┌─────┴──────┐
   ┌────┴───┐   ┌────┴───┐
   │ 하위기회 │   │ 하위기회 │        ← 큰 기회를 작고 풀 수 있는 단위로
   └────┬───┘   └────────┘
   ┌────┴────┬─────────┐
 ┌─┴──┐  ┌──┴─┐   ┌───┴──┐
 │해결책│  │해결책│   │ 해결책 │       ← SOLUTION: 한 기회에 3개씩 비교
 └─┬──┘  └────┘   └──────┘
   │
 ┌─┴─────────┐
 │ 가정 테스트  │                     ← ASSUMPTION TEST: 가장 위험한
 │  (실험)     │                        가정부터 빠르게 검증
 └───────────┘
```

#### ① Outcome (성과 — 뿌리)
- **제품 성과(product outcome)** 여야 한다. 비즈니스 성과나 트래픽 지표가 아니라, **제품 안에서의 고객 행동이나 제품에 대한 감정**을 측정한다.
  - 예: (X) 매출 20% 증가 → (O) 주간 활성 재방문율, 특정 기능 완료율
- 성과는 **발견의 범위(scope)** 를 정해준다. 어떤 기회가 "관련 있는지"를 걸러준다.
- 방향(directional) 자체가 바뀔 때만 새 나무를 그린다. 목표 수치만 조정될 때는 나무를 새로 만들지 않는다.

#### ② Opportunity (기회 — 문제 공간)
- 정의: **"기회란 충족되지 않은 고객의 니즈, 페인 포인트, 또는 욕구(desire)다."**
- 문제만이 아니라 **욕구**도 포함한다. (디즈니랜드에 가고 싶다, 아이스크림이 먹고 싶다, 산악자전거를 타고 싶다 같은 것)
- **반드시 스토리 기반 고객 인터뷰에서 나와야 한다.** 가정으로 지어내면 안 된다. (최소 3~4회 인터뷰 필요)
- 해결책과 구분하는 **결정적 테스트**:
  > "이걸 해결하는 방법이 두 가지 이상 있는가?"
  - 방법이 **하나뿐**이면 그건 기회가 아니라 **변장한 해결책**이다.
- **고객 관점**으로 쓴다. "어떻게 만들지"가 아니라 "고객이 무엇을 필요로 하는지".
  - (O) "저녁 요리할 시간이 없다" ↔ (X) "배달을 주문한다"(이건 해결책)
- 서로 **상호배타적(mutually exclusive)** 이어야 하고, 경험 지도(experience map)를 만들어 상위 순간(moment) → 하위 기회로 **부모-자식·형제 관계**를 이루며 분해된다.

#### ③ Solution (해결책)
- 기능, 제품, 서비스, 워크플로우, 문서 등 무엇이든 될 수 있다.
- 이상적으로는 **하나의 기회를 겨냥**한다 (지속적 배포에 유리).
- 위험이 클 때는 **한 기회에 3개의 해결책**을 뽑아 비교·대조한다.
- 중요한 원칙: **노력/견적으로 미리 컷하지 않는다.** 실현 가능성은 나중에 가정 테스트로 평가한다.

#### ④ Assumption Test (가정 테스트 — 실험)
- 각 해결책이 딛고 선 **밑바탕 가정(assumption)** 들을 뽑아낸다.
- 그중 **가장 위험한(riskiest) 가정**부터 실험으로 검증한다.
- 이 실험이 "어떤 해결책이 그 기회를 가장 잘 풀고, 원하는 성과를 이끄는가"를 판별한다.

### 3. OST를 만드는 단계 (Step-by-Step)

**시작 전 전제조건(prerequisites):**
1. 타깃 고객과 가치 제안(value proposition)에 대한 가설
2. 명확하게 정의된 **하나의** 성과
3. 스토리 기반 고객 인터뷰 3~4회 완료

**만드는 순서:**

| 단계 | 할 일 | 핵심 포인트 |
|------|-------|-------------|
| 1 | 나무 맨 위에 **성과** 배치 | 제품 성과 언어로 |
| 2 | **기회 공간 지도화** | 3~4회 인터뷰 후 초안. "형편없는 첫 초안(crummy first draft)"을 두려워 말 것 |
| 3 | **타깃 기회 하나** 선택 | 작고 풀 수 있는 기회 하나에 집중 |
| 4 | 그 기회에 대해서만 **해결책 브레인스토밍** | 범위를 좁혀야 발산이 유효 |
| 5 | **해결책 3개** 선정 | 비교·대조 |
| 6 | 각 해결책을 **가정으로 분해** | 밑바탕 믿음 나열 |
| 7 | **가장 위험한 가정** 을 실험으로 테스트 | 3개 아이디어 전반에 걸쳐 |
| 8 | **평가·결정** | 다듬기 / 버리기 / 기회 자체를 포기 중 선택 |

### 4. 기회 우선순위 정하기 (Prioritization)

Torres는 **노력(effort)을 기회 우선순위에서 제외**한다. (노력은 나중에 해결책 단계에서 본다.) 대신 네 가지 렌즈로 본다:

| 렌즈 | 질문 |
|------|------|
| **기회 규모(Opportunity sizing)** | 얼마나 많은 고객이, 얼마나 자주 겪는가? |
| **시장 요인(Market)** | 경쟁 포지셔닝에 주는 영향은? |
| **회사 요인(Company)** | 미션·비전·전략 목표와의 정렬은? |
| **고객 요인(Customer)** | 고객에게 얼마나 중요하고, 대안에 대한 만족도는? |

> 💡 **"기회를 너무 일찍 배제하지 말라(Don't rule out an opportunity too early)."**
> 해결책이 생각보다 훨씬 간단할 수 있기 때문이다.

### 5. 지속적 발견(Continuous Discovery)과의 관계

OST는 **매주 이뤄지는 고객 인터뷰**를 전제로 한다. 정적인 산출물이 아니라 **살아있는 문서**다.

```
매주 고객 인터뷰  ──▶  3~4회마다(대략 월 1회)  ──▶  기회 지도 갱신
     (weekly)            새 스토리 반영              (living document)
        │                                               │
        └──────────  단일 스토리에 과잉반응 방지  ◀──────────┘
```

- 3~4회 인터뷰마다(대략 월 1회) 기회 공간을 업데이트한다.
- 지속적 발견을 하면 **단 하나의 스토리에 과잉반응**하는 것을 막을 수 있다.
- 이해관계자 소통은 **청중에 맞춰 디테일 수준을 조절**한다. 누구는 나무 전체가, 누구는 하이라이트만 필요하다.

### 6. 흔한 실수 (Common Mistakes)

- ❌ **인터뷰 없이 기회를 지어내기** — 가정으로 채운 기회 공간
- ❌ **기회와 해결책을 혼동** — "방법이 여러 개인가?" 테스트를 건너뜀
- ❌ **노력을 근거로 기회를 조기 배제**
- ❌ **범위를 하나의 제품 트리오(trio) 밖으로 확장** — 전사 규모 나무는 다루기 힘들어짐
- ❌ **전제조건 생략** — 인터뷰, 성과 명확화, 고객 가설을 건너뜀

## 내가 얻은 인사이트

### 제품/설계 관점

1. **"기회 vs 해결책"을 가르는 단 하나의 질문이 강력하다.**
   - "이걸 푸는 방법이 두 개 이상인가?" — 방법이 하나뿐이면 그건 이미 해결책이다.
   - 이 리트머스 하나로 백로그에 널린 "위장한 해결책"들을 걸러낼 수 있다. 엔지니어링 티켓 다수가 실은 검증 안 된 해결책이라는 걸 드러낸다.

2. **노력을 기회 단계에서 빼는 것이 반직관적이지만 핵심이다.**
   - 보통은 "이거 만들기 어려우니 빼자"로 기회를 죽인다. 그런데 노력은 *해결책*의 속성이지 *기회*의 속성이 아니다.
   - 기회를 살려두면, 예상보다 훨씬 값싼 해결책이 나중에 발견될 수 있다. 문제와 해법의 관심사를 분리(separation of concerns)하는 소프트웨어 설계 원칙과 정확히 같은 사고방식이다.

### 프로세스/조직 관점

3. **성과를 "제품 성과"로 못 박는 제약이 발견의 범위를 준다.**
   - 매출 같은 비즈니스 지표를 뿌리에 두면 나무가 무한 발산한다. "제품 안 고객 행동"으로 좁혀야 관련 기회를 걸러낼 스코프가 생긴다.
   - OKR을 짤 때 Key Result를 제품 행동 지표로 잡아야 하는 이유와 연결된다.

4. **"형편없는 첫 초안"을 허용하는 문화가 전제다.**
   - 3~4회 인터뷰만으로 완벽한 지도를 그리려 하면 아무것도 시작 못 한다. OST는 애초에 매주 갱신되는 **살아있는 문서**로 설계됐다.
   - 이건 문서를 "완성물"이 아니라 "버전 관리되는 코드"처럼 다루라는 뜻이다. 정적 로드맵 문화와 근본적으로 충돌한다.

### 트레이드오프 관점

5. **한 기회에 해결책 3개는 "비용"이 아니라 "보험"이다.**
   - 하나만 파면 확증편향에 빠지기 쉽다. 3개를 나란히 놓고 *가장 위험한 가정*부터 실험하면, 값비싼 완성 전에 헛다리를 걷어낸다.
   - 단, 모든 기회에 3개씩 붙이면 과부하다. **위험이 높을 때만** 다중 해결책을 쓴다는 조건이 실무의 균형점이다.

6. **범위를 "하나의 제품 트리오"로 한정하는 것이 확장성의 열쇠다.**
   - 전사 규모의 나무를 그리려는 유혹이 크지만, 그러면 다루기 불가능해진다.
   - 이건 마이크로서비스에서 서비스 경계를 팀 경계에 맞추는 것(Conway's Law)과 닮았다. 발견의 단위도 결국 **책임질 수 있는 조직 단위**에 묶여야 유지된다.
