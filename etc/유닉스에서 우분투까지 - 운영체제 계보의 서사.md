### 2. Unix의 탄생 (1969~1970s)

벨 연구소(AT&T Bell Labs)의 **Ken Thompson**은 거대 프로젝트 Multics가 무너진 뒤 시간이 남았다. 비싼 메인프레임에서 *Space Travel* 게임을 돌리다가, 버려진 구형 **PDP-7**에 이식하려 했고 — 이 실용적 필요가 더 단순한 OS 설계로 이어졌다.

- 1969년 말, **Dennis Ritchie**와 함께 Unix 개발 시작
- 이름은 프로그래머식 유머: Multics를 빗댄 **"Unics"** → **Unix**
- **1973년 C 언어로 재작성** 한 것이 결정적 전환점

> Unix와 C는 너무 밀접하게 얽혀, 이후로는 OS와 언어가 서로의 필요에 맞춰 함께 진화했다.

여기서 태어난 철학이 전체 서사를 관통한다 → **"do one thing and do it well"** (한 가지를 잘하라).

---

### 3. GNU 프로젝트 — 자유라는 철학 (1983)

**Richard Stallman**의 GNU 프로젝트는 기술이 아니라 *철학적 전환*이었다. "사용자는 컴퓨터를, 그리고 그 위의 소프트웨어를 원하는 대로 쓸 자유가 있어야 한다."

- AT&T의 상업 라이선스를 우회하기 위해 Unix 유틸리티들을 **하나씩 자유 소프트웨어로 다시 작성**
- 전략적으로 **커널은 맨 마지막으로 미룸** (완성된 도구 위에서 최적화하려는 의도)
- 커널은 Unix 철학에 맞춰 **마이크로커널** 구조를 택함 → 이 선택이 발목을 잡게 된다

```
GNU가 만든 것:   컴파일러(gcc), 셸(bash), 코어유틸 ... ✅ 거의 다 완성
GNU가 못 만든 것: 커널(Hurd)                          ❌ 계속 지연
                                          ↑
                              여기 빈 자리에 Linux가 들어온다
```

---

### 4. Linux 커널의 등장 (1991)

1991년 8월 25일, 헬싱키 대학의 21세 학생 **Linus Torvalds**가 올린 전설적인 메시지:

> "Hello everybody out there using minix – I'm doing a (free) operating system (just a hobby, won't be big and professional like gnu)..."
> (그냥 취미예요. gnu처럼 크고 전문적이진 않을 거예요...)

- 두 달 뒤 **Linux 0.0.1** 공개
- Torvalds는 원래 이름으로 **"Freax"** 를 원했지만, 관리자 Ari Lemmke가 FUNET에 올리며 붙인 **"Linux"** 가 굳어짐
- 핵심은 기술적 참신함이 아니라 **타이밍** — 당시 상업 Unix 진영은 표준/라이선스 분쟁("Unix Wars")으로 분열돼 있었다

---

### 5. GPL 라이선스라는 결정적 한 수 (1992)

Torvalds가 Linux를 **GNU GPL v2** 로 재배포한 것이 전환점이었다. 이것이 GNU 개발자들을 끌어들여 *완전한 OS*를 완성하게 만들었다.

```
        GPL (copyleft)              vs           BSD (permissive)
  ┌──────────────────────┐              ┌──────────────────────┐
  │ 개선하면 반드시       │              │ 기업이 가져가서       │
  │ upstream으로 환원     │              │ 독점 제품으로 흡수 OK │
  │ → 개발자/사용자 이익   │              │ → 기여 없이 fork 가능 │
  │   이 정렬됨           │              │                      │
  └──────────────────────┘              └──────────────────────┘
            │
            └─> "GNU 도구 + Linux 커널 = GNU/Linux" 라는 완성된 운영체제
```

이 재라이선싱 후 GNU 엔지니어들이 Linux 커널을 다듬어 수많은 GNU 패키지와 호환되게 만들었다. **빈 커널 자리(GNU)** 와 **갈 곳 없던 커널(Linux)** 이 GPL을 통해 맞물린 순간이다.

---

### 6. 배포판의 시대와 데스크탑의 부상 (1990s)

커널과 도구가 있어도, 일반인이 쓰려면 **묶어서 설치 가능한 형태(배포판)** 가 필요했다.

| 배포판 | 연도 | 특징 |
|--------|------|------|
| S.u.S.E. | 1992 | 독일발 전문 리눅스 지원 |
| Slackware | 1993 | Patrick Volkerding, 단순함·안정성 |
| **Debian** | 1993 | Ian Murdock, 자유 소프트웨어 헌신 (→ 훗날 Ubuntu의 토대) |
| Red Hat | 1994 | Marc Ewing, 상업/엔터프라이즈 지향 |

- **1997 RPM 패키지 매니저** → 의존성 문제 해결
- **1999 Red Hat IPO** → 리눅스가 엔터프라이즈 솔루션으로 공인됨
- GUI 데스크탑 등장: **KDE(1996), GNOME(1997)** — 하지만 리눅스는 여전히 *서버 OS*였고, 데스크탑은 Windows에 한참 뒤처짐

---

### 7. Ubuntu — 대중화의 순간 (2004)

보안회사 Thawte를 Verisign에 매각한 남아공 사업가 **Mark Shuttleworth**가 그 자금으로 "컴퓨팅 접근성"을 바꾸려 했다. 2004년 10월, "Warthogs" 팀과 함께 **Ubuntu 4.10 "Warty Warthog"** 발표 — *비기술자*를 위해 설계된 Debian 기반 배포판.

Ubuntu의 철학:
- **예측 가능한 6개월 릴리스 주기**
- 현지화(localization)와 접근성
- 기술적 깊이보다 **사용 편의성**
- 기존 자유 소프트웨어 프로젝트의 통합

> 출시 전 어떤 홍보도 없었지만, 설치와 사용이 쉽다는 이유만으로 Ubuntu는 거의 하룻밤 사이에 인기 정상에 올랐다.

Shuttleworth는 개발 자금을 대기 위해 **Canonical Ltd.** 설립. Red Hat(→Fedora), SUSE(→openSUSE)가 커뮤니티 프로젝트를 분리한 것과 달리, Canonical은 **Ubuntu 방향을 직접 통제** — 훗날 논란도 됐지만 초기엔 데스크탑 채택을 가속했다.

---

### 8. 왜 Ubuntu가 중요했나

1. **기술 취향이 아닌 사용자 필요를 가정**: Debian은 자유 소프트웨어 순수주의자, Arch는 통제를 원하는 매니아 대상이었지만, Ubuntu는 *그냥 작동하는 컴퓨터를 원하는 사람*을 겨냥했다.
2. **하드웨어 파트너십**: System76(2005~)이 Ubuntu를 선탑재 → 설치 장벽 제거
3. **실용적 접근성**: GNOME 2(후에 3)를 기본 채택, 마이크로커널 이론이나 미니멀리즘에 대한 *교조적 집착 없이* 상용 수준의 완성도 제공
4. **커뮤니티 모멘텀**: 빠른 성장 → 개발자 유입 → 선순환. 2000년대 중반 Ubuntu는 *"보통 사람을 위한 리눅스"* 의 대명사가 됐다

## 내가 얻은 인사이트

### 아키텍처/설계 관점
1. **"빈 자리"가 생태계를 결정한다**
   - GNU는 모든 도구를 만들고도 커널을 비워뒀고, Linux는 커널만 있고 도구가 없었다. 두 빈 자리가 **퍼즐처럼 맞물려** GNU/Linux가 완성됐다. 완벽한 단일 설계보다, 잘 정의된 경계와 빈 슬롯이 조합 가능성을 만든다.
2. **마이크로커널 vs 모놀리식의 현실적 교훈**
   - GNU Hurd는 "이론적으로 우아한" 마이크로커널을 고집하다 지연됐고, Torvalds의 "일단 돌아가는" 모놀리식 커널이 세상을 차지했다. **설계 순수성보다 출시와 모멘텀**이 이긴 대표 사례.

### 라이선스/거버넌스 관점
1. **라이선스는 기술이 아니라 인센티브 설계다**
   - GPL의 copyleft는 "개선은 반드시 환원"이라는 규칙으로 개발자와 사용자 이익을 정렬시켰다. BSD의 관대함이 기업 독점 fork를 허용한 것과 대비된다. 오픈소스 프로젝트를 시작할 때 라이선스 선택은 곧 **공동체 행동 양식의 선택**이다.
2. **통제와 개방의 트레이드오프**
   - Canonical의 직접 통제는 빠른 데스크탑 채택을 가능케 했지만 훗날 논란(Unity, Mir, 텔레메트리 등)을 낳았다. Red Hat/SUSE의 커뮤니티 분리 모델과 비교하면, **거버넌스 구조가 제품의 운명을 가른다**.

### 제품/대중화 관점
1. **타이밍 > 참신함**
   - Linux는 기술적으로 새롭지 않았지만 Unix Wars로 진영이 분열된 *바로 그 순간* 무료 커널로 등장했다. 시장의 빈틈이 열린 타이밍을 잡은 것이 핵심.
2. **마지막 1마일(설치·사용성)이 채택을 결정한다**
   - 커널·도구·GUI가 다 있어도 데스크탑은 Windows에 밀렸다. Ubuntu가 바꾼 건 본질 기능이 아니라 **"그냥 깔면 된다"는 경험**이었다. 인프라가 아무리 좋아도, 사용자 진입장벽을 낮추는 마지막 단계가 보급을 좌우한다.

---

> **한 줄 서사**: Thompson은 *이식성*, Ritchie는 *명료함*, Stallman은 *자유*, Torvalds는 *적시의 무료 커널*, Shuttleworth는 *설치와 사용의 민주화* 를 풀었다. 이 다섯 조각이 GPL이라는 접착제로 묶여 오늘의 Linux/Ubuntu가 되었다.
